# Phase 24.3 — Chat-Layer Convergence Under the Substrate-Uncontaminated Regime

## §0 — Banner

**The center of gravity has shifted from "fix the chat" to "classify the remaining bottleneck honestly."**

Phase 24.2 §4.3 produced the cleanest measurement surface this project has had since chat shipped: substrate dispatches correctly, doctor surfaces dispatch correctness, graph emission verifies, and the canonical probe runs to `status=ok`. The operator-visible failure that survives that cleanup is no longer existential ("is the architecture fundamentally broken?") — it is narrow ("how should orchestration terminate after successful tool execution?"). Phase 24.3 operates on the newly-uncontaminated regime.

Phase 24.3 is **not** a chat-fix phase. It is a **chat-classification phase** under the A/B/C three-domain decomposition:

- **A. Runtime economics** — model sizing, inference time per turn, KV-cache locality (bounded by Phase 24.1 archaeology).
- **B. Termination/convergence behavior** — agentic-loop shape after a successful tool call.
- **C. Operator-perceived responsiveness** — streaming UX.

It is entirely plausible that A is fixed, C is improved, and B remains partially broken — and **that would still be legitimate progress.** Recent phases have succeeded by narrowing uncertainty without pretending to close neighboring layers. 24.3 keeps operating that way.

## §1 — Load-Bearing Diagnosis

The 24.2 substrate correction produced the §4.3 case verbatim:

```
$ flame_execute_python (via conversational dispatch)
  node_kind="python"
  status="ok"
  elapsed_ms=16
  → real Flame execution reached via the conversational dispatch path
```

…**and** an operator-unusable conversational answer (model loops, budget exhaustion at iter 1 / 120s).

Phase 23.1 author-walk on portofino captured the substrate-uncontaminated breakdown:

```
flame_execute_python (wrapper)  status=ok elapsed_ms=12 code_len=581
tool-call iter=1                prompt_tokens=15372 completion_tokens=194
                                elapsed_ms=71002 status=continuing
tool-call session complete iter=1
                                elapsed_s=120.0 reason=max_seconds
```

The tool wrapper executed in 12ms. The model burned ~71s of LLM inference + ~49s of post-tool-result generation. **The substrate is fast. The model is slow. The orchestration loop terminates by budget exhaustion, not by reaching an answer.**

Three observable failure shapes survive §4.3's cleanup, and they live at three different layers:

| Layer | Failure shape | Phase 24.1 evidence | Phase 23.1 evidence | Phase 24.2 evidence |
|---|---|---|---|---|
| **A. Runtime economics** | Cold-prefix amplifier; ~71s/turn at 15K tokens with `qwen2.5-coder:32b` | warm iter 1 82× speedup (115,633ms → 1,403ms) | 70,990ms LLM inference + 49,000ms post-tool generation | substrate-clean canonical probe still hits 120s budget |
| **B. Termination behavior** | Convergence/termination semantics after successful tool call; model continues generating past the readable answer | 8/8 turns `salvage_fired=true` (prose-embedded JSON) | `status=continuing` after a `status=ok` tool result | model loops within budget, never converges to operator answer |
| **C. Perceived responsiveness** | 120s of operator silence between request and timeout; no observable progress | n/a (timing only) | n/a (timing only) | n/a (timing only) |

Each layer's failure has its own evidence shape, its own bounded fix surface, and its own success criterion. The 24.3 framing's job is to keep those three things distinct.

## §2 — Core Goal

**Produce empirical evidence about each of A, B, and C separately, on the substrate-uncontaminated regime, with success criteria that don't launder one layer's progress into another's success.**

Concretely:

- A is addressed by **default-model bump from `qwen2.5-coder:32b` to `qwen2.5-coder:14b`**, conditional on baseline evidence that the smaller model retains affordance-selection capability under the 58-tool prefix.
- C is addressed by **history-grows SSE streaming over `POST /api/v1/chat`**, which preserves D-03 request semantics and salvage unit-of-meaning at message granularity.
- B is **not addressed structurally in 24.3.** Convergence/termination semantics are observable through the same forcing function but remain a separate fix surface. If A speeds up the runtime enough that B becomes operator-tractable as a side effect, that is a bonus observation, not a success claim.

The architectural invariant this phase introduces: **success criterion stays attached to its native layer. UX wins (streaming) ride as additional wins, not laundered into convergence success.** This invariant applies operationally throughout §4.

## §4 — Forcing Function

### §4.0 — Operating Principle (load-bearing framing language)

> **Streaming exists to expose forward progress, not to redefine conversational success.**

Without this discipline, *"we can now watch the system fail in real time"* could accidentally get misclassified as convergence. The forcing function still requires substrate execution success + readable operator answer + within-budget completion. Streaming changes the observability and tolerability of the path toward that outcome — it does not change the criterion.

This line is the architectural invariant of §4. It governs every success/failure verdict downstream.

### §4.1 — Success Criterion (per-layer, NOT a blob)

```
SUCCESS:
  L1 (substrate):     flame_execute_python emits status=ok at the graph store
  L2 (orchestration): operator receives a readable answer enumerating Reel 1 clips
  L3 (runtime):       total interaction remains within the FB-C 120s inner cap
                      AND time-to-readable-answer < 90s (post-swap target)

ADDITIONAL UX WIN (orthogonal, NOT success-criterion):
  visible forward progress before completion via SSE message events
```

If L1 + L2 + L3 all hold under the post-swap + streaming regime, 24.3 has converged at the layers it scoped. If L1 + L3 hold but L2 doesn't (readable answer never reached; orchestration loops within budget), that is **Layer B residual** — legitimate progress at A + C, surface for a separate Layer B phase. Per §4.0, the ADDITIONAL UX WIN does **not** substitute for L2.

### §4.2 — Failure Criterion (Re-Opens Phase 24.3 Scope)

If the post-swap + streaming canonical probe produces:

- Substrate non-emission (no `status=ok` graph event under any iter)
- OR a regression vs the substrate-uncontaminated 32b baseline (the swap *introduced* a substrate-layer fault)
- OR `salvage_fired` rate dropping below the 32b baseline rate in ways that materially affect tool-call interpretation

…then a layer was either misclassified or the swap interaction with `_adapters.py` salvage was understated. Re-open §3 In/Out scope under that evidence.

### §4.3 — Failure Criterion (Hands Off to Next Phase)

If L1 + L3 hold under the post-swap + streaming regime but **L2 does not** — i.e., dispatch is correct, runtime is within budget, streaming reveals forward progress, but the operator still doesn't receive a readable answer — then Phase 24.3 has *succeeded at A + C* and **Layer B termination/convergence is the next phase**. Successor phase opens against uncontaminated Mode 2 evidence (substrate clean, runtime clean, salvage observable, only termination broken).

That is the cleanest possible handoff shape: each layer's residual failure becomes the next phase's load-bearing question, classified honestly at the layer it lives at.

### §6.4 — Request transport shape **[CONVERGED]** (Sub-Q3.a)

**Q3.a. POST + SSE, or GET + SSE?**

**Resolution:** POST + SSE.

Keep `POST /api/v1/chat` request semantics intact; the response side switches from `JSONResponse` to SSE. The existing handler contract (D-02 validation / D-13 rate limit / D-14 timeout) wires off POST semantics. Migrating to GET would force a contract rewrite — out of scope per §3.2 ("don't pull adjacent-layer work in").

### §6.5 — Streaming event shape **[CONVERGED]** (Sub-Q3.b — load-bearing inversion)

**Q3.b. History-grows streaming (emit complete message objects as produced) or token-delta streaming (emit fine-grained chunks)?**

**Initial drafted position:** token-delta, on architectural-cleanliness grounds (matches OpenAI-style conventions).

**Converged position:** history-grows.

**Why the inversion (load-bearing reasoning):**

The system is **not** operating with clean native tool-call semantics. It is operating with prose-embedded JSON + salvage extraction + message-level reconstruction. Under that operational reality, the salvage unit-of-meaning IS the assistant message text. Phase 24.1 measurement: 8/8 turns showed `raw_tool_calls=0 / parsed_tool_calls=1 / salvage_fired=true`. Bug-D today reasons over message text, not token fragments.

Token-delta streaming would force:

- Incremental parse state
- Partial JSON ambiguity
- Mid-stream salvage heuristics
- Tool-call commitment semantics

…all into 24.3's scope. **Too much conceptual surface for one phase.**

History-grows, meanwhile:

- Preserves D-03 response semantics
- Keeps salvage at message granularity
- Minimizes adapter-layer disturbance
- Constrains streaming to delivery cadence instead of protocol reinterpretation

Token-delta is **deferred, not rejected.** If native `tool_calls` become reliable downstream (salvage stops being load-bearing), operational tradeoffs change completely and token-delta likely becomes the cleaner shape. Reactivation lives at v1.6+ Path A scope per §3.3.

Streaming changes the UX shape of slowness. It does not constitute conversational success. The forcing function requires substrate execution + readable answer + within-budget completion. Streaming progress rides as ADDITIONAL UX WIN at §4.1, orthogonal to the success criterion.

Sharpened into operating-principle framing language at §4.0: *"Streaming exists to expose forward progress, not to redefine conversational success."*

## §7 — Anti-Scope Binding (Explicit)

24.3 explicitly does **not**:

- Rename env vars, ports, or surface contracts.
- Add new endpoints, MCP tools, or env vars.
- Change `forge_bridge.__all__` (currently 19 symbols at PR-3-merge state).
- Modify Bug-D salvage logic body at `_adapters.py:733`. **Path A domain-boundary preservation:** if salvage behavior changes shape under the SSE work, that is a 24.3 *regression*, not a 24.3 deliverable. The salvage unit-of-meaning stays at message granularity.
- Tune KV-cache or instrument the cache substrate (Phase 24.1 closed it).
- Narrow tools, rewrite docstrings, reorder system-prompt content, or otherwise compensate for protocol behavior with hidden cognitive layers (Path C anti-scope from `v1.6-WRITERS-ROOM-CONVERGENCE §7`).
- Address Layer B termination/convergence semantics structurally. If A + C close and B remains, B is its own phase.
- Half-solve cancellation. Streaming-without-interrupt is a complete delivery; cancellation lives in a later phase.
- Touch cloud routing or the cloud agentic-loop bug (`SEED-CLOUD-CHAT-AGENTIC-ROUTER-V1.6+`).

**Path A domain-boundary discipline** (from `THREE-DOMAIN-DECOMPOSITION.md`): Bug-D salvage-on-stream is a **protocol-layer** concern, not a UX-layer concern. If `_adapters.py` salvage logic must change to preserve message-granularity semantics under SSE, the framing must explicitly name:

- **Why** the change is needed (forced by SSE delivery shape interacting with salvage scan boundary)
- **At which layer** it lands (protocol-normalization, not UX-delivery)
- **Under which constraint** (message-granularity preserved end-to-end)
- **What remains intentionally unchanged** (salvage logic body)

Otherwise the archaeology eventually reads as *"SSE accidentally changed protocol behavior."* That failure-mode-shape is what this discipline guards against.
