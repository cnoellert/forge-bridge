## §0 Banner — What Is Different About This Phase

Every phase in the 24.x arc prior to this one targeted deterministic system behavior:
substrate truth, runtime economics, transport latency, termination policy, consumer projection.
Correctness was binary. A probe either fired the trigger or it didn't. The DOM either had the
sibling section or it didn't.

24.6 is different.

The target is **probabilistic affordance-selection behavior inside the model.** The system
already recovers correctly when iter-1 selects the wrong tool. The K=2 termination envelope
fires. The consumer surfaces render it truthfully. Those are solved problems.

What remains is **selection efficiency under an unstable affordance prior.** The model's
selection at iter 1 currently favors `flame_find_media` for the canonical query class. More
importantly, the same mis-selection *recurs at iter 3* even after a successful canonical
recovery at iter 2 — the model does not stabilize its affordance prior across the loop. Two
wasted iterations per probe (~20–60s on warm 14b), not one.

This reframes the pathology from "wrong first guess" to **"unstable affordance prior."** The
distinction matters for measurement: sharpened descriptions might shift the initial selection,
OR they might stabilize the prior across subsequent iterations, OR both. Each is a different
kind of win, and the measurement (§5) is structured to attribute the effect honestly.

This changes the epistemic character of the work:

- Variance matters more than in any prior phase
- Overfitting risk is real for the first time in the 24.x arc
- Measurement discipline must be tighter, not looser
- The pass criterion is a shift in selection likelihood across the loop, not a binary trigger

The framing and measurement sections below are structured accordingly.

## §1 Diagnosis

**Observed pathology (two-rerun stable; richer than initial framing surfaced):**

The canonical-probe captures show the same shape across both reruns:

```
iter=1  tool=flame_find_media       status=tool_error
iter=2  tool=flame_execute_python   status=ok          ← first canonical
iter=3  tool=flame_find_media       status=tool_error  ← affordance relapse
iter=4  tool=flame_execute_python   status=ok          ← K=2 fires
```

The model mis-selects at iter 1, recovers at iter 2, and then **relapses to the same
mis-selection at iter 3** before recovering again at iter 4 and triggering K=2 termination.
Two wasted iters per probe; recovery integrity holds (the termination envelope is correct;
the answer is delivered); selection efficiency is what's degraded.

Two-instance pattern stability across independent reruns separated by a phase boundary is the
named re-eval trigger from §11 item (1) of the 24.5 deferral register. The recurrence-after-
recovery shape is what elevates this from "initial confusion" to **unstable affordance prior**
— a more durable model-behavior property than a one-shot mis-guess.

**Root cause (working hypothesis):** The current descriptions of `flame_find_media` and
`flame_execute_python` do not sufficiently distinguish their capability boundaries for the
canonical query class ("What are the clips on Reel 1"). The model's selection prior resolves
ambiguously toward `flame_find_media` because the canonical query's surface-level vocabulary
("clips on Reel 1") pattern-matches against "find media." The successful iter-2 dispatch of
`flame_execute_python` does not durably update the prior — iter 3 reverts to the same
mis-selection.

**Evidence base:**

- `~/.forge-bridge/measurements/2026-05-18-phase-24-4-rerun/sse-warm-server-log.txt` — full
  per-iter `tool-call` log shows the iter-1/3 mis-select + iter-2/4 recovery sequence
  explicitly; `tools_offered_count=58 iterations=4 transport=sse`.
- `~/.forge-bridge/measurements/2026-05-18-phase-24-5-rerun/cli-warm-stdout.txt` — renders
  the `accumulated_results[]` block with both successful canonical dispatches
  (`flame_execute_python [iter=2]` + `flame_execute_python [iter=4]`); per-iter dispatch
  detail was not captured in this bundle's filtered `server-log-key-lines.txt`, but the
  identical 2-success-at-even-iters pattern is the cli-side projection of the same
  iter-1/3-mis-select shape verified explicitly in 24.4.
- Both reruns: `result_hash 8ca14d1e` byte-stable.

**What the evidence does and does not prove:**

- **Proves:** the pathology is reproducible across reruns; the affordance prior is unstable
  (iter-3 relapse), not just initially wrong; the substrate (recovery, termination,
  projection) is correct.
- **Does not prove:** that description sharpening is the correct intervention — it is the
  working hypothesis. The forcing function (§5) is designed to falsify it under multiple
  failure modes (no effect / overcorrection / substrate disturbance).

## §3 Anti-Scope — Architecturally Rejected

**Tool-list narrowing:** `flame_find_media` is not being removed. The reachable capability
space is identical pre- and post-edit. Any edit that reduces the declared capability surface
of either tool to discourage selection has crossed into narrowing and is rejected.

**Orchestration shaping via description:** This is the failure mode most likely to slip in.
The canonical test: if a *new* description edit contains comparative routing instructions
beyond what the inherited baseline already declares ("use this instead of X"), orchestration
advice ("prefer this tool when the orchestrator has already tried Y"), or prompt-like
tool-selection heuristics, it has crossed the line.

The enforced boundary, verbatim:

> *"Descriptions may clarify intended capability boundaries, but may not contain comparative
> routing instructions, orchestration advice, or prompt-like tool-selection heuristics."*

This sentence is the §3 load-bearing guard. It applies to every *new* word added to either
description in this phase.

**Inherited-baseline clarification (load-bearing):**

The pre-existing `flame_find_media` docstring at `forge_bridge/tools/project.py:316-333`
already contains a `Use this tool ONLY when… / Do NOT use this tool for…` block declaring
comparative capability boundaries against `flame_list_libraries`, `forge_list_media`,
`forge_list_published_plates`, and `forge_list_shots`. The pre-existing `execute_python`
docstring at `forge_bridge/tools/utility.py:56-162` similarly positions itself as "the
universal Flame introspection and automation surface" and includes guidance routing
destructive mutations through the staged-ops platform.

> *"This rule applies to new additions; the existing 'Use this tool ONLY when… / Do NOT use
> this tool for…' blocks are inherited baseline structures to refine narrowly within, not
> architectural forms being removed in this phase."*

This phase is **inheriting an already-comparative descriptive regime and attempting to
sharpen it more causally** — not inventing comparative routing language. The distinction is
load-bearing for honest framing archaeology: pretending the baseline were a tabula rasa
would create fake architectural purity retroactively. The inherited regime is a precondition
24.6 operates within, not a property 24.6 endorses globally or forever.

**Catalogue sweep:** Sweeping the full `flame_*` catalogue weakens causal clarity. If iter-1
or iter-3 selection shifts after a multi-tool edit, the change cannot be attributed to the
specific ambiguity being targeted. Single-perturbation discipline requires the two-tool
narrow edit. Catalogue sweep is explicitly deferred to §11 carry-forward pending 24.6
measurement results.

## §5 Measurement — Forcing Function

**Forcing-function metric (multi-dimensional, reflects the unstable-prior diagnosis):**

Across M=5 warm canonical reruns post-edit, the affordance prior shifts toward
`flame_execute_python` and/or stabilizes across the loop, without any other architectural
mutation.

The §0 reframe enables two distinct kinds of "win" — initial-selection improvement and
prior-stability improvement — and the measurement attributes each separately rather than
collapsing them.

**What to capture per probe:**

| Metric | Purpose |
|---|---|
| iter-1 tool choice | **Initial prior** — did the iter-1 selection distribution shift? |
| iter-3 tool choice (when iter-1 was wrong) | **Prior stability** — does the loop teach? Does the model's prior update durably after the iter-2 canonical recovery? |
| Total iterations to termination | Efficiency signal — did wasted iters disappear? K=2 ideally fires at iter ≤ 3 if the prior stabilizes |
| Wall-clock to termination | Operator-time signal — the operator-visible cost the phase exists to reduce |
| Recovery success when iter-N still wrong | Falsification guard — see below |
| `result_hash` | Substrate stability — must remain `8ca14d1e` byte-identical |

**The recovery-path falsification guard:**

If iter-1 or iter-3 selection shifts toward `flame_execute_python` but recovery degrades
when the selection is still wrong, the description edit overcorrected — it confused the
model about `flame_execute_python`'s capability rather than clarifying `flame_find_media`'s
boundary. Degraded recovery is a hard failure condition, not a soft warning.

**Cold probes:** Anti-scope verification only, same as 24.4 and 24.5. Cold path must show
no behavioral drift. Pass criteria below are not evaluated on cold probes.

**Pass criterion (multi-dimensional; any-of-three is a win, all-of-three is the full success):**

1. **Initial-prior shift:** iter-1 picks `flame_execute_python` in ≥4 of 5 warm canonical
   reruns, AND
2. **Prior stabilization:** in any probe where iter-1 still picks `flame_find_media`, iter-3
   does NOT also pick `flame_find_media` (the prior updates after iter-2 recovery), AND
3. **Substrate integrity:** `result_hash 8ca14d1e` is stable across all 5 probes AND
   recovery path remains intact in any probe where selection is still wrong.

A probe set that satisfies (1) without (2) is a real but partial win — initial selection
improved but the underlying affordance prior is still unstable. A probe set that satisfies
(2) without (1) is also a real win — the prior stabilizes across the loop even if the
opening guess didn't shift. Both are reportable outcomes; both inform whether 24.7 should
extend the catalogue sweep.

**Failure criterion:**

- iter-1 AND iter-3 selection unchanged after edit (description change had no effect on
  either initial prior or stabilization), OR
- Recovery degrades in any probe where selection is still wrong (edit overcorrected), OR
- `result_hash` drifts from `8ca14d1e` (substrate disturbed).
