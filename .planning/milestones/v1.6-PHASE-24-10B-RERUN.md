Observed:

| Probe | Route | Arguments at trace | Outcome |
|---|---|---|---|
| versions | PR20 forced | `{"params": {}}` | inner validation error: `params.sequence_name` required |
| rename | LLM path | none | local env failure: `ollama` package missing |
| segments | LLM path | none | local env failure: `ollama` package missing |
| start frames | PR20 forced | `{"params": {}}` | inner validation error: `params.sequence_name` required |

The old top-level wrapper failure is gone on the forced probes:

```text
params
  Field required
```

The new failure is deeper and more precise:

```text
params.sequence_name
  Field required
```

That means the wrapper-normalization boundary is now crossed on forced chat
execution. The remaining forced-path failure is no longer wrapper serialization;
it is deterministic parameter materialization. PR20 forced execution bypasses
the LLM, so the model never gets the chance to emit the flat
`{"sequence_name": "30sec_21"}` payload that 24.10 normalized successfully at
the adapter boundary.

## §4 Disposition

24.10b closes the normalization topology gap.

It does not yet close the full operator-facing probes because those probes now
hit the next boundary:

- forced path needs deterministic extraction/resolution for `sequence_name`, or
- wrapper-schema tools with unresolved required params should fall through to
  the LLM path instead of being forced directly.

The evidence is cleanly differentiated:

- wrapper serialization failure: fixed
- invocation-path bypass: fixed
- sequence-name materialization under forced execution: still open
- local LLM-path probes: blocked in this environment by missing `ollama`

The next phase should not return to prompt/inventory experiments. It should
decide the ownership boundary for required-parameter materialization in PR20:
deterministic extractor/resolver versus no-force-on-unresolved-required-params.
