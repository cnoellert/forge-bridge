# v1.6 Operability — Graph-Native Internal Runtime

> **What this artifact is.** A milestone-level architectural framing. It precedes `v1.6-REQUIREMENTS.md` and `v1.6-ROADMAP.md`; those derive from this, not the other way around. The framing exists because v1.6's load-bearing decision is *not* which surfaces to ship — it's the data shape every surface emits. Getting that data shape wrong forces a v2.x rewrite. Getting it right means v1.7 and v2.0 are surface evolutions, not substrate rewrites.
>
> **What this artifact is not.** Not a roadmap. Not a phase plan. Not a requirements list. Those land downstream of this. Open questions are *intentionally open* at the framing stage so the writer's-room convergence pass can refine them; closing them prematurely is the failure mode.
>
> **How to read it.** §1–§2 surface the prior art (two existing seeds that already partially framed this) and the architectural reframe. §3–§7 are the load-bearing technical decisions. §8 maps every existing surface onto the new model. §9–§10 cover the runtime loop and the vocabulary update. §11 scopes v1.6 in/out. §12 lists open questions for writer's-room. §13 is the deferral register. §14 is counts + references — archaeology-grade facts checked against the live codebase at framing time.

## §1 — Prior Art: Two Seeds, One Conversation

This framing does not invent the graph-native model. It surfaces and extends two strategic seeds planted nine days before this artifact was drafted, in a single architecture conversation on 2026-05-04 immediately after the PR40-42 single-runtime consolidation. The seeds are load-bearing inputs to v1.6, not background references.

### §1.1 — `SEED-NODE-SCHEMATIC-V1.6+` (107 lines, 2026-05-04)

The seed file lives at `.planning/seeds/SEED-NODE-SCHEMATIC-V1.6+.md`. It frames bridge as **the node schematic for the pipeline layer that has never had one** — a web-served visual graph of entities (Project, Sequence, Shot, Version, Media, Tool, Recommendation, AI service) with edges showing dependencies, hand-offs, and live events. Read by a Flame, Nuke, or Houdini artist, the schematic is instantly legible: every operator in the post-production stack already speaks node-graph fluently.

The seed's load-bearing claims, restated for v1.6 framing:

1. **The dependency graph already exists.** `forge_bridge/store/` has carried it since v1.0 — entities, relationships, events, registry. What's been missing is rendering, not substrate. The schematic is the visual face of existing data, not a new data layer.

2. **Node kinds are already differentiated implicitly.** The seed §"How Existing Surfaces Map" distinguishes:
   - *Deterministic dialog* (`/api/v1/exec`) as "keyboard interface to the schematic" — pick a node, instantiate, run.
   - *LLM dialog* (`/api/v1/chat`) as "the foundry where new node types are authored" — output goes through staged-ops approval, then enters the registry.
   - *Macros* (PR33-34) as "subgraphs — like Group nodes — composition of existing nodes saved as a reusable unit. The persistence in PR34 is the equivalent of saving a Gizmo to disk."
   - *Tool types vs tool instances* as "node types in palette vs node instances in the live graph."
   - *VOCABULARY.md Entities + Traits + Roles* as "the schematic's type system."

3. **The three-tier loop is the unified runtime model.** Verbatim: *"the deterministic path uses what's in the graph; the LLM path grows what's in the graph; staged-ops gates what becomes part of the graph."* This is the chat-as-node insight stated as an architectural claim, nine days early.

4. **Real-time events over historical completeness is the v1.6 wedge.** *"The first version of the schematic doesn't need to render every entity in the project. It needs to render what just happened."* Live event rendering, simple force-directed layout, read-only — these are the scope-discipline choices the seed already made.

5. **Vocabulary as forcing function.** *"Vocabulary clarity that came from doc-writing alone has limits; vocabulary clarity forced by rendering is much harder to fudge. If a node type can't be drawn distinctly, it isn't actually a distinct node type."* This is the architectural pain the framing commits v1.6 to taking.

The seed's activation triggers were:
- v1.6 milestone opens. ← **This trigger is firing.**
- External demo/explanation needed.
- Adoption friction surfaces ("artists don't get what bridge is").
- Vocabulary stabilizes enough to render.
- Post-publish triage demo prepared.

**At v1.6 framing time, the trigger is active.** The seed's six open questions (rendering library, scale handling, read-only vs limited mutation, tool types vs instances rendering, staged-ops surfacing, event firehose vs scoped subscription) are *open questions for v1.6 writer's-room*. This framing does not close them — it carries them forward as §12 open questions.

### §1.2 — `SEED-FLAME-CHAT-FOUNDRY-V1.6+` (111 lines, 2026-05-04)

Sibling seed, same conversation, at `.planning/seeds/SEED-FLAME-CHAT-FOUNDRY-V1.6+.md`. Frames a second right-click menu item in Flame ("Forge: Ask…") that hits `/api/v1/chat` rather than `/api/v1/exec`. Conceptually: the deterministic dialog is the keyboard, the Ask dialog is the foundry where new node types are authored, the schematic is the canvas. Three surfaces, one model.

Load-bearing claims for v1.6:

1. **The bootstrap loop has been partial for months.** The foundry seed §"Bootstrap Loop Completion" enumerates a 10-step end-to-end flow that's been waiting on the foundry surface. Eight of the ten steps already exist (deterministic resolution PR15-29, synthesizer v1.0 infra, staged-ops v1.4, chat endpoint FB-D, registry promotion v1.0). The missing two are the foundry surface itself and the failure-handoff from deterministic to foundry. The bootstrap loop closes when the foundry surface ships.

2. **Without the foundry, the registry can only get better at things it already knows about.** Verbatim from §"Why This Matters." This is the structural reason the deterministic surface alone is insufficient — it's a consumer, not a producer.

3. **The foundry is also a node kind.** A chat turn is structurally identical to a synthesis turn at the runtime layer: the LLM receives input, decides what to do, may call existing nodes, may author new ones. The difference is intent and downstream gating, not graph shape. The foundry seed §"Response shape distinguishes" enumerates four response classes (direct answer, tool execution result, proposed tool, synthesized+approved+executed) — these are output kinds of a chat-as-node, not separate runtime concepts.

The foundry seed explicitly cross-references SEED-NODE-SCHEMATIC and says they should be planned together. v1.6 framing honors that: schematic + foundry + runtime are co-load-bearing for v1.6, not three separate phases.

### §1.3 — What the Seeds Did *Not* Frame

Two architectural decisions the seeds did not pre-decide, and which v1.6 framing must:

1. **The seeds frame an *entity graph* (Project → Sequence → Shot → Version → Media + AI services + Recommendations).** This is the schematic's render target. They do *not* explicitly frame an **operation graph** — chains, chat sessions, macros, tool invocations as graph nodes. The entity graph is *what's being operated on*; the operation graph is *what's being performed*. Houdini's natural separation: data graph vs node graph. The schematic eventually renders both. The runtime executes the operation graph. v1.6 framing must name this two-graph model and define how they couple. See §3.

2. **The seeds frame node kinds implicitly** (type vs instance, tool vs macro vs subgraph) but do not enumerate them as a closed set with explicit control-flow extensions (`ForEach`, `Switch`, `Conditional`). v1.6 framing closes the kind enumeration upfront to keep the v1.7 runtime tractable. See §4.

These two extensions are the *added* architectural content this framing brings. Everything else folds the seeds forward as written.

## §2 — The Architectural Reframe

> **One sentence.** v1.6 ships **graph-shaped observability and persistence** — every surface emits the same node/edge/graph data structure, wrapped around the *existing, still-linear* executor (chain engine, chat handler, tool registry) — so v1.7 can promote the executor itself to a graph runtime and v2.0 can surface the node editor as the user-visible product identity.

> **Boundary discipline — what v1.6 is and is not.**
>
> **v1.6 IS:** a node-record schema (§5), a proto-node emission contract every surface honors (§7), a JSONL graph store the learning pipeline can read as a graph (§5.5), and a vocabulary update (§10) that names existing structures by their graph-shaped identities. The data shape becomes graph-shaped; the *observability* of every operation becomes graph-shaped; the *persistence layer* becomes graph-shaped.
>
> **v1.6 IS NOT:** a graph executor. The chain engine at `/api/v1/exec` still runs chains linearly. The chat handler still runs `complete_with_tools` as it does today. Macros still expand through `_macros.py`. Tool dispatch still goes through the existing MCP registry. **No node is executed by a "graph runtime" at v1.6.** What's new is that every executor *emits* records describing its work in graph terms, into a shared store the learning pipeline reads as a graph. The runtime is graph-*shaped*, not graph-*native*.
>
> The promotion of the executor itself to graph-native is **explicitly deferred to v1.7** (§11.2). Treat that boundary as load-bearing: future contributors reading "v1.6 ships graph-native …" anywhere in this artifact should read it as a synecdoche for "graph-shaped data + graph-shaped persistence + linear executors emitting into them," and not as a claim that the executor itself is a graph runtime. Where this artifact uses the shorthand "graph-native internals," it means the data + persistence + observability layer — not the executor.

### §2.1 — What Changes at v1.6

External surfaces — CLI, chat, Console, MCP — feel like discrete commands and conversational turns, same as v1.5. Internally, **the executors stay linear** (chain engine, chat `complete_with_tools` loop, macro expander, MCP dispatch) but every invocation now produces a **node record** matching the v1.6 schema. Every chat turn produces a **subgraph record** describing the tool calls it made. Every macro expansion produces a **named subgraph record**. Every chain execution produces a **linear subgraph record**. The records share a schema; they're emitted to a shared JSONL store; the learning pipeline consumes them as a graph mining *substrate*. Mining itself — the algorithm that promotes recurring subgraph shapes to named tools — ships in v1.7 (§6.2, §11.2).

External vocabulary does *not* change at v1.6. README, CLAUDE.md, and user-facing docs still describe bridge as protocol-agnostic middleware with a canonical vocabulary. The graph-shaped internals are an implementation reality, not a marketing claim.

### §2.2 — What Changes at v1.7

The internal proto-graph becomes the canonical runtime. The chain engine at `/api/v1/exec` becomes a graph executor — chains are linear graphs, chat sessions are LLM-authored subgraphs, macros are named saved subgraphs, registered tools are atomic nodes, raw Python invocations are Python-kind nodes. The learning pipeline switches from threshold-based JSONL promotion to **structural subgraph mining**: recurring subgraphs with stable input/output ports get collapsed into named tools and added to the registry. The promotion path becomes provable rather than heuristic.

External vocabulary still doesn't change at v1.7. Operators see no node editor, no graph view. They see faster convergence in chat, cleaner tool promotions, more useful execution history — but they don't see graphs.

### §2.3 — What Changes at v2.0

The schematic ships per `SEED-NODE-SCHEMATIC-V1.6+`. The Console gains a graph view. The Flame menu gains a "show schematic" item. Graphs become *user-visible* artifacts. Operators can edit, replay, fork, export. The CLI gains `fbridge graph` subcommands. The external product identity updates: bridge becomes the *graph-native pipeline runtime*. The two-year arc from communication-bus framing to graph-native framing closes at v2.0.

### §2.4 — Why This Sequencing

Three reasons the runtime-first sequence is the honest one, rather than shipping the schematic at v1.6 directly:

1. **The user-visible promise costs more to deliver than to claim.** Shipping a node-editor UI at v1.6 commits us to feature parity with Houdini/ComfyUI's interaction affordances on a timeline that does not allow it. Shipping the internal runtime first means v2.0's editor builds on a load-bearing substrate already in production use.

2. **The promotion story sells the architecture.** A learning pipeline that mines structural subgraph patterns and promotes them to named tools is a much better demo of "bridge learns the pipeline" than a node editor with no real graphs in it yet. v1.7's promotion story is what makes v2.0's schematic worth rendering.

3. **The substrate forces vocabulary clarity before the rendering does.** Per `SEED-NODE-SCHEMATIC-V1.6+` §"Forcing Function on Vocabulary": building the runtime against a fuzzy vocabulary is harder than building the schematic against it. v1.6 takes the vocabulary pain. v2.0 just renders what v1.7 has already disambiguated.

The architectural pain is taken upfront in v1.6. The product win is taken at v2.0. v1.7 is the bridge between them — and the milestone where the promotion story closes the bootstrap loop the foundry seed describes.

### §2.5 — Two Runtime Lanes: Conversational and Operational

The Phase 24.1 measurements (preserved as operational archaeology at `.planning/COLD-START-INVESTIGATION.md` and `.planning/PROTOCOL-VS-SUBSTRATE-INVESTIGATION.md`; synthesized at `.planning/THREE-DOMAIN-DECOMPOSITION.md`) surfaced an architectural distinction worth folding into v1.6 framing language before implementation work deepens around the current assumptions.

The chat / tool-execution / dispatch path is not one continuous runtime workload. It is two distinct lanes with different economics, semantics, and correctness requirements:

**Conversational lane:**
- low-latency direct inference
- small context
- no graph emission
- no salvage normalization
- no operational provenance
- no dispatch substrate
- optimized for responsiveness

**Operational lane:**
- large tool envelope
- protocol normalization (see §9.5)
- graph emission (§3, §5, §7)
- dispatch substrate
- reachability semantics
- staged-ops compatibility (§9.3)
- provenance + replayability
- optimized for operational correctness, not minimum latency

This is a runtime distinction, not a UI distinction or a layering convention. The two lanes impose fundamentally different constraints on what the system optimizes for. Code paths, prompt shapes, instrumentation, latency budgets, and failure-mode handling that make sense in one lane may be actively wrong in the other. Lane collapse — treating a conversational-lane improvement as also applying to the operational lane, or vice versa — is the failure mode this framing guards against.

Bridge is no longer best described as "a chat surface with optional tool execution." It is more accurately described as:

- a low-latency conversational runtime
- plus
- a graph-native operational runtime

— sharing a common dispatch substrate (§3, §9) but operating under different constraints and optimization goals.

**Runtime philosophy — deterministic identity over aggressive minimization.** The Phase 24.1 cache-locality work demonstrated that stable large prefixes are operationally acceptable; generation cost dominates warm-path latency; narrowing instability can cost more than it saves. The result: **deterministic runtime identity matters more than aggressive token minimization.** This is runtime philosophy, not implementation detail — it constrains how v1.6 (and later) optimization work weighs token cost against operational determinism. Implementation specifics (specific keep_alive values, cache timings, token counts, sorting strategies) live in the operational archaeology artifacts referenced above, not in this framing.

## §3 — The Two-Graph Model

Bridge maintains **two coupled graphs**, both versioned, both legible, both eventually renderable. They are conceptually distinct and should not be confused.

### §3.1 — Entity Graph

The entity graph is **what's being operated on** — the canonical pipeline data the bridge already tracks since v1.0. Lives in `forge_bridge/store/` (verified at framing time: `models.py`, `repo.py`, `session.py`, `memory.py`, `staged_operations.py`, `migrations/`). Tracked entity types per `docs/VOCABULARY.md`: Project, Sequence, Shot, Version, Media, Stack, Layer, Asset. Edge types: depends-on, derived-from, produced-by, located-at, related-to.

The entity graph is what `SEED-NODE-SCHEMATIC-V1.6+` proposes to render. It's the data layer. Operations *produce, modify, and traverse* it but are not part of it.

At v1.6, the entity graph is **read by** the operation graph and **mutated by** the operation graph. Mutations are events on the entity graph. The entity graph itself is unchanged by v1.6 framing — it works fine; we just give the operation graph a clean coupling surface to it.

### §3.2 — Operation Graph

The operation graph is **what's being performed** — the sequence and composition of operations the bridge executes. New at v1.6. Each node represents an operation (a tool invocation, a Python execution, a chat turn, a control-flow construct). Edges represent dataflow between operations and control-flow sequencing. Subgraphs are named, reusable, and may be promoted to atomic nodes in the registry.

The operation graph is what's been *implicit* in chains, macros, chat sessions, and execution-log JSONL until now. v1.6 surfaces it as a first-class data structure.

### §3.3 — How They Couple

Coupling happens at **node execution boundaries**:

- An operation node may **read** from the entity graph (e.g., `flame_list_desktop` queries Flame, which has its own entity-shaped state; the operation node's output is structured entity data).
- An operation node may **mutate** the entity graph (e.g., `forge_create_shot` creates a new Shot entity; the operation node's effect is a graph mutation).
- An operation node may **emit events** on the entity graph (e.g., a staged-ops approval emits a `staged_operation.approved` event that downstream consumers can subscribe to).

The operation graph references entity graph nodes by stable identifier (the existing entity ID). The entity graph does not reference operation graph nodes — entities don't carry per-operation provenance directly; provenance is recoverable by querying the operation graph for nodes that referenced an entity.

This asymmetry is intentional: it means the entity graph is *stable under operation graph replay*. You can replay a subgraph against the entity graph and the replay does not pollute the entity layer with replay metadata.

### §3.4 — Both Graphs, One Schematic Eventually

`SEED-NODE-SCHEMATIC-V1.6+` proposes a unified visual rendering for v2.0. The schematic renders both:

- Entity nodes as the **substrate** — Project, Sequence, Shot, Version, Media positioned as the structural layout.
- Operation nodes as **annotations and live events** — appearing transiently as operations execute, persisting when operations have left durable mutations (a new Shot node appearing has the operation node that created it attached as provenance).

This is the same model Houdini uses to render geometry processed by a node network: you see the geometry (entity-shaped data) and the network (operations producing it) co-rendered, distinguishable by viewport, both editable, coupled at execution.

For v1.6, both graphs exist as data structures. Neither is rendered yet. The schematic is v1.7+ or v2.0.

## §4 — Node Kinds: Closed Enumeration at v1.6

v1.6 closes the node kind list. v1.7 may add. The closure is the architectural pain we took upfront — without it, the runtime fans out indefinitely.

### §4.1 — Why Closed

Three reasons:

1. **Tractable runtime.** A graph executor needs to know how to schedule, retry, parallelize, and serialize every kind. Open-ended kinds means open-ended runtime work.
2. **Tractable promotion.** Subgraph mining detects recurring patterns. Patterns over an unbounded kind alphabet are noisier than patterns over a closed one.
3. **Tractable schematic.** v2.0's node editor needs to render and edit every kind. Open-ended kinds means an open-ended visual vocabulary.

Houdini does this. SOPs are not an open-ended set; they're enumerated. Custom operations exist (HDAs) but they're *named subgraphs*, not new primitive kinds. We adopt the same pattern.

### §4.2 — The v1.6 Kind List

Eight kinds, three groups:

**Atomic operations (3 kinds)** — single-step, non-recursive, no internal subgraph:

1. **`tool`** — invocation of a registered tool by name. Inputs: the tool's declared parameters. Output: the tool's declared return value. Provenance: tool name + version. Maps directly to the existing MCP tool registry. Example: `forge_get_shot(shot_id="abc")`.

2. **`python`** — execution of a Python snippet against a target runtime. Inputs: the snippet body + the target runtime selector (initially `flame` only, may grow). Output: captured stdout, stderr, and a structured result if the snippet writes one. Provenance: snippet hash + runtime selector. Example: arbitrary Flame Python via the `:9999` hook.

3. **`constant`** — a literal value baked into the graph. Inputs: none. Output: the constant. Used for parameterizing subgraphs at instantiation. Example: a fixed reel name passed into a parameterized macro.

**Compound operations (3 kinds)** — node has an internal subgraph; outer view is atomic:

4. **`chain`** — sequential execution of contained nodes, data flowing left-to-right. Inputs: the chain's declared parameters. Output: the last node's output. Used for the existing `/api/v1/exec` chain engine. Example: the current `->` separator chain.

5. **`chat`** — an LLM turn with input message, session context, and tool budget. Internally a subgraph of `tool` and `python` operations the LLM authored during the turn. Outputs: final text + the authored subgraph as provenance. Used for `/api/v1/chat` and the v1.6 CLI REPL. Example: `chat("show me Reel 1's clips")` becomes a chat node whose internal subgraph is `[flame_exec_python("clips_on_reel_1")]`.

6. **`subgraph`** — a named, registered subgraph instantiated at this site. Inputs: the subgraph's declared input ports. Output: the subgraph's declared output ports. Provenance: subgraph name + version. This is the **macro replacement**: macros are subgraphs in vocabulary terms; the PR33-34 macro registry becomes the subgraph registry. Example: `subgraph("nightly-publish-pipeline")(shot_id="abc")`.

**Control flow (2 kinds)** — graph structure, not data:

7. **`foreach`** — iterate over an input collection, instantiating a body subgraph per element. Inputs: collection + body subgraph reference. Output: collection of body outputs. Houdini ForEach Begin/End in one node. Example: `foreach(shots, body=publish_pipeline)`.

   **2026-05-20 annotation:** this entry is **SUPERSEDED** as an
   in-v1.6 implementation target. 25.x empirical discovery showed that
   topology-expanding foreach requires typed ports, step-body grammar,
   and per-iteration trace rendering. The original entry remains as
   archaeology; the revised position lives in
   `.planning/seeds/SEED-TYPED-PORTS-FOREACH-PHASE-N-V1.6+.md`.

8. **`switch`** — route execution to one of N branches based on a predicate. Inputs: predicate value + branch subgraphs. Output: selected branch's output. Conditional is a degenerate 2-branch switch.

### §4.3 — What's Not in v1.6

- **`parallel` / `fanout`.** v1.6 is sequential. Parallel execution is v1.7 — meaningfully changes the runtime's concurrency model and isn't load-bearing for any v1.6 surface.
- **`loop` / `while`.** Recursion without bounded iteration is a footgun. `foreach` over an explicit collection is the safe-by-default loop primitive. Unbounded loops are v1.7 with budget enforcement.
- **`event` / `trigger`.** Reactive node kinds (fire when X happens) are v1.7+ — they require subscription semantics the entity graph's event stream needs first.
- **`approval` / `gate`.** Staged-ops approval is *not* a node kind in v1.6 — it's an attribute on certain node kinds (specifically `subgraph` instances flagged as mutating). The approval surface is the existing staged-ops platform; the graph just labels nodes as `requires_approval=true`. Promoting approval to a first-class kind is a v1.7 conversation.

### §4.4 — Mixing Fine and Coarse Within One Graph

The kind enumeration is explicitly heterogeneous: a single graph can contain both atomic and compound nodes side by side. A chain may contain a tool node, a chat node, a foreach over a subgraph, and a python node. The runtime treats them uniformly at the scheduling layer; the kinds differ in execution semantics, not in scheduling.

This is the answer to the *"is it fine-grained or coarse-grained"* fork from the conversation that produced this framing: it's neither. It's *kind-stratified*. Kinds vary in granularity; the runtime is one runtime; the graph is one graph. Houdini's actual model.

## §5 — Node Schema

The node record is the load-bearing data shape every v1.6 surface emits. Defined here so writer's-room convergence can refine field-by-field rather than re-litigating the overall shape.

### §5.1 — Required Fields (Every Node)

```json
{
  "id":           "<stable opaque identifier; UUIDv7 recommended for sortability>",
  "schema_version": "1.6.0",
  "kind":         "tool | python | constant | chain | chat | subgraph | foreach | switch",
  "name":         "<kind-specific identifier; tool name for `tool`, subgraph name for `subgraph`, omitted for `python`/`constant`/control-flow>",
  "inputs":       { "<port_name>": <value or {ref: <node_id>.<port>}>, ... },
  "outputs":      { "<port_name>": <value or null until executed>, ... },
  "params":       { "<param_name>": <value>, ... },
  "started_at":   "<ISO 8601, null until execution starts>",
  "ended_at":     "<ISO 8601, null until execution ends>",
  "status":       "pending | running | succeeded | failed | cancelled | skipped",
  "error":        { "kind": "<string>", "message": "<string>", "traceback": "<string or null>" } | null,
  "provenance":   { "surface": "cli | chat | console | mcp | hook", "request_id": "<string>", "user": "<string or null>" }
}
```

**2026-05-20 annotation:** `foreach` remains in the historical enum
above as archaeology, but its v1.6 implementation target is superseded
by `.planning/seeds/SEED-TYPED-PORTS-FOREACH-PHASE-N-V1.6+.md`.
The schema definition will be redone with typed-port substrate
underneath in Phase N.

### §5.2 — Kind-Specific Extensions

- **`tool`** adds `tool_version`, `tool_origin` (`core | synth | consumer`), and the tool's declared parameter schema by reference.
- **`python`** adds `code_hash`, `runtime` (`flame | local | …`), `captured_stdout`, `captured_stderr`.
- **`chain`** adds `nodes: [<node_id>, …]` — the contained node IDs in execution order.
- **`chat`** adds `messages: [<role, content>]`, `model`, `provider`, `tool_calls: [<node_id>, …]` — the authored subgraph by reference, `iterations`, `final_text`.
- **`subgraph`** adds `subgraph_name`, `subgraph_version`, `body: <subgraph_definition_ref>`.
- **`foreach`** adds `collection_ref`, `body_subgraph_ref`, `iterations: [<node_id>, …]` — one body instantiation per element.
- **`switch`** adds `predicate_ref`, `branches: { "<case>": <subgraph_ref>, … }`, `selected_branch`.

**2026-05-20 annotation:** the `foreach` extension line is superseded
as current implementation guidance. Phase N will define this shape from
typed ports, loop-scoped context, collect merge validation, and
per-iteration trace rendering; see
`.planning/seeds/SEED-TYPED-PORTS-FOREACH-PHASE-N-V1.6+.md`.

### §5.3 — Edge Schema

Edges are not first-class records in v1.6 — they're *implied* by `inputs.<port>.ref` pointing to another node's output. This keeps the storage shape compact and the dataflow trivially serializable. v1.7 may promote edges to first-class records if conditional/transformed dataflow becomes necessary; v1.6 ships the simpler model.

### §5.4 — Graph as Container

A graph record is just a node list + metadata:

```json
{
  "id":             "<graph_id>",
  "schema_version": "1.6.0",
  "name":           "<optional name; required for saved subgraphs>",
  "root_node":      "<node_id of the root; required for execution>",
  "nodes":          { "<node_id>": <node_record>, ... },
  "metadata":       { "created_at": "...", "created_by": "...", "tags": [...] }
}
```

Subgraphs are graphs. Named saved subgraphs are graphs with `name` populated and a stable registry entry. Promotion is moving a graph from anonymous (chat-authored, ad-hoc chain) to named (registered, reusable).

### §5.5 — Storage

v1.6 default: **JSONL append-only**, one node per line, separate file per graph, under `~/.forge-bridge/graphs/<graph_id>.jsonl`. Mirrors the existing execution-log pattern at `forge_bridge/learning/execution_log.py` (verified at framing time: 347 LOC). The learning pipeline already knows how to consume this shape.

v1.7 may migrate to a graph-aware DB layer (likely Postgres rows extended from the existing `staged_operations` schema in `forge_bridge/store/`). v1.6 stays JSONL — the migration path is intentional and low-risk because JSONL is structurally a subset of any DB representation.

### §5.6 — What's Deliberately Missing from the Schema

- **No `parent_node` field.** Nesting (subgraph membership) is recovered by looking up the subgraph's body, not by upward pointers. Keeps records flat.
- **No execution metrics in the base record.** Latency, tokens, cost go in a sibling `metrics` field or a separate metrics stream — load-bearing for production but not for graph correctness.
- **No human-readable rendering hints.** Position, color, icon are v2.0's schematic concerns, not v1.6's runtime concerns.

These omissions are intentional — they keep the v1.6 schema small enough that adding them in v1.7/v2.0 is additive, not breaking.

## §6 — Subgraph Promotion: Structural Collapse, Not Threshold

The learning pipeline's job changes **across v1.6 and v1.7, in that order, with v1.7 doing the heavier lift**: from today's heuristic threshold ("if this code pattern runs 5 times, promote it") to **structural subgraph mining** ("if this subgraph shape recurs with stable input/output ports, promote it"). This section frames the criteria and the data substrate; v1.7's planning will design and ship the mining algorithm itself.

> **Boundary discipline.** At v1.6 the pipeline gets a *graph-shaped substrate to read from* — node records with explicit kinds, ports, and dataflow. The pipeline's pattern-matching logic at v1.6 stays close to today's heuristic threshold; what changes is that the same threshold logic now operates over node records instead of code strings, so it produces strictly cleaner candidates than the text-similarity path. **Structural subgraph mining as an algorithm — detecting recurring shapes, isolating parametrized ports, deciding promotion — ships in v1.7** (§11.2). The v1.6 work is making the algorithm *possible* by emitting the right substrate, not running it.

### §6.1 — Today's Pipeline (Verified at Framing Time)

Existing files (line counts as of 2026-05-14):

- `forge_bridge/learning/execution_log.py` — 347 LOC. JSONL execution log at `~/.forge-bridge/executions.jsonl`. The source of truth.
- `forge_bridge/learning/synthesizer.py` — 416 LOC. LLM-driven tool synthesis from observed patterns.
- `forge_bridge/learning/manifest.py` — 77 LOC. Tool manifest with provenance.
- `forge_bridge/learning/probation.py` — 95 LOC. Probationary tool gating before full registry promotion.
- `forge_bridge/learning/watcher.py` and `forge_bridge/learning/storage.py` — watcher polling + SQL mirror.

The pipeline's existing logic: count occurrences of similar code patterns in the execution log; when count exceeds threshold, invoke the synthesizer to draft a tool; route through probation; promote to registry on success.

### §6.2 — What Changes at v1.6 (Substrate Only)

The execution log becomes a graph log. Each entry is a node record per §5. The pipeline still reads JSONL and still runs today's threshold-based heuristic; what changes is the *input shape*. Where the pipeline previously counted text-similar code strings, it now counts records whose `kind` and port structure are explicit and machine-readable. Same heuristic, strictly better inputs.

A "shape" is a subgraph signature: ordered list of node kinds + their input/output port names + dataflow edges between them. Two subgraph instances have the same shape if their signatures match. v1.6 **records emissions in a form that makes shape comparison trivial**; it does not ship the shape-comparison-and-promotion algorithm. The §6.3 promotion criteria are the *target* the v1.7 algorithm meets; the v1.6 heuristic still uses today's count-by-similarity logic against the new record format.

### §6.2.1 — What Changes at v1.7 (Algorithm)

v1.7 replaces the heuristic with the structural-mining algorithm: detect recurring shapes across emitted records, isolate stable input/output ports, propose subgraph names, route through quarantine, register on approval. The §6.3 criteria become enforced at promotion time rather than aspirational.

The v1.6/v1.7 split is deliberate: emit the right records first, mine them second. Shipping the mining algorithm against today's text-string records would not produce meaningfully different results. Shipping it against v1.6's structured records does — but only after enough records have accumulated to validate the algorithm empirically (§13.2 trigger condition).

### §6.3 — Promotion Criteria (Strong Recommendation, Open for Writer's-Room)

A subgraph is **eligible for promotion** if all hold:

1. **Recurrence count ≥ N** (default N=5, matching today's threshold for vocabulary continuity).
2. **Structural stability** — the subgraph shape has been observed unchanged across all N occurrences. If the shape varies (extra nodes, reordering, port changes), it's a candidate for *parametrized* promotion (treat varying ports as parameters) or skip.
3. **Named-port stability** — inputs and outputs are typed consistently across occurrences. A subgraph that takes `shot_id: str` in some occurrences and `shot: Shot` in others is two different subgraphs.
4. **No frame-coupled Python** — `python` nodes referencing local Python frames (closures, locals not passed as params) cannot be promoted; they're not pure under replay. Flag for the synthesizer to rewrite, or skip.
5. **No staged-ops-mutating nodes without explicit gating** — a subgraph containing a node that mutates the entity graph through staged-ops gets the `requires_approval` attribute carried forward when promoted.

### §6.4 — Promotion as Graph Collapse

The act of promotion is: take the subgraph definition, register it in the subgraph registry under a name (LLM-proposed or operator-named), record its signature, and rewrite future occurrences of the same shape as `subgraph` kind nodes referencing the registered name. Earlier occurrences in the execution log are *not* rewritten — provenance is preserved.

This is the structural elegance the fine-grained decision bought. Promotion is a graph operation, not a code-rewrite operation. The "promoted tool" is the subgraph itself, accessible by name, replayable, editable in v2.0's schematic.

### §6.5 — Why This Subsumes Macros Cleanly

Macros (PR33-34, `forge_bridge/console/_macros.py`, 135 LOC, persisted at `~/.forge_macros.json`) are explicitly named saved chains. In the new model, a macro is a named saved subgraph of kind `chain`. The existing `register_macro(name, chain)` / `get_macro(name)` / `expand_macro(message)` API becomes a thin adapter over the subgraph registry. Storage migrates from `~/.forge_macros.json` to the same JSONL store as other subgraphs, with the macro registry as a view.

This migration is *not* required for v1.6 to ship. Macros can keep their existing file and existing code path; the framing simply names them as a subset of the subgraph model. Vocabulary leads; code follows. The migration would be a clean fast-follow in v1.6.x or v1.7.

## §7 — Proto-Node Emission Contract

Every v1.6 surface emits node records per §5. This section names the surfaces, names what each emits, and names the contract that makes the records cross-surface consistent.

### §7.1 — Producing Surfaces

- **CLI** (`fbridge run`, `fbridge exec`, `fbridge flame-exec`, `fbridge chat`/`fbridge ask`) — each invocation produces one root node + any nested children. Emitted on completion (success or failure).
- **Chat endpoint** (`/api/v1/chat`) — each chat turn produces one `chat` node whose internal subgraph contains the tool calls the LLM authored. Already produces a `tool_trace` structure in `ChatTurnResult`; the v1.6 work is shaping that into proto-node records.
- **Chain engine** (`/api/v1/exec`) — each chain produces one `chain` node containing the per-step `tool` nodes.
- **Console exec view** (new at v1.6) — each manual exec produces a `python` or `tool` node depending on what the operator submitted.
- **MCP tools** — each tool invocation produces a `tool` node. The MCP server records emission as a side effect, identical shape to CLI/chat-originated invocations.
- **Flame hook** (`POST :9999/exec`) — each Flame Python execution produces a `python` node with `runtime: flame`. The hook itself emits or the calling surface emits depending on which side of the wire owns the record; framing recommends the *calling* surface emits to keep the hook stateless.

### §7.2 — Consuming Surfaces

- **Learning pipeline** — at v1.6, consumes node records using today's threshold-based heuristic (now running over structured inputs instead of code strings — same algorithm, cleaner inputs per §6.2). **Structural subgraph-pattern detection ships in v1.7**, against the substrate v1.6 has been emitting for ≥30 days (§6.2.1, §13.2).
- **Console execs view** — currently renders flat execution log; v1.6 makes it render node records grouped by graph.
- **Future schematic** (v2.0) — renders both entity and operation graphs; the v1.6 records are its input.
- **`fbridge doctor`** — may surface graph store health (record count, last write time, parseability) per the §13 diagnostics extension.

### §7.3 — Contract Invariants

Every emitted record must:

1. **Match the §5 schema for its declared kind.** Schema-violating records are rejected at emission; the emitter raises rather than silently drops.
2. **Carry stable provenance.** `provenance.surface` is one of the named producers; `provenance.request_id` is unique per top-level invocation; `provenance.user` is best-effort (often null in v1.6 pending SEED-AUTH).
3. **Reference other nodes by ID only.** No embedded sub-records inside a record (e.g., a `chain` node lists its children by ID; child records exist separately in the same store).
4. **Be append-only.** Records are immutable once emitted. Status updates (pending → running → succeeded) happen by emitting *new* records that supersede prior state — or by writing a sibling `status_event` record (v1.6 framing leaves this choice for writer's-room — see §12).
5. **Survive replay.** Replaying a graph against the same inputs should produce structurally identical (but new-ID-bearing) records. Non-determinism (LLM nondeterminism, time-dependent tools) is flagged in the record so the replay surface can warn.

### §7.4 — What Goes Wrong If We Don't Enforce This

Bridge has been carrying a `tool_trace` shape inside `complete_with_tools()` that's chat-handler-specific. CLI invocations produce a different shape (call_wrapper's timeline+events). The Console execs view produces a third shape. The learning pipeline reads a fourth (JSONL executions). At v1.5, these are four separate data flows that don't compose.

The proto-node emission contract is the moment we unify them. Without enforcement, v1.7's graph runtime gets four legacy shapes to migrate, and the subgraph mining algorithm has to handle each — which is exactly the v2.x rewrite the framing is intended to prevent.

## §9 — The Three-Tier Loop in Runtime Terms

`SEED-NODE-SCHEMATIC-V1.6+` §"How Existing Surfaces Map" stated the loop in graph terms: deterministic *uses*, generative *grows*, staged-ops *gates*. v1.6 binds this to runtime contracts.

### §9.1 — Deterministic Tier (Uses)

Surfaces: `fbridge run`, `fbridge exec`, `fbridge flame-exec`, the chain engine, the macro expander, the Console exec view, the MCP tool registry consumed by external clients.

Semantics: every invocation consumes named entries from the registry — registered tools, named subgraphs, named macros. The invocation produces node records but does not extend the registry. The runtime is a *reader* of the registry, a *writer* of the operation graph.

Failure mode: the named entry doesn't exist. The deterministic tier returns a structured error; the operator decides whether to escalate to the generative tier.

### §9.2 — Generative Tier (Grows)

Surfaces: `fbridge chat`, the chat endpoint `/api/v1/chat`, the Flame foundry dialog (per SEED-FLAME-CHAT-FOUNDRY), the synthesizer pipeline.

Semantics: every chat turn may author new nodes (`tool` invocations of existing tools, `python` snippets, ad-hoc compositions). The chat node's internal subgraph is the proposal. If the subgraph contains only existing tool invocations, the turn is *use*, not *growth* — it executes within the existing registry. If the subgraph contains Python or ad-hoc compositions, it's *growth* — eligible for subgraph mining and promotion downstream.

The generative tier is the *writer* of the operation graph at points where the registry doesn't yet cover the operator's intent. Per the foundry seed: "Without an LLM entry point in Flame, the registry can't grow from the surface where artists actually work."

### §9.3 — Staged-Ops Tier (Gates)

Surfaces: the existing v1.4 staged-operations platform — `staged_operations` table in `forge_bridge/store/`, MCP tools (`forge_list_staged`, `forge_get_staged`, `forge_approve_staged`, `forge_reject_staged`), the Console staged view.

Semantics: a generative tier proposal that mutates the entity graph (creates a Shot, renames Media, publishes a Version) routes through staged-ops. The operation node carries `requires_approval=true`. Until approved, the node's `status` is `pending`. Approval transitions the status to `running`; execution proceeds.

The staged-ops tier is the *gate* between proposal and effect. It does not write to the operation graph itself — it controls when proposed operation nodes get to execute.

### §9.4 — The Bootstrap Loop, Closed

Per `SEED-FLAME-CHAT-FOUNDRY-V1.6+` §"Bootstrap Loop Completion": the 10-step end-to-end loop. v1.6 closes it.

1. Operator invokes deterministic dialog with a command. → CLI or Console.
2. Deterministic path resolves and runs against the registry. → Existing chain engine.
3. If deterministic resolution fails, the failure surface offers escalation to generative.
4. Generative tier receives the original intent. → Chat endpoint or Flame foundry dialog.
5. LLM authors a subgraph: existing tool invocations, ad-hoc Python, or proposed new tool.
6. Subgraph executes; if it mutates entity graph, staged-ops gates approval.
7. Operator approves or rejects. Approved nodes execute.
8. Recurring subgraphs across multiple operators/sessions become candidates for promotion.
9. Promoted subgraphs enter the named subgraph registry; available to the deterministic tier as `subgraph` kind nodes.
10. Next operator invocation: deterministic tier resolves the named subgraph directly.

At v1.6, steps 1-7 are end-to-end usable. Step 8-10 are *substrate-ready* — the runtime emits records the learning pipeline can mine, but the structural-subgraph-mining algorithm itself ships in v1.7. v1.6's promotion path is still threshold-based, just operating over node records instead of code strings.

This is the operational-maturity-not-architectural-completeness shape the framing commits to: v1.6 makes the loop usable end-to-end; v1.7 makes the promotion structurally clean; v2.0 surfaces the schematic.

### §9.5 — Protocol Normalization

The Phase 24.1 protocol-path instrumentation (preserved at `.planning/PROTOCOL-VS-SUBSTRATE-INVESTIGATION.md`; synthesized at `.planning/THREE-DOMAIN-DECOMPOSITION.md`) revealed an operational reality worth naming explicitly in v1.6 framing: under qwen2.5-coder:32b with large MCP tool envelopes, the model reasons correctly about the desired operation but consistently fails to serialize that reasoning into Ollama's native structured `tool_calls` payload. Semantically-correct operation intent is present every turn; the protocol-serialization shape is not.

The Bug-D salvage path (introduced at Phase 16.2 as recovery for malformed model output) is now operationally the *primary* protocol adapter under this configuration. Earlier framing implicitly treated salvage as a degraded fallback; the measurements show it functions as the canonical normalization layer between provider output and dispatch.

Architecturally:

- **Provider-native tool protocols are ingestion formats.** They are one possible shape model output arrives in.
- **The canonical internal object is normalized operation intent** — operation, tool name, arguments, provenance — provider-agnostic.
- **Protocol normalization is a first-class runtime layer**, not an emergency recovery path.

This has direct implications for the generative tier (§9.2) and the proto-node emission contract (§7): the protocol-normalization layer is what maps raw provider output to the canonical operation-intent object that emits as a node record. v1.6 implementation work in this area treats salvage as designed normalization, not transient compensation; provider-native parsing as ingestion, not canonical representation.

**Out of v1.6 scope:** formalizing the protocol-normalization layer as a named module, defining the canonical operation-intent type, or rewriting salvage as designed primitive. Those land in the four directional paths opened at `.planning/THREE-DOMAIN-DECOMPOSITION.md` §"What this artifact opens" (Paths A + B) and route through v1.6 phase planning, not this framing. The framing's contribution is to constrain how those paths get discussed — provider-agnostic operation intent, not provider-native payload shape.

The gap between each tier is deliberate; it lets the architecture mature in production before the marketing claim attaches to it, and it keeps each milestone's promise sized to what that milestone actually delivers.

## §12 — Open Questions for Writer's-Room Convergence

These are the decisions deliberately *not* closed in framing. Writer's-room converges them before v1.6 phases open.

### §12.1 — Schema-Level

1. **`status` transitions: in-place mutation or append-only event log?** The framing leans append-only (sibling `status_event` records) for replay correctness, but in-place mutation is simpler. Decide based on whether v1.6 replay is a feature.
2. **`provenance.user` capture in absence of auth.** SEED-AUTH is v1.6+ but probably not v1.6.0. What goes in `provenance.user` today? `os.getuser()`? The IP-based rate-limit key? Null? Affects subgraph mining (does "same user across sessions" weight promotions?).
3. **`metrics` field placement.** Sibling to base record or separate stream? Sibling is simpler; separate is cleaner for query workloads. v1.6 default is sibling unless an argument lands for separation.
4. **Node ID format.** UUIDv7 vs ULID vs monotonic-counter. UUIDv7 is sortable + standard; the framing leans here.
5. **Schema versioning policy.** v1.6.0 ships `schema_version: 1.6.0`. Compatibility policy through v1.7 / v2.0 — strict, lenient, deprecation periods?

### §12.2 — Storage

6. **JSONL file granularity.** One file per graph (framing default) vs one file per day vs one file per surface. Per-graph is cleanest for replay; per-day is cleanest for log management. Pick one.
7. **Retention policy.** Append-only logs grow forever. v1.6 default: no rotation. v1.7+: rotation policy. Confirm v1.6 doesn't need it.
8. **Concurrency.** Multiple surfaces emit concurrently. JSONL append is atomic per line on POSIX; verify on the target platforms (Linux flame-01, macOS portofino). Fallback if not.

### §12.3 — Surface Integration

9. **Chat endpoint response shape.** Add `node_record` + `subgraph` to the response (framing default) or keep response as-is and emit records server-side only? Affects API surface; either is defensible.
10. **CLI exit codes for graph operations.** Existing CLI has 0/1/2/3 exit codes (ok/fail/unreachable/timeout). Graph operations introduce new failure modes (subgraph not found, emission failed). Extend the code set or fold into existing?
11. **MCP tool emission point.** Server-side (FastMCP-wrapping) or client-side (in each tool's implementation)? Server-side is DRY; client-side is more flexible. Framing leans server-side.

### §12.4 — Subgraph + Promotion

12. **Promotion criteria N (recurrence count).** Framing default N=5 matching today's threshold. Empirical evidence from v1.6.0 deployment may suggest higher or lower.
13. **What counts as "same shape" for promotion.** Strict structural match vs structural match modulo named parameter renames vs canonicalized form? Framing leans strict; writer's-room may relax.
14. **Subgraph name authorship.** LLM-proposed name + operator-rename window? Operator-only? Auto-generated `subgraph_a73f`? Affects vocabulary cleanliness in the registry.
15. **Macro migration timing.** v1.6.0 (vocabulary only) vs v1.6.x (vocabulary + code) vs v1.7 (full migration including file format)?

### §12.5 — Foundry

16. **Foundry inclusion in v1.6.0 or v1.6.1.** Framing recommends v1.6.1 to keep v1.6.0 small; foundry seed is willing to slip.
17. **Foundry system-prompt bias.** Per foundry seed §"Open Questions": should the chat endpoint reward "synthesize a tool" over "give a conversational answer"? Affects v1.6 chat-endpoint behavior broadly.

### §12.6 — Scope-Adjacent

18. **`fbridge doctor` graph-store check.** Add a `graph_store` row (file existence, last write time, parseability sample)? Framing leans yes; aligns with Phase 23's diagnostics-as-architectural-teaching pattern.
19. **Console execs view rendering.** Current flat log → grouped-by-graph: how aggressive? Show only graphs with multiple nodes? Always group? Affects v1.6.0 UI work scope.
20. **CLI `fbridge graph` subcommand sketch.** Even though full graph CLI is v2.0, should v1.6 ship a minimal `fbridge graph list` / `fbridge graph show <id>` for debugging? Could be 50 LOC of read-only commands and significantly aid v1.6 development.
