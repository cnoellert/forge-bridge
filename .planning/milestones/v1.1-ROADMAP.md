# Milestone v1.1: projekt-forge Integration

**Status:** ✅ SHIPPED 2026-04-19
**Phases:** 4-6
**Total Plans:** 13 (4 + 5 + 4)

## Overview

Where v1.0 stood forge-bridge up as a standalone canonical package, v1.1 delivered the payoff: **projekt-forge now consumes forge-bridge as a pip dependency** instead of duplicating its source. Three sequential phases hardened forge-bridge's public API (Phase 4), rewired projekt-forge's imports to the pip surface (Phase 5), and wired forge-bridge's learning pipeline into projekt-forge's LLM config, per-project log paths, and storage callback (Phase 6). Each phase shipped independently — v1.0.1 (Phase 5 prep), v1.1.0 (Phase 6 API release), and v1.1.1 (PATCH-01 `__version__` exposure) — with annotated tags on origin and wheel + sdist assets on the v1.1.0 and v1.1.1 GitHub releases. Post-shipping live UAT against a real Ollama on assist-01 verified cross-process log isolation, router routing, and enriched-prompt end-to-end delivery.

## Milestone Summary

**Decimal Phases:**

- None. All v1.1 phases were planned integer phases (no urgent-insertion decimals).

**Patch Releases:**

- **v1.0.1** (2026-04-15, Phase 5-00): Upstream 4 projekt-forge fixes into forge-bridge (query_lineage/query_shot_deps/media_scan builders, entity_list narrowing, ref_msg_id correlation fallback, timeline T0 gap-fill). Prerequisite for Phase 5 Wave B pip adoption.
- **v1.1.0** (2026-04-17, Phase 6-03): Public API additive bump — 4 new symbols (`ExecutionRecord`, `StorageCallback`, `PreSynthesisContext`, `PreSynthesisHook`); `__all__` grew 11 → 15; annotated tag on origin; wheel + sdist on GitHub release.
- **v1.1.1** (2026-04-19, PATCH-01): Expose `forge_bridge.__version__` via `importlib.metadata` — one-line fix + regression test, full release ceremony.

**Key Decisions:**

- **Clean break on API renames** — no aliases, no `_module_level_synthesize` — pre-1.0 philosophy extended into v1.1 since projekt-forge is the only consumer and aliases are dead weight
- **Injectable `LLMRouter` with arg→env→default precedence** — consumers need deterministic config without env-var side effects
- **`register_tools()` post-run guard + public `startup_bridge`/`shutdown_bridge`** — lifecycle contract for pluggable consumers; prevents silent no-op registrations
- **`ExecutionLog.set_storage_callback()` is per-instance**, sync/async detected at registration via `inspect.iscoroutinefunction`, dispatch after `fcntl.LOCK_UN`, callback failure isolated (WARNING logged, JSONL write unaffected)
- **`SkillSynthesizer.pre_synthesis_hook` is additive-only** — base `SYNTH_SYSTEM`/`SYNTH_PROMPT` never replaced; hook failure falls back to empty `PreSynthesisContext` and synthesis continues
- **SC #3 scope-reduced to "log-stream mirror"** — `_persist_execution` is a logger-only stub in projekt-forge; SQL backend (EXT-03) deferred to v1.1.x or v1.2 (the `ExecutionRecord` contract is stable so the callback body swaps cleanly later)
- **Non-goal: no hot-reload of `LLMRouter`** — build once at consumer startup, restart to pick up config changes
- **Non-goal: no shared-path JSONL writers** — `fcntl.LOCK_EX` serializes writes but `_counters` is per-process state, so two processes on the same file would duplicate-promote. SC #1 proves **path isolation**, not shared-path concurrency.
- **Minor-version bump ceremony** — barrel re-export → pyproject.toml → regression test → annotated tag on main → push — locked at v1.1.0 and reused at v1.1.1, reusable for v1.2+

**Issues Resolved:**

- Fork divergence between forge-bridge and projekt-forge — 179 `from forge_bridge.*` imports + 8 bare-module refs + 30+ `patch()` targets all rewritten to pip surface (Phase 5)
- Duplicate tool trees (catalog, orchestrate, scan, seed) deleted from forge-bridge, surfaced via `register_tools()` from projekt-forge (Phase 5)
- `_startup`/`_shutdown` private lifecycle renamed to `startup_bridge`/`shutdown_bridge` with injectable args (Phase 4)
- LLM routing config in projekt-forge previously came from env vars; now from `forge_config.yaml` via DI (Phase 6)
- Cold-install smoke tests expected `forge_bridge.__version__` (standard convention) — fixed in v1.1.1 PATCH-01 (Phase post-close)
- Flame hook deployment on this machine — `scripts/install-flame-hook.sh` + GitHub wheel asset close the "standalone installer" gap
- Code-review WR-03: `sys.path.insert()` hack in `tests/conftest.py` moved to idiomatic `[tool.pytest.ini_options] pythonpath = ["."]` in `pyproject.toml`

**Issues Deferred:**

- **EXT-01** (Shared synthesis manifest between repos) — scope unclear; revisit after EXT-02 clarifies what metadata needs sharing
- **EXT-02** (Tool provenance in MCP annotations) — the `.tags.json` sidecars are being produced by Phase 6-02; consumer-side surfacing punted to v1.2
- **EXT-03** (SQL persistence backend for `ExecutionLog` via `typing.Protocol` + Alembic chain) — punted to v1.2 per SC #3 scope-reduction decision
- **WR-01** (async storage-callback failure-path tests) — advisory finding from code review; punt to v1.2 opener as hygiene
- **WR-02** (`ExecutionRecord` docstring vs `mark_promoted` schema-drift risk) — advisory; punt to v1.2 opener
- **3 live-stack human-UAT items** persist in `06-HUMAN-UAT.md` marked `complete` (all passed on 2026-04-18 against assist-01's Ollama running `qwen2.5-coder:32b`)
- **README conda-env guidance** (mirror projekt-forge's "create forge env → pip install -e ." pattern) — flagged during install/deploy discussion on 2026-04-19; small polish item for v1.2 opener

**Technical Debt Incurred:**

- `_persist_execution` in projekt-forge is a logger-only stub — EXT-03 in v1.2 swaps the body for a real DB write (the `ExecutionRecord` contract is stable, so the swap is isolated to one function)
- `tests/conftest.py` site-packages fixture in projekt-forge is intentionally strict (fails if the repo is editable-installed) — guard works as designed but couples local dev loop to pip release cadence
- projekt-forge's `pyproject.toml` needs `[tool.hatch.metadata] allow-direct-references = true` to resolve the forge-bridge git-URL pin; this is declared in projekt-forge, not forge-bridge, but worth noting for any future consumer adopting the same pin pattern
- projekt-forge declares `aiosqlite` only as a transitive test dep — surfaced during v1.1.0 human-UAT when running tests from the conda `base` env. Projekt-forge concern, not forge-bridge, but the discovery documents a real footgun.
