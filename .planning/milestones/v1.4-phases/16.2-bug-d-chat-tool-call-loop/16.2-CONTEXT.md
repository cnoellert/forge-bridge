## Implementation Decisions

All four decisions below are **locked** — derived directly from the four strong-reco recommendations the user accepted in a single batch on 2026-04-28.

### G-1: Diagnosis methodology — bottom-up, test-first

- **D-01 (locked):** Diagnosis order is bottom-up against the three plausible loci (Ollama HTTP response → adapter parse → router loop → handler dispatch → UI render). The lowest layer where the loop terminates without iterating is where the bug lives, and starting at the bottom gets the cheapest definitive answer. The router/handler/UI loci are unlikely given the symptom (loop returns after one Ollama round-trip with the JSON as terminal content).

- **D-02 (locked):** Test-first methodology. Each diagnostic step lands as a failing test before any fix is applied. Concretely, the planner builds the following sequence:
  1. **Capture step (one-shot, no commit):** On assist-01, capture Ollama's raw HTTP response body to a chat-with-tools call that reproduces Bug D. This may be done via `OLLAMA_HOST=...`+ verbose curl, `mitmproxy`, or simply enabling debug logging on `OllamaToolAdapter.send_turn()`. The output is the canonical fixture for D-04.
  2. **Adapter unit test (RED):** Write a failing test in `tests/llm/test_ollama_adapter_parse.py` (or extend an existing adapter test file if one exists — planner's call) that feeds the captured response shape to `OllamaToolAdapter.parse_response()` and asserts a non-empty `tool_calls` list. This RED commit locks the diagnosis as a regression guard before the fix lands.
  3. **Fix (GREEN):** Whatever the captured fixture reveals — most likely a fallback text-parser in the Ollama adapter when structured `tool_calls` is empty but `content` matches a tool-call JSON shape, OR an Ollama client request-side change to force structured tool emission, OR both. The planner picks based on what step 1's evidence shows.

- **D-03 (locked):** **Strong working hypothesis (must verify before locking the fix):** The Ollama adapter at `forge_bridge/llm/_adapters.py:395` reads `message.get("tool_calls") or []` from Ollama's response. qwen2.5-coder:32b is known to emit tool calls as JSON-shaped *text* in `message["content"]` when its tool-format negotiation falters — bypassing Ollama's structured `tool_calls` field. The adapter then sees zero tool calls → router terminates at `router.py:435` (`if not response.tool_calls:`) → returns the raw JSON-as-text as the terminal answer. The < 5 s response time on the assist-01 UAT, combined with the JSON shape, matches this exactly. **Phase 16.2 must verify this hypothesis with the captured fixture before applying any fix; if the fixture shows a different failure mode, the planner re-opens this gray area.**

- **D-04 (locked):** Captured fixture lives in the test file as a verbatim `_OLLAMA_RESPONSE_TEXT_TOOL_CALL` constant or fixture file. Operator notes (date captured, model version, prompt used) live alongside as comments. This is reproduce-from-disk evidence that future regressions can replay deterministically.

### G-2: Strategy B test-strengthening — two-pronged

- **D-05 (locked):** Adapter-level unit test (per D-02 step 2) is one of the two prongs. Lives in `tests/llm/test_ollama_adapter_parse.py`. Always-on (no `FB_INTEGRATION_TESTS` gate — uses captured fixture, no live Ollama).

- **D-06 (locked):** `tests/integration/test_chat_endpoint_live.py::test_chat_canonical_uat_prompt_under_60s` gets two new assertions on top of the existing structural ones:
  1. **Reject raw tool-call JSON as terminal content.** Match against a regex like `^\s*\{\s*"name"\s*:` (and similar permissive variants for whitespace / pretty-printed JSON). If the assistant's terminal `content` matches, FAIL with a message that explicitly cites Bug D so future regressions are diagnosed at the assertion site.
  2. **Assert the loop iterated.** The response `messages` list MUST contain at least one `role="tool"` turn AND at least two `role="assistant"` turns (one with `tool_calls`, one terminal text). This proves `complete_with_tools()` actually executed the tool and re-prompted the LLM with the result. The bare minimum healthy shape is: `[user, assistant(tool_call), tool, assistant(text)]` — 4 messages. Assertion: `len([m for m in messages if m["role"] == "tool"]) >= 1 and len([m for m in messages if m["role"] == "assistant"]) >= 2`.

- **D-07 (locked):** Reject any "natural prose detection" heuristic (e.g., counting word boundaries, asserting non-JSON via heuristic content shape). Too brittle. The two assertions in D-06 are surgical: they fail on the exact symptom of Bug D (JSON-as-terminal-text + no iteration) and don't false-positive on edge-case but legitimate responses (e.g., a one-word "yes" answer to a follow-up).

### G-3: Fresh-operator UAT path — D-14 soft-PASS with stricter technical floor

- **D-08 (locked):** Phase 16.2 closes with `Outcome: PASS with deviations` per Phase 10.1 / 16.1 D-14 precedent IF AND ONLY IF all three of the following hold:
  1. Strategy B E2E (with D-06's strengthened assertions) PASSES on assist-01 with live `qwen2.5-coder:32b` on Ollama.
  2. Adapter-level unit tests (per D-05) PASS in the default `pytest` run on dev and on assist-01.
  3. CN/dev manual walkthrough of the canonical CHAT-04 prompt at `http://localhost:9996/ui/chat` produces a useful natural-language answer (not raw JSON, not refusal, not loop_budget). Walkthrough is recorded in `16.2-HUMAN-UAT.md` with a screen capture (image attached or pasted verbatim text from the chat panel).

- **D-09 (locked):** D-08 is the **technical floor** — none of the three conditions can be skipped. Phase 16.1's deviation was the absence of (3); the additional fail was that Bug D itself meant (3) couldn't have been recorded as PASS even if attempted. Phase 16.2 cannot repeat that pattern. The deviation acceptable here is "no fresh artist; CN/dev re-test against the strengthened automated floor" — and the floor must hold.

- **D-10 (locked, ratification path):** If a fresh artist becomes available before v1.4 milestone close, the operator replays the canonical CHAT-04 prompt at `http://localhost:9996/ui/chat` and their observation is appended to `16.2-HUMAN-UAT.md` as **ratification**. This does NOT re-open the phase. Ratification strengthens the close-with-deviation record; it does not gate it.

- **D-11 (locked, rejected paths):** Reject "hard-block until fresh artist available" — calendar already held hostage by Phase 16.1's identical constraint, another stall serves nothing. Reject "accept CN/dev manual walkthrough alone without the strengthened technical floor" — that was the gap Bug D opened.

### G-4: Scope boundary — Bug D only

- **D-12 (locked):** In scope: Bug D root-cause investigation per D-01..D-04, fix per D-03 (or whatever the captured fixture reveals), Strategy B test strengthening per D-05..D-07, fresh-operator UAT path per D-08..D-11.

- **D-13 (locked, out of scope):** Health strip "Degraded" rendering with mixed red/green dots seen on the chat panel during 16.1's UAT is OUT of scope. On bare assist-01 (no Flame on `:9999`, no projekt-forge bridge on `:9998`), the health probes for those backends SHOULD return unreachable; "Degraded" with reds for those services is the correct signal. If during 16.2 investigation we discover a dot is actually wrong (green-vs-red inverted, wrong service mapped, etc.), that becomes a separate Phase 16.3 / v1.4.x issue, NOT a scope expansion of 16.2.
