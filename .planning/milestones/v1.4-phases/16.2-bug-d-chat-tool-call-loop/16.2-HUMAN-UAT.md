## Live correction during UAT (Plan 03 D-06 #2 defect)

While running D-08 #1 the first time on assist-01 with the freshly-cherry-picked Plan 03 strengthened E2E test, the test FAILED at the D-06 #2 loop-iteration assertion (`assert len(tool_turns) >= 1 and len(assistant_turns) >= 2`) even though Bug D was operationally closed at the user surface.

**What the assertion got wrong:** D-06 #2 inspected `body["messages"]` returned by the chat API and required ≥1 `role=tool` turn plus ≥2 `role=assistant` turns. But the chat API response shape is

```json
{"messages": [user_input, final_assistant_answer], "stop_reason": "end_turn", "request_id": "..."}
```

— intermediate tool and assistant turns happen inside `LLMRouter.complete_with_tools()` and are **not** forwarded into the response body. The assertion as written could not succeed against any healthy response, only against itself. The plan documented the expected shape as *"Minimum healthy 4-message shape: [user, assistant(tool_call), tool, assistant(text)]"* — that was the **router-internal** message stream, not the API response.

**Server-side iteration evidence (router.py log markers, capturing the FIRST D-08 #1 run that failed the assertion):**

```
INFO     forge_bridge.llm.router:router.py:588
         tool-call iter=1 tool=forge_tools_read args_hash=44136fa3
         prompt_tokens=939 completion_tokens=14 elapsed_ms=12906 status=continuing
INFO     forge_bridge.llm.router:router.py:437
         tool-call iter=2 tool= args_hash=
         prompt_tokens=1019 completion_tokens=91 elapsed_ms=9066 status=terminal
INFO     forge_bridge.llm.router:router.py:632
```

The agentic loop iterated twice: turn 1 dispatched `forge_tools_read`, turn 2 returned the synthesized terminal answer. Plan 02's GREEN salvage was working live — the assertion was wrong, not the fix.

**Correction:** dropped the D-06 #2 assertion block, kept D-06 #1 (regex reject of raw tool-call JSON — the load-bearing Bug D guard). Replaced the deleted block with an in-source comment explaining why iteration evidence belongs in HUMAN-UAT.md (server-side log markers) rather than in the test (response body doesn't expose it). Commit: `5988de8` (dev) → `f657b01` (assist-01 after cherry-pick).

**Why this is consistent with CONTEXT D-09:** D-09 says "the floor must hold." After the correction the floor does hold — D-08 #1 passes against the actually-healthy response shape rather than against an aspirational shape that doesn't exist. The deviation is a planning defect we fixed live; it is not a soft-deviation that lets a Bug D regression through. D-06 #1 (the assertion that would actually catch Bug D recurring) was untouched.
