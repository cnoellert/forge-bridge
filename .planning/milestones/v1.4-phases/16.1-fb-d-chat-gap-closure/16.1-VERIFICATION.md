### Gaps Summary

**Plans 01–04 shipped clean.** All four Wave 1/2 technical must-haves are satisfied:

- Bug C (49-tool hang) is fixed by the backend-aware filter in `_tool_filter.py`, wired into `chat_handler`, with 503 fallback and `tools_offered_count` log telemetry.
- Bug A class prevention (TemplateResponse drift) is locked by the parameterized UI-render guard (`test_ui_handlers_render.py`) and the Starlette migration + pin drop.
- Bug B class prevention (lost `llm_router=` kwarg) is locked by the boot-wiring smoke test (`test_lifespan_wiring.py`) and the `_canonical_console_read_api` module global.
- Bisection evidence (D-06) is committed — 9 JSONL measurements, clean threshold at tool_count=20–30. Strategy B chat E2E (D-07 #3) exists with real LLMRouter and real mcp.list_tools().

**Plan 05 did not satisfy CHAT-04.** Two distinct failure conditions:

1. **Operator deviation:** CN/dev substituted for a fresh artist. No fresh artist was available. Per D-14, a deviation alone would qualify for `PASS with deviations` if the underlying answer was useful. It was not.

2. **Bug D (new structural failure):** The chat surface terminated after the LLM's first-turn tool call, returning the raw tool-call JSON `{"name": "forge_tools_read", "arguments": {"name": ""}}` as the assistant's response text. The tool was never executed. No tool result was fed back to the LLM. No synthesized natural-language answer was produced. This is distinct from Bugs A/B/C — it is in the tool-call execution / loop continuation / rendering pathway, which Phase 16.1 did not modify.

**Strategy B test gap (follow-up for Phase 16.2):** `test_chat_canonical_uat_prompt_under_60s` PASSED at 2.02s on assist-01 at the same time as the UAT FAILED. The test asserts `stop_reason == "end_turn"`, `len(content) >= 40`, and `elapsed < 60`. All three are satisfied by the raw-JSON response Bug D produced. The assertion must be strengthened in Phase 16.2 so it cannot pass on a tool-call JSON string.

**Root cause for Phase 16.2 (three candidate loci documented in HUMAN-UAT.md):**
- (a) `LLMRouter.complete_with_tools()` may exit the loop after one Ollama turn when a tool call is emitted rather than continuing the execute-tool → re-prompt cycle
- (b) The chat handler may not be wired to call `mcp.call_tool()` and feed results back into `complete_with_tools()`
- (c) The UI may be receiving a structured tool-call response body and rendering it verbatim as text

**Action:** Open Phase 16.2. v1.4 milestone close remains blocked until Phase 16.2 closes Bug D and a fresh-operator UAT records `Outcome: PASS` (or `PASS with deviations` per D-14 with non-must-have deviations only).
