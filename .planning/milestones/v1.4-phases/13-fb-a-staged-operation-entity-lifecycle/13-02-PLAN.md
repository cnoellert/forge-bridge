Ship the `StagedOperation` application class — the single source of truth for
the JSON shape that FB-B's MCP tools and HTTP routes will return verbatim
(STAGED-06 zero-divergence test anchor). This class must override the
`entity_type` property to return the literal string `"staged_operation"` —
the #1 silent-bug risk in the entire phase, because `BridgeEntity` derives
`entity_type` from `cls.__name__.lower()` which would yield `"stagedoperation"`
and break the `EntityRepo.save` discriminator write at `repo.py:254`.

Purpose: Plan 03's repo state machine constructs `StagedOperation` instances
on every fetch and serializes them on every save. Plan 04's tests assert
`fetched.proposer`, `fetched.parameters`, `fetched.status` round-trip cleanly.
Without this class shipped first, neither Plan 03 nor Plan 04 has anything to
import. FB-B (Phase 14) imports this class verbatim.

Output: One new module (`forge_bridge/core/staged.py`) and one barrel update
(`forge_bridge/core/__init__.py`). Per Claude's Discretion in CONTEXT.md, the
class lives in its own module (not inside `entities.py` which is already 22KB)
to keep the original-vocabulary entities focused and to give FB-B a clean
import line.
