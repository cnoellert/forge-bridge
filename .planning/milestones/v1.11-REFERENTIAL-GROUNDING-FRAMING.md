# v1.11 Referential Grounding — Milestone Framing (cycle-1)

> **Theme.** Turn the artist's *references* into the *identifiers the resolved tool
> needs.* v1.10 got reads to resolve to a single tool; they still don't answer
> because the parameter layer never grounds "shot 10 on 30sec_edit 21" into a
> `shot_id`. v1.11 is the **nearest hill** the v1.10 close named and ranked —
> the wall standing immediately behind the resolver wall DI.2 took down.
>
> **One-line thesis.** A read that resolves to one tool should get that tool's
> required parameters **grounded from the prompt + accessible substrate state** —
> not abort with `Field required`. Referential grounding is the difference between
> *"the system chose `forge_get_shot`"* and *"the system answered the question."*

## Why this is the next milestone (not chosen — measured)

The DI.2 T1 baseline (33 samples) already ranked it. Of the 9 dogfood failures, the
3 DI.2 made *reachable* (R8/R9/R10) now resolve to a single tool and then abort at
**parameter resolution** — `forge_get_shot` requires `shot_id`, derivable from "shot
10" + the sequence but never derived. R7 dead-ends one layer over on the same axis
(no project scope → `MULTIPLE_PROJECTS`). This is the **param-resolution successor**
of the decomposed `SEED-COMPILE-QUALITY`, and it is the one with the most reachable,
best-evidenced win: **the reads that v1.10 unblocked are one resolver away from
actually answering.**

## What exists (grounded — this is an extension, not new architecture)

The resolution substrate is already real (`_tool_chain.py`):

- **`_PR25_CHAINS`** — a registry mapping tool → `{requires, resolver}`. Today it
  registers resolvers for **`project_id`** (`_resolve_project_id`), **`sequence_name`**
  (`_resolve_sequence_name`, via 24.11 `resolve_query_entities`), and **`sequence_name`
  + `prefix`** (`_resolve_rename_params`). That's the whole coverage.
- **`resolve_required_params`** — precedence **explicit (PR28) > memory (PR26) >
  deterministic resolver (PR25)**; three-valued resolver contract (`str` single match
  → memory-write + merge; `list[dict]` → `__disambiguation__` sentinel; `None` →
  leave for the `MISSING_*` graceful contract). The explicit-first invariant is
  load-bearing and must not invert.

So v1.11 **extends a working substrate** — register more tools, add resolvers — in the
same shape the project already uses. (Substrate, not rewrite.)

## What's missing (the grounded gap)

1. **No shot resolver, and `forge_get_shot*` are unregistered.** "shot 10 on
   30sec_edit 21" → `shot_id` has no path: there is no `_resolve_shot_id` and
   `forge_get_shot` / `_shot_stack` / `_shot_versions` / `_shot_lineage` aren't in
   `_PR25_CHAINS`, so `resolve_required_params` returns params unchanged and the tool
   aborts. **This is R8/R9/R10's exact downstream abort.**
2. **No session / project scope.** Project-scoped reads with >1 project dead-end on
   `MULTIPLE_PROJECTS` (the disambiguation sentinel surfaced raw) instead of resolving
   to a current-project default or a conversational "which project?". **This is R7.**

## Scope — phase lean (for the room to ratify)

Two phases, evidence-ordered (nearest, most-reachable first):

- **RG.1 — Entity/shot resolution (the reachable win).** Add `_resolve_shot_id`
  (ordinal/name reference + sequence context → `shot_id`, via the existing
  `resolve_query_entities` path or a deterministic substrate lookup) and register the
  `forge_get_shot*` family in `_PR25_CHAINS`. Acceptance: R8/R9/R10 **complete
  end-to-end** (resolve → execute → answer), measured against the DI.2 baseline corpus.
- **RG.2 — Session / project scope.** A current-project context so project-scoped
  reads don't dead-end; resolve-to-default or conversational disambiguation (outcomes,
  not raw `MULTIPLE_PROJECTS`). Acceptance: R7 resolves.

The **DI.2 T1 baseline reads are the built-in acceptance corpus** — measure-first
continuity: re-run them; the RG-reachable reads should move from `chain_aborted` to
answered.

## Success criterion (per-layer, honest — no overclaim)

`[[feedback-distinct-success-criteria-per-adjacent-layer]]`:

- **RG succeeds if** the reads that DI.2 resolved-to-a-tool now **complete and answer**
  (R8/R9/R10 end-to-end; R7 resolves project). That is the parameter-grounding layer's
  native success.
- **RG does NOT claim "chat is useful."** Compile-shape (R2/R5/R6 — malformed/
  over-injected chains), answer-pass fidelity (R3), and the dormant context-eligibility
  seam remain explicitly out of scope. "Make chat useful" is still several milestones;
  this is the next one, not the last.

## Constraints (binding)

- `forge_bridge.__all__` stays **19**; no new external libraries.
- **Substrate-extension, not rewrite** — register tools + add resolvers in the
  `_PR25_CHAINS` shape; preserve the explicit>memory>resolver precedence (do NOT
  invert the explicit-first invariant).
- **Deterministic-first** — grounding stays deterministic where the substrate allows;
  the resolver is a confidence gate, not a guesser ([[feedback-arbitration-boundary-discipline]]).
- **Reads-side** — DI.1's gate untouched; a mutation that grounds its params still hits
  the gate. RG only makes *reads* answer.
- **Measure-first** — the DI.2 baseline corpus is the acceptance instrument; size each
  phase against it, and let a baseline gate any contingent work
  ([[feedback-measure-first-gate-rung-not-built]]).

## Non-goals (named, deferred — not rejected)

- **Compile-shape quality** (malformed/over-injected chains) — separate SEED successor.
- **Answer-pass fidelity** (refuse-on-missing-field, error-envelope triage) — known
  since CR.1.
- **Context-eligibility** (the dormant seam: tools needing *unavailable* context) —
  instrumented, graduation deferred to data.
- **Artist UAT** — re-enters scope only after RG lands (it was struck precisely until
  the compile/param layer moved).

## Open questions for discuss (leans given)

- **Q-RG.1 — ordinal vs name references.** Does "shot 10" mean the 10th shot
  (positional) or a shot named "10"/"shot_010"? *Lean: reuse the 24.11
  `resolve_query_entities` substrate; ground positional-vs-name from the existing
  resolver categories rather than inventing a new heuristic. Grounding read required
  at plan time.*
- **Q-RG.2 — sequence context propagation.** `forge_get_shot` needs the shot within a
  sequence; the sequence ("30sec_edit 21") must resolve first, then scope the shot
  lookup. *Lean: two-key resolution (sequence_name → shot_id), mirroring the existing
  `_resolve_rename_params` multi-param pattern.*
- **Q-RG.3 — session/project scope mechanism.** A current-project default vs a
  conversational "which project?" (outcomes-not-tools, per DI.2's T5 surface). *Lean:
  prefer a session-current-project default with conversational fallback; do not leak
  raw `MULTIPLE_PROJECTS`.*
- **Q-RG.4 — phase split.** RG.1 (shot/entity) then RG.2 (session scope), or combined?
  *Lean: split — RG.1 is the higher-reachable win and lands first; RG.2 follows.*
