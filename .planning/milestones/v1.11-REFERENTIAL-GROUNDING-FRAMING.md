# v1.11 Referential Grounding — Milestone Framing (cycle-2)

> **Theme.** Turn the artist's *references* into the *identifiers the resolved tool
> needs.* v1.10 got reads to resolve to a single tool; they still don't answer
> because the parameter layer never grounds "shot 10 on 30sec_edit 21" into a shot
> identity. v1.11 is the nearest hill the v1.10 close named.
>
> **One-line thesis (still right after DT's probe).** A read that resolves to one
> tool should get its required parameters **grounded from the prompt + accessible
> substrate state.** Referential grounding is the difference between *"the system
> chose `forge_get_shot`"* and *"the system supplied it an identifier."*
>
> **What cycle-2 corrects (DT's probe).** The cycle-1 claim that RG.1 makes
> *R8/R9/R10 complete end-to-end via forge shot resolution* was an **over-claim** —
> grounded against the resolver *mechanism* but not against the actual shot rows.
> Read against the data, the reachable win is **per-attribute and crosses two
> substrates.** That correction reshapes the milestone; the thesis survives, the
> scope sharpens.

## The reframe DT's probe forced (cycle-2 — read the rows before ranking the hill)

"shot 10 on 30sec_edit 21" is a **Flame timeline reference** (sequence + ordinal),
not a forge entity key — and **which substrate answers depends on the *attribute*
asked, not the entity.** Against the live forge shot data (DT-grounded):

| Read | Attribute | In forge shot attrs `{cut_in,cut_out,sequence_id}`? | Where the answer lives |
|---|---|---|---|
| **R10** duration | `cut_out − cut_in` | **yes** | forge-answerable **if the reference grounds** |
| **R9** timewarp | — | **no** | **Flame only** (`flame_get_sequence_segments`) — `forge_get_shot` cannot answer with *any* `shot_id` |
| **R8** path | — (Locatable → locations) | no (locations join) | forge-locations-join **or** Flame — *uncertain* |

**Consequence:** the reachable win is **not 3 uniform reads.** It is ~**1 clean
forge win (R10)**, **1 that is structurally a Flame read (R9)** no amount of
grounding makes `forge_get_shot` answer, and **1 uncertain (R8)**. Setting
acceptance at "R8/R9/R10 complete via forge resolution" would fail for *substrate*
reasons, not grounding reasons. DT's count (~1.5 of 3 through forge) is the honest
starting estimate — **to be locked by the substrate-locating pass below before
scope freezes.**

**This promotes Creative's dormant seam.** R9 is a *live, observed* instance of the
**context-eligibility** seam Creative named last turn as dormant-and-unevidenced:
right entity, right reference, *wrong substrate for the attribute.* DT's probe moved
it from dormant to observed. That is a clean reason to keep RG **narrow** (below).

## What exists (grounded — the registry extension still holds)

The resolution substrate is real (`_tool_chain.py`): `_PR25_CHAINS` (tool →
`{requires, resolver}`; today project_id / sequence_name / rename only),
`resolve_required_params` with **explicit > memory > resolver** precedence and a
**three-valued resolver contract** — `str` (a grounded identifier), `list[dict]`
(disambiguation *among known candidates*), `None` (give up to the `MISSING_*`
graceful contract). Registering a tool + adding a resolver is genuine substrate
extension. *That mechanism claim was correct in cycle-1 and stands.*

## What's missing (the grounded gap, re-scoped)

1. **Ordinal grounding is the load-bearing missing piece** (cycle-1 Q1, promoted).
   Sequence grounding already exists (`_resolve_sequence_name`, 24.11 — it grounds
   "30sec_edit 21" and serves the Flame sequence tools). The gap is "shot 10" →
   a shot/segment index — and the **real data makes it non-deterministic**:
   `tst_010`, `tst_100`, `tst_110` all loosely match "10"; there is no naming
   convention that makes "shot 10" resolve deterministically. This is a *scope
   decision*, not a discuss footnote.
2. **Forge-side sequence scoping is a dangling reference** (cycle-1 Q2, promoted).
   Shots carry a `sequence_id` attribute, but there is **no sequence entity_type in
   forge and no sequence-name→id resolver** — so even forge-side scoping can't map
   "30sec_edit 21" → `sequence_id`. The Flame path (`_resolve_sequence_name` +
   `flame_get_sequence_segments`) is the real grounding route, not forge.
3. **Per-attribute substrate routing** (R9, maybe R8) — *belongs to the
   context-eligibility seam, NOT to RG* (see boundary). Naming it here so RG does
   not silently absorb it.
4. **No session / project scope** (R7) — project-scoped reads dead-end on
   `MULTIPLE_PROJECTS`. A different problem class (which *project universe*).

## The boundary RG must defend (Creative — the milestone's success criterion)

Same architectural shape that has run through the whole arc — **each layer must
refuse to become the layer above it.** DI.1 refused to let safety become usability;
DI.2 refused to let arbitration become recompilation; **RG must refuse to let entity
resolution become query interpretation.**

| In scope — RESOLUTION | Out of scope — INTERPRETATION |
|---|---|
| "shot 10" → `shot_id`/segment index | understanding what the user *meant* |
| "30sec_edit 21" → `sequence_name` | rewriting ambiguous requests |
| "hero version" → `version_id` | choosing between plausible interpretations |
| ordinal *within an already-resolved collection* | re-compiling bad chains |
| — | **per-attribute substrate routing** (→ eligibility seam, not RG) |

**Enforcement hook (Orch — structural, not just doctrinal).** The boundary is
already encoded in the resolver contract: a resolver returns `str` / `list[dict]` /
`None` — **none of those three is a reinterpretation.** A resolver *cannot* return
"maybe you meant…". As long as RG resolvers stay inside the three-valued contract,
the mini-compiler drift Creative warns about is structurally blocked (the RG analog
of DI.2's "selection picks from the provided set, never authors prose"). The one
new temptation — per-attribute routing — is explicitly handed to the eligibility
seam so RG stays a *grounder*, never a *router*.

## Scope — phase lean (re-shaped; for the room to ratify)

- **RG.1-T1 — substrate-locating pass (measure-first; FIRST; the RG equivalent of
  DI.2's T1).** For each reachable read, locate where the answer lives — forge
  attribute / forge locations join / Flame segments — against **live pg + Flame**,
  re-confirming DT's table and resolving R8. Output: the *true* reachable-forge-win
  count and a **test-data-quality assessment** (the DI.2 corpus grades grounding
  against `tst_NNN` + surface-noise references; RG needs ≥1 read whose reference
  *cleanly* grounds in the data, or acceptance fails for data reasons). **Scope does
  not freeze until this lands.**
- **RG.1 — reference grounding (narrow).** Ground the timeline reference (sequence
  via 24.11 + **ordinal grounding**, the missing piece) to a shot identity, and
  register the forge read tools that can then answer a forge-held attribute.
  Acceptance: the cleanly-groundable, **forge-answerable** read completes
  end-to-end (R10 the candidate, pending the pass). RG grounds the reference; it
  does **not** route attributes across substrates.
- **RG.2 — session / project scope (R7).** Current-project context so project-scoped
  reads resolve (default + conversational "which project?", outcomes-not-tools) —
  never raw `MULTIPLE_PROJECTS`. A distinct problem class; lands after RG.1.
- **Per-attribute routing (R9, maybe R8) → the now-observed context-eligibility
  seam.** NOT folded into RG. Its own future phase/milestone, now corpus-evidenced.

## Success criterion (per-layer, honest — corrected)

- **RG succeeds if** a cleanly-groundable, forge-answerable read **completes
  end-to-end** (reference → identifier → execute → answer) — at least R10, measured
  against a corpus containing ≥1 cleanly-groundable reference.
- **RG does NOT claim "R8/R9/R10 all complete."** R9 is structurally a Flame read
  (eligibility seam); R8 is pending the pass. RG grounds references; it does not make
  `forge_get_shot` answer a Flame attribute.
- **RG does NOT claim "chat useful."** Compile-shape, answer-pass, per-attribute
  routing, and session scope (beyond RG.2) remain out of scope.

## The handoff lesson (v1.10 → v1.11 — DT, explicit)

**This is the 4th time this arc a load-bearing premise flipped on a probe** (the
9-tool match · "builtins pass none" · DI.2's arbitration-vs-working-reads · now RG's
reachable-win). Each time the claim that would have *shaped the work* was correct in
mechanism and wrong in the specific data, until the rows got read. **Lesson: ground
the reachable-win against real data rows before ranking the hill** — and for
consumer-fronted entities, that means reading the *consumer/pg* data model, not just
the bridge substrate (the exact reason cycle-1 missed it). Candidate methodology
(a data-layer manifestation of [[feedback-ground-specs-in-actual-files]]); author at
milestone close.
