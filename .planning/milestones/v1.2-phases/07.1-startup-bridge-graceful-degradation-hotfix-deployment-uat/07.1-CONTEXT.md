# Phase 07.1: startup_bridge Graceful Degradation Hotfix + Deployment UAT - Context

<domain>
## Phase Boundary

**What this phase delivers:** A forge-bridge v1.2.1 release whose `python -m projekt_forge --no-db` (and equivalently `python -m forge_bridge`) boots cleanly on a machine with no listener on the standalone forge-bridge WebSocket port (:9998) — serving `tools/list` and `tools/call` over MCP stdio with only `flame_*`/`synth_*` tools functional (forge_* tools degrade per the existing warning log). Plus a re-UAT of PROV-02 _meta envelope end-to-end using a **real MCP client** (the user's Claude Code session registered with projekt-forge as an MCP server), replacing Phase 07-04's raw JSON-RPC harness + monkey-patched launcher.

**Why this phase:** Phase 7 (PROV-02) shipped v1.2.0 publicly, but the UAT exposed a latent deployment-blocking bug in `forge_bridge.mcp.server.startup_bridge`. The existing try/except at that function (lines 140-147 of the installed 1.2.0) only wraps `wait_until_connected()`, not `_client.start()` — which means a ConnectionRefusedError from `ws_asyncio.connect()` propagates up through the MCP server's lifespan and prevents the process from ever serving tool requests. The docstring and warning log at those same lines promise graceful degradation ("forge_* tools will fail. flame_* tools still work if Flame is running."); the code violates that contract. Any user on a machine without the standalone WS server on :9998 (i.e., a local-dev workstation; a machine where the PostgreSQL-backed forge-bridge server isn't running) will hit this today.

Phase 7-04's UAT worked around this with a runtime monkey-patch in `/tmp/run_pf_mcp.py` that wraps `startup_bridge` with an exception-swallowing shim. That made the UAT "pass" but it wasn't actually testing the deploy path — it was testing our patched environment. Phase 07.1's job is to remove that workaround by **fixing the latent bug**, then re-verify PROV-02 through a real MCP client session with no patches of any kind.

</domain>

<decisions>
## Implementation Decisions

### Fix Shape (startup_bridge)

- **Move the try/except** in `forge_bridge/mcp/server.py:startup_bridge` to wrap BOTH `await _client.start()` AND `await _client.wait_until_connected(timeout=10.0)`. Currently only `wait_until_connected` is guarded; `start()` raises first and escapes.
- **On exception:** log the same warning message already present ("forge_* tools will fail. flame_* tools still work if Flame is running."), then call `await _client.stop()` inside a nested try/except (idempotent; safe on partially-started client), then set `_client = None` so the `shutdown_bridge()` global guard (`if _client:`) cleanly skips on shutdown.
- **Behavior contract preserved:** on successful connect, behavior is unchanged. On any connection failure, MCP server boots with `_client = None`, serves `tools/list`/`tools/call` for `flame_*` + `synth_*` tools; `forge_*` tools fail at call-time with the pre-existing ConnectionError path (no new behavior invented). `shutdown_bridge()` is a no-op instead of calling stop() on a half-dead client.

### Regression Test Scope

- **New test file:** `tests/test_mcp_server_graceful_degradation.py` (or similar — planner's call).
- **Nyquist gate:** the test MUST fail against v1.2.0 unpatched AND pass against the v1.2.1 fix. Without this check, we can't trust the fix.
- **Test mechanics:** set `FORGE_BRIDGE_URL` environment variable to a known-unreachable localhost port (use `socket.socket()` to bind+close a throwaway port to find a free-then-closed port, OR hardcode an unused port like `:65432` if portable). Call `startup_bridge()` directly → assert no exception propagates AND `_client is None` AND a warning was logged. Then call `shutdown_bridge()` → assert no exception.
- **Optional end-to-end test:** spawn `mcp.run()` in a thread with the bad URL + bounded test harness that sends one `tools/list` request and asserts a response. More valuable but more complex. Planner's call whether to include.

### Version Management

- **Version bumps:** `pyproject.toml` `1.2.0` → `1.2.1` (PATCH). Also update `tests/test_public_api.py` version guard string (Phase 7-04 already has a line asserting `__version__ == "1.2.0"` — change to `"1.2.1"`).
- **Semver rationale:** this is a pure bugfix with zero API changes. No new public symbols, no signature changes, no `__all__` delta. PATCH release is correct.
- **Release ceremony:** commit with `chore(release): bump version 1.2.0 → 1.2.1 — startup_bridge graceful degradation hotfix`, tag `v1.2.1` (annotated), push branch + tag, create GitHub release with wheel + sdist built via `python -m build`. <specifics>
## Specific Ideas

- **The fix is ~10-15 lines changed in one function.** It is not a refactor. Planner should resist scope creep toward "while we're in there" cleanup of neighboring code.
- **Suggested atomic commit shape:** (1) RED: add failing test (commits before fix lands; test asserts ConnectionRefusedError does NOT escape startup_bridge when :9998 is dead; MUST fail against 1.2.0). (2) GREEN: apply the try/except move + `_client = None` nulling + stop() call. (3) chore(release): version bump 1.2.0 → 1.2.1 + test_public_api.py version string update. Three commits total in forge-bridge. Then one commit in projekt-forge for the re-pin.
- **The UAT step REQUIRES a Claude Code restart by the user.** This is a workflow discontinuity that plans should explicitly call out as a handoff checkpoint, not paper over. Plan task for 07.1-UAT needs to (a) stop before the restart with clear instructions, (b) write a concrete resumption note that the NEXT Claude session can pick up from zero context.
- **Do NOT plan around a threshold-crossing auto-synthesis flow.** It doesn't exist in v1.2.0. `ExecutionLog.record()` returns True at threshold, but nothing wires that to `SkillSynthesizer.synthesize()`. Direct invocation of `synthesize()` is the current production path. `/tmp/ollama_synth_uat.py` already proves this path end-to-end.
- **Do NOT plan the PostgreSQL schema fix** for the standalone forge-bridge WS server on :9998. That's orthogonal; forcing users to run :9998 for MCP-only mode is the opposite of the design intent.

</specifics>
