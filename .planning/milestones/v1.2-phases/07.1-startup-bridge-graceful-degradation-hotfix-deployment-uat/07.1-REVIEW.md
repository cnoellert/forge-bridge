# Phase 07.1: Code Review Report

**Reviewed:** 2026-04-21
**Depth:** standard
**Files Reviewed:** 4
**Status:** issues_found

## Summary

The Phase 07.1 hotfix is a narrow, correct, and defensible change. The revised
`startup_bridge()` in `forge_bridge/mcp/server.py` now wraps both
`_client.start()` and `wait_until_connected()` in a single try/except, performs
best-effort cleanup in a nested try/except, and nulls out `_client` so
`shutdown_bridge()` becomes a no-op. The AsyncClient contract supports this
usage: `_connect()` fails before `self._ws` is assigned, so the subsequent
`stop()` call's `if self._ws and …` guard correctly short-circuits on a
partially-started client. The regression test is a true nyquist gate — RED at
v1.2.0 because `_client.start()` raises outside the original `try/except`, GREEN
at v1.2.1 because the expanded `try` now covers it. Version handling (pyproject
1.2.0 → 1.2.1 and the public-API version-guard update) is consistent across both
files.

Findings below are defensive-depth observations, not defects that would regress
the hotfix. Two warnings call out behaviour-drift risk between the mock-based
test and the real code path, plus a latent `_lifespan`/degraded-mode coupling
question. Three info items cover a deprecated asyncio idiom, a test-assertion
over-narrowing, and a documentation-drift guard.

## Warnings

### WR-01: Test mock does not simulate the real `start()` failure shape

**File:** `tests/test_public_api.py:130-151` (and by extension
`tests/test_mcp_server_graceful_degradation.py:51-78` via real socket)

**Issue:** `test_startup_bridge_injection` uses `AsyncMock()` for
`mock_client.start`, `wait_until_connected`, and `stop`. All three return
cleanly. That exercises the "happy path" dispatch of the injected URL kwarg but
tells us nothing about failure semantics. The dedicated graceful-degradation
test
(`test_mcp_server_graceful_degradation.py::test_startup_bridge_graceful_degradation_on_dead_port`)
is the real gate and correctly uses a bound-then-closed socket to produce a
genuine `ConnectionRefusedError` from `websockets.asyncio.connect()` — good.

However, there is no test that simulates the _second_ failure mode the v1.2.1
try/except now covers: `_client.start()` succeeds (so `self._ws` is set) but
`wait_until_connected()` raises (e.g. `asyncio.TimeoutError`). In that
scenario the subsequent `_client.stop()` inside the except block actually does
real work — it closes `self._ws`, cancels `self._recv_task`, and rejects
`self._pending`. If any of those raised asynchronously the outer `except
Exception: pass` would swallow it, but no test verifies the code takes the
cleanup branch at all. A server-side slow-handshake scenario (bridge listening
but not completing hello/welcome within 10s) is a plausible real-world
deployment failure we do not have coverage for.

**Fix:** Add a second mock-based test in
`tests/test_mcp_server_graceful_degradation.py` that injects an `AsyncClient`
whose `start()` succeeds but whose `wait_until_connected()` raises
`asyncio.TimeoutError`. Assert: (a) `startup_bridge()` does not raise, (b)
`mock_client.stop` was awaited exactly once, (c) `_client is None` afterwards.

```python
async def test_startup_bridge_graceful_degradation_on_handshake_timeout(monkeypatch):
    from unittest.mock import AsyncMock, MagicMock, patch

    mock_client = MagicMock()
    mock_client.start = AsyncMock()  # succeeds
    mock_client.wait_until_connected = AsyncMock(side_effect=asyncio.TimeoutError())
    mock_client.stop = AsyncMock()

    with patch("forge_bridge.mcp.server.AsyncClient", return_value=mock_client):
        await mcp_server.startup_bridge()

    mock_client.stop.assert_awaited_once()
    assert mcp_server._client is None
```

### WR-02: `_lifespan` unconditionally sets `_server_started = True` after a degraded startup

**File:** `forge_bridge/mcp/server.py:73-74`

**Issue:** After the v1.2.1 hotfix, `startup_bridge()` returns normally even
when the forge-bridge connection failed and `_client` is now `None`. The
lifespan then unconditionally sets `_server_started = True` (comment: "D-14:
trips the register_tools() guard"). That means downstream callers of
`register_tools()` cannot distinguish "MCP server is live with a healthy
bridge" from "MCP server is live in degraded mode (forge_* tools will fail at
call time)". The v1.2.1 docstring promises "forge_* tools will fail. flame_*
tools still work if Flame is running." — that is the intended behaviour, but
`_server_started = True` is now an ambiguous signal.

This is not a bug today: the only consumer of the flag is the post-run guard
in `register_tools()`, which correctly wants `True` whenever the server is
actually running regardless of bridge health. But future code that reads
`_server_started` as "bridge is up" would be wrong, and there is no compile-time
enforcement of the distinction.

**Fix:** Either (a) add a comment at line 74 making the semantic explicit, e.g.
`_server_started = True  # D-14: server is running; does NOT imply bridge is connected (see get_client())`,
or (b) introduce a second module-level boolean `_bridge_connected` and set it
inside `startup_bridge()` only on the success path. Option (a) is lowest-risk
for a patch-release review scope; option (b) is the structurally correct fix
for a future minor release.

## Info

### IN-01: Deprecated `asyncio.get_event_loop()` in test-fixture teardown

**File:** `tests/test_mcp_server_graceful_degradation.py:44`

**Issue:** `asyncio.get_event_loop().run_until_complete(...)` is deprecated
since Python 3.10 and scheduled to be removed in future versions. In Python
3.12+ it emits a `DeprecationWarning`; in 3.14+ calling it when no event loop
is running raises. The fixture is defensive cleanup, so in practice this
branch never fires (the v1.2.1 contract guarantees `_client is None` after
both success and failure), but the dead code is a forward-compatibility hazard
that also contradicts Python's own recommended pattern.

**Fix:** Replace with `asyncio.run(...)` or simply delete the branch since
`_client` will always be `None` by the time the fixture runs. Minimal patch:

```python
if mcp_server._client is not None:
    try:
        asyncio.run(mcp_server.shutdown_bridge())
    except Exception:
        pass
```

### IN-02: Over-narrow warning-substring assertion couples test to log string

**File:** `tests/test_mcp_server_graceful_degradation.py:74-78`

**Issue:** The test asserts the warning contains the substring "Could not
connect to forge-bridge". That literal string is in exactly one place
(`server.py:143`) and any future refactor of the warning wording (e.g. a
switch to structured logging with different top-line wording) silently breaks
the regression test even though the actual graceful-degradation behaviour is
unchanged. For a nyquist-gate test, coupling to an implementation-detail log
string is stronger than the contract warrants.

**Fix:** Assert on a more stable observable — either (a) the log record's
logger name + level (already available in `caplog.records`), or (b) the fact
that at least one WARNING with `logger.name == "forge_bridge.mcp.server"`
fired, without pinning the message text. Example:

```python
server_warnings = [r for r in caplog.records
                   if r.levelno == logging.WARNING
                   and r.name == "forge_bridge.mcp.server"]
assert server_warnings, (
    f"Expected at least one WARNING from forge_bridge.mcp.server. "
    f"Saw records: {[(r.name, r.levelno, r.message) for r in caplog.records]!r}"
)
```

### IN-03: Version-coupling across three call sites with no single source of truth

**File:** `tests/test_public_api.py:179-185` (and `pyproject.toml:7`,
`forge_bridge/__init__.py` via `importlib.metadata`)

**Issue:** `test_package_version` hard-codes the literal `'version = "1.2.1"'`
in a substring match against `pyproject.toml`. Every release bump now requires
editing two files in lockstep (pyproject + this test), plus the comment block
above it (lines 168-177, which narrates the version history by hand). This is
a self-inflicted merge-conflict and forgotten-update hazard — the v1.2.1 bump
itself required a matched edit here. `test_version_attribute_exposed` already
does the right thing by asserting
`forge_bridge.__version__ == _pkg_version("forge-bridge")`, so the hard-coded
literal in `test_package_version` is redundant with the stricter dynamic
check.

**Fix:** Replace the literal-string assertion with a regex that validates
*format* (semver), not a specific value, since the dynamic metadata test
already guards the actual value:

```python
import re

def test_package_version_is_semver():
    """pyproject.toml declares a valid semver; actual version value is
    guarded by test_version_attribute_exposed against importlib.metadata."""
    pyproject = Path(__file__).parent.parent / "pyproject.toml"
    content = pyproject.read_text()
    assert re.search(r'^version = "\d+\.\d+\.\d+"', content, re.MULTILINE), (
        "pyproject.toml must declare a semver version string"
    )
```

This eliminates the lockstep-update requirement and removes a class of
forgotten-bump release defects.
