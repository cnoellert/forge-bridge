# Phase N+ — The Commit Primitive: the Preview→Apply Seam

## Architectural Thesis (carried verbatim from 25.0–Phase N)

> **25.x is not about making the model smarter. It is about making
> execution structure explicit. Reliability comes from making execution
> structure explicit, not from making the model smarter. When the system
> fails, ask what structure is missing before asking what intelligence
> is missing.**

## What Opens Phase N+

Phase N closed with five graph primitives — filter, if-gate, select,
foreach, collect — and a typed-port substrate contract. Every one of
those primitives shares a property Phase N did not name until its close:
**they are all referentially transparent with respect to the domain.**
`dry_run` flows through all of them. Nothing mutates Flame. The entire
25.x + Phase N substrate is a *preview substrate* — the chain, to date,
is read-only.

`commit` is the primitive that crosses that line. It takes a reconciled
aggregate manifest and causes a real domain mutation. It is not a sixth
topology operation. It is the substrate acquiring a **second mode** —
the preview→apply seam. Phase N+ is not "add the commit node." It is
"the substrate becomes read-write, and `commit` is the single
controlled boundary where that happens."

The commit primitive was deferred from Phase N with its structural
questions seeded in `SEED-COMMIT-PRIMITIVE-PHASE-N+.md`. This phase
answers them.

## Substrate Mode: Preview and Apply

The substrate has two modes, and Phase N+ names both first-class:

- **Preview mode** — the five existing primitives. `dry_run` is the
  default substrate posture, expressed as a directive carried through
  the chain. Preview is not a primitive and Phase N+ does not add one.
- **Apply mode** — entered only through `commit`. `commit`'s category is
  **domain-applying**: it is a terminal node, not a topology transform.
  The topology vocabulary (preserving, affecting, expanding, converging)
  describes data-shape transforms; `commit` does not transform shape, it
  applies a reconciled plan to the domain.

Naming the mode split makes the seam visible to future contributors and
to anti-scope enforcement. The substrate is not gaining a second
apply-mode primitive in this phase.

## What Commit Is — and Is Not

`commit` does not invent or infer mutations. It does not execute tools
in the agent-runtime sense. **It reconstructs a declared apply
invocation from a self-describing aggregate preview manifest, and
dispatches it through an explicitly declared authority boundary.**

The semantic center of the primitive is **manifest → apply-call
reconstruction**. `commit` validates a declared mutation contract,
reconstructs the apply call from the resolved plan the aggregate
manifest carries, and dispatches it. That is constrained, inspectable,
constitutional behavior — not generalized orchestration intelligence.
The framing holds this line deliberately, because "commit executes"
drifts toward agent-runtime semantics the substrate has never had.

## The Apply Mechanism: Declaration Wrapping Mechanism

The seed framed the apply question as a binary — re-invoke with
`dry_run=false`, *or* call a domain-specific apply tool. The substrate
had already collapsed that binary. Phase 25.0 made preview a `dry_run`
mode on mutation tools: `flame_preview_rename` is a compatibility shim
that delegates to `flame_rename_shots` with `dry_run=True`. Preview and
apply are not separate tools — they are separate modes of one tool.

The corrected answer, preserved verbatim as the phase's load-bearing
phrase:

> **3b's declaration wrapping 3a's mechanism.**

- **The mechanism (3a):** apply is `dry_run=False` re-invocation of the
  same tool, with the apply parameters reconstructed from the resolved
  plan.
- **The discipline (3b):** the tool's manifest *self-describes* its
  apply-counterpart. `commit` reads that declaration; it does not infer
  or hard-code it.

The apply-counterpart is **not fundamentally about symbol identity** —
it is about a declared authority transition. The counterpart is often
the same tool in apply mode. The framing fences two failure futures:
a counterpart-tool registry (`preview_foo`/`apply_foo` symbolic sprawl),
and implicit replay magic (hidden mutation authority in execution
state). The declaration lives on the manifest the tool emits; not in a
side registry, not in chain-execution state.

## The Mutation Manifest Contract (load-bearing)

`commit`'s input is a mutation manifest. The contract:

**Vocabulary — contract versus artifact.** This framing uses two
coherent vocabularies, intentionally distinct. The *contract* language
("mutation manifest contract," "aggregate manifest," "per-iteration
manifest content") describes what the artifact must satisfy. The
*artifact* identifier — the value the manifest carries in its `type`
self-description field, the code-level name — is `mutation_plan`. A
contributor reads: the body tool emits per-iteration manifest content
conforming to the mutation manifest contract; `collect` aggregates it
into a `mutation_plan` at `commit`'s input. Both vocabularies hold; the
split is specification versus realized thing.

**Emission and aggregation.** "Manifest" names content at two levels,
and the framing distinguishes them. Each body-tool invocation in
`dry_run` mode emits **per-iteration manifest content** — the resolved
plan and intent parameters for the segments that invocation would
rename. `collect` aggregates iterations of that content into the
**aggregate manifest** `commit` consumes: per-change records flow into
`iterations[]`, converged intent and apply-target context surface as
the reconciled top-level. The mutation manifest contract holds at both
levels — the per-iteration content the body tool emits, and the
aggregate the typed-port contract binds to `commit`'s input. `commit`
does not consume body-tool output directly; it consumes `collect`'s
aggregate. A direct body-tool→`commit` chain bypasses the aggregation
layer and is not a supported shape.

**Two layers of content.** The manifest carries (a) the **original
intent parameters** — what was requested — as provenance and context,
and (b) the **resolved plan** — what the preview actually computed — as
the legally authoritative apply layer. `commit` dispatches the resolved
plan. An operator reads both, to see what was asked versus what the
substrate resolved it into.

**The resolved plan is authoritative — not raw input.** The manifest
records what the preview *computed and the operator reviewed*, not
instructions for recomputing it later. This is binding: **the apply
produces what the preview showed.** Raw-parameter replay is unsafe as
constitutional behavior the moment preview and apply are separated in
time — state may drift, defaults may resolve differently, selection
sets may change. `flame_rename_shots` already performs state-aware
default resolution (padding width adopted from existing sequence
grammar); a manifest recording raw input could preview `padding=3` and
apply `padding=4` against drifted state. Recording the resolved plan
makes the manifest a faithful, auditable, replayable artifact —
inspectable weeks later without the original environment state.

**Full structured content, not a delta.** The manifest records the
complete resolved plan, not a diff from defaults. Delta encoding opens a
"what counts as default" ambiguity surface; the full structured plan is
unambiguous and self-contained.

**Hard floor — segment identity.** `flame_rename_shots` reconstructs
segment targeting from a four-part key: `track_idx | record_in |
seg_name | source_name`. The aggregate manifest must preserve that key
tuple per change, or `commit` cannot reconstruct `selected_segments` for
the apply call. This is not optional contract surface.

**Self-description.** The manifest declares its own type (`mutation_plan`),
its originating capability, and its apply-counterpart. This is the 3b
declaration layer — it is what makes the manifest the legal bridge
between preview and apply.

## Commit's Input Binding

`commit` consumes the aggregate manifest — both layers of `collect`'s
output:

- **`iterations[]`** — per-change provenance. The atomic units: each
  per-segment proposed rename, carrying the four-part segment key.
- **reconciled top-level** — apply-target context: which sequence, which
  scope. What the chain agreed on across iterations.

"Apply these renames" (`iterations[]`) "to this sequence" (reconciled
top-level). `commit`'s typed input port binds to `collect`'s output
topology — the aggregate manifest specifically, not per-iteration
content and not a generic manifest shape. Step A's Drop policy — which
moved varying scalars out of the reconciled top-level into
`iterations[]` — shaped this contract before `commit` existed. Named
here as architectural archaeology, not coincidence.

## Commit's Failure Modes (first-class)

`commit` has three named failure modes. None is silently best-effort:

1. **Manifest invalid** — the aggregate manifest does not satisfy the
   contract.
2. **Apply-counterpart not declared** — the aggregate manifest carries
   no declared apply-counterpart; `commit` cannot dispatch what was not
   declared.
3. **Plan/state mismatch** — the resolved plan no longer matches current
   domain state (a target segment was deleted, moved, or renamed by
   another actor since preview). `commit` **fails fast and refuses to
   apply**; the operator re-previews. `commit` does NOT adaptively
   re-resolve the plan to match changed state — that is the precise
   drift the resolved-plan contract exists to prevent.

## The Three Seed Questions — Answered

1. **What does commit dispatch internally?** It re-invokes the declared
   apply-counterpart — for rename, `flame_rename_shots` with
   `dry_run=False` and parameters reconstructed from the aggregate
   manifest's resolved plan. Not a separate apply tool.
2. **Does commit require typed-port input matching?** Yes. `commit`
   declares a typed input port; `collect`'s aggregate-manifest output
   must be compatible. `commit` is a chain primitive in the dispatch
   sense and the typed-port contract applies.
3. **Does commit require a domain-specific apply capability per body
   tool?** The apply-counterpart is *declared by the tool*, on its
   per-iteration manifest content — but the counterpart is frequently
   the same tool in apply mode. The seed's binary collapsed: declaration
   wraps mechanism.

The seed's Q3 was right on axis (implicit vs explicit) and wrong on
layering — it missed that mechanism and discipline live on different
layers and compose. This is named here as a **seed-correction-by-
substrate**: a contributor reading the seed must find the corrected
answer.

## Anti-Scope

Phase N+ does **not**:

- Introduce a preview primitive. Preview is `dry_run`, a directive. The
  symmetry-pressure to add a preview counterpart to `commit` is the kind
  of over-engineering anti-scope exists to fence.
- Break or subsume the directive form (`rename shots ... commit`). The
  directive form and the bare-step form are **sibling expressions** of
  domain-apply intent, with crisp non-overlapping semantics: the
  directive form resolves at apply time (un-previewed apply); the
  bare-step form applies a reviewed resolved plan (previewed apply).
  Neither subsumes the other.
- Support a direct body-tool→`commit` chain. `commit` consumes
  `collect`'s aggregate manifest; the aggregation layer is not optional.
- Generalize the manifest contract across other mutating body tools.
  `flame_rename_shots` only. Broader adoption is a future second-consumer
  phase — that is where the manifest-as-first-class-artifact argument
  earns its evidence.
- Build a counterpart-tool registry or any substrate-wide "all tools
  must emit manifests" interface. The contract is declared per-tool, not
  enforced substrate-wide.
- Establish manifest schema/version identity, cross-domain schema
  guarantees, transactional or rollback semantics, or approval-workflow
  semantics. The manifest contract is *expected to evolve*; Phase N+
  establishes its *legality*, not its final shape. Contributors must not
  treat first-motion manifest fields as frozen.
- Engage chat/exec liberalization.
- Consume Item 15 (compare) or Item 17 (state-aware default resolution).
- Build a graph runtime, scheduler, or async fanout. The seam phrase
  holds: **new dispatch path, not a new executor.**

## Canonical Probe

The probe exercises the preview→apply seam end-to-end: a direct
(sequence-global) `flame_rename_shots` discover producing a mutation
manifest, an if-gate on the resolved plan, and the bare-step `commit`
form verifying that manifest and dispatching the declared apply.

```bash
# Substrate anchor — deterministic, LLM-free:
fbridge exec --json "rename shots on 30sec 21 with prefix genesis dry_run -> if(resolved_plan exists) -> commit"

# Operator-path — natural-language composition through the chat endpoint:
fbridge chat --verbose --trace "rename shots on 30sec 21 with prefix genesis dry_run -> if(resolved_plan exists) -> commit"
```

Pass criterion: `flame_rename_shots` discover resolves the whole sequence
(sequence-global numbering) and emits a `mutation_plan` manifest; the
if-gate opens on the non-empty resolved plan; `commit` validates the
manifest, re-dispatches `flame_rename_shots` in verify mode,
structurally compares the fresh resolved plan against the held one, and
on a clean match dispatches the apply. The mutation lands in Flame. The
probe verifies plan/state-match fail-fast as negative-space: a manifest
whose resolved plan no longer matches current state must raise
`PLAN_STATE_DRIFT`, not adaptively re-resolve. The exec run anchors
substrate correctness independently of operator-surface composition; the
chat run proves the same constitutional flow composes through the
natural-language operator interface.

Note — foreach + collect + commit composition: the original probe reached
for the full graph composition (foreach-wrapped discover, collect
aggregation, then commit). Phase N+ does not demonstrate that composition
end-to-end: `flame_rename_shots` is a sequence-global tool, and foreach
provides per-segment slices, so foreach-wrapped discover cannot produce
correct sequence-global numbering. Demonstrating foreach + collect +
commit requires a body tool that is both foreach-natural and
manifest-emitting — a future second-consumer-phase concern, sibling to
the C-1 `intent_parameters` deferral.

This is the probe the seed's Step B originally attempted with bare
`commit` — it failed because `commit` was not yet a primitive. Phase N+
makes it pass.
