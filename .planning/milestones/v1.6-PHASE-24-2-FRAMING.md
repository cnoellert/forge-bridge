# Phase 24.2 — Doctor Truth ↔ Effective Dispatch Truth

## §0 — Banner

**The architecture did not fail. Operational configuration escaped architectural visibility.**

The chat path is currently broken on portofino because the running daemon dispatches to the wrong port — and **`fbridge doctor` reports green anyway**. Doctor validates substrate *existence*. Doctor does not validate substrate *usability from the active dispatch path*. That distinction is now operationally load-bearing.

Phase 24.2 closes this one gap, surgically. It does not rename env vars. It does not generalize doctor across surfaces. It does not begin chat-layer convergence work.

## §1 — Load-Bearing Diagnosis

Yesterday's warm-prefix repeat of `"What are the clips on Reel 1"` produced 5 `flame_execute_python` invocations across 8 iters — *all* with `status=transport_error` in 10-14ms, RST shape — while `fbridge doctor` continued reporting `flame_bridge ok / running http://127.0.0.1:9999`.

Four operational facts were *simultaneously true* on portofino at that moment:

1. The Flame hook process was alive on `:9999`.
2. A direct `bridge.execute(...)` probe from the operator shell succeeded.
3. `fbridge flame-exec` (the operator-side surface shipped at `c091e5e`) succeeded.
4. The conversational dispatch path silently targeted the wrong endpoint and produced transport_error on every Flame call.

Doctor saw facts 1-3 (its own world). Doctor was blind to fact 4 (the daemon's world). Both worlds compute their dispatch target by reading `FORGE_BRIDGE_PORT` *in their own process environment* — operator shell vs. launchd process. The two envs disagreed. The two computations diverged. Doctor reported the operator's truth as if it were the daemon's truth.

This is not a process-liveness-vs-dispatch-correctness gap (the earlier framing). It is a **config-context divergence**: doctor re-derives effective config from its own env; the daemon dispatches from a different env; doctor never observes the daemon's effective config. The two can drift silently and indefinitely.

That is the architectural lesson. The fix follows from naming it.

## §2 — Core Goal

**Align doctor truth with effective dispatch truth.**

Doctor stops re-deriving dispatch config and instead *exercises the daemon's actual dispatch surface*. A round-trip probe through the running daemon eliminates the config-context divergence by construction — there is no second world to disagree.

The architectural invariant this phase introduces: **the health surface reflects daemon-observed dispatch truth, not independently reconstructed local truth.** This invariant applies equally to CLI doctor and Console mirror (§6.3). Shell-derived config remains *diagnostic context* for divergence detection; it is never authoritative.

This is a one-row, one-probe change inside `_check_flame_bridge()`. It is not a doctor rewrite. It is not a new doctrine layered over the existing doctor. It is the existing doctor row, refactored from "probe the substrate I think the daemon is using" to "ask the daemon to round-trip through the substrate it's actually using."

### §6.1 — Probe Shape **[CONVERGED]**

**Q1. Through which existing daemon endpoint should the round-trip probe go?**

**Resolution:** `POST :9996/api/v1/exec` with `text="flame_ping"`. Doctor parses the PR31 envelope; the `flame_ping` body itself already returns the daemon's effective `bridge.BRIDGE_URL` (utility.py:292) — the load-bearing field for divergence detection — alongside `connected` / `version` / `project`.

**Why this is the right probe:**

1. **Same path as chat handler, minus LLM.** `_execute.py:20-21` docstring: *"Mirrors chat tool snapshot: list_tools then filter_tools_by_reachable_backends (same order as `chat_handler`)."* The chain engine and chat handler share dispatch infrastructure by construction — exercising `/api/v1/exec` exercises what the chat path actually does, without the agentic-loop noise on top.
2. **Daemon-process env is authoritative by construction.** When the daemon's `flame_ping` calls `bridge.execute_json(...)`, it reads `bridge.BRIDGE_URL` from the daemon process's environment. That URL is then echoed in the response body. Doctor never re-derives — doctor consumes.
3. **Divergence-detection field already exists.** `flame_ping` returns `bridge_url` in its JSON body (utility.py:292). Doctor compares this daemon-reported URL against doctor's own `config.flame_bridge_url()`. Convergent: row is OK. Divergent: row is WARN with the divergence rendered per §6.2's structured-status shape.
4. **`flame_available` already encoded.** `flame_ping` returns `connected: bool` based on whether the `flame` module imported and `flame.projects.current_project` resolved. Preserves the existing `_check_flame_bridge()` `flame_available` semantics without re-implementing them.
5. **No protocol invention.** No new endpoint, no new MCP tool, no new env var, no new config surface. The probe leverages four existing surfaces (`/api/v1/exec`, chain engine, `flame_ping` MCP tool, `bridge.BRIDGE_URL` echo) — none introduced for doctor.

**Rejected alternatives carried forward:**

- `POST /api/v1/chat` — exercises LLM. Wrong probe shape: doctor validates *transport*, not *agentic loop*.
- New dedicated `/api/v1/doctor/dispatch-probe` endpoint — synthetic health endpoint creating exactly the "doctor green / actual path broken" asymmetry 24.2 exists to eliminate.
- `POST /api/v1/exec` with `text="execute_python pass"` — `flame_execute_python` works as a probe but doesn't echo `bridge_url`, requiring doctor to either trust the tool ran-or-didn't-run binary signal (loses divergence detection) or hit a second endpoint (architecturally messier). `flame_ping` returns the diagnostic field as a side effect of the call; preferred.

**Convergence binding:** The probe shape locks to `flame_ping`. If implementation surfaces an edge case where `flame_ping`'s tool-filter narrowing is non-deterministic (e.g. another tool's name matches the word "ping"), the resolution is to extend the chain engine's filter discipline, NOT to re-open Q1 toward a synthetic endpoint.

### §6.2 — Truth-Asymmetry Row Rendering **[CONVERGED]**

**Q2. How does the new doctor row surface the divergence when doctor's re-derived config and daemon's effective config disagree?**

**Recommendation:** single row, with richer structured truth and an operator-action-bearing fix line. Decomposition is rejected unless implementation pressure genuinely forces it.

The architectural problem is not "there are two Flame bridge systems" — it is "one operational surface is observing the wrong configuration world." A two-row decomposition (`flame_bridge_process_liveness ok` + `flame_bridge_dispatch fail`) would visually legitimize the split reality instead of diagnosing it. The operator experience must read as: *one Flame bridge surface / one effective dispatch truth / one diagnosis when the worlds diverge* — not as two semi-independent realities accidentally encoded into the UX.

Cold operators do not need abstract status labels, inferred remediation, or symbolic divergence states. They need:

- **what is wrong** — surfaced as the daemon-observed dispatch failure
- **which world is authoritative** — daemon-effective config (shell-derived is diagnostic context only)
- **what exact action repairs it** — operator-action-bearing fix line carries the load

Conceptual row shape:

```
flame_bridge   FAIL
dispatch target mismatch
daemon=:9998 shell=:9999
→ unset FORGE_BRIDGE_PORT in launchd env and restart daemon
```

The single-row preference holds unless convergence surfaces concrete implementation pressure that proves decomposition unavoidable. A two-row split is the path toward `--strict` second-order abstraction (out of scope §3.3) and toward [`feedback-explicitly-unbound-vs-implicitly-rejected`]-violating rendering shapes. Convergence may overturn only on empirical operator-illegibility evidence, not on canonical-row-taxonomy preference.

### §6.3 — Console Mirror Scope **[CONVERGED]**

**Q3. Does the Console health view need parallel divergence rendering, or does it suffice for Console to surface the same row data the CLI does?**

**The Console mirror is bound by the same architectural invariant as the CLI doctor (§2):** the health surface reflects daemon-observed dispatch truth, not independently reconstructed local truth. If Console re-derives `flame_bridge_url()` from its own env (today: `forge_bridge/console/read_api.py:_check_flame_bridge` does exactly this), the system *recreates the exact pathology 24.2 is trying to eliminate* — just at a different surface.

**Resolution:** Console's `_check_flame_bridge` is re-routed through the same daemon-routed dispatch probe the CLI uses. The Console process never independently computes the Flame bridge dispatch target. The row data flows from daemon truth → CLI doctor + Console mirror as siblings, neither authoritative over the other.

This is not "Console mirrors CLI." This is "Console and CLI both consume the daemon's effective dispatch truth." The distinction matters because the second framing makes the invariant violation visible if a future surface (Flame foundry UI, MCP resource, etc.) attempts to re-derive locally.

### §6.4 — Probe Failure Mode Discipline **[CONVERGED]**

**Q4. When the daemon is unreachable (the doctor probe can't even hit `:9996`), the new probe can't run. How does truth authority degrade under partial substrate failure?**

**Resolution:** four operational states map cleanly to four `flame_bridge` row outcomes. The invariant § 2 holds under degradation — doctor never falls back to a re-derived local probe; daemon truth or no truth.

| State | Probe outcome | Row | Fix line |
|---|---|---|---|
| **Convergent** | `flame_ping` returns `connected=true` AND daemon's `bridge_url` matches doctor's `config.flame_bridge_url()` | `flame_bridge OK / dispatching {url}` | — |
| **Divergent** | `flame_ping` returns `connected=true` (or `connected=false`) but daemon's `bridge_url` differs from doctor's re-derived URL | `flame_bridge WARN / dispatch target mismatch / daemon={A} shell={B}` | `→ unset FORGE_BRIDGE_PORT in launchd env and restart daemon` (or operator-specific; documented in TROUBLESHOOTING.md) |
| **Daemon-observed failure** | `flame_ping` returns `connected=false` AND URLs converge | `flame_bridge FAIL / dispatching {url} but {reason}` | `→ start Flame / open a project / check the bridge hook is loaded` (reason-derived) |
| **Daemon unreachable** | Doctor can't reach `:9996` (HTTP error / timeout on `/api/v1/exec`) | `flame_bridge UNKNOWN / daemon unreachable — see mcp_http` | `→ see mcp_http row` (link, don't duplicate) |

**Truth-authority degradation principle:** under any condition where daemon truth is unavailable, doctor reports `UNKNOWN` — not a re-derived local truth. The shell-derived `config.flame_bridge_url()` is *diagnostic context* (used to populate the `shell={B}` field in the Divergent state for divergence detection) — it is **never** the authoritative answer to "is dispatch working." The invariant § 2 is binary: daemon truth, or no truth at this surface.

**Explicit rejection:** no "diagnostic-mode local probe" with a "this is shell-derived, not authoritative" label as a fallback. That pattern smuggles the pathology back in under a different name. Under daemon unreachability, the operator sees `flame_bridge UNKNOWN` and follows the `mcp_http` thread to recover daemon reachability first.

**Sub-state preserved from pre-refactor doctor:** the daemon-observed-failure state subsumes the existing `running, no flame module` row outcome (Flame hook up, `flame` module not importable) — daemon's `flame_ping` returns `connected=false` with the reason field naming the cause. The pre-refactor doctor's three-way taxonomy (`ok` / `unreachable` / `running, no flame module` / `http {status}`) collapses cleanly into the four-state table: convergent-OK / daemon-observed-FAIL with reason-derived fix line / UNKNOWN-via-mcp_http / new WARN state for divergence.

## §7 — Anti-Scope Binding (Explicit)

This phase resolves one specific operational asymmetry: doctor speaking from the operator-shell world while dispatch executes from the daemon-launchd world. It does not introduce generalized distributed config reconciliation, service discovery, config orchestration, or runtime topology management. Those patterns are not indicated by the evidence and are not in scope.

The following binding holds across the phase until the canonical-probe re-fire either confirms or refutes the diagnosis. Bound to prevent scope drift during execution:

- No env-var renames. (§3.2)
- No new env vars introduced. The probe uses existing config surfaces.
- No semantic-layer changes. Docstrings, tool descriptions, system prompts, tool ordering — frozen for the duration of 24.2.
- No model swaps. Local Ollama `qwen2.5-coder:32b` remains the canonical probe model.
- No new MCP tools registered.
- No additional doctor rows beyond the refactored `flame_bridge` row.
- No additional cleanups of unrelated doctor wording, formatting, or status taxonomy in the same commit. Doctor refactor is surgical; cosmetic drift handled separately if at all.

The binding loosens only if the canonical-probe re-fire produces evidence that re-frames the diagnosis (§4.2 / §4.3).
