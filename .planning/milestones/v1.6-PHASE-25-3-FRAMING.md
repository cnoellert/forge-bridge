# Phase 25.3 — Positional Resolver Stabilization + Generic Select

## Architectural Thesis

25.x is not about making the model smarter.
It is about making execution structure explicit.

Reliability comes from making execution structure explicit,
not from making the model smarter.

When the system fails, ask what structure is missing
before asking what intelligence is missing.

## What The 25.x Arc Has Established

```text
25.0  graph substrate exists          (preview as universal mutation gate)
25.1  topology-preserving primitive   (FilterNode — list[X] -> list[X])
25.2  conditional control flow        (IfGateNode — gate open / gate closed)
25.3  interpretation structure        (positional anchoring + generic select)
```

Two graph primitives are in production. The resolver binds five
categories deterministically. The canonical probe shape is established:
enumerate -> operate -> gate -> commit.

The substrate is now rich enough that resolver ambiguity has become
the limiting factor, not missing primitives.

That is the signal that Item 18 has reached its maturation condition.
Both filter (25.1) and if/then (25.2) are landed. The heuristic
compensation Item 18 named is now removable — not because it has been
convenient to remove, but because the graph primitives it was waiting
on exist.

## The Phase 25.3 Thesis

Phase 25.3 stabilizes interpretation structure without introducing
typed topology expansion.

The two problems 25.3 solves are structurally aligned:

**Positional resolver stabilization (Item 18):**
The resolver currently matches directives (`commit`, `dry_run`, format
modifiers) anywhere in step text via word-boundary regex. This is
match-anywhere heuristic compensation for a substrate that once lacked
filter and conditional primitives. Those primitives exist now.
Positional anchoring replaces the heuristic with bounded structural
extraction: directives are valid only in their structural positions.

**Generic select (Item 12):**
The resolver has no way to express "which item inside this collection
did the operator mean?" — resolution by identity, not by predicate.
Select is the topology-preserving primitive that closes this gap.

Both are the same class of problem: interpretation-boundary
stabilization. Neither requires typed fan-out semantics. Neither
pressures port compatibility. Neither introduces topology expansion.

## Why Foreach Is Not 25.3

foreach is deferred because it is the first primitive whose
correctness depends on typed graph contracts.

Not because it is too large.

Without typed ports, foreach collapses into one of three unacceptable
shapes:

1. magic — infer item shape dynamically and hope
2. rigid — hard-code supported upstream collection types
3. fragile — fail unpredictably on unfamiliar upstream topology

None of those satisfy 25.x substrate discipline.

foreach is Phase N's opener. Phase N opens because foreach forces
typed ports. That is the structural argument that makes typed ports
unavoidable — not a future enhancement decision.

```text
25.3  interpretation structure stabilizes

Phase N opens:
      foreach + collect  (topology-expanding — requires typed ports)
      typed ports
      chat/exec split
      port compatibility
      macro bridge semantics
```

## Item 18 Scope Boundary

25.3 lands lightweight positional anchoring.

In scope:
- `commit` valid only in terminal directive position
- `dry_run` valid only in modifier position (not inside a noun phrase
  or prefix value)
- Format modifiers anchored to terminal clause
- Explicit precedence ordering for directive extraction
- Stop-word / phrase-boundary stabilization
- Bounded positional interpretation for all five existing resolver
  categories

Out of scope:
- Formal grammar
- Parser ASTs
- Recursive syntactic structures
- Typed interpretation at parse time
- Structural validation trees
- Generalized parsing framework

The litmus test:

> If the resolver still fundamentally operates as ordered bounded
> extraction passes, you are in 25.3 scope.
>
> If you are building a generalized syntax system, you have crossed
> into Phase N.

That seam must be named verbatim in commit messages when touching
resolver code.

## Generic Select — Topology Contract

Select is topology-preserving:

```text
filter  list[X] -> list[X]   narrow by predicate
select  list[X] -> list[X]   narrow by identity
foreach list[X] -> ...       topology expansion — Phase N
```

Select's output is always `list[X]`, even when operator semantics
feel singular. A successful single-item select returns a single-item
collection. `list[X] -> X` would introduce scalar values into a graph
that operates uniformly on collections — a scalar/collection
distinction that is a typed-port question, deferred to Phase N.

SelectNode does not guess.

## Select Identity Contract

25.3 select performs deterministic identity resolution only.

v1 identity semantics: ordered field-try against canonical identity
fields. SelectNode tries declared fields in declared order and matches
on the first field present in the collection entry.

No fuzzy matching. No semantic similarity. No ranking. No partial token
retrieval. No vector retrieval. No LLM-assisted selection.

Select is not retrieval. Select is identity resolution.

Canonical identity fields (v1), tried in this order:
1. `seg_name`
2. `shot_name`
3. `sequence_name`
4. node name
5. clip name

Field-order fallback rationale: upstream collection entries do not
carry uniform identity fields. `proposed_changes` entries from
`flame_set_segment_attribute` carry `segment` not `seg_name`;
role-assignment dry_run carries `segment` and `index` not `shot_name`.
SelectNode does not normalize upstream shapes — it tries canonical
fields in declared order and matches on the first present. This is
deterministic (same input -> same match, every time) without requiring
upstream normalization. Upstream normalization (approach B) and
emitter-declared identity fields (approach C, typed-port adjacent) are
both deferred to Phase N and later.

Resolver behavior:
- Zero matches -> structured error (`IDENTITY_NOT_FOUND`)
- Multiple matches -> structured ambiguity error (`IDENTITY_AMBIGUOUS`)
- SelectNode does not guess in either case.

`is_select_step` anchoring contract:
`is_select_step` matches `^\s*select\s+` — anchored at start of step
text, parallel to `is_if_step`'s `^\s*if(?:\s*\(|\s+)`. This prevents
`select` tokens in incidental prose positions from stealing dispatch.

Select input topology after mutation-preview steps:
Select operates over the active execution manifest, not the original
upstream collection. After a `dry_run` step followed by an if-gate, the
active graph object is the execution manifest. Identity resolution
therefore occurs against manifest-produced entities (e.g., entries in
`proposed_changes`) rather than raw segment dicts. This preserves
manifest-first architecture established in 25.2.

Anti-scope for select:
- No fuzzy identity resolution
- No ranked selection
- No semantic retrieval
- No scalar output — always returns `list[X]`

## The 6th Resolver Protocol Category

```text
1. entity values       -> sequence_name, reel_name, library_name
2. numeric directives  -> increment, padding, start
3. intent modifiers    -> dry_run (preview intent)
4. filter predicates   -> flat FilterPredicate AST     (25.1)
5. if-gate predicates  -> flat FilterPredicate AST     (25.2)
6. select identity     -> identity resolution target   <- NEW in 25.3
```

## Anti-Scope

- Do not introduce typed ports. Typing is Phase N.
- Do not implement foreach or collect. Phase N.
- Do not build a generalized step parser or grammar tree. That crosses
  the Item 18 litmus test into Phase N territory.
- SelectNode dispatch shape parallels filter/if-gate exactly:
  `_maybe_execute_select_step` returns
  `{result, extracted_context, tool='graph_select', params}`.
  No new chain dispatch shape. SelectNode is a topology-preserving
  sibling of FilterNode and IfGateNode, not a re-architecture vector.
- No fuzzy or semantic identity resolution in SelectNode.
- SelectNode never returns a scalar — always `list[X]`.
- Do not normalize `proposed_changes` entry shapes across tools. Field
  normalization is Phase N substrate work (approach B deferred).
- Do not address the post-commit chat-handler hang (bounded defect 2
  from 25.2). Separate investigation.

## Forcing Function

**Probe 1 — positional anchoring (pathological case):**

```bash
fbridge chat "rename shots with prefix commit"
```

Pass criterion: `commit` in prefix-value position does NOT fire the
commit directive. Step executes as a dry_run or returns without
mutation. This is the true pathological shape — `commit` as the sole
trailing token where match-anywhere fires incorrectly today.

Positive coverage probes:

```bash
fbridge chat "rename shots with prefix genesis commit"
fbridge chat "rename shots with prefix genesis dry_run"
```

Pass criteria: `commit` in terminal directive position fires correctly;
`dry_run` fires correctly as intent modifier.

**Probe 2 — select identity resolution:**

```bash
fbridge chat "get segments on 30sec 21 -> select genesis_0010 -> rename with prefix archive commit"
```

Pass criterion: select resolves `genesis_0010` by exact identity match
from upstream collection; rename operates on the selected single-item
collection only; one rename lands in Flame.

**Canonical probe — full chain:**

```bash
fbridge chat "get segments on 30sec 21
  -> rename shots with prefix genesis dry_run
  -> if(proposed_changes exists)
  -> select genesis_0020
  -> rename shots with prefix archive commit"
```

Pass criterion: gate opens; select resolves `genesis_0020` from the
active execution manifest (proposed_changes entries) using ordered
field-try; archive rename commits for that entry only; `--trace` shows
all five steps correctly; one rename lands in Flame.
