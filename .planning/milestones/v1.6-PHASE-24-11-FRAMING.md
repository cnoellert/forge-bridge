## §0 Root Cause

24.10 closed the wrapper serialization boundary:

- flat provider-style arguments are normalized to `params` wrappers
- direct registry invocation works
- forced chat, chain-step, and CLI invocation now share the same normalizer

The remaining ecological failure is one level higher. Forced chat execution
selects the correct tool but reaches FastMCP with:

```json
{"params": {}}
```

FastMCP then correctly reports:

```text
params.sequence_name
  Field required
```

That is no longer a wrapper-schema problem. It is a deterministic entity
materialization problem at the query boundary.

---

## §1 Hypothesis

Forge already has enough local convention to resolve common operator entity
mentions before the LLM sees the prompt:

- Flame sequence names use underscore-separated canonical forms.
- Operators naturally say or type space-separated forms.
- `30sec 21`, `30sec21`, and `30sec_21` refer to the same canonical sequence.
- Reel names follow the same normalization pattern.

The LLM should handle intent and tool selection. A resolver layer should handle
canonical entity extraction from the natural language query.

If the chat handler enriches the query with resolved entities before tool
execution, the same ecological probes should cross the `sequence_name` boundary
and return a real Flame response, even if the response is `Sequence not found`.

---

## §2 Scope

Single perturbation:

- add a query-time resolver module
- call it from `forge_bridge/console/handlers.py`
- inject resolved context into the user message before the LLM path
- pass resolved entities into the existing forced execution path

Minimum viable resolved entities:

- `sequence_name`
- `reel_name`
- `prefix` / `shot_name` pass-through when explicitly expressed

Out of scope:

- tool descriptions
- system prompt
- model selection
- adapter normalization
- inventory order
- provider-specific tool-call behavior

---

## §3 Injection Shape

The LLM-facing user message is enriched as deterministic context, not synthetic
conversation history:

```text
[Resolved entities from query]
sequence_name: "30sec_21"  (normalized from "30sec 21")

User query: Give me the versions on the sequence 30sec 21
```

Forced execution uses the same resolved entity map as explicit user parameters
before calling FastMCP.

---

## §4 Forcing Function

Run the same four ecological operator probes:

```bash
fbridge chat "Give me the versions on the sequence 30sec 21"
fbridge chat "Rename the shots on 30sec 21 to genesis, 4-digit padding, by 10s"
fbridge chat "Get the segments on the sequence 30sec 21"
fbridge chat "Preview the start frames for 30sec 21"
```

**H1 pass criterion:** any target tool fires with `sequence_name` populated and
returns a real Flame response. `Sequence not found` is acceptable because it
means the resolver crossed the validation boundary and Flame is answering.

**H0 failure criterion:** `params.sequence_name` is still missing after resolver
injection.

---

## §5 Invariants

- No description changes.
- No system prompt changes.
- No model swap.
- No adapter changes.
- No permanent registry reorder.
- Resolver behavior must be deterministic and idempotent.
