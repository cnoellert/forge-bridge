## What 25.2 empirically established

### The second graph primitive — IfGateNode

`forge_bridge/graph/if_gate.py` adds `IfGateNode` and `parse_if_step`
alongside `is_if_step`. The primitive reuses `FilterPredicate` exactly
— filter and if-gate differ in graph semantics, not predicate
representation. `IfGateNode.run()` accepts a previous execution
manifest (the `__previous_result__` from the chain context), evaluates
the predicate against it, and returns an augmented manifest with two
new keys:

- `execution_state: "passed"` or `"skipped"` — the gate decision
- `if_gate: { matched: bool, predicate: dict }` — predicate echo

The gate outcome propagates through `_step.py:_maybe_execute_if_step()`
into `extracted_context["__if_gate_skip_next__"]`. The chain engine
`_engine.py:31-35` reads this flag on the next iteration: if true, the
downstream step is suppressed — its result is the inherited manifest
plus `execution_state: "skipped"` and `skipped_step: <step_text>`.
The step is written to the chain trace regardless; topology is
preserved even for suppressed steps.

### The 5th resolver protocol category — if-gate predicates

The resolver protocol now has five categories:

```text
entity values       → sequence_name, reel_name, library_name
numeric directives  → increment, padding, start
intent modifiers    → dry_run (preview intent)
filter predicates   → flat FilterPredicate AST     (25.1)
if-gate predicates  → flat FilterPredicate AST     ← NEW in 25.2
```

Filter and if-gate share the `FilterPredicate` AST representation but
resolve through separate resolver keys (`filter_predicate` /
`filter_error` vs `if_predicate` / `if_error`) and dispatch through
separate `_maybe_execute_*` functions in `_step.py`. The AST is
shared; the graph semantics are distinct.

### `execution_state` as a coherent multi-source signal

Phase 25.2 names a new substrate-layer signal for the first time.
`execution_state` appears in two structurally distinct sites:

| Source | File | Line | Value produced |
|--------|------|------|----------------|
| Gate decision | `forge_bridge/graph/if_gate.py` | 73 | `"passed"` or `"skipped"` on the gate step's result |
| Suppression effect | `forge_bridge/console/_engine.py` | 31-35 | `"skipped"` on the downstream step's result |

Both use the same key name and value vocabulary (`"passed"` /
`"skipped"`) but they describe different things:

- The gate step's `execution_state: "passed"` means the predicate
  matched; execution continues downstream.
- The gate step's `execution_state: "skipped"` means the predicate
  missed; the engine will suppress the next step.
- The downstream step's `execution_state: "skipped"` (set by the
  engine, not by `IfGateNode`) means this step was suppressed by
  an upstream gate miss.

The `skipped_step` key disambiguates the two "skipped" meanings at
the trace layer: the gate step never has `skipped_step`; the
suppressed step always does.

**The signal is coherent across sources** — an operator reading the
chain trace sees a complete picture of gate decision + suppression
effect without ambiguity. The `--trace` render layer
(`_maybe_render_gate_step` in `chat.py`) uses exactly this
distinction: `"skipped_step" in result` identifies a suppressed step;
`result.get("if_gate")` identifies a gate step.

### Framing amendment — `"passed"` not `"executed"`

The Phase 25.2 framing-as-written used `"executed"` as the affirmative
`execution_state` value. The code is more correct: `if_gate.py:73`
uses `"passed"`. `"passed"` is semantically precise — the predicate
passed, so execution continues. `"executed"` would conflate "the step
ran" with "the gate matched," which are distinct ideas. Future
contributors should trust the code value (`"passed"`) over any
planning-document framing that says `"executed"`.

## Open bounded defects

### Bounded defect 1 — `commit` / `dry_run` resolver match-anywhere

**Location:** `forge_bridge/llm/resolver.py` — the `\bcommit\b` and
`\bdry[_ -]?run\b` regexes are word-boundary anchored but not
positionally anchored. They match anywhere in step text.

**Risk:** A step like `rename shots with prefix genesis commit` would
have `commit` matched by the word-boundary regex even if the intent
is a prefix value, not a commit directive.

**Bound:** Pathological in practice only if an operator uses `commit`
or `dry_run` as a literal prefix value. Currently hypothetical; not
observed in any production probe.

**Maturation condition:** Positional resolver vocabulary — the same
maturation condition named in §11 Item 18. When the resolver gains
positional anchoring, this defect closes automatically. Not a
separate fix; it is a consequence of Item 18 landing.

### Bounded defect 2 — post-commit chat-handler hang

**Observed:** PASS-direction canonical probe. After the gated commit
step lands (24 renames confirmed in Flame), `fbridge chat` does not
return. The mutation succeeded; the chain trace is complete. The hang
is at the chat-handler / LLM-router post-chain response-finalization
path, not in the chain engine or gate logic.

**Bound:** NOT a 25.2 substrate defect. The chain engine, IfGateNode,
and the gate-decision render are all correct. The hang is a
pre-existing condition made observable by the 25.2 probe for the
first time (prior probes used dry_run mode, which has a simpler
finalization path).

**Maturation condition:** Investigation of the post-commit
response-finalization path in `forge_bridge/console/handlers.py`.
Candidate causes: LLM router attempting a follow-up inference turn
after the chain completes; response path not closing cleanly on
chain-only responses; timeout misconfiguration on the commit path.

**Separate phase.** Does not touch the graph substrate.

## §11 future-pattern note

**General field-mapping shape from the `sequence_name` fix.**

Commit `88d2a3c` fixed a gap where `sequence_name` was not resolving
for the downstream commit step when it followed a gate step. The fix
added a post-resolve fallback in `execute_chain_step` at the existing
`UNRESOLVED_KEY in params` branch: when the unresolved key is
`sequence_name`, the dispatcher inspects `__previous_result__["sequence"]`
(or `["sequence_name"]`) and re-resolves with the injected value.

This is distinct from the `extracted_context` propagation done by
filter and if-gate primitives — this is a dispatcher-layer safety net,
not a graph-node extraction. The two mechanisms coexist: graph nodes
propagate fields through `extracted_context`; the dispatcher has a
fallback for when propagation doesn't cover a required resolution
param.

Future primitives that produce manifests containing resolvable names
should consider both mechanisms: `extracted_context` propagation for
reliable forward flow, and awareness that the dispatcher fallback
exists as a safety net. If a future probe surfaces a resolution gap
despite correct `extracted_context` propagation, the dispatcher
fallback is the correct fix site — not the graph node.

Not an active deferral. The next primitive that encounters this shape
will promote it.
