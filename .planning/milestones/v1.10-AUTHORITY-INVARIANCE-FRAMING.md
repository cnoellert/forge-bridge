# v1.10 Authority Invariance — Routing Must Not Alter Authority or Eligibility

> **What this artifact is.** Cycle-3 writing-room framing for v1.10, drafted
> from the CR.1 close + dogfood corpus and reframed through the room's
> cross-voice fold (DT grounding / Creative experience / Orch synthesis).
> Compact per [[feedback-cadence-artifacts-shrink-to-load-bearing]].
>
> **What this artifact is not.** Not a roadmap, not a phase plan. Not a
> reopening of substrate authority, the ratify/assent discipline, or the CR.1
> answer-pass — all stay closed and unregressed. v1.10 touches the **routing
> layer at the compile→dispatch seam**, nothing downstream of a
> correctly-routed step.
>
> **The one-line thesis.** There is **one seam, two symptoms, one invariant.**
> The CR.1 dogfood read as "dispatch is broken" — but the room's fold found the
> deeper shape: **the routing layer has accumulated responsibilities that
> exceed routing.** It now silently decides whether a mutation reaches the
> authority chain, whether a read reaches execution, and whether a user sees
> progress or paralysis. Both failures violate a single invariant:
>
> > **Operator authority and execution eligibility must not depend on which
> > routing path was selected.**
>
> Hole B says *authority changes when routing changes.* Resolver paralysis says
> *eligibility changes when routing cannot establish a path.* v1.10 restores
> the invariant.
>
> **The sharpest statement of the finding (cycle-4):** the mutation classification
> already exists and is correct — `flame_set_start_frames` was registered
> `readOnlyHint:False`. *The system knew the tool was mutating and still let a
> routing path bypass authority enforcement.* This is not missing metadata; it is
> an invariant the routing layer was permitted to ignore.

## The transition

v1.6 made the substrate operable; v1.7 shipped the artist surfaces; v1.8
projected authority to the Console; v1.9 stood up the read answer-pass as a
pressure instrument and **ran it** (CR.1 dogfood, 2026-06-01). The instrument
did its job: the corpus's dominant signal is not answer quality. Every tool
that *executed* returned correct data (iteration `6`, batch groups, reels). The
failures cluster at the compile→dispatch seam — and when the room looked at
*why two unlike failures (an authority breach and a usability dead-end) emerge
from the same physical location*, the answer was the routing layer doing more
than routing. That diagnosis, not "dispatch integrity," is the milestone.

## One invariant, two symptoms (grounded — live reads 2026-06-01)

The deepest statement (Creative→Orch, cycle-5): **this is a consistency problem,
not a classification one. The system already knows which tools mutate — it simply
isn't behaving consistently with what it knows.** `readOnlyHint` is declared and
correct (`registry.py`, incl. `flame_set_start_frames` → `False` at `:716`), but
the execution edges never consult it. Two symptoms fall out: authority changes
with route (Symptom 1), and eligibility changes with route (Symptom 2).

### Symptom 1 — authority changes when routing changes (SAFETY)

**Three execution edges can run a tool; none consults read-vs-mutate.** The
cycle-3/4 framing named two and undercounted — DT cycle-5 grounded a third. Counts
are load-bearing: it is **three sites**, and a guarantee that gates only two ships
with a hole.

- **1A — the commit gate (R4/R5).** `run_compile_branch` routes to the
  mutating-preview regime on one condition: does compiled step text contain a
  `commit` token? (`_chat_compile.py:158`, `graph_contains_commit_node`, def at
  `graph/commit.py:75/93`.) Classification *correct*; the model emitted `commit`
  for a read → an artist sees a **"Ratify & Apply" prompt** for a question.
  Gated, but mis-routed. (Routing decision — not itself a dispatch.)
- **1B — the forced-tool dispatch (R11), NO gate.** When narrowing yields one
  survivor, `_execute_forced_tool` (`handlers.py:528`) calls the tool **directly**
  at **`:720-722`** (`raw = await _mcp_server.mcp.call_tool(...)`) — no commit
  gate, no `would_mutate`, no preview, no ratify. The only prior stops are
  param-resolution failures (`:603`/`:678`), not safety guards. R11
  (`"duration in frames"`) mis-selected `flame_set_start_frames` (a mutation) and
  the forced path **executed it directly**; only a missing param stopped the
  write. *Reachable, not theoretical:* the window is narrow (single mutation
  survivor with resolvable params) but R11 demonstrated selection + execution.
- **1C — the chain dispatch (DT cycle-5), NO gate.** `_step.py:409` —
  `raw = await mcp.call_tool(tool_name, params)`, wrapped only in try/except. The
  chain-step path executes tools itself. A mutation tool in a **non-commit** chain
  (1A routed it to *execute*, not *preview*) runs here with no ratify. **This is
  the chain-path twin of 1B**, and the original framing missed it entirely.

**This is the invariant broken — uniformly.** The fix is not three special-case
patches; it is **one invariant applied at every execution edge: may this tool
execute, given its declared authority class?** — at `:158` (route mutations to
preview, never execute), `:720-722` (1B), and `:409` (1C). `_execute_forced_tool`
and the chain executor were both conceived as routing mechanics; the moment a
mutation tool reaches either, they became **alternate authority paths with no
authority** (the A.2 AssentRecord chain bypassed). v1.10 makes dispatch consistent
with what the registry already declares. *Per Creative:* DI.1 asks **"may this
tool execute?"**; DI.2 asks "which tool should execute?" — adjacent questions,
separable promises.

### Symptom 2 — eligibility changes when routing cannot establish a path (USABILITY)

When a compiled step token matches ≠1 tool, the resolver hard-stops
(`_step.py:326-337`) with `tool_selection_ambiguous` and leaks an internal hint
(*"Step matched 4 tools… use a more specific verb/noun"*). It **already carries
the candidate list** (`:334`) — the material to disambiguate — but instead of
arbitrating, it dies. Three dogfood reads (path / timewarp / duration of a shot)
hit this. The documented [[feedback-pre-orchestration-resolution-paralysis]]
shape: a strict exact-match resolver downstream of an ambiguity-prone compile.
Whether a read is *eligible to execute at all* depends on whether routing could
confidently choose — which is the invariant violated from the other side.

## Doctrine (carried, sharpened — not reopened)

- **The invariant is the milestone's law.** *Operator authority and execution
  eligibility must not depend on which routing path was selected.* Authority
  (who may authorize a mutation) and eligibility (whether a read may execute)
  are properties of the **request**, never of the **route**. Enforced at every
  routing site — the commit gate (1A), the forced-tool call at `:720-722` (1B),
  and the resolver (Symptom 2).
- **The arbitration layer is control-flow, not meaning**
  ([[feedback-orchestrator-control-flow-not-meaning]]). Selecting among
  candidate tools — or asking the operator a *structured* choice — is dispatch
  arbitration. It MUST NOT author prose, synthesize intent, or invent a tool
  outside the candidate set. The 24.4 guard stands.
- **A blocked tool must fail toward understanding, not refusal** (Creative —
  *phase-level acceptance criterion, not implementation detail*). Once consumer
  tools default mutating-until-annotated (Decision 2), the redirect is **part of
  the product, hit systematically** — every time an integrator forgets an
  annotation, not occasionally. *"Operation blocked."* makes the phase feel
  broken; *"I can't safely determine whether this modifies project data — here's
  what I can inspect instead."* makes it feel cautious. Same safety outcome,
  opposite experience. The redirect fires for an **unbounded, consumer-defined
  set**, so "fail toward the read" must be robust, not tuned to the 11-read
  corpus.
- **The cut line is unchanged** (v1.9): explain facts that exist; invent
  nothing. v1.10 adds no new model authority; if it uses the model at all
  (Symptom 2 disambiguation), the model *selects from a provided set*.
- **Measure-first.** The intervention's effect is only legible against a
  *contemporaneous* failure-shape baseline
  ([[feedback-baseline-drift-invalidates-controls]]) — re-run the dogfood reads
  on current code before any fix; capture failure-shape, not just counts
  ([[feedback-failure-shape-stability-as-disposition-evidence]]).

## One invariant, two enforcement bars (§consequence-analysis)

The two symptoms share the invariant but are held to different bars — and the
room must vote on that:

| | Symptom 1 (authority) | Symptom 2 (eligibility) |
|---|---|---|
| Violation | authority changes with route | eligibility changes with route |
| Worst case | 1B/1C: a read **executes a mutation directly, no ratify** (forced *or* chain path; 1A: one careless ratify away) | an artist gets a confusing dead-end and gives up |
| Bar | a **guarantee** — deterministic, tested invariant; "best-effort" is unacceptable for a safety property | **best-effort** — measurably fewer dead-ends; arbitration may legitimately still miss |
| Trust model | must NOT depend on a second LLM pass (don't guard a safety property with a probabilistic classifier) | MAY use a bounded LLM selection among candidates (control-flow) |

**Consequence:** Symptom 1 leads and is non-negotiable. Prompt-hardening of
`build_compile_system_prompt` (the loose `commit` instruction at `:82-86`) is a
**mitigation, not the guarantee** — it reduces bad compiles probabilistically;
the guarantee is deterministic and lives below the model, at the routing sites.

## The spine

**v1.10 restores authority invariance at the compile→dispatch seam: operator
authority and execution eligibility become properties of the request, enforced
identically regardless of which routing path was selected.** Symptom 1 (the
authority guarantee — *one invariant at all three execution edges* `:158` /
`:720-722` / `:409`) is the load-bearing half; Symptom 2 (eligibility arbitration
at the resolver) is the usability half. The answer-pass (CR.1) is left intact;
v1.10 ensures more reads *reach* it correctly-routed and unambiguous.

**This is an enforcement-and-invariance milestone, not a classification one**
(cycle-4, room-confirmed). The mutation substrate already exists, curated and
live (`registry.py`), and was correct on the exact path that breached. The
project simply failed to make routing subordinate to it. So the work narrows to
three things — **(1) enforce the one invariant at all three execution edges**
(`_chat_compile.py:158` route, `handlers.py:720-722` forced 1B, `_step.py:409`
chain 1C); **(2) close the absent-annotation set at the registration boundary**
(`registry.py:127` — extend PROV-04's fail-safe to `user-taught`, so consumer
tools default mutating-until-annotated; this is ~0 work on builtins, which are
69/69 annotated, and it eliminates the consumer-path exposure an audit can't see);
**(3) regression-lock the invariant** — and explicitly NOT: invent a classifier,
migrate annotations, or reconstruct orphan logic.

**One sentence of explicit honesty (carry into every derived artifact):** *Forge
promotes MCP read/write annotations from advisory client metadata into
routing-authority enforcement semantics.* The promotion is a conscious
architectural adoption, not semantic abuse, **because** the annotation surface is
curated, comprehensive on the builtin surface, fail-safe for synthesized tools
(PROV-04), and already proven correct on the exact mutation path that exposed
Hole B (`flame_set_start_frames`, `registry.py:716`).

The remaining CR.1 §7 items (session/project scope; answer-pass
refuse-on-missing-field + error triage) are **not** in the headline. The
corpus-instrument capture-seam fix is named as *enabling tooling* (Q-DI4)
because measuring either symptom honestly needs the abort/preview seams the
current instrument is blind to.

## Open questions for discuss (leans given)

- **Q-DI1 — RESOLVED (cycle-4): the mutation substrate already exists live; this
  is not a classification milestone.** The cycle-3 framing (and an early Orch
  grep) asserted the annotation surface was "not runtime-reachable / must be
  built." **That was wrong — corrected by grounding the specific claim.** The
  live canonical home is `register_tool(annotations=...)` → `mcp.add_tool`
  (`registry.py:65-70, 132-137`), and `register_builtins` (`registry.py:290-1000+`)
  **populates it comprehensively and by hand** — `readOnlyHint` True/False plus
  `destructiveHint`, with reasoning comments (`:670`). Decisively, the exact tool
  that breached (Hole B / R11), `flame_set_start_frames`, is registered
  **`readOnlyHint: False`** at `:716` — *the system already knew it was a
  mutation.* The orphan `server.py` is a **stale duplicate, not the source of
  truth**; the live registry is. So the room does NOT have to answer "how do we
  classify mutation." It only has to answer **"why is routing allowed to ignore
  the classification that already exists?"** — which is the invariant, not a
  build dependency.
  *Resolution → (A), strongly* (DT + Creative converged): **promote the existing
  MCP read/write annotations from advisory client metadata into routing-authority
  enforcement semantics.** Building a second mutation-flag surface (B) would
  duplicate a complete, curated surface and create exactly the dual-authority /
  synchronization problem the room rejects. The framing carries one sentence of
  explicit honesty (below) so future contributors know these hints are no longer
  UI-only.
  *Remaining sub-decisions for discuss are small:* fail-closed semantics (absent
  or non-read ⇒ treated as mutating) and the audit's completeness bar.
- **Q-DI2 — Symptom 2 arbitration mechanism.** *Lean: extend the existing
  narrowing machinery (reachability / PR14 / PR20 at `_step.py`); add a bounded
  candidate-selection step (deterministic ranking first; one LLM selection among
  the ≤5 candidates only if needed); NEVER leak "matched N tools."* **Creative:**
  the disambiguation must not leak **tool identifiers** either — `flame_get_clip`
  vs `forge_get_shot` is the same developer-facing disease one layer up. If the
  artist must choose, the choice is in task terms, not tool names.
- **Q-DI3 — does v1.10 touch the compile prompt?** *Lean: yes, as secondary
  mitigation only* — tighten the `commit` instruction opener
  ([[feedback-rhetorical-position-as-architectural-control-surface]]) — but the
  ratifiable deliverable is the deterministic guarantee, not the prompt.
- **Q-DI4 — corpus-instrument capture seam.** Fold the capture-on-abort/preview
  fix in as measurement tooling, or hold it? *Lean: fold the minimal version* —
  without it the milestone cannot measure Symptom 1's "reads no longer reach
  mutation" or Symptom 2's "fewer dead-ends" against the corpus.
- **Q-DI5 — baseline.** Re-run the 11 dogfood reads on current `main` for a
  contemporaneous failure-shape baseline before any change. *Lean: yes,
  prerequisite.* (Daemon torn down at CR.1 close; same stdio-held-open bring-up.)

## Constraints (binding)

- **Public surface frozen:** `forge_bridge.__all__` stays **19**; no new external
  libraries. **Milestone number vs package version are decoupled** (Orch): the
  *milestone* is `v1.10`; the *package* stays on the `1.5.x` line — consistent
  with v1.5–v1.9 all riding package `1.4.1 → 1.5.1`. `v1.10` implies no `1.10.0`
  bump; discuss sets the actual patch/minor.
- **Authority invariance becomes a *tested invariant*** at all three execution
  edges — the milestone's acceptance criterion. A read cannot reach a mutation by
  *any* route (forced, chain, or commit-mis-route); eligibility does not depend on
  routing confidence.
- **Consumer-defined tools default mutating-until-annotated** (`registry.py:127`
  boundary fix). The absent set becomes ∅ universally; no annotate-before-enforce
  flag-day. A consumer read tool falls to the safe side (blocked/redirected,
  consumer-fixable via `annotations={"readOnlyHint": True}`), never a silent
  mutate.
- **A blocked tool fails toward understanding, not refusal** — phase-level
  acceptance criterion (the redirect is product, hit systematically).
- **No regression of the CR.1 answer-pass** or the preview→ratify→apply path; the
  guarantee *prevents reads from entering* the mutating path, it does not modify
  the mutating path.
- **No new model authority.** Arbitration selects; it never authors.
- **Reads-only stays reads-only.** Nothing here touches `AssentRecord`, ratify,
  or apply semantics.
