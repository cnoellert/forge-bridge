# Phase 10.1: Code Review Report

**Reviewed:** 2026-04-23
**Depth:** standard
**Files Reviewed:** 11
**Status:** issues_found

## Summary

Phase 10.1 closes the artist-UX gap for the Phase 10 web UI: it fixes the D-39
shell-duplication render bug (both the `hx-boost` nav-link path and the
`htmx.ajax` chip/query-console path), adds a server-derived Status chip column,
demotes developer telemetry via a caption, pins the quarantine-surface data
contract with a dedicated test, and adds two Playwright regression tests.

Security posture is appropriate for the localhost-only (`127.0.0.1:9996`),
read-only surface:

- No new HTTP mutations, no auth removal, no new secrets.
- The one user-reachable filesystem read (`_read_synth_source`) already has a
  path-traversal guard (SYNTH_ROOT containment check on the resolved path).
- Template rendering uses Starlette's `Jinja2Templates`, which enables HTML
  autoescape by default — all `{{ ... }}` interpolations into HTML body text
  are safely escaped.

No critical issues found. Two warnings concern attribute-context escaping in a
couple of templates (latent — not exploitable given current call sites, but
likely to bite later) and one unused-variable in `_render_error`. Five info
items cover minor cleanups.

Overall: the fixes land cleanly, tests are well-structured and tightly scoped
to the D-39/D-40/D-44 contracts, and the architectural notes in
`_derive_tool_status` and `_tools_preset_chips` are exemplary for future
maintainers.

## Warnings

### WR-01: Chip `tokens` are interpolated into a single-quoted JS string without JS escaping

**File:** `forge_bridge/console/templates/fragments/query_console.html:33`

**Issue:** The chip button renders a click handler as:

```
@click='stuffAndSubmit("{{ chip.tokens }}")'
```

Jinja2 HTML autoescape protects the *attribute quoting* (single quote `'`
wrapping the handler value), but the string interior is still a JavaScript
string literal wrapped in double quotes. If any future call site produces a
`tokens` value containing a literal double-quote (`"`) or a backslash (`\`),
the JS string will be broken or open to injection *inside the Alpine
expression*. Current callers (`_tools_preset_chips`, `_execs_preset_chips`,
`_manifest_preset_chips`) all hardcode safe tokens (`origin:synthesized`,
`promoted:true`, `hash:abcd`, etc.), so this is latent, not exploitable.

This is the kind of coupling ("template-safe only because every current caller
happens to be safe") that silently breaks when a Phase 11/12 chip gets a
`label` like `he said "hi"` or a dynamic token sourced from manifest data.

**Fix:** Pass the chip tokens through Jinja2's `tojson` filter so the JS literal
is produced defensively (tojson auto-wraps in quotes and escapes interior
characters per JSON rules, which are a strict subset of JS string literal
rules):

```html
<button type="button"
        class="chip"
        @click="stuffAndSubmit({{ chip.tokens|tojson }})">
  {{ chip.label }}
</button>
```

Note the outer double-quoted `@click` attribute; `tojson` emits a JSON string
with double quotes, and Jinja2 HTML autoescape will convert those to `&#34;`
for attribute safety — Alpine parses the attribute value after unescape, so the
expression ends up as `stuffAndSubmit("origin:synthesized")` in memory, which
is what we want.

### WR-02: `title` attribute renders full `code_hash` without an explicit safety note

**File:** `forge_bridge/console/templates/fragments/tools_table.html:24`

**Issue:** `<span class="mono" title="{{ tool.code_hash }}">{{ tool.code_hash[:8] }}</span>`
interpolates `tool.code_hash` into a double-quoted HTML attribute. Autoescape
handles HTML entities, so this is safe today. However, `code_hash` is produced
by trusted pipeline code (sha256 hex), so the attribute will always be
`[0-9a-f]{64}`. If a future refactor ever allows a non-hex value to reach this
template (e.g., an error placeholder), attribute-context escaping is the only
line of defense.

Additionally, `tool.code_hash[:8]` uses Python slice syntax in Jinja — this is
valid Jinja but uncommon enough that a reader may mistake it for a Python
expression leaking through. It's also safe against `None` only because the
surrounding `{% if tool.code_hash %}` guard prevents the slice from ever running
on `None`. That guard is the load-bearing check — it must not be removed.

**Fix:** Add a short comment pinning the invariant, and consider using Jinja's
more idiomatic `truncate(8, True, "")` or a dedicated server-side short-hash
field:

```html
{# code_hash is a pipeline-produced sha256 hex (64 hex chars); guard above
   ensures None cannot slip through — do not remove the {% if %} guard. #}
{% if tool.code_hash %}
<span class="sep">·</span>
<span class="mono" title="{{ tool.code_hash }}">{{ tool.code_hash[:8] }}</span>
{% endif %}
```

No code change is strictly necessary, but an explicit invariant comment would
prevent a silent regression if someone removes the guard.

## Info

### IN-01: `_render_error` keyword arg `status` is referenced via `status_code`, not `status`

**File:** `forge_bridge/console/ui_handlers.py:62-67`

**Issue:** The helper signature is `_render_error(request, template, message, status: int)` and passes `status_code=status` to `TemplateResponse`. This is correct — but the name `status` shadows HTTP-response-status-code terminology and common usage of `status` elsewhere in the file (e.g., `_derive_tool_status` return values `"active"/"loaded"/"quarantined"`). A reader scanning for the word `status` will get false hits.

**Fix:** Rename the parameter to `status_code` to match what it is actually assigned to:

```python
def _render_error(request: Request, template: str, message: str, status_code: int) -> HTMLResponse:
    return request.app.state.templates.TemplateResponse(
        template,
        {"request": request, "message": message, "active_view": None},
        status_code=status_code,
    )
```

### IN-02: `ui_index_handler` parameter `request` is unused

**File:** `forge_bridge/console/ui_handlers.py:172-174`

**Issue:** The handler receives `request` but never references it, and always
returns a 302 redirect. The parameter is mandatory for the Starlette ASGI
handler signature, so it can't be dropped, but a comment or `_ = request`
assignment would signal intent to linters and readers.

**Fix:** Add a one-line comment clarifying the signature requirement:

```python
async def ui_index_handler(request: Request) -> HTMLResponse:  # noqa: ARG001 (Starlette signature)
    """Redirect /ui/ -> /ui/tools (the default view)."""
    return RedirectResponse(url="/ui/tools", status_code=302)
```

Or if ruff is not configured for ARG rules, just leave as-is; this is purely
stylistic. Not a correctness issue.

### IN-03: Imports inside handler functions

**File:** `forge_bridge/console/ui_handlers.py:265-266, 323`

**Issue:** `ui_execs_handler` and `ui_exec_detail_handler` do `from dataclasses import asdict` and (in one case) `from forge_bridge.console.handlers import _parse_pagination, _parse_filters` inside the function body. This is typically done to avoid circular imports, but `dataclasses.asdict` has no circular-import risk — it's stdlib.

**Fix:** Hoist the stdlib import to the module top:

```python
from dataclasses import asdict
```

Leave the `forge_bridge.console.handlers` import in-function if there's a
confirmed circular-import problem; otherwise hoist both.

### IN-04: `_filter_tools` does case-insensitive match on `readonly` but exact match on `origin`/`namespace`

**File:** `forge_bridge/console/ui_handlers.py:118-142`

**Issue:** `readonly` compares `.lower() != readonly.lower()`, but `origin`
and `namespace` use exact `!=`. Users filtering with `origin:Synthesized` (mixed
case) will get empty results where `readonly:True` (mixed case) works. This is
a consistency issue, not a bug — UI grammar is documented to be case-sensitive
for values in D-08. But the inconsistency with `readonly` is surprising.

**Fix:** Either document "readonly is the only case-insensitive filter" or make
all value comparisons case-insensitive for UI-grammar-reachable filters:

```python
if origin and t.origin.lower() != origin.lower():
    continue
if namespace and t.namespace.lower() != namespace.lower():
    continue
```

This is an Info-level call — the tests explicitly use lowercase values so no
regression risk exists. Surface for D-08 grammar decision.

### IN-05: Duplicate fixture logic between the two Playwright regression tests

**File:** `tests/test_ui_chip_click_regression.py:64-200`, `tests/test_ui_nav_swap_regression.py:74-200`

**Issue:** `_find_free_port`, `_make_fake_read_api`, and `live_console_server`
are copied verbatim between the two test modules. This is documented as an
intentional choice ("module-local copies, following the existing convention"),
but it means a fixture-shape change (e.g., a new ToolRecord field like
`quarantined`) has to be applied in two places. Given the quarantine-surface
test explicitly guards against new fields, drift seems unlikely, but the
maintenance cost is real.

**Fix:** Promote the three helpers into `tests/conftest.py` or a new
`tests/_live_console_fixtures.py`. Leave the test assertions in their own
files — only share the harness code. This would be Info-level cleanup for
a future wave; current Phase 10.1 decision to duplicate is explicit and
documented in both test-file docstrings.
