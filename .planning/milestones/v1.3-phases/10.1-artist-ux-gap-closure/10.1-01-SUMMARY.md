## Triage outcome confirmed from code read

**Outcome: (b) "no signal exists because quarantined tools are filtered out of the registry."**

Evidence chain:
1. `forge_bridge/learning/probation.py:71-88` — `ProbationTracker.quarantine()` moves the source file from `~/.forge-bridge/tools/synth/` to `~/.forge-bridge/quarantined/` AND calls `mcp.remove_tool(tool_name)`.
2. `forge_bridge/learning/watcher.py:278-293` — the watcher's scan loop iterates `seen - current_stems` (tools whose files disappeared), calls `mcp.remove_tool(stem)`, and — when `manifest_service is not None` — schedules `manifest_service.remove(stem)` on the event loop. The watcher treats quarantine as a filesystem event: the file vanished from the scan dir, so the manifest drops the record.
3. `forge_bridge/console/read_api.py:111-113` — `get_tools()` returns `self._manifest_service.get_all()`. There is no separate zombie category, no filter on a `quarantined` field, no alternate read path.
4. `forge_bridge/console/manifest_service.py:39-60` — `ToolRecord` has NO `quarantined: bool` and NO `status: str` field. The third pinning test asserts this.

Result: the Tools view's Status-chip variant set reduces from 3 (`active` / `quarantined` / `loaded`) to 2 (`active` / `loaded`) for Phase 10.1. Plan 10.1-03 inherits this constraint.
