### The topology vocabulary — NEW project vocabulary entry

This is the first instance in the codebase of an explicitly-named
graph-substrate property:

**Filter is topology-preserving:**
```
list[X] → list[X]
```
It narrows a collection while preserving semantic level.

**Foreach + collect are topology-bridging** (deferred, §11 item 14):
```
list[X] → list[Y]
```
They introduce fan-out and/or semantic transformation.

The PR #21 body documents only the topology-preserving half — the
foreach half lands in PR body framing when item 14 phases. That's
correct prescriptive-vs-predictive discipline.

The negative space in PR #21's vocabulary section is precise:

> *"Filter never transforms semantic layer, bridges collection type,
> introduces fan-out, or performs iteration semantics."*

This implicitly references what foreach would do without naming the
future primitive. Substrate boundary defined by what filter is NOT,
not by gesturing at what comes next.

| Boundary | Rule | First instance |
|----------|------|----------------|

| Flame-native vs substrate-native | conversion at tool-output boundary; substrate never sees Flame-native | `timecode_to_frames()` |

`timecode.py` is the first utility. It carries empirically-verified

### The boundary inversion catch — first instance of active testing under new process

The timecode utility landed initially as `e4a9c51` with a subtle
defect: the conversion was placed INSIDE the Flame execution body
(the `bridge.execute_json` f-string template), not in the Python
wrapper. The Flame body executes in
`/opt/Autodesk/shared/python/forge_bridge/` which lacks
`forge_bridge.utils` — so the conversion would have failed at
runtime in Flame even though all unit tests passed.

The bug surfaced via ecological probe (the canonical Probe 3
attempt). The fix landed as `9ec391e` and moved conversion to the
Python wrapper layer.

The architectural significance: **the original commit violated
the architectural rule it stated in its own `utils/__init__.py`
docstring.** The rule said "utilities called at tool-output
boundaries" — meaning the Python wrapper layer. The commit placed
the call inside the Flame execution body — the INPUT boundary,
where Flame produces values. The fix didn't just make it work in
Flame; it restored the architecture to match its own stated rule.

This catch coincided with the room's commitment to a new
**active-testing process** for the code-handoff cycle. The active-
testing process commits the writing-room reviewer to:

1. Verify the diff matches the spec
2. Read the actual code at touched lines + adjacent context
3. Trace f-string body / wrapper boundary on Flame-touching tools
4. Run focused tests locally if possible
5. Run static checks
6. Look for sibling instances of the same pattern
7. Flag adjacent risks even if not in the commit
8. Name missed catches explicitly rather than gloss over them

`9ec391e` is the first commit landed under the new process — the
verification report named what was checked (sibling-bug scan, local
test execution, structural placement verification) rather than only
verifying the diff. Discipline-policy enforcement test pattern
extended to a 4th instance (`assert "forge_bridge.utils" not in
flame_body`) protects the specific boundary going forward.

## The successful defense — first instance of discipline operating during framing

The PR C testing arc surfaced a specific architectural pressure that
the room caught during framing, not after code landed.

### What happened

Probe 3 was originally framed as:

```
list reel contents on Sequences -> filter -> rename shots
```

That chain accidentally crossed abstraction layers. The intended
semantic was:

```
enumerate sequences → filter sequences → foreach(sequence → rename(sequence))
```

Without `foreach` as a substrate primitive (named in §11 item 14, not
yet implemented), the only way to make the original Probe 3 work
would have been to bake foreach semantics into `flame_rename_shots`
— making rename auto-loop over collection inputs.

### The pressure shape

A proposed fix arrived during the testing arc:

> *"When chain is `list reel contents -> filter -> rename shots` and
> rename can't resolve sequence_name from upstream, inject the filter
> collection as selected_segments and have rename loop over each
> sequence."*

That fix was operationally correct — it would make Probe 3 work. But
it would silently introduce foreach semantics into a domain mutation
tool. That is exactly the graph-collapse pattern 25.x has been
defending against: hidden iteration semantics inside a domain verb.

### The discipline functioning correctly

The room recognized the shape during framing discussion. It named:

1. **The larger pattern being held open**: foreach semantics
2. **The architectural maturation condition**: §11 item 14 (foreach +
   collect primitive) landing as its own substrate

It then steered to **(A) reshape Probe 3** instead of baking foreach
into rename. The revised Probe 3 uses `get segments on 30sec 21`
which is the correct enumeration-substrate-level entry point —
segments in, segments filtered, segments mutated. No topology
crossing required.

### Room-coined load-bearing observation

The room called this transition explicitly:

> **"This is the first successful defense of the graph-layer hierarchy
> against convenience pressure. That is a maturity signal, not a
> failure."**

What it means architecturally:

```
Filter composes with downstream mutation only when the
upstream enumeration topology matches the downstream
mutation scope. Filter is topology-preserving — it narrows
collections, it does not transform semantic level. Sequence
collections and segment collections are different graph
topologies. Filter does not bridge between them. Foreach will.
```

This is now part of PR #21's body framing — durably encoded in
public archaeology, not just internal discussion.
