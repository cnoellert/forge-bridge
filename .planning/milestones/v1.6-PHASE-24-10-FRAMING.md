## §0 Root Cause

Wire archaeology confirmed the cold-node failure is not a tool-selection,
inventory-position, or prose-shaping problem.

The model emits provider-native flat arguments:

```json
{"sequence_name": "30sec_21"}
```

`invoke_tool()` passes those arguments directly through:

```python
raw = await mcp.call_tool(name, arguments=args)
```

FastMCP validates against wrapper-schema tools that require:

```json
{"params": {"sequence_name": "30sec_21"}}
```

The validation boundary rejects the flat payload because `params` is missing.
The earlier visible `{}` collapse is therefore a boundary artifact, not proof
that the model emitted no arguments.

---

## §1 Hypothesis

The right boundary for correction is the adapter/invocation boundary, not the
prompt layer.

If `invoke_tool()` detects that a selected MCP tool has a required `params`
wrapper schema and the provider emitted flat arguments, it should normalize:

```json
{"sequence_name": "30sec_21"}
```

to:

```json
{"params": {"sequence_name": "30sec_21"}}
```

before calling FastMCP.

---

## §2 Scope

Single code perturbation:

- `forge_bridge/mcp/registry.py`
- `invoke_tool()` and local schema-inspection helpers only

Implementation requirements:

- Detect wrapper requirement by inspecting the actual tool schema.
- Preserve already-wrapped calls without double wrapping.
- Convert `{}` to `{"params": {}}` for wrapper-schema tools.
- Leave non-wrapper tools unchanged.
- Emit DEBUG observability when normalization applies.

Out of scope:

- Tool descriptions
- System prompt
- Inventory order
- Provider adapter parsing
- Tool schemas or signatures
- Public API surface

## §3 Forcing Function

Run the same four ecological operator probes:

```bash
fbridge chat "Give me the versions on the sequence 30sec 21"
fbridge chat "Rename the shots on 30sec 21 to genesis, 4-digit padding, by 10s"
fbridge chat "Get the segments on the sequence 30sec 21"
fbridge chat "Preview the start frames for 30sec 21"
```

**H1 pass criterion:** any target tool fires with populated `params` and
returns a real Flame response. One working tool is architecturally sufficient.

**H0 failure criterion:** validation errors persist, proving the wrapper schema
is not the issue and further adapter archaeology is required.
