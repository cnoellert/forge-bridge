# Phase N — Typed Ports + Foreach + Topology-Expanding Execution

## Architectural Thesis (carried verbatim from 25.0–25.3)

25.x is not about making the model smarter.
It is about making execution structure explicit.

Reliability comes from making execution structure explicit,
not from making the model smarter.

When the system fails, ask what structure is missing

## Why Typed Ports Are Not Optional Here

Without typed ports, foreach collapses into one of three unacceptable
shapes:

1. **magic** — infer item shape dynamically and hope
2. **rigid** — hard-code supported upstream collection types
3. **fragile** — fail unpredictably on unfamiliar upstream topology

None satisfy the substrate discipline established in 25.x. The topology
vocabulary makes the distinction explicit:

```
filter  list[X] → list[X]                    topology-preserving  (25.1)
select  list[X] → list[X]*                   topology-preserving  (25.3 — see note)
if-gate manifest → manifest                   topology-affecting   (25.2)
foreach list[X] → list[IterationResult]       topology-expanding   ← Phase N
collect list[IterationResult] → list[Y]       topology-converging  ← Phase N
```

**Select topology note — conceptual vs runtime contract:**
The *conceptual* topology of select is `list[X] → X` — it resolves a
single item by identity. The current runtime contract (25.3) returns
`list[X]` (single-item collection) to preserve chain uniformity and
defer scalar-vs-collection execution semantics. Phase N is where typed
ports make scalar-vs-collection execution semantics *expressible* as a
distinct substrate concept. Whether select scalarizes at runtime remains
an implementation decision once that substrate exists — the architectural
win at Phase N is that the substrate can finally express the distinction.
This is the 25.3 deferral decision carried forward explicitly.

**foreach = fanout. collect = reconciliation boundary.**
foreach expands topology: one collection becomes N iteration results.
collect converges topology: N iteration results become one reconciled
collection. Without collect, foreach has no reconciliation surface.
Without foreach, collect has nothing to converge.

## How the Substrate Earned Typed Ports

The project is discovering its abstractions through substrate pressure,
not importing them prematurely. Typed ports arrive as a consequence of
graph pressure:

- filter pressure → manifest propagation as the chain's currency
- manifest propagation → gate semantics operating on manifest structure
- gate semantics → select identity operating on manifest-produced entities
- select identity → trace rendering anchored on step kind not metadata
- resolver stabilization → interpretation boundaries explicit enough to
  distinguish what is structurally bound from what remains natural language

Each phase of the 25.x arc removed one degree of implicit interpretation.
Typed ports are what's left when all the heuristic compensation is gone
and the graph still needs to know what it's handling across topology-
expanding boundaries. This ordering is correct and load-bearing.

## The Typed Port Contract (load-bearing)

A typed port is not documentation metadata or an annotation hint.
**It is a runtime compatibility contract between graph nodes.**

A node declares:
- accepted input topology (what collection shapes it will receive)
- emitted output topology (what collection shapes it produces)
- manifest/item collection shape (field names and types carried)

**Typed-port validation is incremental and local.** Immediately before
dispatching step N, the chain engine validates step N's declared input
topology against the emitted topology from step N-1. This is wire-time
relative to the next dispatch edge — not a separate graph-construction
phase, not a preflight compilation step.

An incompatible topology wired between nodes is a **chain-wire
compatibility error** — caught at the dispatch edge before step N
executes. Not mid-execution. Not at chain initialization.

This is the mechanism that makes foreach substrate-grade: the body step
declares what item type it accepts; the foreach dispatch validates that
the upstream collection's emitted item type matches the body's declared
input type before the first iteration executes.

**The chain engine remains the executor throughout Phase N. Typed-port
validation is a new check inside the existing dispatch loop —
new dispatch path, not a new executor.**

## The Three Typed-Port Questions Phase N Must Answer

**1. What is the item type inside a foreach iteration?**
The loop body receives one item from a collection. Without typed ports,
the body either hard-codes the expected shape or fails on mismatch.

**2. How does loop-scoped context propagate to body steps?**
Each iteration has its own execution context. The current chain engine's
context propagation is manifest-based and flat (`_engine.py:28, 78-79`
confirms: single context dict rebuilt per step from `extracted_context +
__previous_result__`). foreach requires scoped context that does not
leak between iterations. This is a real substrate change.

**3. How are N iteration results reconciled back into a collection?**
collect is foreach's reconciliation boundary. Without typed output
topology declared on the body step, collect cannot validate that all N
`IterationResult` outputs have compatible shape before converging them
into `list[Y]`.

## Phase N Scope (full enumeration)

**1. Typed ports — minimum viable contract**
Input/output port type declarations on graph nodes. Minimum viable shape:
sufficient to support foreach body step validation and collect
reconciliation. Does NOT attempt to generalize to arbitrary graph typing,
polymorphic node algebra, or future DAG semantics. Those are Phase N+.

Commit 2 establishes the contract. Commit 3 pressure-tests it under
topology-expanding conditions. The causal order matters: substrate
contract first, topology-expanding primitive second — but Commit 2 is
bounded to the minimum shape foreach needs, not pre-generalized.

Chain-wire compatibility checks occur immediately before dispatch of
each step. Incompatible topology raises `ChainWireCompatibilityError`
at the dispatch boundary.

**2. foreach — Phase N initial scope: one body step**
First topology-expanding primitive. `ForEachNode` in
`forge_bridge/graph/foreach.py`. Takes a collection, executes one body
step per item, emits `list[IterationResult]`.

**Phase N initial foreach scope is one body step only.** A
chain/subgraph body (e.g. `foreach(filter(...) -> rename(...))`) is
Phase N+ scope — it introduces termination bounding, local manifest
scoping, and nested collect reconciliation out of scope for the initial
primitive.

**foreach body steps dispatch through the existing step pipeline** using
the same resolver and execution surfaces as top-level steps, with the
iteration-scoped context substituted for chain-scoped context. foreach
is not a mini-language. It is a scoped reuse of the existing execution
substrate.

Dispatch via `_maybe_execute_foreach_step` in `_step.py` —
**new dispatch path inside the existing chain engine, not a new
executor.**

Grammar: `is_foreach_step` anchored `^\s*foreach\s*\(`. Parens contain
a single step, not a predicate. `parse_foreach_step` extracts the body
step text. Per established pattern (filter, select, if-gate all bundle
grammar with the primitive in `graph/*.py`), grammar lands with the
primitive in Commit 3.

**Note on `_chain_parse.py`:** balanced-paren extraction for the foreach
body step likely touches `_chain_parse.py` as well as `_step.py`. This
is a likely touch surface pending confirmation once body-parse shape is
known. Do not assume `_step.py`-only changes.

**3. collect — substrate-generic reconciliation**
`CollectNode` in `forge_bridge/graph/collect.py`. Consumes
`list[IterationResult]`, produces reconciled `list[Y]`.

**collect is substrate-generic.** It does not have named-field semantics
for `proposed_changes`, `execution_state`, or any other domain field.
Those fields appear in the canonical probe because that is what the probe
exercises — they are not special cases in the collect implementation.

Generic reconciliation rules:
- list fields: merge by declared topology rule
- scalar fields: reconcile by compatibility rule
- incompatible field topology: raises `ChainWireCompatibilityError`

The canonical probe exercises these generic rules against rename-pipeline
manifests. The domain fields are the test vehicle, not the
implementation target.

**4. Per-iteration trace render — render-layer only**
Today's `--trace` is one line per step. foreach introduces N body
iterations. Per-iteration rendering in `forge_bridge/cli/chat.py` —
**render-layer-only; zero chain engine changes.**

**Carry-forward from 25.3:** `_maybe_render_*` functions anchor on step
kind, not metadata presence. Iteration results carry upstream metadata
via context propagation — the foreach render path must not fire for
non-foreach steps inheriting foreach metadata. The **12th
discipline-policy enforcement test** asserts this negative space.

**5. Chat/exec split — bounded first motion**
Phase N does not fully liberalize chat interpretation.

> **Sequencing constraint (load-bearing):**
> The first motion is typed execution contracts for foreach.
> The chat/exec split initially surfaces only where typed-port validation
> already exists. Execution structure stabilizes first.
> Interpretation liberalization follows behind it.

At Phase N scope:
- **Exec surface:** chain-wire compatibility errors at the foreach/collect
  boundary. Does NOT generalize to all step types in Phase N.
- **Chat surface:** typed port contracts validate inferred port mappings
  at the foreach/collect boundary. Natural language interpretation of
  non-topology-expanding steps is unchanged from 25.x.

Full chat/exec split across all steps is Phase N+ scope.

**6. Port compatibility**
Chain-wire compatibility checking at the foreach/collect dispatch
boundary. Caught at the dispatch edge before iteration begins.

**7. Macro bridge semantics**
Named subgraphs with declared port types. Closes the macro/subgraph
migration named in v1.6-FRAMING.md §6.5.

Macros inherit the typed-port substrate established in Commit 2; no
separate macro-runtime migration commit exists in Phase N. The Phase N
motion is vocabulary alignment and typed-port adoption, not a new
execution surface.

## Anti-Scope

- **No formal parser ASTs or recursive grammar.**
- **No parallel/fanout execution.** Sequential foreach only. v1.7.
- **No unbounded loops.** foreach over explicit collections only.
- **No chain/subgraph foreach body in Phase N.** One body step only.
- **No schematic rendering.** v2.0.
- **No graph runtime executor.** This is the most important boundary:

  > **new dispatch path, not a new executor.**

  This phrase is load-bearing throughout Phase N and should appear in
  commit messages, close artifacts, and future seeds. It prevents
  premature drift toward DAG runtimes, schedulers, async graph execution,
  or distributed runners before typed iteration is proven stable.

- **No preflight graph compilation.** Typed-port validation is
  incremental and local — per dispatch edge, not a construction phase.
- **Do not rename `ChainWireCompatibilityError` to
  `GraphConstructionError`.** The former is accurate; the latter implies
  Shape B.
- **collect is not domain-aware.** No named-field semantics for
  `proposed_changes`, `execution_state`, or any rename-pipeline concept.
- **Typed-port compatibility validation does not subsume primitive semantic
  validation.** Compatibility errors and primitive runtime errors remain
  distinct diagnostic classes.
- **No "resolver category 7."** foreach body parsing is nested step
  parsing, not value resolution. Grammar lands with the primitive.
- **No full chat liberalization in Phase N.** Bounded first motion only.
- **Do not reopen select identity deferrals** unless foreach body
  execution surfaces a new forcing function.
- **No compare primitive (Item 15).** Still deferred.

## Canonical Probe

The probe must pressure collect reconciliation semantics, not just
foreach iteration.

```bash
fbridge chat --trace "get sequence segments on 30sec 21 -> foreach(rename shots with prefix genesis dry_run) -> collect -> if(proposed_changes exists)"
```

Pass criterion: foreach iterates over all N segments returned by the
upstream enumeration step, each body step producing a dry_run
`IterationResult` carrying `proposed_changes`. collect converges N
`IterationResult` manifests using generic topology rules — list fields
merged, scalar fields reconciled (varying scalar fields omitted from
the reconciled top-level manifest per `fix(graph): scalar variance is valid collect state`), compatible topology confirmed. If-gate opens on
the collected `proposed_changes`; gate decision renders in `--trace`
along with foreach per-iteration scoped lines and the collect step.
Mutation (`commit` step) is deferred to Phase N+ per
`.planning/seeds/SEED-COMMIT-PRIMITIVE-PHASE-N+.md`. The Step B rerun
empirically established that collect reconciles structural evidence for
gate semantics; it does not materialize executable mutation plans.
Verbatim N recorded in Commit 5 evidence.

Negative-space probe (chain-wire compatibility invariant): deliberately
wire a body step that emits incompatible topology.
`ChainWireCompatibilityError` must raise at the dispatch edge — before
any iteration executes.
