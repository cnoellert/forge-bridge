# Phase 24.4 — Layer B Termination Semantics Under Successful Execution

## §0 — Banner

**The orchestrator acquires termination authority — and only termination authority.**

Phase 24.3's rerun bundle captured the cleanest Layer B isolation evidence this project has produced: across three canonical-probe firings, the substrate dispatched `flame_execute_python` SIX TIMES with `status=ok`. Each invocation returned the canonical Reel 1 clip list to the LLM. The model received the answer SIX TIMES. It synthesized a readable conversational response ZERO TIMES.

That observation transforms D-07 from an observability primitive into a policy-boundary question:

> **What authority does repeat-call recognition confer?**

Phase 24.4's answer is bounded: **termination authority, not semantic authority.** When the orchestrator observes K identical successful canonical dispatches (same tool, same args, same result, all `status=ok`), the orchestrator terminates the loop deterministically and exposes the accumulated tool results verbatim in a structured terminal envelope. It does NOT synthesize conversational text. It does NOT inject prompts. It does NOT coerce the model semantically. It does not even attempt to answer the user. It says, in structured form: *here is what the model produced; here is why I stopped iterating; here is the orchestration state at termination.*

This is the first phase in the 24.x sequence where the orchestrator gains decision authority that the model previously held alone. The discipline of the framing is to keep that authority precisely scoped.

## §1 — Load-Bearing Diagnosis

The 24.3-rerun §4.3 case (per `.planning/milestones/v1.6-PHASE-24-3-RERUN.md`) demonstrated:

```
JSON cold  : 1 status=ok substrate emission, 0 readable answers, 120.094s wall-clock
JSON warm  : 2 status=ok substrate emissions, 0 readable answers, 120.062s wall-clock
SSE warm   : 3 status=ok substrate emissions, 0 readable answers, 120.039s wall-clock
─────────────────────────────────────────────────────────────────────────────────
TOTAL      : 6 status=ok dispatches with canonical result, 0 conversational answers
```

The 14b SSE warm probe is the load-bearing measurement:

```
iter=2  flame_execute_python   status=ok   581 bytes result
iter=4  flame_execute_python   status=ok   581 bytes result  ← identical to iter 2
iter=6  flame_execute_python   status=ok   376 bytes result  ← different (variant query)
```

Iters 2 and 4 are byte-identical successful canonical dispatches separated by a (correctly-blocked) wrong-affordance iter 3. The model received the same canonical answer twice within the same conversation and did not terminate. The current D-07 site at `forge_bridge/llm/router.py:639-662` does NOT fire on this pattern — its `repeat_key = (tool_name, args_canonical)` matches at iter 4, but the existing threshold (`>= 3`) prevents action until a third identical dispatch.

The pathology is **structurally orchestration-layer**. The substrate dispatched correctly (`status=ok` at substrate boundary, 14-15ms latency, 581 bytes of clip-list payload). The runtime returned each iter within budget. The streaming layer exposed the path of failure observably. **The only layer where the failure lives is orchestration termination semantics.**

Phase 24.3 §4.3 binding satisfied verbatim:

> *"If L1 + L3 hold under the post-swap + streaming regime but L2 does not — i.e., dispatch is correct, runtime is within budget, streaming reveals forward progress, but the operator still doesn't receive a readable answer — then Phase 24.3 has succeeded at A + C and Layer B termination/convergence is the next phase."*

24.4 is that next phase. The empirical precondition is the 6-status=ok-zero-answers archaeology of `v1.6-PHASE-24-3-RERUN.md`.

## §2 — Core Goal

**Promote orchestration-observed K-fold canonical recurrence into a deterministic termination trigger that returns a structured terminal envelope without semantic coercion of the model.**

Concretely:

- Extend D-07's trigger predicate from `(tool_name, args_canonical) >= 3` to a **parallel trigger** keyed on `(tool_name, args_canonical, result_canonical)` with threshold `>= 2`, firing only when all three match AND `status=ok`.
- When the new trigger fires, the orchestrator terminates the `complete_with_tools` loop deterministically and returns an HTTP 200 with a structured `orchestration_terminated` envelope.
- Preserve existing D-07 behavior for the failed-affordance-repetition pathology (parallel triggers, different authorities — see §6.4).
- Verify the change modified ONLY orchestration authority, NOT model behavior, by comparing the first 3 iterations against the 24.3-rerun bundle byte-for-byte.

The success of Phase 24.4 is **operationally observable** and **archaeologically distinct**: a canonical-probe re-fire under the new policy should terminate at iter 4 in roughly 30-50s (compounding Layer A's per-iter speedup with Layer B's deterministic stop) and return a terminal envelope whose `trigger` field is `"k_fold_canonical"` — legible without code inspection.

## §3 — Scope

### §3.1 — IN scope

1. **Trigger primitive** at the orchestration layer (`LLMRouter.complete_with_tools()`):
   - Detect K-fold `(tool_name, args_canonical, result_canonical, status=ok)` recurrence.
   - K=2 (resolved at §6.1).
   - Result hash computed over post-sanitization, post-truncation bytes (the bytes the LLM actually sees per `_sanitize_tool_result`).

2. **Termination authority** at the orchestration layer:
   - On trigger, terminate the loop deterministically.
   - No further `complete_with_tools` iterations occur.
   - No further LLM invocations occur.

3. **Structured terminal envelope** (HTTP 200, `orchestration_terminated` payload):
   - Required fields: `status`, `trigger`, `reason`, `iterations`, `accumulated_results`.
   - `accumulated_results` exposes every successful tool result verbatim.
   - Envelope shape clearly distinguishes orchestration-decided termination from model-decided termination.

4. **Telemetry**:
   - New log status `status=orchestration_terminated` (analog to existing `status=repeat_blocked`).
   - Trigger predicate components observable at telemetry — no semantic coupling.

5. **C-layer anti-scope verification**:
   - Canonical-probe re-fire bundle demonstrating identical first-3-iter footprint vs 24.3-rerun.
   - Identical substrate dispatches, identical args, identical results, up to the trigger point.

### §3.2 — OUT of scope (explicitly deferred)

1. **Affordance-selection regression** — iter-1 model picks `flame_find_media` instead of `flame_execute_python` despite 23.1 docstring rewrites. Separately-bounded; could feed into a future phase. NOT contaminated into 24.4.

2. **Orchestrator-side answer synthesis** — orchestrator does not produce conversational text from accumulated tool results. The chat handler / Console UI / CLI may surface the envelope to the user downstream; that is a separate UX phase, not 24.4 scope.

3. **Model-side behavior modification** — no prompt change, no system message change, no temperature change, no tool-list narrowing, no semantic shaping of any kind. The model still does what it currently does up to the K=2 trigger.

4. **D-07 unification** — existing D-07 (failed-affordance repetition) and new trigger (successful-canonical recurrence) remain parallel triggers with different authorities. Not unified into a single primitive. See §6.4.

5. **Cross-provider termination semantics** — Anthropic-routing has its own loop characteristics; the daily chat path is Ollama via `sensitive=True`. 24.4 ships the policy on the Ollama path; Anthropic-path verification is bounded out.

6. **Token-delta streaming** — still deferred per 24.3 §3.3 anti-scope (orthogonal; native `tool_calls` reliability unchanged).

7. **K-tuning beyond K=2** — adaptive thresholds, model-class-conditioned K, per-tool K — all deferred. 24.4 ships one K value across all cases.

8. **Result-hash normalization beyond exact bytes** — semantic equivalence (e.g., reordered JSON fields, whitespace-insensitive comparison) is anti-scope. Hash the bytes the LLM actually sees, byte-exact, post-sanitization.

## §4 — Trigger Primitive (CONVERGED)

The new trigger fires when:

| Predicate component | Source | Notes |
|---|---|---|
| `tool_name` matches | `ToolCall.tool_name` | Existing D-07 component |
| `args_canonical` matches | `json.dumps(arguments, sort_keys=True)` | Existing D-07 component (`router.py:634`) |
| `result_canonical` matches | post-sanitization, post-truncation bytes shown to LLM | **NEW** — bytes returned by `_sanitize_tool_result(tool_result, max_bytes=effective_max_bytes)` |
| `status == "ok"` (substrate-level) | tool call did not raise; `is_error=False` on the `ToolCallResult` | **NEW** — restricts trigger to successful dispatches |
| K-fold count `>= 2` | Per-tuple counter (analog to `seen_calls`) | **NEW** threshold; existing D-07 uses `>= 3` and stays unchanged |

The trigger is **observation-layer-clean**:

- No semantic coupling. The orchestrator does not interpret the tool result; it hashes the bytes.
- No prompt shaping. The orchestrator does not write to the conversation; it observes recurrence.
- No model coercion. The orchestrator does not inject messages until termination occurs.

The asymmetry vs existing D-07 (which fires on FAILED repetition) is intentional. Existing D-07 sees a model stuck on a wrong affordance; the new trigger sees a model stuck on a right one. Different pathologies, different evidence shapes, different orchestration responses.

### §4.1 — Result-hash semantics

Hash the **exact post-sanitization, post-truncation bytes** that flow back to the LLM via the conversation history. Specifically: the `content` field of the `ToolCallResult` after `_sanitize_tool_result(..., max_bytes=effective_max_bytes)`.

Rationale: the orchestration layer must reason over the **model's epistemic view**, not raw substrate payloads. If sanitization or truncation alters bytes that reach the LLM, those are the bytes that drive convergence behavior — and therefore the bytes whose recurrence signals settled truth.

Exact-bytes hashing (not semantic equivalence) is intentional and binding:

- If args_hash matches but result_hash diverges (different bytes shown to model), state is changing or substrate variance is present — iteration may still be legitimate.
- If both match — settled truth is being re-served, and termination authority becomes eligible.

### §4.2 — Trigger key uniqueness

The trigger key is `(tool_name, args_canonical, result_canonical)`. The existing D-07 key is `(tool_name, args_canonical)`. They occupy separate counters; firing one does not increment the other.

Concretely: a `seen_canonical_results: dict[tuple[str, str, str], int]` map analog to `seen_calls`, scoped to the same `complete_with_tools` invocation.

## §5 — Authority Shape (CONVERGED)

When the trigger fires:

1. **Loop termination is deterministic.** No further iterations of `complete_with_tools` execute. No further LLM invocations occur. No further tool dispatches occur.

2. **Terminal envelope is returned via HTTP 200** with a structured `orchestration_terminated` payload:

   ```json
   {
     "status": "orchestration_terminated",
     "trigger": "k_fold_canonical",
     "reason": "Tool <tool_name> dispatched successfully <K> times with identical canonical arguments and identical canonical result. Loop terminated by orchestration policy.",
     "iterations": <iter_count>,
     "accumulated_results": [
       {"tool_name": "...", "args_hash": "...", "result_hash": "...", "content": "<verbatim>", "iter": N},
       ...
     ]
   }
   ```

3. **No synthesized conversational text.** The orchestrator does NOT produce an `assistant` message wrapping or paraphrasing the accumulated results. The terminal envelope is the entire response.

4. **No prompt injection.** The orchestrator does NOT add a "you have the answer, please summarize" message to the conversation history. The trigger fires AFTER the LLM has had its turn and the orchestrator has decided to stop.

5. **Trigger field is load-bearing archaeology.** Naming `trigger = "k_fold_canonical"` keeps future firings legible without code inspection. If subsequent phases introduce additional triggers (deferred per §3.2.7), each gets its own taxonomic name.

### §5.1 — Why no orchestrator-side synthesis

Orchestrator-side synthesis would be hidden convergence shaping by another name. The orchestrator producing prose — even verbatim wrappings of tool results — drifts toward semantic authority. 24.4 ships **termination authority only**; the downstream surface (chat handler, Console UI, CLI) decides how to present the envelope to the user. That decision is a separate UX phase, intentionally unbound at this framing.

### §5.2 — Why HTTP 200, not HTTP 504

The orchestration succeeded at its scope: it observed K-fold canonical recurrence and made a deterministic decision. The semantic value of the conversation may be incomplete (model failed to synthesize), but the orchestration layer did not fail — it terminated by policy. HTTP 504 (`request_timeout` envelope) would misclassify a policy-layer success as a transport-layer failure and degrade the archaeology of future firings.

## §6 — Authority Location (CONVERGED)

The new trigger lives at the existing D-07 site: `forge_bridge/llm/router.py:639-662` inside `LLMRouter.complete_with_tools()`.

Reasons:

- **Orchestration semantics belong in the orchestration layer.** Not adapter protocol parsing (`forge_bridge/llm/_adapters.py`), not transport handling (`forge_bridge/console/handlers.py`), not registry execution (`forge_bridge/mcp/`), not UI shaping (`forge_bridge/console/app.py`).
- **D-07 is the policy-boundary site already.** The site reads tool dispatch state, decides whether to continue or interfere with the loop, and emits status telemetry. Extending it with a parallel trigger is layer-discipline-preserving.
- **Same telemetry shape, parallel taxonomy.** Existing D-07 emits `status=repeat_blocked`; new trigger emits `status=orchestration_terminated`. Future readers searching the logs for either status see the same shape.

24.4 promotes D-07 from:

> "repeat-call observation/blocking"

to:

> "repeat-call observation + deterministic termination authority"

Same code site. Expanded authority. Existing behavior preserved as a parallel trigger.

## §10 — Anti-Scope Binding (Explicit)

The architectural integrity of the 24.x decomposition depends on Layer B's termination policy NOT collapsing into prompt shaping. The anti-scope is therefore binding at framing time, not at execution time.

### §10.1 — Explicitly forbidden

1. **Orchestrator-side answer synthesis.** No prose generation from accumulated tool results. The orchestrator MUST NOT write a sentence that paraphrases or interprets a tool result.

2. **System message modification.** The system prompt (if any) MUST NOT be altered to nudge convergence. No "you have the answer now, please summarize" injections at any point in the loop.

3. **Mid-loop prompt injection.** No synthetic assistant or user messages added to the conversation history between iterations to influence model behavior. The existing D-07 synthetic message ("Try different arguments…") stays as-is for its pathology; the new trigger MUST NOT add any prompt.

4. **Tool-list narrowing.** No reduction in the 58 tools offered. No filtering of available tools mid-loop.

5. **Semantic equivalence in result-hash.** Hash the exact bytes. No normalization, no whitespace-insensitive comparison, no field-reordering, no semantic dedup.

6. **Temperature / sampling-parameter changes.** Model invocation parameters MUST remain byte-identical to 24.3 close state.

7. **Cross-provider drift.** No reach into Anthropic adapter, no cloud-routing modifications, no provider-specific termination heuristics.

8. **Hidden convergence shaping under any disguise.** If a proposed implementation requires the orchestrator to interpret tool result content, the proposal is out of scope. If a proposed implementation requires the orchestrator to write to the conversation, the proposal is out of scope.

### §10.2 — The discipline

24.4 ships **one** new policy primitive. Anti-scope discipline is the cost of preserving the 24.x decomposition's clarity. The operator's framing names the temptation explicitly:

> *"Do NOT let termination policy collapse into prompt shaping. The project is finally legible because: substrate truth stabilized, runtime economics isolated, responsiveness isolated, termination pathology isolated. Protect that decomposition."*

§10.1 operationalizes that protection.
