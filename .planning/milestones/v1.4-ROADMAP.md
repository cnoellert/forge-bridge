## Overview

Extended forge-bridge with the human-in-the-loop primitives projekt-forge v1.5 needs to become a thin Flame-side consumer: a `staged_operation` entity with proposed→approved→executed/rejected/failed lifecycle, an MCP + HTTP read/write surface for list/approve/reject, an agentic `complete_with_tools()` coordinator on `LLMRouter`, and a chat endpoint that binds it all together for the Web UI and external Flame-hooks consumer.

Phase letter scheme `FB-A..FB-D` was preserved (NOT renumbered) because projekt-forge v1.5 declared the alias as required deps; numeric phase IDs are internal plumbing for `find-phase`, state tracking, and directory naming. Phases 13 (FB-A) and 15 (FB-C) shipped in parallel — no shared dependency. 14 (FB-B) depends on 13; 16 (FB-D) depends on 15.

Two decimal gap-closure phases were inserted mid-milestone per the Phase 10/10.1 precedent: 16.1 closed three Phase 16 deploy bugs (chat tool-list hang, Starlette TemplateResponse signature drift, lifespan wiring guard); 16.2 closed Bug D, the chat surface returning raw tool-call JSON as the assistant's terminal text instead of executing the tool and synthesizing.

### Phase 15 (FB-C): LLMRouter Tool-Call Loop

Hard iteration cap, wall-clock cap, repeat-call detection, ingest-time result truncation, sanitization boundary on tool results, and a recursive-synthesis guard ship in the same phase.
