## §1 — Schema Cluster (Q1–Q5)

### Q1 — `status` transitions: in-place vs append-only

**Position:** **v1.6.0 permits in-place mutation at the storage layer while preserving append-oriented lifecycle semantics at the operational layer.** Full append-only persistence remains a v1.7+ reconsideration point once replay and graph-runtime requirements harden.

**Why:** v1.6 ships substrate emission, not replay. Full append-only persistence at the storage layer would pay a real cost on every JSONL reader (re-derive current status from event sequence) and the use case that justifies the cost (deterministic replay) is v1.7-scoped per framing §11.2. Today's `forge_bridge/learning/execution_log.py` (347 LOC) treats records as immutable observations; introducing append-only-mutation semantics at the storage layer in v1.6.0 risks shipping a hybrid where some fields mutate (`status`) and others don't (everything else).

But the runtime semantics are already drifting toward append-oriented lifecycle thinking even where the storage layer temporarily permits mutation. v1.6 carries `graph_id` correlation, `started → transport_error` event pairs, replay-oriented provenance, graph-event archaeology, and operational lifecycle traces. The storage-layer simplification is a Phase 24 implementation win, **not a philosophical divergence** from the direction the substrate is already evolving toward.

> **Architectural observation worth naming explicitly:** operational semantics are becoming event-oriented *before* execution becomes graph-native. This ties together the substrate's graph events, provenance, replayability, `transport_error` archaeology, protocol normalization, doctor surfaces, and `graph_store` diagnostics — and explains why the current substrate decisions look the way they do. Decisions like Q1 (in-place storage), Q18 (graph_store doctor row), and Path C governance norms (anti-scope binding with §7.3 trigger conditions) all reflect this asymmetric maturation: lifecycle thinking lands first; full append-only persistence and graph-native execution land later, against evidence triggers.

**Counter:** If v1.6.x surfaces in-session replay (chat REPL restore from last graph), in-place storage will need migration. Migration is bounded — reader logic changes, not schema shape. Append-only is additive: nothing prevents v1.7 from introducing sibling `status_event` records alongside the mutating `status` field without breaking v1.6 readers. The boundary between operational-semantics-event-orientation (already present at v1.6) and storage-layer-append-only (deferred to v1.7) is the right trigger boundary.

### Q2 — `provenance.user` capture pre-auth

**Position:** Three-source resolution, evaluated in order:
1. **Local-host surfaces** (CLI, MCP stdio, Console direct): `os.getuser()`.
2. **HTTP surfaces** (`/api/v1/chat`, `/api/v1/exec`): the existing IP-based rate-limit key.
3. **Otherwise:** `null` (explicit, not faked).

**Why:** All three values are non-load-bearing for v1.6.0 promotion criteria (subgraph mining doesn't weight user identity); they become load-bearing at v1.7+ if cross-session promotion weighting wants user-identity input. Best-effort capture today preserves the option without making `SEED-AUTH-V1.5` a v1.6 dependency. The three-source split mirrors the actual surface architecture: local surfaces are OS-bound; HTTP surfaces are IP-bound; pre-auth + ambiguous paths are explicitly null.

**Counter:** `os.getuser()` inside MCP stdio captures the *bridge process* user, not the LLM-agent caller. This is a known imperfection — the best signal local-host MCP has pre-auth. Documented at the schema field's comment, not silently glossed.

### Q3 — `metrics` field placement: sibling or separate stream

**Position:** **Sibling at v1.6.0**, with a documented size cap (≤ 5 numeric fields). Promote to separate stream when fields exceed the cap.

**Why:** Sibling keeps record-write atomicity simple — one append per node, all node-related data co-located. Separate-stream is cleaner for query workloads (production observability, aggregations) but those workloads don't exist at v1.6.0 — they're v1.7+ when graph runtime starts producing production traffic patterns.

**Counter:** Unbounded metrics-field growth bloats every node read for consumers that only need structural data. The size cap is the mitigation; documented in the schema docstring.

**Defer:** "Promote to separate stream" addition to v1.7 unless cap is exceeded earlier.

### Q4 — Node ID format

**Position:** **UUIDv7.**

**Why:** Sortable (creation-time monotonic within the same millisecond), standard (RFC9562), python ecosystem support exists (`uuid6` external dep on 3.11, stdlib `uuid.uuid7()` on 3.13+). ULID is functionally equivalent but non-standard. Monotonic counter forces global state and breaks concurrent emission across surfaces.

**Counter:** Python 3.11 (the pinned floor) requires a small external dep. The project already carries dozens of deps; one more is not material. Migration to stdlib `uuid.uuid7()` is a future patch when the Python floor moves.

### Q5 — Schema versioning policy

**Position:** **Lenient, additive-within-minor.** `schema_version: "1.6.0"` field present on every record. Readers tolerate unknown fields (log at debug-level). Major-version mismatch hard-rejects. Breaking changes carry a deprecation period documented in the milestone's release notes.

**Why:** v1.6 is a 6–8 week arc with substrate emission at week 1–2 and full universalization at week 5–6. Mid-arc additive fields are likely. Strict versioning would make every additive field break older readers within the same minor.

**Counter:** Lenient policy means a v1.6.0 reader silently misses new v1.6.1 fields. Mitigation: readers log unknown-field-encountered at debug-level; consumers opt into strict mode if they need all-or-nothing semantics.

## §2 — Storage Cluster (Q6–Q8)

### Q6 — JSONL file granularity

**Position:** **Per-graph at v1.6.0** (`~/.forge-bridge/graphs/<graph_id>.jsonl`), with year-month sharding (`graphs/<YYYY-MM>/<graph_id>.jsonl`) introduced mechanically when any single directory exceeds ~1500 files.

**Why:** Per-graph aligns the read unit with the natural cognitive unit. Per-day mixes orthogonal graphs into one file, forcing every read to filter by graph_id. Per-surface segments by emitter (CLI vs chat) but graphs cross surfaces (chat-authored chain executed via chain engine emits from two surfaces into one graph).

**Counter:** File count grows linearly. At production rates (~50 graphs/day per heavy user), that's ~18K files/year per user. Year-month sharding is the standard mitigation; not v1.6.0-blocking.

### Q7 — Retention policy

**Position:** **No rotation at v1.6.0.** Manual-cleanup recipe in `docs/TROUBLESHOOTING.md` if operator pressure surfaces; rotation as policy is v1.7+.

**Why:** v1.6.0 doesn't have enough adoption to surface rotation pressure. Premature rotation policy is design-against-fiction. Append-only JSONL is also cheap on disk (~5KB/node typical); even a heavy user's annual log is bounded at sub-GB.

**Counter:** Long-running operator workstations could accumulate hundreds of MB. Mitigation: opt-in cleanup recipe ("delete files older than N days") in v1.6.x if surfaced.

### Q8 — Concurrency / POSIX append atomicity

**Position:** **Verify both target platforms during Phase 24** (Linux flame-01 + macOS portofino) as an explicit acceptance criterion. If either platform fails atomicity for typical record sizes (≥ 512 bytes), fall back to fcntl advisory locking — a ~10 LOC addition with no behavior change.

**Why:** POSIX `O_APPEND` is atomic per-write within `PIPE_BUF` (4096 bytes on Linux; **512 on macOS**). Typical node records exceed 512 bytes, so macOS atomicity is *not* guaranteed by POSIX alone. This is the kind of architectural-correctness gap that should be surfaced empirically rather than assumed.

**Mitigation options ranked:** (a) per-graph file means typically single-emitter — strongest path; (b) fcntl advisory locking for cross-surface emission into the same graph; (c) line-buffered + explicit flush; (d) write-to-temp + rename (heaviest).

**Verdict:** Phase 24 acceptance criterion. Choose (a) + (b) if (a) alone proves insufficient under cross-surface emission.

**Verification status (resolved-operational-assumption archaeology):**

- **macOS portofino (Darwin, PIPE_BUF=512): VERIFIED 2026-05-15** at `tests/test_graph_emit_atomicity.py::test_concurrent_multiprocess_append_preserves_record_integrity`. **This was the harder of the two platforms** — PIPE_BUF=512 with payloads intentionally biased to 600 / 1500 / 3000 bytes (all above PIPE_BUF), 8 workers × 100 records (= 800 concurrent appends to a single shared file). Five invariants held cleanly: no torn writes, no merged lines, no truncated JSON, no dropped records (exact count preservation), content integrity intact (per-worker filler-character invariant — strongest interleave detector — held across all 800 records).
- **Linux flame-01 (PIPE_BUF=4096): VERIFIED 2026-05-15** under the `forge` conda env at `/home/cnoellert/Documents/GitHub/forge-bridge`. Test runs identically (same fixture, same payload distribution, same invariant checks); 1 passed in 2.81s. Expected outcome held — Linux PIPE_BUF=4096 substantially exceeds typical record sizes — but verification was the acceptance gate, not the expectation.

**Operational interpretation:** the append-only JSONL substrate has now been operationally verified under concurrent multiprocessing pressure on **both production platforms**, rather than relying on inherited POSIX assumptions. The convergence-anticipated failure mode on macOS — torn writes when records exceed PIPE_BUF — did *not* manifest under aggressive multi-process pressure on portofino, and the easier Linux case passed cleanly on flame-01. Mitigation option (b) (fcntl advisory locking) stays a **deferred-not-rejected** seed at v1.6.0 — not forgotten, not rejected, deferred because the operational trigger condition never manifested; re-evaluable if (1) the test ever fails on either platform under future stress, or (2) cross-surface emission lands and produces a multi-graph-writer pattern not exercised by this test. **This is operationally-verified truth across both production platforms, not inherited POSIX folklore.**

## §3 — Surface Integration Cluster (Q9–Q11)

### Q9 — Chat endpoint response shape

**Position:** Chat response carries `{graph_id, root_node_id}` (references, not full record body). Full node records are retrievable via a new read endpoint `GET /api/v1/graph/<graph_id>`. Emission is server-side.

**Why:** Reference-only keeps response size bounded — existing chat response shape stays mostly intact. Full-record-in-response would inflate response size for chat clients that only need the assistant's final text. Server-side-only emission means UI must reconstruct graph state from chat tokens, which is fragile.

**Counter:** Requires the `GET /api/v1/graph/<graph_id>` endpoint to land at v1.6.0 — ~50 LOC addition. Adds surface but is the right architectural shape for the inspectability pattern Phase 22's Recipe 6 established.

### Q10 — CLI exit codes for graph operations

**Position:** Extend the existing locked set (`0` ok / `1` fail / `2` unreachable / `3` timeout) with:
- **`4` graph_not_found** — graph_id referenced doesn't exist on disk
- **`5` emission_failed** — substrate write failed (disk full, permission, lock contention)

Reserve `6`–`9` for future graph-specific failure modes.

**Why:** Folding new failure modes into existing codes loses signal. Extending preserves the contract and adds discriminating signal for shell scripts.

**Counter:** Each new code is a new contract. Cost: documented in CLI `--help`; tested in `tests/console/test_cli_doctor.py` (or parallel test file).

### Q11 — MCP tool emission point

**Position:** **Server-side wrapping at the registry layer.** Each MCP tool dispatch goes through a registry-wrapping layer that emits the node record, mirroring the sanitization pattern already established in Phase 7 + FB-C's `_sanitize_patterns.py` consolidation.

**Why:** Server-side enforces the emission contract uniformly across all registered tools (canonical and synthesized) and removes per-tool emission boilerplate. Each MCP tool today is already wrapped by the registry layer for sanitization; emission is the same architectural pattern. Client-side per-tool emission would duplicate the wiring 60+ times (current tool count) and miss synthesized tools loaded via the watcher.

**Counter:** Server-side wrapping requires every tool dispatch to go through the wrapper, including synthesized tools loaded post-startup. Acceptance criterion at the registry layer; verified by Phase 24 test pinning that a watcher-loaded synthesized tool emits records.

## §4 — Subgraph + Promotion Cluster (Q12–Q15)

### Q12 — Promotion N

**Position:** **N=5 at v1.6.0**, matching today's threshold for vocabulary continuity. Flagged for empirical re-evaluation in v1.6.1 based on promotion-candidate-count telemetry.

**Why:** Holding N constant while substrate changes (records → graph) disentangles attribution if promotion behavior shifts post-v1.6.0. Changing both substrate and threshold simultaneously makes any observed delta ambiguous.

**Counter:** If structured records produce substantively more (or fewer) false positives than text-similarity records, N=5 may need adjustment quickly. Mitigation: telemetry on promotion candidate count post-v1.6.0; adjustment lands in v1.6.1 if needed.

### Q13 — Same-shape definition

**Position:** **Strict structural match modulo named-parameter renames.** Full canonicalization (port reordering equivalence, control-flow restructuring tolerance) is v1.7 algorithm scope per §11.2.

**Why:** v1.6's promotion logic remains the heuristic threshold per §6.2's boundary discipline. The cleaner inputs (structured records) get cleaner matches without changing the matcher itself. Parameter-rename tolerance — lowercase + strip + whitespace-normalize — significantly improves heuristic signal without committing to canonicalization. Full canonicalization is the v1.7 algorithm's job per §6.2.1.

**Counter:** "Strict modulo rename" still produces false negatives when same-shape subgraphs differ in argument-type wrapping (e.g., `shot_id: str` vs `shot: Shot`). Acceptable v1.6.0 floor; v1.7 fixes.

### Q14 — Subgraph name authorship

**Position:** **LLM-proposed + operator-rename window + auto-generated fallback.**
- LLM proposes a name during synthesis.
- Operator has a window (default 24h, configurable via env var) to rename before the name freezes in the registry.
- If LLM proposal is rejected (validation failure, naming collision) and operator doesn't rename, fallback is `subgraph_<8-hex-chars>` derived from the subgraph's structural hash.

**Why:** LLM names are typically reasonable but not always; auto-generated fallback keeps the registry usable when the LLM produces noise. The rename window gives operator authority without blocking promotion.

**Counter:** 24h is arbitrary. Configurable resolves the arbitrariness without ceremony.

### Q15 — Macro migration timing

**Position:** **Vocabulary at v1.6.0; code at v1.6.x; file format permanent** per §13.4.

**Why:** Vocabulary alignment ("macros are named saved subgraphs") clarifies the model immediately and lands in `docs/` + CLAUDE.md ground-truth. Code migration is mechanical but touches `forge_bridge/console/_macros.py` (135 LOC) and the `~/.forge_macros.json` reader/writer; splitting that work into v1.6.x cleanly separates clarification from implementation.

**Counter:** Vocabulary-without-code can be confusing in documentation. Mitigation: docs explicitly say "vocabulary update at v1.6.0; code converges at v1.6.x; file format readable indefinitely."

## §6 — Scope-Adjacent Cluster (Q18–Q20)

### Q18 — `fbridge doctor` graph-store check

**Position:** **Yes, ship in Phase 24** alongside the `flame-exec` CLI command. **The graph_store doctor row is the first operator-visible surface of the graph-native runtime.**

**Why:** Phase 23 established diagnostics as an architectural teaching surface, not merely operational garnish — degraded-tolerant rows that teach the substrate model (Commit A `6446356` + Commit B `7786c83`). At this point in the milestone arc, graph emission, normalization boundaries, provenance, dispatch substrate, and `transport_error` traces are **core runtime concepts**, not optional observability extras. A `graph_store` row teaches "graph emission is the substrate v1.6 is built on; here's whether it's healthy" — exactly the kind of architectural surface the doctor should render once the substrate ships. Cheap (~50 LOC mirroring Phase 23's postgres-probe pattern). Classifies degraded-tolerant per operational-survivability invariant: a missing graph store doesn't make the runtime non-usable, just non-introspectable.

This is the doctrinal continuation of Phase 23's doctor-as-architectural-teaching pattern into v1.6 — the same posture that gave Phase 23 its substrate/consumer 4-state truth table now teaches the runtime's lifecycle-event orientation through a single new row.

**Counter:** None. The strengthening here is doctrinal — doctor-as-architectural-teaching continues into v1.6 unchanged from Phase 23.

### Q19 — Console execs view rendering

**Position:** **Group-by-graph when `graph_id` is present; fall back to flat for legacy execs without `graph_id`.** Surface the legacy section under an explicit header ("Pre-v1.6 executions") to make the mixed-mode display intentional rather than visually inconsistent.

**Why:** v1.6 records carry `graph_id`; legacy records don't. Conditional grouping handles both cleanly. Always-group breaks legacy display. Show-only-multi-node is confusing during the migration window when most execs have 1 node each.

**Counter:** Hybrid view (some grouped, some flat) is visually inconsistent. Mitigation: explicit legacy-section header gives the inconsistency a clear semantic.

### Q20 — `fbridge graph` debug subcommand sketch

**Position:** **Ship minimal read-only at v1.6.0:** `fbridge graph list` (recent graphs by mtime) + `fbridge graph show <id>` (pretty-printed node list). Document as "debug" surface with stability-warning prose in `--help`. v2.0's user-facing graph CLI is additive on top.

**Why:** ~50 LOC, read-only, significantly aids v1.6 development — operator + developer can inspect graphs without writing one-off scripts. The "user-facing graph CLI" deferral to v2.0 is about *mutation* commands (create, delete, rename) — read-only inspection is debugging scaffolding that v1.6 development already needs.

**Counter:** A v1.6.0 CLI surface that v2.0 expands is back-compat debt. Mitigation: stability-warning in `--help` ("debug subcommand, contract may change at v2.0"). v2.0 expands the subcommand tree without renaming the existing `list` / `show` verbs.

## §7 — Path C Promotion: Domain-Boundary Discipline as Forward-Governance Norm

The THREE-DOMAIN-DECOMPOSITION's "What this artifact opens" §C names this work as **immediate-applicable** process/discipline. This section promotes it from open path to governance norm. Paths A / B / D remain v1.6+ architectural scope.

### §7.1 — The three domains, restated for governance use

| Domain | Owns | Measured by |
|---|---|---|
| Runtime economics | prompt size, KV-cache locality, generation latency, tool-prefix stability, keep_alive | warm-call latency, prefix-stability rate, model-unload incidence |
| Protocol serialization | provider tool_calls field, prose-embedded JSON, Bug-D salvage | parse-success rate, salvage-fire rate, content-shape distribution |
| Dispatch substrate | `flame_execute_python` invocation, graph-event emission, transport-error visibility | graph-emission integrity, dispatch-error rate independent of protocol-layer state |

### §7.2 — Governance norms

1. **Each layer has its own success metric.** Optimizations are evaluated against the layer they target, not cross-layer. A change in salvage strategy is evaluated against parse-success metrics, not warm-call latency. A change in cache TTL is evaluated against warm-call latency, not graph-emission count.
2. **Cross-layer effects are surfaced explicitly when they occur.** If a salvage change measurably affects dispatch (e.g., changes the rate at which `graph_id` is captured), that effect is documented in the commit body + investigated as a discrete cross-layer interaction — not absorbed silently under a single "the runtime is more reliable now" narrative.
3. **Investigative phases honor the decomposition.** A phase that touches one layer doesn't simultaneously refactor another. If entanglement is unavoidable, the phase is renamed and re-scoped to make the cross-layer scope explicit.
4. **Carry the anti-scope binding forward.** From THREE-DOMAIN-DECOMPOSITION §"Anti-scope binding," strengthened by this convergence: at v1.6, **no more cache tuning** (runtime economics substantially resolved), **no more instrumentation** (Pass 1 sufficient), **no more narrowing experiments** (narrowing instability can cost more than it saves), **no more prompt surgery** (prompt churn is no longer existential). Those were correct investigative steps. They answered their questions and are closed.

### §7.3 — When the binding loosens

The four "do NOT" items above are bound at v1.6 phase scope. They re-open under specific evidence-trigger conditions:

- **Cache tuning re-opens** if warm-call latency regresses ≥ 50% across a stable measurement set without an attributable code-path change.
- **Instrumentation re-opens** if a new failure shape surfaces that Pass 1 cannot diagnose (e.g., dispatch-layer concurrency anomaly invisible to current logs).
- **Narrowing re-opens** if prompt-eval cost dominates warm-call latency at production tool counts (current measurement: 58 tools at 2026-05-14 evening; budget breach would need to be empirically demonstrated, not assumed).
- **Prompt surgery re-opens** if a model upgrade (e.g., default model bump per `SEED-DEFAULT-MODEL-PERFORMANCE-V1.6+`) makes prompt-shape behavior measurably different from qwen2.5-coder's observed shape.

These triggers are **intentionally unbound pending evidence**, not implicitly rejected. Future-Gate-X evaluation per `feedback_deferral_first_class_governance`.
