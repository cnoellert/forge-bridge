# Phase 25.0 — The SOP Graph Layer

## Architectural Thesis

25.x is not about making the model smarter.
It is about making execution structure explicit.

Reliability comes from making execution structure explicit,
not from making the model smarter.

When the system fails, ask what structure is missing
before asking what intelligence is missing.

This principle retroactively explains every phase from 24.2 through 24.11.
Each intervention moved responsibility out of probabilistic cognition
and into deterministic structure. The model's job gets narrower and
more tractable at each step. The system gets more reliable not because
the model improved but because there's less for the model to get wrong.

25.0 is the first phase where the operator surface genuinely
expands instead of merely becoming more reliable.

That is a real transition point.

The phase map that frames this moment:

```text
24.x                -> prove the substrate works
bounded-defect arc  -> prove the substrate stays clean
format_result PoC   -> prove the substrate composes
PR25 strengthening  -> prove execution paths converge
25.0                -> prove the substrate enriches into
                       a SOP graph
```

## What The 24.x Arc Established

The substrate is sound. Retrieval works. Mutation works. Format works.
The execution graph exists in embryo:

- chain syntax is wire
- tools are nodes
- resolver binds entity ports deterministically
- flame_execute_python is the VEX escape hatch

The natural evolution is from:

```text
text -> single tool call
```

to:

```text
text -> execution graph
```

The nodal structure is already present in how operators describe their
work. Verbs are operations. Nouns are entities. Modifiers are port
values. Sequencing language is wiring. The system's job is to make
that structure explicit and executable.

## The Houdini Boundary

The SOP graph is deep enough.

flame_execute_python is VEX — the escape hatch when the node network
can't express the operation. The verb catalogue is the SOP graph —
named operations that map onto how operators think about their work.

The design stops at the level where an operator would naturally name
the operation. It does not decompose into primitives the operator
would not use in a sentence.

## Aggressive First Step

The insight: every mutation verb already contains the logic to
determine what it would change. It just executes rather than
returning a diff. Surfacing that diff is a small delta from the
existing implementation.

The proposed primitive: preview as a standard output mode on every
mutation verb. Same input ports. Same scope resolution. Same logic.
Stops before writing. Returns a structured proposed-change list.

```text
preview rename shots on 30sec 21 with prefix genesis
-> [{seg_idx, current, proposed}, ...]
```

Why this is the right aggressive first step:

1. Delivers immediate operator value — see what's going to change
   before it changes. Reviewable, forwardable, supervisor-approvable.
2. Solves the scope ambiguity problem partially — preview output
   exposes which version and track the system is targeting. Wrong
   scope becomes visible before execution, not after.
3. Chains into format immediately:

   ```text
   preview rename on 30sec 21 prefix genesis -> format as table
   ```

   Produces a change manifest the operator can review or forward.
   Real production utility today.
4. Does not require select, filter, foreach, or collect first.
   Works with existing tools. Buys time to design those primitives
   properly.
5. Encodes a design constraint that applies to all future mutation
   verbs: every mutation verb must have a corresponding preview
   with identical input ports.

The mutation gate pattern this establishes:

```text
preview [mutation] -> operator reviews -> [mutation]
```

Making the implicit explicit, but honestly bounded:

25.0 establishes observability before mutation, not enforced
approval semantics.

The enforced invariant form — mutation execution requires a
matching approved preview — is a stateful, approval-aware,
replay-aware, manifest-aware system. That is outside current
scope and must not be implied by the framing language.

The convention 25.0 establishes is:

```text
preview before mutation is the recommended operator workflow
```

NOT:

```text
preview before mutation is a substrate invariant
```

False architectural claims are poison later. The honest version is
still powerful.

## Remaining Primitives

select — explicit scope binding. Required before mutation verbs with
ambiguous scope (version, track, segment range). The operator's
equivalent of clicking the right track in Flame's UI before applying
an operation.

filter — subset a collection. Required for "only the comp segments",
"only sequences with raw plates", "only shots on version 1". Without
filter, collection queries can't be scoped.

foreach + collect — iteration over enumerated collections with merge
output. The Houdini foreach network in collect mode. Required for
"segments on all sequences", "rename all sequences with prefix X".
collect is the merge node that gathers iteration outputs.

compare — state comparison. Required for "are we using the most
current version of graphics", "which shots have changed since
yesterday". Currently only answerable through flame_execute_python.

## What 25.x Is Not

- Not a prompt engineering phase
- Not a model swap phase
- Not an orchestration topology phase
- Not VEX — flame_execute_python already covers that

25.x is the SOP graph layer. Named operations with typed ports.
Explicit wiring. Preview before mutation. The operator's natural
language maps directly onto the graph without translation.

## Preview / Dry_Run Language Boundary

preview = operator language (what the operator says)
dry_run = execution modifier (what the resolver sets)

These are not the same abstraction and must not collapse into one.

The resolver bridges them:

```text
operator intent -> execution modifier
```

Natural language stays human-shaped.
Execution stays machine-shaped.

This is "move ambiguity upward, move determinism downward" applied
to the preview primitive specifically.

The resolver currently handles:

1. entities — sequence_name, reel_name
2. numeric directives — increment, padding, start
3. format directives — email/table/bullets
4. intent modifiers — preview/dry_run, new in 25.0

At some point `resolve_query_entities()` becomes a deterministic
semantic projection layer, not merely entity extraction. That
broadening is named here as trajectory, not as 25.0 scope. Resolver
architecture cleanup is explicitly deferred.

## Anti-Scope

- Do not add preview dialogs or confirmation UI — preview is a
  structured data output, not a UI pattern.
- Do not make preview tool-specific beyond the first pass — the goal
  is a universal primitive, not per-tool variants.
- Do not implement select, filter, foreach, or collect in this commit
  arc — those are named next primitives, not 25.0 scope.
- Do not modify flame_execute_python — it is the escape hatch, not a
  target for this arc.
- Do not change the chain syntax — preview slots into the existing
  chain as a standard node.
- dry_run eligible tools are structural diff mutations:
  flame_rename_shots, flame_set_start_frames, flame_assign_roles,
  flame_set_segment_attribute. These produce deterministic state-delta
  representations. The diff is computable without executing the
  operation.
- dry_run is not eligible for side-effect mutations:
  flame_publish_sequence, flame_reconform_sequence,
  flame_replace_segment_media, flame_render_batch. These produce
  operational consequences that cannot be simulated as a diff without
  executing the operation. Extending dry_run to these tools is
  explicitly out of scope and would turn preview into a speculative
  execution engine.

## §11 Deferral Register

Item: Diff-shaped payload format rendering

Trigger: After first dry_run diff is fed into format as table/email/bullets,
observe whether output reads as a change manifest or a data dump. If it reads
as a data dump, a format_diff verb class or format-aware rendering hints are
needed. Proposed-change manifests are a cognitively different artifact class
from production summaries: before/after emphasis, risk visibility, scope
clarity, approval-oriented language. Defer until empirical evidence from the
25.0 forcing function.

Item: Resolver as deterministic semantic projection layer

Trigger: When a third intent modifier class beyond preview/dry_run is added.
At that point `resolve_query_entities()` warrants architectural treatment as a
projection layer, not just a collection of regex helpers. Individual resolver
functions become exceptional adapters, not the default mechanism. Defer until
the pattern recurs a third time.

Item 16: if/then conditional primitive

Trigger: second operator query requiring conditional branching to express
correctly. Currently only answerable through flame_execute_python.
Priority: higher than foreach — conditionals are required before iteration
is fully expressive.
Note: consumes compare output naturally.
compare -> if/then is the canonical pattern for state-dependent mutation
workflows.

Item 17: state-aware default resolution as a named pattern

Trigger: second parameter on any tool where current operational state should
influence parameter default selection at execution time. padding width is the
first instance. When second instance occurs, promote as a named substrate
primitive with explicit contract.

Current instance: padding width inferred from existing shot names in
flame_rename_shots. Regex `_(\d+)(?:_|$)` extracts numeric width; unanimous
agreement overrides `params.padding`; mixed or absent state falls back to
Field default.

Immediate sibling candidates (not yet addressed):

- flame_set_start_frames.default_frame: Field(1001) — if existing shots have
  a consistent start frame, silent override may apply.
- flame_rename_shots.increment + .start — both could be inferred from
  existing numbering grammar if present.

Wording that must survive promotion: "preserve coherent existing structure
unless operator explicitly overrides. State-aware deterministic defaulting,
not inference."

Item 18: Implicit positional heuristic replacement

Added 2026-05-20 from the `5675e59` ecological probe on `30sec_21`.
Distinct category from items 13 + 16 — those name MISSING primitives
that future phases ADD; this names TRANSITIONAL COMPENSATION that future
phases REMOVE.

Current shape:
    Heuristic interpretation compensates for missing filter/conditional
    graph primitives. Mutation tools (e.g., `flame_rename_shots`) use
    positional convention (`tracks[0] = background = shots`) as an
    implicit scope-binding heuristic. The 2026-05-20 probe surfaced that
    operators naturally express scope as predicate filtering ("NOT on
    v0.0", "NOT single-frame graphic", "NOT gap-effect"), not as
    positional reference. The heuristic is brittle by definition — it
    requires operator-side sequence prep (in the present case, manually
    removing the reference track) to align Flame state with the
    heuristic's assumptions.

Replacement trigger:
    Once filter + if/then primitives (items 13 + 16) can express the
    same operator intent structurally.

Deferred target:
    Remove heuristic interpretation in favor of explicit graph
    predicates and execution gates. Heuristic remains as a sensible
    default for the clean-sequence case; filter expression becomes the
    surface for everything else.

Architectural note:
    This is transitional ambiguity compensation, not a permanent
    execution primitive.

Why explicitly named (room discipline):
    Transitional structure — intentionally temporary, behaviorally
    load-bearing, and likely to survive accidentally — tends to become
    permanent convenience behavior unless named as debt with a removal
    condition. Prior sibling instances in this project: legacy preview
    tools (now delegated through `dry_run`), resolver-specific adapters
    (item 7), PR21 priority pairs (item 4), lexical-specificity shims
    (item 6). Item 18 is the first 25.0-era explicit instance of this
    discipline applied to a substrate-level positional heuristic,
    structurally distinct from the missing-primitive class (13 + 16 ADD
    capability; 18 names compensation that will be REMOVED when 13 + 16
    mature).

Evolutionary arc preserved by explicit naming:

```text
24.x       explicit deterministic structure
25.x       explicit graph structure
item 18    temporary ambiguity bridge until graph structure
           fully subsumes it
```

The four-section carrier structure above (Current shape / Replacement
trigger / Deferred target / Architectural note) is precedent for future
"transitional prosthetic" entries. When the next transitional structure
needs explicit naming, mirror this shape so the category remains legible
to future contributors without re-deriving the framing.

## Forcing Function

Primary probe:

```bash
fbridge chat "preview rename shots on 30sec 21 with prefix genesis -> format as table"
```

Pass criterion: structured diff output rendered as a human-readable
table showing current -> proposed shot names. Producer-grade. No Flame
state changed.

Secondary probe:

```bash
fbridge chat "preview rename shots on 30sec 21 with prefix genesis"
fbridge chat "rename shots on 30sec 21 with prefix genesis"
```

Pass criterion: preview shows the diff, rename executes correctly
against the same scope the preview targeted.
