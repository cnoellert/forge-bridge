## §2 Failure-shape stability — the load-bearing signal

The strongest evidence for H0 is not the binary "did the call succeed?" answer. It's
the QUALITATIVE SHAPE of the failure.

| Pre-24.7 failure shape | Post-24.7 failure shape | Delta |
|---|---|---|
| Empty `{}` params payload | Empty `{}` params payload | **identical** |
| 0.0s elapsed validation failure | 0.0s elapsed validation failure | **identical** |
| No partial argument binding | No partial argument binding | **identical** |
| No malformed-but-attempted synthesis | No malformed-but-attempted synthesis | **identical** |
| Daemon rejects at FastMCP schema layer | Daemon rejects at FastMCP schema layer | **identical** |

The intervention produced **zero observable behavioral delta** at the cold-node
invocation layer. Not a partial improvement that didn't cross the threshold — no
improvement at all.

**Why this matters epistemically:** a partial-success-that-doesn't-quite-work shows up
as gradations (some warm probes succeed; arg shape closer but not perfect; first call
fails but repair loop succeeds; etc.). The 24.7 result shows none of these gradations.
The model appears not to be incorporating the exemplar into its argument-synthesis
pathway at all — the exemplar is in the docstring the daemon serves, but the model is
not behaving as if it has read it for the purpose of constructing the wrapped `params`
structure.

This is qualitatively different from 24.6's result. 24.6 produced a strong distributional
shift (5/5 warm probes shifted iter-1 selection; topology collapse — no iter-3 relapse).
24.7 produced **no distributional shift at any rate**. The interventions operate on
entirely different model-behavior dimensions, and the description layer reaches only
one of them.

## §4 The architectural-layer map (operator-articulated)

The 24.6 + 24.7 result set, taken together, sharpens the project's understanding of
which control surfaces actually shape which model behaviors:

| Behavioral surface | Description shaping effectiveness | Established by |
|---|---|---|
| Tool selection / affordance routing | **Effective** | 24.6 |
| Argument serialization / wrapper synthesis (cold nodes) | **Ineffective** | 24.7 H0 |
| Repair-loop state incorporation | **Ineffective** | `forge_list_shots` probe (within 24.7 dogfood scope) |
| Cold-node invocation bootstrapping | **Likely requires structural context outside the tool surface** | 24.7 H0 + cross-tool generalization |

This map is the phase's primary architectural deliverable. It does not say "descriptions
don't matter" (24.6 demonstrated the opposite, and PROMOTED the
`[[feedback-rhetorical-position-as-architectural-control-surface]]` memory on that
evidence). It says: **the description layer has a multi-register structural surface
where some registers control some behaviors and other registers do not reach other
behaviors.**

Description shaping appears to control where the model's PRIOR PROBABILITY MASS sits —
which is exactly where affordance selection lives. Description shaping does NOT appear
to provide the structural-context concreteness the model needs to synthesize a wrapped
`params` payload for a tool it has never successfully invoked before. The model needs
something OPERATING ABOVE THE LOCAL TOOL DESCRIPTION LAYER to bootstrap cold-node
invocation.

Candidate escalation surfaces (named for the next phase's framing space, not for 24.7
execution):

- Orchestrator-level structural bootstrapping (system message tuning, prompt-prefix
  injection)
- Dynamic repair injection (orchestrator catches validation failure and reformulates
  the call with the correct shape)
- Explicit tool-call scaffolding (orchestrator constructs a partial tool-call template
  and asks the model to fill it)
- Synthetic successful-call priming (orchestrator pre-seeds the conversation with one
  successful invocation per cold tool so the model has session-history pattern to draw
  from)
- Another mechanism operating ABOVE the local tool description layer entirely

24.7 deliberately did NOT explore any of these — that would have violated FRAMING §3
anti-scope. The architectural escalation is justified BY EVIDENCE, not by the
temptation to mix-fix during execution. Per
`[[feedback-anti-scope-discipline-under-pressure]]`, the §3 guard held throughout
24.7's execution; partial-failure pressure didn't pull the intervention into
orchestrator territory mid-phase. That discipline preserves the architectural finding's
integrity.

Without the substantive 24.3–24.6 work — substrate truthfulness, observability,
provenance, termination semantics, architectural legibility — today's H0 disposition
would not be possible to make confidently. The same `{}` empty-call failure shape

The 24.x arc's apparent over-investment in substrate before behavioral work IS the
reason 24.7 can produce a sharp H0 with cross-tool consistency. Per
`[[feedback-provenance-precedes-behavioral-interpretation]]` (promoted at PR #10): the

the operator confirmed daemon restart before probing, and `install_provenance` from
PR #10 was the surface where that confirmation was made.

This validates the earlier arc's ordering. **Topology / observability / provenance /
termination first; behavioral fixes after.** Without the foundation, the behavioral
findings would be unanchored.

The recursive methodology archaeology: 24.6's empirical demonstration of rhetorical-
position-as-control-surface was the framing antecedent for 24.7's discriminator design
(the project knew description-layer shaping COULD work for some surfaces, so the
question for 24.7 was "does it work for THIS surface" — a sharp falsifiable question,
not a fuzzy "let's try harder").

## §6 What this rerun does NOT prove (anti-overclaim)

Per `[[feedback-operational-maturity-not-completeness]]` and the 24.6 §6 epistemic-
honesty pattern:

- **Does NOT prove that ALL description-layer interventions are useless for cold-node
  invocation.** A more elaborate intervention (e.g., multi-exemplar / per-failure-mode
  exemplar / runtime-templated exemplar) was not tested. 24.7 tested the
  single-3-part-exemplar template across 4 tools; it tested ONE point in the
  description-layer intervention space.
- **Does NOT prove that the 2 modified-but-unprobed tools (`flame_rename_shots`,
  `flame_set_start_frames`) also fail.** Per §3 honest scope disclosure, those tools
  remain unprobed; their disposition is explicitly unbound.
- **Does NOT prove the model is unable to ever invoke these cold tools.** Operator can
  manually scaffold the call (e.g., by quoting the wire shape directly in chat); the
  H0 disposition is about UNPROMPTED, NATURAL-LANGUAGE-DRIVEN invocation behavior.
- **Does NOT prove the candidate escalation surfaces in §4 will work.** Those are
  framing candidates for a future phase; 24.7 doesn't endorse any specific one.
- **Does NOT close §11 item (1) further** (24.6's affordance-selection partial closure
  remains PARTIAL CLOSURE; 24.7 doesn't touch that disposition).
- **Does NOT close any of the 10 other §11 items inherited from 24.6.** 24.7 was
  scoped narrowly to test the description-layer-effectiveness boundary at the
  invocation surface for cold nodes; the §11 register's other items remain in their
  pre-24.7 dispositions.

The phase result is honestly stated as: **the description-layer-as-fix-surface for
cold-node invocation behavior has been falsified at one specific intervention shape
(single 3-part exemplar per tool) across 4 tools.** Stronger claims about the
description layer's complete exhaustion require additional intervention shapes to be
tested. Weaker claims (e.g., "exemplars sometimes work") would deny the cross-tool
consistency of the failure shape.
