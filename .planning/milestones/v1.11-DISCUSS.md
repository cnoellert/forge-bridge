## The enabling finding — and the plan-grounding catch that re-shaped the mechanism

**The namespace marker exists** (`_tool_filter.py:264`, `_NAMESPACE_PREFIXES = {flame,
forge, format, synth}`) — so substrate is a first-class token, no ontology needed. But
running the *actual* `filter_tools_by_message` + `deterministic_narrow` (SR.1-T1 part B,
`UAT/SR.1-T1-substrate-confirm.md`) **overturned cycle-1's "bias in the candidate set"
mechanism:**

- compiled step `forge_get_shot` → narrow keeps the forge_get_shot family; **`flame_get_sequence_segments`
  is dropped** by max-overlap.
- user intent "duration of shot 10 on 30sec_edit 21" → **`flame_get_sequence_segments`
  isn't even in the filter** — the user's words ("shot","duration") share name-tokens
  with forge *entity* tools; the timeline tool shares none.
- only "get segments on 30sec_edit 21" (the tool's own vocabulary) narrows to it.

**The substrate mismatch is lexical.** The right-substrate tool is **not a candidate to
"prefer"** — a within-set bias can't reach it. The usable signal is the **sequence
reference**: both failing phrasings carry "on 30sec_edit 21", but the user's attribute
vocabulary lexically out-votes it. So the mechanism is **sequence-reference-signal
routing** — *a sequence reference present in the read → route to the timeline substrate
(inject/force `flame_get_sequence_segments`), overriding the forge-lexical match.* Coarse
(keyed on sequence-ref presence), deterministic, no map — but **compile-adjacent** (it
overrides the lexically/compiler-chosen tool), which re-opens the mechanism-location
question with real data.

## The boundary to watch (sharper after the catch)

The grounded mechanism (sequence-ref override) **brushes the compile boundary** — it
overrides the tool the compiler/lexical-match chose. Discipline to protect: **SR
overrides only the *substrate* (forge-entity → flame-timeline) on the deterministic
sequence-ref signal; it does NOT repair malformed chains, reinterpret the request, or
resolve segment selection.** It must be a *single deterministic swap* keyed on
"sequence reference present + selected tool is forge-entity," never a general
chain-rewrite. If no sequence reference is present, SR does nothing — **fail safe to
current behavior.** This is precisely why option (b) needs the room: a step-rewrite is
one principled swap away from looking like compile-repair, and the line must be drawn
explicitly.
