## §0 Banner — What Is Different About This Phase

24.7 closed with H0: description-layer shaping at the richest per-tool register
tested in this project (full 3-part invocation exemplars on four cold nodes)
did not bootstrap reliable cold-node argument synthesis. Four target tools
failed identically — empty `{}` params, 0.0s validation error from the daemon,
no partial binding. The failure shape was stable across the set, which classifies
the pathology as structural rather than per-tool: the model lacks a serialization
grammar for the `params` wrapper shape, and the per-tool affordance surface
cannot teach it.

The 4-tier behavioral-surface map captured at the 24.7 cursor
(`[[project-state-2026-05-18-phase-24-7-h0-merged]]`) names this register
boundary: tool selection / affordance routing is shaped by per-tool description
(established 24.6); argument serialization for cold nodes is NOT (established
24.7 H0). The map locates the next intervention one register above the per-tool
description surface — at the orchestrator-level system prompt.

24.8 is the first 24.x phase that operates ABOVE the per-tool affordance
surface. Every prior 24.x phase intervened on substrate, transport, termination
policy, consumer projection, or per-tool description content. 24.8 intervenes
on the system prompt injected into every local-router call. The intervention
is a tool-invocation grammar — declarative, not synthetic — that names the
wrapper shape, the natural-language inference convention, and the repair-loop
pattern, with three concrete examples drawn from the 24.7 failure set.

This is the **minimum-perturbation choice** from the §11 item (11) design
space named at 24.7's H0 disposition: "system-message tuning, orchestrator
few-shot prompting, runtime adaptation, dynamic repair injection, explicit
tool-call scaffolding, synthetic successful-call priming, or another mechanism
operating ABOVE the local tool description layer entirely." System-message
grammar is the thinnest of these — a single declarative addition to one
already-existing prompt, no orchestrator logic change, no runtime adaptation,
no synthesis, no coercion. Per
`[[feedback-orchestrator-control-flow-not-meaning]]`: the orchestrator's
control flow is untouched; the addition is on the declarative grammar surface,
which is a distinct register from the model→tool meaning boundary.

Both H1 (grammar lifts cold-node invocation) and H0 (grammar also insufficient)
are architecturally valuable. H1 names a reachable behavioral dimension at the
orchestrator-grammar register and validates the 4-tier map's prediction that
the description SURFACE has registers above the per-tool level that reach
argument synthesis. H0 narrows the §11 item (11) design space by falsifying
its thinnest intervention and justifies the next step (synthetic successful-call
priming, per operator brief) by evidence rather than by frustration.

## §1 Diagnosis

**Pathology inherited from 24.7 H0 (cross-tool stable):**

Four target tools (`flame_inspect_sequence_versions`, `flame_rename_shots`,
`flame_get_sequence_segments`, `flame_set_start_frames`) all failed identically
under live operator probe after the 24.7 richest-per-tool intervention:

- Tool selection: correct (right tool picked in each case)
- Tool fire: failed at schema validation
- Params: empty `{}`
- Elapsed: ~0.0s (model emits the call without inferring any arguments)
- Repair loop: did not synthesize a corrected second call
- Failure-shape stability: 5-of-5 identical signature (per
  `[[feedback-failure-shape-stability-as-disposition-evidence]]`), qualitatively
  stronger evidence than success-count alone that the per-tool description-layer
  intervention does not enter the model's argument-synthesis pathway

**Working hypothesis (24.8):**

The argument-synthesis pathway is reachable from the orchestrator-level system
prompt but not from per-tool description content. A declarative grammar at the
system-prompt register that names:

1. The `params` wrapper shape explicitly (always wrap; never flat; never empty)
2. The natural-language inference convention (spaces ≡ underscores; infer from query)
3. The repair-loop pattern (extract required-field name from error; include next call)

plus three concrete invocation examples covering the failure cases (sequence-name
inference, multi-arg binding, state-carry across iterations) is sufficient to
lift the four target tools' invocations from `{}` collapse to populated-params
firing.

**What this hypothesis predicts:**

- Empty-call collapse eliminated on all four probes (`{}` → populated `params`)
- `sequence_name` binding correctly inferred from natural-language operator
  query ("30sec 21" → "30sec_21")
- Multi-arg binding correctly populated when the operator query names multiple
  fields ("prefix gen2, increment 10, 4-digit padding" → all three bound)
- Tool fires successfully at the daemon's schema validator on all probes

**What this hypothesis does NOT predict (anti-overclaim per
`[[feedback-operational-maturity-not-completeness]]`):**

- That the underlying tool action (rename, inspect, etc.) succeeds end-to-end
  — Flame-side execution depends on live project state and is not the
  intervention surface
- That the grammar generalizes to tools beyond the four 24.7 target set
  without further probing
- That the model's repair-loop synthesis works perfectly on every error class
  — the example demonstrates the pattern for one common case (`project_id
  required`); other error classes remain untested

**Evidence base for the intervention point:**

- `_DEFAULT_SYSTEM_PROMPT` at `forge_bridge/llm/router.py:155-167` is currently
  domain-only: Flame version, shot naming convention, openclip frame notation.
  It contains no tool-invocation grammar.
- The `FORGE_SYSTEM_PROMPT` env override path is verified at
  `forge_bridge/llm/router.py:363-365`: `os.environ.get("FORGE_SYSTEM_PROMPT",
  _DEFAULT_SYSTEM_PROMPT)`. Operators who override get exactly what they set;
  the 24.8 grammar applies only when the default is in effect.
- Single intervention site: 1 file, 1 string constant. No other file is
  touched by commit 2.

## §3 Anti-Scope — Architecturally Rejected

**This is the load-bearing anti-scope guard for 24.8:**

> *"The grammar is a declarative addition to one already-existing system prompt.
> It is not orchestrator logic; it does not synthesize, wrap, coerce, retry, or
> branch on tool results. The orchestrator's control flow is byte-identical.
> The grammar lives on the model→tool meaning surface; the orchestrator stays
> in its control-flow lane."*

Per `[[feedback-orchestrator-control-flow-not-meaning]]`: when orchestration-
layer authority crosses model-decision boundaries, the orchestrator must stay
in the control-flow lane. The system-prompt grammar is NOT a control-flow
change — it is a declarative shaping of the model's input context. The
distinction is load-bearing: future work might add control-flow interventions
(retry hooks, structured repair injection); those belong in a separate phase
with a different anti-scope shape.

**Specifically rejected during 24.8 execution:**

- Adding orchestrator-side wrapping of empty-params results into synthetic
  retries
- Adding hidden repair-loop injection in the orchestrator (per operator brief:
  "preserves truthful conversational state")
- Adding any synthesis or summarization at the orchestrator layer
- Adding the grammar to multiple prompts (single location: `_DEFAULT_SYSTEM_PROMPT`)
- Sweeping similar grammar additions to tool docstrings ("if it works in the
  system prompt, sprinkle it in tool descriptions too") — per
  `[[feedback-anti-scope-discipline-under-pressure]]`, single-perturbation
  discipline requires the single-site scope; broader sweeps weaken causal
  attribution and reopen the 24.7 register
- Adding more than three examples — operator constraint of three; if one is
  insufficient, three is the discriminator boundary; adding four-plus would
  weaken the experiment's interpretability
- Changing existing lines of `_DEFAULT_SYSTEM_PROMPT` — append-only per
  operator brief
- Modifying the `FORGE_SYSTEM_PROMPT` env override path — operator brief
  explicitly preserves it

**Inherited-baseline clarification:**

- Per-tool docstrings carry their post-24.7 state (full 3-part exemplars on
  the four target tools; D-04 + PR #12 + PR #13 priors on
  `flame_inspect_sequence_versions`) — unchanged. The system-prompt grammar
  augments the tool descriptions rather than replacing them. Per
  `[[feedback-brief-examples-as-behavioral-reference-shapes]]`: preserve
  battle-tested topology; the 24.7 exemplars are operational archaeology.
- The current `_DEFAULT_SYSTEM_PROMPT` content (Flame version / shot naming /
  openclip frame notation) is preserved verbatim. The grammar section appends
  after the closing "Respond with concise, production-ready Python unless
  asked otherwise." line.

## §5 Forcing Function — Pass Criterion

**The forcing function reuses the four operator queries that failed in 24.7.**
This is single-perturbation discipline: the only difference between the 24.7
H0 result and the 24.8 probe set is the system-prompt grammar. The four queries:

1. `fbridge chat "Give me the versions on the sequence 30sec 21"` — target:
   `flame_inspect_sequence_versions`
2. `fbridge chat "Assign shots to 30sec 21 using prefix gen2, increment 10,
   4-digit padding"` — target: `flame_rename_shots`
3. `fbridge chat "Get the segments on the sequence 30sec 21"` — target:
   `flame_get_sequence_segments`
4. `fbridge chat "Preview the start frames for 30sec 21"` — target:
   `flame_set_start_frames` (preview path; live execution out-of-scope for
   the intervention surface)

**Multi-dimensional pass criterion** (per
`[[feedback-distinct-success-criteria-per-adjacent-layer]]` — each dimension
attaches to its native layer):

| Dimension | Required | Native layer |
|---|---|---|
| **Empty-call elimination** | No `{}` collapse on any probe | model→tool meaning |
| **Tool-fire success** | Daemon receives a structurally valid call (passes schema validation) | wire transport |
| **Argument binding** | `sequence_name` value derives from operator's natural-language query | grammar inference |
| **No hallucination** | Model invokes the correct tool rather than fabricating an inline answer | tool selection |
| **No wrong-tool routing** | Each probe fires the named target tool, not an adjacent affordance surface | tool selection |
| **Substrate integrity** | `result_hash` of `flame_execute_python` canonical probe (from 24.6) remains byte-stable `8ca14d1e` | substrate control |

**H1 disposition** — pass on all four probes across all six dimensions:
system-prompt grammar is sufficient to bootstrap cold-node invocation behavior
for the 24.7 failure set; description-SURFACE has reachable registers above
the per-tool level for argument synthesis; 4-tier behavioral-surface map gets
its predicted positive instance at the orchestrator-grammar register.

**H0 disposition** — `{}` collapse persists on any target tool: orchestrator-
level grammar at this minimum-perturbation shape (single declarative section,
three examples) is insufficient; §11 item (11a) closed with H0; next escalation
step within item (11) family is **synthetic successful-call priming** (per
operator brief — already named in §11 register as item (11b); intervention
shape: pre-seed the model's context with successful invocation transcripts
before each session begins).

**Partial-pass dispositions** (per
`[[feedback-explicitly-unbound-vs-implicitly-rejected]]`):

- Some tools pass, others fail → grammar shape works but per-tool variance
  remains; needs per-tool tuning beyond the 24.8 template; opens a
  decomposition question whether per-tool failure correlates with field count,
  field type, or natural-language ambiguity
- Empty-call eliminated but argument-binding wrong → grammar provides wrapper
  shape but not semantic anchor; partial intervention success at the
  serialization register, failure at the inference register

The disposition language at the rerun doc must explicitly distinguish full
pass / partial pass / full failure, per
`[[feedback-operational-maturity-not-completeness]]`.

**If H1 confirms (system-prompt grammar lifts cold-node invocation):**

content. The 4-tier behavioral-surface map gets its first positive instance at
the orchestrator-grammar register, joining the 24.6 selection-layer positive
instance:

| Behavioral surface | Description shaping | Established by |
|---|---|---|
| Tool selection / affordance routing | Effective (per-tool) | 24.6 |
| Argument serialization (cold nodes) — per-tool register | **Ineffective** | 24.7 H0 |
| Argument serialization (cold nodes) — orchestrator-grammar register | **Effective** (if H1) | 24.8 |
| Repair-loop state incorporation | Effective (if H1's state-carry example works) | 24.8 |

The combined finding strengthens the
`[[feedback-description-layer-multi-register-surface]]` memory: not only is
the description layer multi-register, the registers are LAYERED — per-tool
register controls selection but not argument-synthesis; orchestrator-grammar
register controls argument-synthesis where per-tool cannot. The candidate
insight "register-vs-dimension reach is not flat; outer registers reach
behaviors inner registers cannot" becomes eligible for promotion at H1
disposition.

**If H0 confirms (grammar also insufficient):**

The §11 item (11) ladder narrows by one rung. Synthetic successful-call
priming (item 11b) becomes the next intervention shape. The accumulated
falsification evidence (D-04 + PR #12 + PR #13 + 24.7 H0 + 24.8 H0) is the
load-bearing case for declaring the declarative-shaping path family exhausted
and opening control-flow-intervention work (items 11d, 11e, 11f) as the
remaining design space. Each H0 step strengthens the eventual successful
intervention's architectural claim because the design space gets falsified
methodically rather than abandoned to frustration.

The candidate insight "discriminator-probe pattern as generalized methodology
tool beyond §11 item (9)" (carried forward from PR #13 and 24.7) gets its
third within-project instance at 24.8 (regardless of disposition), strengthening
its eligibility for promotion. 24.8 is structurally a discriminator probe at
the orchestrator-grammar-register-effectiveness scope, the third generalized
application after PR #13 (description-layer-intervention-shape scope) and
24.7 (description-layer-reach scope).

**Partial-pass disposition language:**

A partial pass (e.g., warm grammar works but cold daemon does not load it; or
sequence-name inference works but multi-arg binding does not) is not a
"diminished H1" — it is a half-truth that must be classified explicitly per
`[[feedback-distinct-success-criteria-per-adjacent-layer]]` and
`[[feedback-operational-maturity-not-completeness]]`. The rerun doc must
attach each successful dimension to its native layer and each failed dimension
to its native layer; UX wins do not get laundered into convergence success;
control-flow wins do not get laundered into grammar success.

**Either outcome:** the 4-tier behavioral-surface map gets richer; layer
boundaries stay intact; the §11 register's deferred-not-rejected discipline
continues to pay dividends by binding each named intervention shape to its
own trigger condition rather than collapsing the design space into a single
"try everything" phase.
