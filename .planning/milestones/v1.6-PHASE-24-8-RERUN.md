## §2 Withdrawn Findings — Epistemic Accounting

This section is not apology theater. It is epistemic accounting per
`[[feedback-operational-maturity-not-completeness]]` and the discipline that
disposition language must distinguish what the evidence actually supports from
what the mid-phase narrative was building toward.

### §2.1 Withdrawn: "24.8 grammar caused flame_execute_python regression"

**The claim:** during pre-revert probing, my synthetic control query
`"execute python: print('canonical-probe-result-hash')"` produced a `{}` collapse
on `flame_execute_python`. I interpreted this as the grammar regressing a
previously-warm tool — framed as the most architecturally significant finding
of the phase.

**The falsification:** post-revert verification with the SAME synthetic query
produced the SAME `{}` collapse on `flame_execute_python`. Per
`[[feedback-failure-shape-stability-as-disposition-evidence]]`, identical-shape
failure on both sides of the intervention is evidence the intervention did NOT
enter that behavior pathway. The regression was not grammar-caused.

**The root cause of my error:** I synthesized the control query without
grounding it against the actual 24.6 canonical probe. The real 24.6 canonical
query is `"What are the clips on Reel 1"` (verified at
`~/.forge-bridge/measurements/2026-05-18-phase-24-6-rerun/run-probes.sh`, line 7:
`CANONICAL_QUERY="What are the clips on Reel 1"`). My synthetic query was never
the canonical probe; its failure tells us nothing about the canonical-probe
substrate state.

### §2.2 Withdrawn: "24.6 result_hash 8ca14d1e is a valid implicit control"

**The claim:** the 24.6 canonical probe's byte-stable `result_hash 8ca14d1e`
across 5/5 warm probes (documented at
`.planning/milestones/v1.6-PHASE-24-6-RERUN.md`) constituted a stable substrate-
integrity baseline for any subsequent phase to check against.

**The falsification:** post-revert verification with the REAL canonical query
(`"What are the clips on Reel 1"`) produced a 504 timeout / LLM-loop-budget
exhausted after 120s — NOT the 8ca14d1e result_hash. The current substrate
state on portofino does NOT reproduce the 24.6 baseline. Yet the revert proves
24.8's intervention is NOT what changed the substrate state.

**The root cause:** the 24.6 baseline was ecologically valid against a specific
Flame project state on 2026-05-18 ("Reel 1" loaded with the specific 350+ clips
that hashed to 8ca14d1e). The Flame project state, model state, or some other
substrate dimension has drifted in the intervening 24 hours. The drift is
independent of 24.8 — but it ALSO means 24.6's result_hash cannot serve as an
implicit control for subsequent interventions.

### §2.3 What survives from the mid-phase narrative

**Grammar correctness issues** — operator-diagnosed during the revert decision:

- **35-wrapped / 1-flat overgeneralization** (ground-truthed via module
  introspection across `forge_bridge/tools/*.py`): the grammar's "All tools
  require arguments wrapped in a params object" rule is factually false for
  `flame_execute_python(code: str)`, which is the only flat-arg tool in the
  inventory. The grammar was teaching the model an overgeneralization.
- **Literal `params={}` attractor in example 3**: the repair-loop example
  contained `forge_list_projects(params={})` as a literal call shape. The model
  pattern-matched the empty `params={}` from this example and may have produced
  empty `{}` calls for other tools.
- **Dict-syntax vs JSON-shape register mismatch**: `params={field: value}` is
  Python dict literal syntax. MCP tool invocations are JSON-shape on the wire.
  The grammar was teaching a Python syntactic register that the wire-shape
  layer doesn't accept.

These three are real design critiques. They stand independently of the regression
claim that was withdrawn. They explain WHY the grammar entered the wrong register
even if they do not explain HOW (the architectural finding in §4 below).

## §4 Architectural Finding — The Register Mismatch

### §4.1 The 4-tier behavioral-surface map gains a new entry

Carrying forward from 24.7 H0 disposition cursor
(`[[project-state-2026-05-18-phase-24-7-h0-merged]]`):

| Behavioral surface | Description shaping | Established by |
|---|---|---|
| Tool selection / affordance routing (warm) | **Effective** (per-tool register) | 24.6 |
| Argument serialization / wrapper synthesis (cold nodes) — per-tool register | **Ineffective** | 24.7 H0 |
| Argument serialization (cold nodes) — orchestrator-grammar register | **Ineffective at the invocation register; reaches text-completion register only** | **24.8 H0 (new entry)** |
| Repair-loop state incorporation | Ineffective at per-tool register | 24.7 |
| Cold-node invocation bootstrapping | **Likely requires intervention OUTSIDE prompt-surface text entirely** | 24.7 + 24.8 combined |

The combined 24.7 + 24.8 result set demonstrates the description SURFACE has at
least two registers (per-tool, orchestrator-grammar) and BOTH fail to reach the
cold-node invocation register from the text-shaping side. This is a strictly
stronger architectural claim than either phase alone could establish.

### §4.2 Quote-without-operationalize as register-evidence

The probe 3 evidence formalizes a new pattern:

> When a model's text output contains literal fragments of an intervention's
> source material, that is high-confidence evidence the intervention reached
> *some* behavioral register. Whether it reached the *target* register is a
> separate question. Quote-without-operationalize is evidence the intervention
> hit the WRONG register, not evidence the intervention failed to land.

This is the architectural complement to
`[[feedback-failure-shape-stability-as-disposition-evidence]]`. Where failure-
shape stability evidences "intervention did not reach this behavioral pathway,"
quote-without-operationalize evidences "intervention reached this content
context, but the wrong behavioral pathway." Together they triangulate
register-level effectiveness with much higher resolution than either alone.

Candidate memory (single-instance NOVEL within this project):
`feedback-quote-without-operationalize-as-register-evidence`. Eligible for
promotion at next within-project occurrence per the project's two-instance
promotion rule.

### §4.3 What the missing layer is probably NOT

The §11 item (11) design space named at 24.7 H0 disposition — system-message
tuning, orchestrator few-shot prompting, runtime adaptation, dynamic repair
injection, explicit tool-call scaffolding, synthetic successful-call priming —
all operate at the **prompt-surface text layer**. 24.8 just empirically located
that layer as the WRONG layer for cold-node invocation pathology.

Specifically: the grammar-text-echo on probe 3 demonstrates the model CAN ingest
prompt-surface text content, but cannot translate it into invocation behavior.
Adding MORE prompt-surface text (more examples, few-shot variants, priming
sequences) is likely to produce more text-completion artifacts without crossing
the register boundary.

### §4.4 What the missing layer might be

Operator-articulated candidate registers above prompt-surface text:

- **Tool-call channel alignment** — transport/protocol-level intervention
  (e.g. forcing the model into structured tool-call mode vs free-form text;
  function-calling format vs natural-language response; provider-native
  `tool_use` blocks vs orchestrator-side parse)
- **Invocation-mode conditioning** — model-call-mode-level intervention
  (e.g. the difference between chat completion and structured-output modes;
  Anthropic Claude's `tool_choice` vs `auto` vs `required`; Ollama's
  format-constrained output)
- **Runtime state shaping** — orchestrator-runtime intervention (e.g. context
  shaping, prior-state injection, transcript pre-loading with successful
  invocation patterns DELIVERED AS ACTUAL TOOL-CALL HISTORY rather than as
  text examples)

These three live above the prompt-surface text layer. They are not yet
operationally framed; they constitute a new design-space dimension that requires
its own phase opening. They are NOT promoted into the §11 register at this
disposition — the §11 register's (11) parent category needs re-decomposition
before adding new children at this layer.

### §6.2 Baseline-drift invalidation — NEW operational policy

This is the more architecturally significant of the two methodology findings.

**The discipline:** previously-established warm behavior is not a safe implicit
control. The substrate (Flame project state, model state, environment) is
dynamic enough that "the 24.6 canonical probe still works" cannot be assumed
without verification. Without explicit control-probe reruns, intervention
effects cannot be distinguished from substrate drift.

**Today's evidence:** post-revert canonical probe (`"What are the clips on Reel
1"`) timed out at 130s budget exhaustion. The revert proves 24.8 did NOT cause
this drift. Yet without running the canonical probe, we would have continued
attributing post-revert behavior to "substrate restored to 24.6 baseline" —
which would have been wrong.

**Operational implication:** future phases that interpret behavior change
against a prior-phase baseline must include an **explicit control-probe rerun
as a forcing function precondition**. The rerun establishes whether the
substrate has drifted *independently of the intervention being tested*.
Without it, intervention effect and substrate drift become indistinguishable.

**Memory candidate (NEW, single-instance NOVEL):**
`feedback-baseline-drift-invalidation`. Captures the discipline; trigger
condition is "phase imports prior-phase quantitative or behavioral baseline."
Eligible for promotion at next within-project instance.

Both methodology findings will be captured as memory writes after this PR
lands; this disposition doc surfaces them at the project-artifact level first.

## §7 Operational Policy Carry-Forward

Beyond memory, two policies should propagate forward operationally:

### §7.1 Control-probe rerun as forcing-function precondition

For any phase that interprets behavior change against a prior-phase baseline,
the rerun script must include an **explicit control-probe rerun** that
re-establishes (or invalidates) the inherited baseline before any intervention-
specific probes fire. The rerun's output classifies into:

- **Baseline reproduced** → intervention effect can be measured against it
- **Baseline drifted** → drift is the dominant signal; intervention testing is
  postponed until drift is investigated, or recast against current substrate

The control probe must be **grounded against the actual prior-phase
measurement bundle** (per §6.1's extension to canonical probe text), not
synthesized or inferred.

### §7.2 Synthesized-vs-grounded discipline at probe construction

When a phase needs a control probe, baseline reference, or canonical query, the
construction discipline is:

1. Read the prior-phase `run-probes.sh` (or equivalent runbook) directly
2. Read at least one prior-phase `meta.txt` or stdout artifact to confirm
   the canonical query text + expected behavior
3. Only then construct the current phase's verification script

This is the same discipline as
`[[feedback-ground-specs-in-actual-files]]` applied to test-input artifacts
rather than spec-shape artifacts. The same anti-pattern (inference where
verification was available) produces the same class of error.
