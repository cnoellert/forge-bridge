# v1.11 Source-of-Truth Routing — Milestone Framing (cycle-2, redirected)

> **What this is.** v1.11 redirected from *Referential Grounding* after the
> substrate-locating pass showed the reads don't fail for lack of a *parameter* —
> they fail because the system queries **the wrong substrate.**
>
> **One-line thesis.** A question should be answered by the **tool whose substrate
> holds the answer.** "Duration of shot 10 on 30sec_edit 21?" is a Flame *timeline*
> question; the system sent it to a forge *entity* tool that structurally cannot
> answer. v1.11 routes questions to their source of truth.

## Why the redirect (the pass overturned Referential Grounding)

R8/R9/R10 are **substrate-selection failures, not param-grounding failures:**

- **forge cannot answer them.** All 27 shots carry only `{cut_in, cut_out,
  sequence_id}` — `sequence_id` NULL on every shot, no sequence entity, `locations`
  joins media+version (**zero shots**). No timewarp, no path.
- **The Flame timeline tool is already param-ready** — `flame_get_sequence_segments`
  is registered with `sequence_name` resolution (`_tool_chain.py:292`). The parameter
  was never the problem.
- **The dogfood mis-selected `forge_get_shot`** for a timeline question. *That* is
  the failure: wrong substrate. Creative's **context-eligibility** seam, confirmed.

## The reachable win — 2/3, not 3/3 (DT's source-read; one more pre-code flip)

Reading the Flame collector, not inferring it:

| Read | Attribute | `flame_get_sequence_segments` emits it? | Classification |
|---|---|---|---|
| **R8** path | `file_path` | **yes** ✅ | routing win |
| **R10** duration | computed integer-frame duration | **yes** ✅ | routing win |
| **R9** timewarp | — (not in `_collect_segments`; **no read tool exists** anywhere) | **no** | **capability gap — NOT routing** |

**R9 reclassified, by SR's own boundary logic.** A router routes among *known* tools.
For R8/R10 a tool holds the answer → SR routes → win. For R9 **no tool holds the
answer** → SR has nothing to route to → **R9 is a capability gap, not a
substrate-selection failure, and scopes OUT of SR** into a separate "add a
segment-effects/timewarp inspection tool" carry-forward. This is healthier than 3/3:
had SR.1 believed it owned R9, it would have discovered mid-build that a third of its
success criterion depended on functionality that doesn't exist.

**Probe-honesty caveat (DT, reflexive):** R8/R10 are *positive-presence* findings
(the collector emits those fields — strong). R9 is an *absence* claim (grep +
collector-read — the weak direction). **SR.1-T1 must test timewarp specifically
live**; if a live segment surfaces a timewarp via some tool, the absence
false-negatived and R9 returns to scope. *Presence asserted; absence handed to the
daemon.*

## The boundary this milestone defends (the success criterion)

Same layer-refusal shape as the arc. **SR answers "which *substrate* receives this
question," NOT "which tool exposes this *fact*."** Capability discovery is out.

| SR owns | NOT SR |
|---|---|
| "timeline question" → Flame timeline substrate | which tool exposes timewarp (**capability discovery**) |
| "entity question" → forge entity substrate | understanding what the user *meant* (interpretation) |
| route among **known** tools | repairing a malformed chain (compile-shape) |
| — | choosing among *tied* candidates (DI.2 arbitration) |

**The composition this exposes (clean layering):** **SR picks the substrate → DI.2
arbitrates the tool within it → execute, or fail *honestly* as "right substrate, no
capability."** Today R9 is *wrong substrate → wrong tool → wrong answer path*; after
SR it is *right substrate → no capability* — a clean, diagnostic failure. The
four-problem progression the room keeps separating: **DI.1 authority correctness ·
DI.2 eligibility correctness · SR source-of-truth correctness · future capability
coverage (the timewarp tool).**

## The mechanism — COARSE, by substrate (the central decision, resolved toward a lean)

Both voices converge, and DT's timewarp dig is the proof: a reroute keyed on "which
substrate answers this *attribute*" presupposes a **per-tool, per-attribute
answerability map that does NOT exist** — nothing in tool metadata said
`flame_get_sequence_segments` answers duration+path-but-not-timewarp; it took reading
collector source. Unlike DI.1 (where `readOnlyHint` already existed), the SR
classification substrate is absent. So:

- **(a) per-attribute answerability map** — heavier than "route among known tools"
  implies; a new ontology build. **Rejected for SR.1 scope** (it would smuggle a
  big build behind a small-sounding thesis).
- **(b) COARSE route by substrate** — "is this a *timeline* question?" → Flame
  timeline substrate; "an *entity* question?" → forge. A small fixed classification,
  no attribute ontology. **Recovers R8/R10; lets R9 fail honestly** as "no tool on
  the right substrate answers." **This is the SR.1-sized lean.**

*Location* (still a genuine discuss fork): compile-time selection vs a narrow
post-compile answerability reroute. *Lean: post-compile reroute, coarse, deterministic,
below the model* — consistent with DI.1/DI.2. Naming the answerability-map dependency
here so SR.1-discuss picks the mechanism with eyes open.

## Scope — phase lean (for the room)

  read tool) — a future milestone, NOT SR.

  — session context, not substrate routing).

## Success criterion (per-layer, honest)

- **SR succeeds if** timeline questions reach the Flame timeline substrate and **R8 +
  R10 answer** end-to-end (2/3 of the once-"reachable" set), measured against the
  (groundable) DI.2 corpus + loaded sequence.
- **SR does NOT claim** R9/timewarp (capability gap), compile-shape repair, answer-pass
  fidelity, session scope, or "chat useful." A clean source-of-truth-correctness
  milestone, no more.
