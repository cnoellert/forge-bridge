# v1.11 Source-of-Truth Routing — Milestone Framing (cycle-1, redirected)

> **What this is.** v1.11 redirected. It opened as *Referential Grounding* (ground
> the artist's references into the identifiers a tool needs). The substrate-locating
> pass — run early at discuss-open, exactly to ground the reachable-win before
> ranking — **overturned that premise.** The reads don't fail for lack of a
> *parameter*; they fail because the system queries **the wrong substrate.**
>
> **One-line thesis.** A question should be answered by the **tool whose substrate
> actually holds the answer.** "What's the duration of shot 10 on 30sec_edit 21?" is
> a Flame *timeline* question; the system sent it to a forge *entity* tool that
> structurally cannot answer it. v1.11 routes questions to their source of truth.

## Why the redirect (the pass overturned Referential Grounding)

Grounded against the live substrate, R8/R9/R10 are **substrate-selection failures,
not param-grounding failures:**

- **forge cannot answer them.** All 27 shots carry only `{cut_in, cut_out,
  sequence_id}` — no timewarp (R9), no path (R8; the `locations` table joins
  media+version, **zero shots**). And `sequence_id` is **NULL on every shot** with
  **no sequence entity** — so a shot can't even be scoped from "30sec_edit 21".
- **The answer lives on the Flame timeline,** and the right tool —
  **`flame_get_sequence_segments`** — is **already registered and already
  param-ready** (`_tool_chain.py:292`, `sequence_name` via 24.11). Its parameter was
  never the problem.
- **The dogfood mis-selected `forge_get_shot`** — a forge entity read — for a Flame
  timeline question. *That* is the failure: wrong substrate, not missing identifier.

So referential grounding (a `shot_id` for `forge_get_shot`) was solving for the wrong
tool. The grounding half that *does* matter (`sequence_name`) already works. **The gap
is selection: route the question to the substrate that holds its answer.** This is
Creative's **context-eligibility** seam — named dormant, DT-promoted to observed, now
root-cause-confirmed as the live failure.

## The boundary this milestone defends (the success criterion)

Same architectural shape that has run through the arc — **each layer refuses to
become the layer above it.** SR routes among **known tools** to the one whose
substrate answers; it does **not** reinterpret the request, invent capability, or
repair malformed chains.

| SR owns (routing/selection) | NOT SR |
|---|---|
| "timeline attribute" → Flame timeline tool | understanding what the user *meant* (interpretation) |
| "entity attribute" → forge entity tool | rewriting/repairing a malformed chain (compile-shape) |
| select the answer-holding tool among candidates | inventing a tool / a new interpretation |
| ordinal selection *within* the answered set (residual) | choosing among *tied* candidates (that's DI.2) |

**Distinct from its neighbors (so it doesn't absorb them):**
- **vs compile-shape:** R8/R9/R10 compiled to a *well-formed* single valid tool —
  just the wrong substrate. Compile-shape (R2/R5/R6) is *malformed/over-injected*
  chains. Well-formed-but-wrong-substrate ≠ malformed.
- **vs DI.2 arbitration:** DI.2 chose among *tied* candidates once the right set was
  present. SR is upstream of that — it's about whether the answer-holding tool is
  *selected at all*.
- **Control-flow, not meaning** ([[feedback-orchestrator-control-flow-not-meaning]]):
  routing picks an existing tool; it never authors prose or reinterprets.

## The central mechanism question (the milestone's core discuss decision)

The compiler turned the read into the step `forge_get_shot`. So the wrong-substrate
selection happens **at compile time.** The open design question:

- **(a) Compile-time selection** — teach the compiler/selector that timeline-attribute
  questions resolve to Flame timeline tools (entity-attribute → forge). Risk: blurs
  into compile-quality.
- **(b) Post-compile routing** — after a tool is chosen, check whether its substrate
  can hold the asked attribute; reroute to the source-of-truth tool if not. Risk:
  a new routing layer that must not become a shadow compiler.

*Lean (for discuss): (b) framed narrowly* — an **answerability check** that reroutes
among **known tools** when the selected tool's substrate provably lacks the attribute,
staying inside the control-flow/eligibility lane. But this is genuinely the room's
call and it defines the milestone's shape.

## Scope — phase lean (for the room)

- **SR.1-T1 — confirm the source-of-truth (measure-first, FIRST).** Bring the daemon
  up and call `flame_get_sequence_segments` against the loaded `30sec_edit 21`;
  confirm a segment carries duration / timewarp / source-path (the one claim still
  *inferred* — standard timeline semantics, but ground it before building). Locks the
  reachable win (potentially 3/3 via one Flame tool, vs DT's forge-side ~1.5/3).
- **SR.1 — timeline-vs-entity routing (the reachable win).** Route R8/R9/R10's
  timeline-attribute questions to `flame_get_sequence_segments` (sequence already
  grounds; residual: ordinal "shot 10" → segment selection). Acceptance: R8/R9/R10
  **answer** against the loaded sequence.
- **R7 / session-project scope — flagged ADJACENT, likely not SR.** R7
  (`MULTIPLE_PROJECTS`) is "which project *universe*", a session-context problem, not
  substrate-routing. Candidate for its own small phase or a different milestone; do
  not fold it into SR's routing thesis.

## Success criterion (per-layer, honest)

- **SR succeeds if** well-formed reads reach the **answer-holding** tool — R8/R9/R10
  answer via the Flame timeline source of truth, measured against the (now
  groundable) DI.2 corpus + loaded sequence.
- **SR does NOT claim** compile-shape repair, answer-pass fidelity, session scope, or
  "chat useful." Still several milestones; this is the substrate-selection one.

## The handoff lesson (now 5 flips — and one was mine)

**Five times this arc a load-bearing premise flipped on a probe** (9-tool match ·
"builtins pass none" · DI.2 arbitration-vs-working-reads · RG reachable-win ·
RG-is-actually-routing). The discipline: **ground the reachable-win against real
substrate AND real state before naming the milestone.** New this round — *the probe
itself can be the unreliable instrument*: my Flame traversal false-negatived a
sequence the operator could see on the desktop. Ground-truth beats your own probe;
when a probe contradicts the operator's direct observation, suspect the probe.
Candidate methodology (data-layer + instrument-reliability manifestations of
[[feedback-ground-specs-in-actual-files]] / [[feedback-audit-before-overfit]]); author
at milestone close.
