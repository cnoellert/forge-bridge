## §0 Banner — What Is Different About This Phase

Every phase in the 24.x arc prior to 24.6 targeted deterministic system behavior:
substrate truth, runtime economics, transport latency, termination policy, consumer projection.
Correctness was binary.

24.6 crossed into probabilistic affordance behavior, but at the **selection** layer:
WHICH tool the model picks across the registered catalogue. Rhetorical sharpening of two
warm/high-history tool descriptions (`flame_execute_python`, `flame_find_media`) shifted
the iter-1 selection distribution and eliminated the iter-3 relapse pattern. The PROMOTED
memory `[[feedback-rhetorical-position-as-architectural-control-surface]]` captures the
generalized principle from that work.

24.7 is structurally distinct again.

The target is **invocation behavior at cold affordance nodes** — tools the model has no
successful-invocation session history for. The pathology is not which tool gets picked
(selection layer) but whether the picked tool gets invoked correctly (invocation layer).
The model selects the right tool, then collapses to `{}`, or hallucinates the answer
inline, or fails to bind natural-language entities to the required arguments, or fails
to execute the repair loop after the daemon returns a validation error.

The epistemic character of this phase is shaped by an empirically-located boundary:

**The post-24.6 dogfood window (PRs #9–#13) accumulated evidence that description-layer
shaping alone is insufficient for cold affordance nodes.** Three increasingly structural
description-layer interventions on `flame_inspect_sequence_versions`:

1. **D-04** — field description relaxation (`Natural-language variants are acceptable.`)
2. **PR #12** — prose usage example in opener (`Call with the sequence name from the user's query (e.g. sequence_name="30sec_21")`)
3. **PR #13** — minimal structural exemplar (`Example call: {"params": {"sequence_name": "30sec_21"}}`)

all failed against the live operator probe. The PR #13 discriminator fired H0:
description-layer shaping at the thinnest structural register is empirically insufficient
for cold-node invocation bootstrapping.

24.7 tests the **upper bound of the description layer** with a deliberately richer
intervention: full 3-part invocation exemplars (natural-language query → exact wire-shape
call → compact response shape) embedded directly at each cold node's affordance surface.

If 24.7 works, the upper bound is established — description-layer interventions CAN
bootstrap cold-node invocation behavior, provided the exemplar carries enough structural
concreteness. The candidate principle "structural exemplars vs prose" held at the PR #13
cursor gets validated.

If 24.7 fails, the description layer is empirically exhausted as a fix surface for this
class. The architectural escalation to orchestrator-level bootstrapping (system message,
few-shot prompting at the orchestrator layer, runtime adaptation) becomes JUSTIFIED BY
EVIDENCE rather than tempted by frustration. The accumulated dogfood-window data becomes
the load-bearing case for whatever phase opens next.

Both outcomes are architecturally valuable.

## §1 Diagnosis

**Observed pathology (cross-tool stable):**

The operator-facing failure modes accumulated through ecological dogfood after 24.6
convergence are *not* topology failures. They are affordance-inhabitation failures —
the model selects the right tool and then cannot inhabit the invocation correctly.

Five concrete instances surfaced through live operator chat:

| Tool | Failure mode |
|---|---|
| `flame_inspect_sequence_versions` | `{}` empty-call collapse; 0.0s elapsed; never reaches inference; daemon rejects at schema validation |
| `flame_rename_shots` | Model hallucinated JSON output inline instead of invoking the tool |
| `flame_get_sequence_segments` | Model routed to `forge_list_shots` (wrong affordance surface) instead of Flame sequence inspection |
| `forge_list_shots` | Model received explicit repair instructions ("project_id is required") yet failed to synthesize the corrected second call |
| `flame_execute_python` | Succeeds consistently — abundant successful-invocation patterns in-context |

The asymmetry between `flame_execute_python` (works) and the four failing tools (don't)
is the load-bearing data point. The model's session history is full of successful
`flame_execute_python` invocations from the 24.6 measurement bundles + every subsequent
chat probe; it has *zero* successful invocations for the failing cold tools. The
description layer's information content is similar across the set; the model's behavioral
asymmetry is dramatic.

**Working hypothesis:** Cold affordance nodes require at least one structurally concrete
successful-invocation pattern at the description surface before the model can reliably:

- synthesize the wrapped `params` request structure FastMCP expects
- bind natural-language entities from the user query to required arguments
- distinguish adjacent affordance surfaces (`flame_get_sequence_segments` vs `forge_list_shots`)
- execute repair loops after validation failures (binding the daemon's error back into
  a corrected second call)

**Evidence base (ecological — earned through dogfood after 24.6 convergence):**

- 5-PR dogfood window outputs documented across cursors:
  `[[project-state-2026-05-18-pr12-inspect-versions-merged]]`,
  `[[project-state-2026-05-18-pr13-discriminator-armed]]`
- PR #13 discriminator-probe H0 outcome (operator-confirmed at phase-24.7 framing handoff):
  minimal structural exemplar insufficient for cold-node bootstrapping
- Cross-tool failure-mode taxonomy (above) demonstrates this is a class problem, not a
  per-tool bug — four distinct tools, four distinct concrete failure shapes, one
  underlying invocation-layer pathology

**What the evidence does and does not prove:**

- **Proves:** description-layer shaping at the minimal-structural register (PR #13
  one-line wire-shape template) does not bootstrap cold-node invocation behavior; the
  failure class spans at least 4 cold tools with distinct concrete failure shapes
- **Does not prove:** that RICHER structural exemplars (full 3-part invocation patterns)
  will also fail. The 24.7 forcing function (§5) is designed to falsify the working
  hypothesis by giving the description layer its strongest possible intervention shape
  and observing whether the pathology persists

## §3 Anti-Scope — Architecturally Rejected

**This is the load-bearing anti-scope guard for 24.7:**

> *"The exemplars live inside the local affordance surface of each tool description only.
> They are not orchestrator-level few-shot prompts, not system-message content, not
> runtime adaptation. The perturbation is localized to the cold-node affordance surfaces
> themselves."*

The temptation that 24.7 must resist is **architectural escalation under partial-failure
pressure**. If H1 holds (exemplars bootstrap reliable invocation), no escalation needed.
If H0 holds (exemplars also fail), the next phase justifies escalation. Pre-emptive
escalation during 24.7 execution erodes the discriminator. Per
`[[feedback-anti-scope-discipline-under-pressure]]`: when partial-failure surface becomes
visible mid-execution, out-of-scope mitigation modes become psychologically tempting.
The §3 guard above is the load-bearing protection.

**Specifically rejected during 24.7 execution:**

- Adding tool-selection routing hints inside exemplars ("prefer this tool when X")
- Adding orchestration-layer affordance shaping (system message tweaks, prompt-prefix
  injection) "to help the exemplars land"
- Adding multiple exemplars per tool ("if one is good, three is better") — single
  exemplar per tool preserves the perturbation's structural clarity
- Sweeping the exemplar pattern across non-target tools — single-perturbation discipline
  requires the 4-tool scope; broader sweeps weaken causal attribution
- Changing the underlying tool schemas or signatures while editing the docstrings —
  description-only change at every site

**Inherited-baseline clarification:**

The four target tools' current docstrings contain the post-D-04 + post-PR-#12 +
post-PR-#13 state on `flame_inspect_sequence_versions` (field description relaxed; prose
example present; minimal structural exemplar present). The other three target tools
(`flame_rename_shots`, `flame_get_sequence_segments`, `flame_set_start_frames`) carry
their pre-dogfood-window docstrings unchanged.

24.7 adds the **3-part exemplar block** to all four tools. Where prior interventions
already exist on `flame_inspect_sequence_versions`, the 3-part exemplar augments them
rather than replacing them — the priors are part of the structural archaeology of the
docstring's evolution. Per
`[[feedback-brief-examples-as-behavioral-reference-shapes]]`: preserve battle-tested
topology; the prior PRs are part of the operational archaeology, not cosmetic debris to
remove during the next intervention.

## §5 Forcing Function — Pass Criterion

24.7 is the second phase in this project's history (after 24.6) that requires
**distributional correctness** rather than binary correctness. The success criterion is
multi-dimensional per `[[feedback-distinct-success-criteria-per-adjacent-layer]]`.

**M-warm probes per target tool (M=3):** after daemon restart loads the 24.7 exemplars,
fire 3 distinct natural-language operator queries per tool against the warm daemon. Total
warm probes: 12 (3 × 4 tools).

**M-cold probes per target tool (M=1):** after daemon restart loads the 24.7 exemplars
AND before any warm chat history accumulates, fire 1 cold probe per tool. Total cold
probes: 4 (1 × 4 tools).

Cold-vs-warm distinction is load-bearing: the working hypothesis says cold nodes are
where description-layer shaping struggles. If exemplars work under warm conditions but
fail under cold conditions, the result is half-true and the §11 disposition reflects
that nuance.

**Multi-dimensional pass criterion:**

| Dimension | Required | Cold | Warm |
|---|---|---|---|
| **Empty-call elimination** | No `{}` collapse on any probe of any target tool | 4/4 | 12/12 |
| **Tool-fire success** | Daemon receives a structurally valid call (passes schema validation) | 4/4 | 12/12 |
| **Argument binding** | Bound `sequence_name` value derives from the operator's natural-language query | 4/4 | 12/12 |
| **No hallucination** | Model invokes the tool rather than fabricating an answer inline | 4/4 | 12/12 |
| **Substrate integrity** | `result_hash` of `flame_execute_python` canonical probe (from 24.6) remains byte-stable `8ca14d1e` | (control) | (control) |

**Failure conditions (any one triggers H0 disposition):**

- Empty-call collapse on any cold probe AND any warm probe
- Tool-fire success rate < 4/4 cold or < 12/12 warm
- Substrate integrity broken (24.6 canonical probe result_hash drift)
- Hallucination on any probe

**Partial-pass conditions (H1 with caveats):**

- Warm probes pass; cold probes fail → exemplars require successful-invocation history
  to activate (warm-only bootstrapping; cold remains pathological)
- Some tools pass, others fail → exemplar shape works but per-tool variance remains;
  needs per-tool tuning beyond the 24.7 template
- Empty-call eliminated but argument-binding wrong → exemplar provides shape but not
  semantic anchor; partial intervention success

The disposition language at the rerun doc must explicitly distinguish full pass / partial
pass / full failure, per `[[feedback-explicitly-unbound-vs-implicitly-rejected]]` and
`[[feedback-operational-maturity-not-completeness]]`.

## §10 Architectural Significance — Conditional

**If H1 confirms (description layer can bootstrap cold-node invocation with rich exemplars):**

24.6 established that *rhetorical ordering* can alter affordance-selection behavior for
warm nodes. 24.7 extends the principle: *structural exemplars* can bootstrap invocation
behavior for cold nodes.

The combined finding: **LLM tool reliability is shaped not only by semantic capability
descriptions, but by the topology and concreteness of invocation patterns exposed at the
affordance surface.** This is a strictly stronger architectural claim than "documentation
quality matters." It says the description SURFACE has multiple structural registers
(rhetorical ordering, prose framing, structural exemplars) and each register controls a
different behavioral dimension.

The `[[feedback-rhetorical-position-as-architectural-control-surface]]` memory would be
joined by a sibling: `feedback-structural-exemplars-bootstrap-cold-invocation` (or
similar). Both would name the description layer as a multi-register architectural surface
rather than a flat prose channel.

**If H0 confirms (rich exemplars also insufficient):**

The description layer is empirically exhausted as a fix surface for cold-node
invocation. The accumulated dogfood-window evidence (D-04 + PR #12 + PR #13 + 24.7) is
the load-bearing case for architectural escalation. The next phase frames orchestrator-
level bootstrapping (system message, few-shot at orchestrator layer, runtime adaptation)
as JUSTIFIED BY EVIDENCE rather than tempted by frustration.

The methodology archaeology becomes: "the project iteratively located the upper bound of
description-layer effectiveness and crossed it deliberately, with falsification evidence
at every intermediate step." Negative result; high architectural value; opens the next
layer's framing with clean justification.

**Either outcome:** layer boundaries stay intact. The discriminator design discipline
from §11 item (9) is now applied at a second scope (description-layer effectiveness
boundary, vs item (9)'s cache-vs-prior-shift confound), strengthening the candidate
insight "discriminator-probe pattern as generalized methodology tool beyond §11 item (9)"
held at the PR #13 cursor.
