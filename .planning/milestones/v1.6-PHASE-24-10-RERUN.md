Therefore the fix is validated at the intended adapter boundary, while the
ecological chat forcing function is blocked by a second invocation boundary and

## §1 What Changed

`forge_bridge/mcp/registry.py` now normalizes arguments immediately before the
FastMCP call:

```python
args = _normalize_args(name, args, available)
raw = await mcp.call_tool(name, arguments=args)
```

The normalizer is schema-driven:

- wrapper-schema tool: required top-level `params` with `properties.params.$ref`
  to a nested object in `$defs`
- already wrapped: pass through unchanged
- flat-native schema: pass through unchanged
- empty flat args for wrapper-schema tool: `{"params": {}}`

It logs applied normalization at DEBUG:

```text
invoke_tool: flat→params normalization applied for {name}: {flat_args}
```

failure:


This confirms the flat→params normalization works at the adapter boundary.

## §5 Disposition

H1 is confirmed for the adapter-boundary hypothesis:

> flat provider-native arguments can be normalized to the wrapper schema and
> then reach Flame successfully.

H1 is not yet confirmed for the full `fbridge chat` ecological path because
the current route can bypass `invoke_tool()` before the LLM emits arguments.

Next work should explicitly decide whether the forced-execution path should:

1. route through `invoke_tool()` so all tool execution shares one adapter
   boundary, or
2. apply the same schema-driven normalization at every direct FastMCP
   `call_tool()` site.

That is a separate perturbation from 24.10's promised `invoke_tool()`-only
change.
