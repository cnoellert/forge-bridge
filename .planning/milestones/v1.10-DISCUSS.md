## The enabling finding (grounded this session)

**(A) has no plumbing gap — the annotation is already readable at every
enforcement site.** `cli/discover.py:_tool_record` (`:95-110`) already reads
`readOnlyHint` off `getattr(tool, "annotations", …)`. The forced-tool `tool`
(`_execute_forced_tool`) is a survivor of `filter_tools_by_message(tools, …)`
(`_step.py:251`) — i.e. one of the same annotated objects `mcp.list_tools()`
returns (`handlers.py:1296`). So the guard reads `tool.annotations.readOnlyHint`
*off an object already in hand* at all three execution edges (commit-route
`:158`, forced dispatch `:720-722`, chain dispatch `_step.py:409`). The work is
enforcement, not plumbing.

## Resolutions

- **Q-DI1 (residuals) — RESOLVED.**
  - *Reachability:* confirmed above. No new accessor needed.
  - *Canonical read path:* `tool.annotations` (matches what `register_tool`
    writes via `mcp.add_tool(annotations=…)`). **Audit item:** a second surface
    reads `meta.readOnlyHint` as a string ("true"/"false") in `cli/tools.py:120`
    — the enforcement MUST read `.annotations` (the write side), and the audit
    confirms the two don't disagree.
  - *Fail-closed:* **absent OR non-read ⇒ treated as mutating.** A tool executes
    as a read *only if* affirmatively `readOnlyHint == True` — a forgotten
    annotation fails safe (over-block a read), never open (run a mutation).
  - *Absent set + sequencing — RELOCATED to the registration boundary (DT cycle-5,
    supersedes the earlier annotate-before-enforce sequencing).* The absent set is
    **not** the builtins — they are **69/69 annotated** (0 absent) — it is
    **consumer `user-taught` tools**: `register_tools` (`registry.py:140/180`)
    registers with no annotations and PROV-04 (`:127`) fires only for synth, so
    consumer tools land `readOnlyHint`-absent — and a bridge-side audit **cannot
    see them** (registered at the consumer's import time). The fix is one line at
    `:127` — extend PROV-04's fail-safe to `user-taught` — making the absent set
    **∅ universally**. Then fail-closed has no flag-day and annotate-before-enforce
    is moot: registration *guarantees presence* rather than an audit checking for
    absence. Trade (named): a consumer read tool defaults mutating-until-annotated
    → safe side, consumer-fixable via `annotations={"readOnlyHint": True}`.
- **Q-DI2 — arbitration mechanism. RESOLVED to a concrete hook.** `_step.py`
  already runs `deterministic_narrow(filtered, step_text)` at `:260` when
  `len(filtered) > 1`; the hard-stop at `:326` fires only if it *still* >1. So
  the work is: (a) strengthen `deterministic_narrow`; (b) add a **bounded
  candidate-selection** step before the hard-stop (deterministic ranking first;
  one LLM selection among the ≤5 candidates only if needed — control-flow, not
  meaning); (c) if it still can't resolve, a **task-term** "did you mean?" to the
  operator — **never** the tool-identifier leak (`flame_get_clip` vs
  `forge_get_shot`) nor "matched N tools." Best-effort bar.
- **Q-DI3 — prompt mitigation. RESOLVED: secondary only.** Tighten the `commit`
  instruction opener in `build_compile_system_prompt` (`_chat_compile.py:82-86`)
  to reduce bad compiles, but it is *not* the guarantee — the deterministic guard
  is. Rides in the authority phase, low cost.
- **Q-DI4 — corpus-instrument capture. RESOLVED: fold the minimal version.**
  Capture must also fire on `preview_emitted` / `chain_aborted` / forced-tool
  error (with an outcome tag), or the milestone can't measure "reads no longer
  reach mutation" or "fewer dead-ends." Enabling tooling, lands in the baseline
  step.
- **Q-DI5 — baseline. RESOLVED: prerequisite.** Re-run the 11 dogfood reads on
  current `main` for a contemporaneous failure-shape baseline before any change
  ([[feedback-baseline-drift-invalidates-controls]]). First task of the first
  phase.

## Phase decomposition (lean — for the room/roadmap to ratify)

The safety asymmetry drives the split: the guarantee leads and is sequenced
first; the best-effort usability work follows.

- **DI.1 — The authority guarantee (load-bearing, non-negotiable).**
  baseline (Q-DI5) + minimal capture seam (Q-DI4) → **close the absent set at the
  boundary** (`registry.py:127`, one line — Decision 2) → **enforce the one
  invariant at all three execution edges**: commit-route (1A, `_chat_compile.py:158`),
  forced dispatch (1B, `handlers.py:720-722`), chain dispatch (1C,
  `_step.py:409`) — a read intent cannot reach a mutation by *any* route →
  **regression-lock** the invariant (the tested acceptance criterion). Prompt
  mitigation (Q-DI3) rides here. **Acceptance criterion (Creative, first-class):
  a blocked tool fails toward understanding, not refusal** — the redirect is
  product (it fires systematically for the unbounded consumer-tool set), so it
  must be robust, not corpus-tuned.
- **DI.2 — Eligibility arbitration (usability, best-effort).** Q-DI2: strengthen
  `deterministic_narrow` + bounded candidate-selection + task-term disambiguation
  before the `:326` hard-stop; kill the "matched N tools" and tool-identifier
  leaks. Shares `_step.py` with DI.1's 1C gate but at non-colliding regions
  (`:260/326` arbitration vs `:409` dispatch, ~70 lines apart) — *"which tool?"*
  vs *"may it execute?"*. Rides after DI.1.

Corpus capture (Q-DI4) and baseline (Q-DI5) are *enabling tooling* inside DI.1,
not their own phase. Two phases, sequenced DI.1 → DI.2.
