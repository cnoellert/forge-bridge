# v1.5 Legibility — Milestone Close Synthesis

> **What this artifact is.** A milestone-arc archaeology synthesis closing v1.5 under writer's-room cadence. It is the durable record of what v1.5 shipped, what it deferred, what changed mid-milestone, what methodology evolved, and how v1.6 inherits the residual obligations. It does NOT follow the standard `vX.Y-MILESTONE-AUDIT.md` substrate shape because v1.5 did not run on the standard `vX.Y-REQUIREMENTS.md` + `vX.Y-ROADMAP.md` substrate; running ceremony-shaped audit against writer's-room work would produce false structure. See §9.
>
> **Voice discipline binding.** Every "substrate operationally mature" claim in this document is paired with consumer-shape + consumption-conditions per the §5.1 voice constraint developed at A.5.3.2 phase close. No paragraph asserts that no-further-work-in-this-area is justified; every closure claim is explicitly scope-bound. The architectural test from §5.1 ("could a future contributor reading this paragraph cold conclude no further work in this area is justified?") was applied to every section during drafting; victory-lap-shaped phrasings were rewritten to handoff-shape.
>
> **What this artifact is not.** Not an audit. Not a retrospective excuse. Not a victory lap. The milestone shipped real work; the milestone also did not ship a non-author UAT walk. Both facts are stated plainly. The synthesis exists because the difference matters — and the difference is exactly what carries forward to v1.6 as the next opening forcing function.

## §1 — What v1.5 was

### §1.1 — Original scope as articulated at milestone open

v1.5 Legibility opened 2026-04-30 immediately after v1.4.x Carry-Forward Debt closed at tag `v1.4.1`. The milestone's stated theme (CLAUDE.md, PROJECT.md, both still verbatim in those files at synthesis time):

> Make forge-bridge usable by its first daily user — close the gap between what's shipped (19 phases, 5 surfaces, ~40k LOC) and what a person can sit down and actually use without re-deriving the deployment topology each time.

Four phases were named at open:

- **Phase 20** — Reality audit + canonical install (`docs/INSTALL.md`, README install refresh, CLAUDE.md ground-truth, `install-flame-hook.sh` default pinned to `v1.4.1`). Requirements: INSTALL-01..04 + DOCS-02.
- **Phase 21** — Surface map + concept docs (`docs/GETTING-STARTED.md`, README "What This Is" rewrite, projekt-forge relationship documented). Requirements: DOCS-01, 03, 04.
- **Phase 22** — Daily workflow recipes (6 step-by-step recipes for daily tasks). Requirements: RECIPES-01..06.
- **Phase 23** — Diagnostics + recovery (`docs/TROUBLESHOOTING.md`, doctor polish if recipe authoring surfaced gaps). Requirements: DIAG-01..05.

### §1.2 — Original forcing function (verbatim from the source artifacts)

Two source artifacts state the forcing function. Both still read this way at synthesis time:

**CLAUDE.md** (v1.5 milestone context):
> Forcing function: a non-author UAT must pass on a fresh conda env.

**PROJECT.md** (v1.5 milestone open, 2026-04-30):
> Forcing function: if the install doc doesn't work end-to-end on a clean machine, we don't ship it. Phase 20 will likely surface deployment gaps that get fixed in-flight (and may add code-fix plans alongside doc plans).

The two phrasings are close but not identical. CLAUDE.md uses the synthesis-shorthand "non-author UAT"; PROJECT.md uses the operationally-grounded "end-to-end on a clean machine." The synthesis-shorthand is what subsequent in-flight discovery progressively softened against (§2); the operationally-grounded version was substantively satisfied by Phase 20.1's walk-validation on flame-01 + portofino (§3.1).

### §1.3 — Original key constraints

Three constraints were stated as binding at open:

1. **Legibility, not features.** No new external libraries.
2. **Public `forge_bridge.__all__` stays at 19** unless an install audit surfaces a genuine need.
3. **All v1.5+ planted seeds defer to v1.6+.**

All three held through close. Public `__all__` is 19 at synthesis time (verified by AST parse of `forge_bridge/__init__.py`); no new external libraries shipped in v1.5; the planted-seed register grew from ~15 at open to 36 at close (all v1.6+-tagged or earlier-V1.5-tagged-deferred, no v1.5-active-scope seeds remain unresolved).

### §1.4 — What the milestone became

v1.5 opened as a four-phase legibility milestone. v1.5 closed as a six-phase milestone (4 + 2 decimal patches: 20.1 and 23.1) covering:

- Runtime survivability (installability) — Phase 20 + 20.1
- Operational walkthrough discipline — Phase 21
- Progressive-trust operator legibility — Phase 22
- Recovery semantics + observability convergence — Phase 23
- Canonical introspection surface + chat-path convergence — Phase 23.1

The decimal-patch additions were not a scope expansion; both decimal patches landed as in-flight ship-blocker responses to gaps surfaced during the milestone's authored writer's-room cadence (D-04 decimal-patch precedent established Phase 10.1 in v1.3.1, exercised again at 20.1 and 23.1). The four originally-named phases all closed against their stated requirements; the two decimal patches close gaps that the original four-phase frame would have left as ship-blockers.

## §2 — The methodology shift mid-milestone

### §2.1 — The shift, in one sentence

The original forcing function was right for the framing v1.5 was given. The milestone evolved through operational discovery into a project that proved something larger than the original framing tested for. Author-walk validation + explicit residual seeding became the accepted sufficiency standard during execution, durably recorded in three independent seed files. This synthesis reckons with that evolution explicitly rather than papering over it.

This is not "we skipped UAT." It is: *the milestone changed what needed proving*.

### §2.2 — The evolution trail (four in-flight artifacts, not one closing act)

The forcing-function softening was not a single act at milestone close. It was a four-step in-flight progression captured contemporaneously across three independent seed files + one terminal cursor. Each step was a phase-close governance act in real time, not a retroactive synthesis act.

**Step 1 — Phase 20 close (2026-05-12).** Planted `SEED-PHASE-20-NON-AUTHOR-UAT-V1.6+`. The author performed `20-HUMAN-UAT.md` walk on flame-01 + portofino; fresh non-author UAT explicitly seeded forward. Phase 20 closed on author-walk evidence + the seed.

**Step 2 — Phase 20.1 close (2026-05-12).** Planted `SEED-PHASE-20.1-ARTIST-UAT-V1.6+`. The author walked Linux + macOS gates with 4-walker partition + idempotency rerun; real-artist fresh-machine walk explicitly seeded forward. Phase 20.1 closed on operational-maturity-sufficient-for-handoff language (per `project_state_2026_05_12_phase_20_1_closed_walk_validated.md` cursor verbatim).

**Step 3 — Phase 21 close (2026-05-13).** Planted `SEED-PHASE-21-NON-AUTHOR-UAT-V1.6+`. The seed file itself, at 76 lines, is load-bearing archaeological evidence for this synthesis — it explicitly distinguishes "gate not fully closed" from "blocking ship" and articulates the methodology reasoning for why author cold-read is concrete-but-weaker-than-ideal evidence. Verbatim from the seed:

**Step 4 — Phase 23.1 terminal cursor (2026-05-14).** The phrase "v1.5 milestone-close evaluation now unblocked" appears in both the STATE.md cursor and the terminal memory cursor. By Phase 23.1 close, the forcing-function softening had already happened three times in three independent seed files; the 23.1 terminal cursor surfaces it as a single phrase rather than a single act.

### §2.3 — Why this is governance, not retroactive synthesis

Three properties of the evolution trail distinguish it from retroactive excuse-making:

1. **The softening happened in-flight, durably recorded.** Three seed files were authored during the phases that surfaced the gaps. The seed files were committed contemporaneously with the phase-close cursors. No retroactive editing of those seed files is necessary to support this synthesis; the seeds say what they say at HEAD.

2. **Each softening act was explicit, not implicit.** The Phase 21 seed contains the phrase "without blocking v1.5 ship" verbatim. The Phase 20.1 cursor contains the phrase "INSTALL-01 v1.5 ship-blocker requirement MET per operator declaration" verbatim. The Phase 23.1 cursor contains the phrase "yes-with-known-residuals" verbatim. Each act named what it was doing while doing it.

3. **The residual register grew with each act.** Each phase that softened the forcing function paid into the residual register with a named seed file. Three non-author UAT seeds (Phase 20, 20.1, 21) ride forward together. The deferral is operationally bounded, scope-named, and trigger-conditioned. Per [[feedback-deferral-first-class-governance]]: deferral language preserves maneuverability; the seeds are governance acts, not latent debt.

### §2.4 — What changed at the milestone, in framing-terms

v1.5 was articulated at open as an **installability + legibility** milestone with non-author UAT as the forcing function. v1.5 closed as a **legibility milestone where installability is one of six legibility surfaces tested**.

Those framings are related but not identical. The original framing collapsed installability and legibility into one forcing function. The mature framing distinguishes them and treats them as separate properties — both load-bearing, but separately measurable.

The milestone evolved because the work surfaced this distinction. Phase 20.1 proved installability under author-walk + idempotency rerun. Phases 21/22/23 proved legibility under author-walk + writer's-room cadence + explicit residual seeding. Phase 23.1 proved chat-path operational correctness on the production routing path. Each phase validated a separate legibility surface, against a sufficiency standard accepted in-flight.

The non-author UAT remains the only valid test of reader-comprehension legibility for a non-author audience, by definition. The milestone proves the other five legibility surfaces (installability, operational walkthrough, progressive trust, recovery, chat-path) under the sufficiency standard the work itself surfaced. The non-author UAT carries forward as v1.6's opening forcing function (§7.5, §8.2).

### §2.5 — A literalist reading would invert the methodology

A literalist reading of the original CLAUDE.md forcing-function sentence would say: "v1.5 cannot close until a non-author UAT passes." That reading would ignore:

- Phase 20.1's walk-validation on two reference operator machines with idempotency rerun
- Phase 21's author cold-read with one finding surgically corrected
- Phase 22's 6 recipes shipped against real production friction
- Phase 23's TROUBLESHOOTING.md with doctor-as-architectural-teaching
- Phase 23.1's author-walk on portofino converging end-to-end on the canonical regression query
- Three explicit seeds planted forward as governance acts
- v1.6's framing artifact already drafted as the architectural anchor for what comes next

It would also invert the methodology the project has consistently used: artifact shape matches developmental reality, not process template. Holding v1.5 open indefinitely waiting for a non-author UAT — for an indeterminate duration, against an undefined trigger — would muddy the architectural history. v1.6 is already framed around exactly the kinds of operational truths v1.5 surfaced (substrate/consumer separation, degradation tolerance, graph-runtime provenance, runtime observability, canonical introspection surfaces, promotion provenance). Continuing the work is more legible than waiting on it.

## §4 — What v1.5 proved operationally

Six milestone-level truths surfaced through the milestone work. Each is paired with its consumer-shape + consumption-conditions per the §5.1 voice constraint.

### §4.1 — The bridge can degrade without collapsing

The five degraded-tolerant doctor rows (storage_callback / llm_backend.cloud / llm_backend.local / postgres-storage-side / jsonl-first-run) encode the operational-survivability invariant: a degraded subsystem the bridge can survive without gets WARN; a degraded subsystem the bridge cannot survive without gets FAIL. Phase 23 made this invariant testable at the doctor surface.

*Consumer shape:* an operator running `fbridge doctor` against a bridge with Anthropic unconfigured sees WARN, not FAIL, and knows the bridge is usable for local-Ollama-only workflows. An operator running against a bridge with Ollama unhealthy under sensitive=True production sees FAIL and knows the chat path is broken.

*Consumption conditions:* the invariant holds at the surfaces Phase 23 authored against. Subsystems Phase 23 did not author against (the WS bus on `:9998` graceful-degradation per Phase 07.1; the learning pipeline's dormant-without-consumer pattern) inherit the invariant by pattern-correspondence but were not retested under Phase 23 scope. Further degraded-subsystem additions in v1.6+ inherit the invariant as the documented expectation; any new doctor row that violates it is a v1.6 surface defect.

### §4.2 — Docs can drive observability evolution

The Phase 23 Commit B postgres row is the architectural manifestation. Writing the troubleshooting recipe for Postgres-unreachable surfaced that the doctor had been telling operators about `storage_callback` without ever telling them about `postgres`, conflating cluster ownership with schema ownership at the diagnostic surface. The fix added a row to the substrate, not just to the docs.

*Consumer shape:* a future docs-authoring pass that surfaces a substrate observability gap is licensed to fix the substrate, not just the docs — the docs are a forcing function on the substrate, not an after-the-fact narration of it.

*Consumption conditions:* the pattern holds when the docs author is also (or directly authorized to be) the substrate author. v1.5 ran with this property; v1.6+ phases that decouple docs authorship from substrate authorship should preserve the pattern explicitly via PR-level scope grants, not assume it.

### §4.3 — Substrate/consumer separation is stable across multiple subsystems

The Learning pipeline and the Staged-ops platform are both substrate-not-producer at v1.5 close. Both ship recording / approval machinery; both stay dormant on a stock install until a consumer (projekt-forge in production) wires the produce-side surface. Phase 22 canonicalized this distinction at the documentation surface for both subsystems. Phase 23 reinforced it at the doctor surface for the storage_callback / postgres row split. Phase 23.1 surfaced a third instance: `flame_execute_python` is substrate (the bridge ships the tool; the LLM is the consumer that authors against it).

*Consumer shape:* a v1.6+ subsystem author can rely on substrate/consumer separation as a working architectural pattern across at least three independent subsystems (learning, staged-ops, flame-execute-python). New v1.6 subsystems that adopt the pattern have substrate-shape precedent to reference.

*Consumption conditions:* substrate-not-producer requires the substrate to ship complete + dormant + ready, AND requires a consumer to wire it. The Learning pipeline and Staged-ops platform are dormant on a stock install. `flame_execute_python` is technically dormant differently — it's wired at MCP registration time, but useful only when an LLM agent invokes it. The pattern generalizes; the dormancy shape varies.

### §4.4 — The runtime can support progressive trust

Phase 22's 6 recipes form a progressive-trust operational arc: setup → external-agent wiring (Claude Desktop) → observe substrate behavior (tool synthesis happening live) → drive substrate behavior (multi-step Flame automation via chat) → review what substrate proposed (staged-ops approval/rejection) → inspect what substrate promoted (manifest browsing). Each step extends the operator's trust surface incrementally; no recipe asks for trust the prior recipe hasn't earned.

*Consumer shape:* an operator working through the recipes in order is trained on the substrate's progressive-trust contract as a side effect of doing daily work. v1.6+ surface additions can extend this arc (e.g., schematic-render in v1.6 NODE-SCHEMATIC scope) without reauthoring the trust contract.

*Consumption conditions:* the arc was authored under writer's-room cadence against real production friction. A v1.6 surface that breaks an earlier recipe's trust contract (e.g., introducing autonomous mutation under a surface that previously asked for approval) would be a regression even if the new surface ships cleanly on its own.

### §4.5 — The universal introspection surface is viable once surfaced correctly

`flame_execute_python` existed pre-v1.5 as `execute_python(params: ExecutePythonInput)` in `forge_bridge/tools/utility.py`. The substrate was complete. Two architectural fixes made it viable for LLM consumption: (1) MCP registration + canonical-introspection docstring repositioning; (2) signature flatten so the LLM-visible JSON schema is parseable by the chat model. The combination converted a dormant function into the canonical Flame introspection surface — "the universal Flame introspection and automation surface" / "reflective surface of Flame itself."

*Consumer shape:* an LLM agent receiving the MCP tool list at registration time can reach the `flame_execute_python` tool with a flat 2-parameter call shape, escalate to it when no narrow `flame_*` tool fits, and receive the full BridgeResponse shape (stdout + stderr + result + error + traceback) for next-iteration loop reasoning. The author-walk on portofino verified this end-to-end on the canonical regression query.

*Consumption conditions:* viability is verified under local-32b production routing (sensitive=True). Verification under cloud routing (sensitive=False) is blocked by `SEED-CLOUD-CHAT-AGENTIC-ROUTER-V1.6+`. Viability under operator-authored mutation queries (main_thread=True) is asserted by the tool surface but not exercised in 23.1's author-walk; main_thread=True is operator-author burden, not LLM-author burden at v1.5.

### §4.6 — Author-walk validation + explicit residual seeding is a coherent sufficiency standard

The synthesis-shorthand interpretation of the original forcing function would have been: "non-author UAT must pass." The operational interpretation that emerged through the milestone is: "author-walk validation + explicit residual seeding establishes scope-bound sufficiency for ship; the unseeded residual surface remains the next opening forcing function." Four phases (20, 20.1, 21, 23.1) closed under this standard with author-walk evidence + named seed forward.
