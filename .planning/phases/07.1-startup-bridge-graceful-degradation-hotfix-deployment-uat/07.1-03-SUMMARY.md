## Decisions Made

### Editable-shadow remediation (plan-approved Option A)

**Found during:** Pre-flight sanity check of `pip show forge-bridge` before editing projekt-forge's pin.

**Issue:** The `forge` conda env had an editable install of forge-bridge (`Editable project location: /Users/cnoellert/Documents/GitHub/forge-bridge`). Per plan Step 4 and the D-06 / WR-03 guard, an editable shadow would make `direct_url.json` reference a local `file://` URL, violating the plan's identity-lock acceptance criterion ("`direct_url.json` references `@v1.2.1`").

**Resolution (orchestrator-approved Option A):**
1. `pip uninstall -y forge-bridge` to clear the editable shadow.
2. Proceed with the pin edit + reinstall per the plan.
3. Verify `direct_url.json` references the v1.2.1 tag post-reinstall (confirmed: `requested_revision=v1.2.1, commit_id=abd047c`).

**Rationale (per orchestrator):**
- Shadow was introduced during earlier development (forge-bridge was `pip install -e .`'d during Phase 07.1-01 TDD work, plausibly).
- HEAD of the forge-bridge repo is `f069407` (docs-only commit ahead of the v1.2.1 tag at `abd047c`) — so the shadow's source code was byte-identical to what the tag install would resolve. Remediation is a clean no-behavior-change operation, not a substantive code swap.
- Leaving the shadow in place would have produced a `direct_url.json` with `"dir_info": {"editable": true}, "url": "file:///Users/cnoellert/Documents/GitHub/forge-bridge"` — failing identity-lock AC5 and leaving Plan 04's UAT testing a local-checkout drift rather than the released tag.

### Resolved apparent-shadow scare after reinstall

**Found during:** Step 4 identity verification, first run from `/Users/cnoellert/Documents/GitHub/forge-bridge/` CWD.

**Issue:** `python -c "import forge_bridge; print(forge_bridge.__file__)"` returned `/Users/cnoellert/Documents/GitHub/forge-bridge/forge_bridge/__init__.py` — appearing to show the shadow was still active despite a clean uninstall + reinstall.

**Root cause:** Python's `sys.path[0]` is the empty string (CWD). When the command is run from the forge-bridge repo, the local `forge_bridge/` subdirectory is resolved before site-packages. This is NOT an editable install, NOT a `.pth` file — it's just the standard Python CWD-in-path behavior. Running the identical command from `/tmp` or from `/Users/cnoellert/Documents/GitHub/projekt-forge` (where pytest runs) correctly resolves to site-packages.

**Remediation:** No remediation needed. Re-ran verification from neutral CWDs to confirm the real install.

**Not a deviation** — just a verification-methodology adjustment. The install itself is correct on every CWD except the forge-bridge repo root.
