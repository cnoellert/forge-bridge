# Phase 23.1: Chat Convergence Ship-Blocker (v1.5 milestone-close gate)

**Gathered:** 2026-05-14
**Status:** Ready for execution
**Trigger:** Phase 23 dogfood UAT surfaced a v1.5 ship-blocker that v1.5's Legibility framing did not anticipate but is structurally adjacent — the chat surface that v1.5 made *legible* turns out to be *operationally broken* in a way that documentation alone cannot fix. Decimal-patch precedent: Phase 20.1 (install bootstrap ship-blocker, opened the same way after Phase 20 walks surfaced the gap).

---

## §1 — Phase Boundary

Close two narrowly-scoped v1.5 ship-blocker gaps surfaced during Phase 23's authored writer's-room cadence:

1. **The chat surface can't converge on simple introspection queries.** The dogfood query "What are the clips on Reel 1" hits the chat handler's 120s inner-loop budget without producing an answer. Root cause: the local 32b model has no escape hatch — it can attempt narrow `flame_*` tools (`flame_list_desktop` returns counts but not clip names; `flame_context` may be filtered out by PR14's keyword pre-filter) and loops on whichever it picks. Bridge has the substrate (POST `:9999/exec` accepts arbitrary Flame Python) but doesn't expose it as an MCP tool the LLM can author against.

2. **The CLI's failure-mode message is structurally misleading.** When the chat endpoint returns 504 after exhausting the 120s server-side LLM budget, the CLI retries with whatever total-budget time remains and then reports `"LLM request timed out after 8.6s"` from [forge_bridge/llm/call_wrapper.py:291](forge_bridge/llm/call_wrapper.py:291) and [forge_bridge/llm/call_wrapper.py:471](forge_bridge/llm/call_wrapper.py:471). The 8.6s reads as a threshold; it's actually budget exhaustion downstream of a slow server-side timeout. Operator concludes the client is being too aggressive when in fact the server already gave up.

These two are coupled at the dogfood-failure surface: gap #1 produces the slow server-side timeout; gap #2 misreports it. Fixing #1 alone leaves the misreporting in place for every future timeout; fixing #2 alone doesn't make chat usable. Both ship in 23.1.

**Forcing function:** the same dogfood query that surfaced this phase (`fbridge chat "What are the clips on Reel 1"`) must converge to a structurally correct answer end-to-end against a live Flame on portofino or flame-01. If it doesn't, 23.1 doesn't close, v1.5 doesn't earn its milestone-close evaluation, and the conversation we just had goes back to "bridge is shipped but unusable."

## §3 — Implementation Decisions

### §3.1 — Why utility.py, not a new module

The `forge_bridge/tools/` package has six modules today (project, timeline, batch, publish, utility, reconform, switch_grade). `flame_exec_python` is the smallest possible new tool — adding a `forge_bridge/tools/exec.py` for one function inflates the surface for no gain. `utility.py` already houses `flame_ping`, which is the closest cousin (low-ceremony, escape-hatch-shaped). The new function lives there. If/when v1.6 grows other escape-hatch tools, refactor at that point.

### §3.2 — Why `bridge.execute()`, not `bridge.execute_json()`

`execute_json` requires the snippet to print exactly one JSON object/array and crashes the caller if it doesn't. The LLM is going to write snippets that print free-form text, dictionaries-as-repr, multi-line output, or nothing at all. `execute()` returns the full `BridgeResponse` (stdout + stderr + result + error + traceback) and lets the tool surface render all of it — that's strictly more legible to the model in the next loop iteration than a JSON parse error.

The tool's output format is the JSON-serialized response shape:
```json
{
  "stdout": "...",
  "stderr": "...",
  "result": ...,
  "error": null | "...",
  "traceback": null | "..."
}
```
Stable, kind-aware (the model can tell `error != null` means a Flame Python exception), useful in chat tool-trace rendering downstream.

### §3.3 — Why `main_thread=False` default

99% of LLM-authored uses of `flame_exec_python` are introspection (reading reel contents, querying project metadata, listing batch nodes). The Flame hook's `main_thread=True` schedule is only needed for `set_value` / `create` / `delete` operations. Defaulting to False keeps the common path fast and prevents the LLM from inadvertently mutating Flame state without explicit param-setting. If the LLM needs `main_thread=True`, it asks for it explicitly in the tool call — and that explicit request is auditable in the chat tool trace.

### §3.4 — Why three docstring examples and not more or fewer

One example reads as a hint; the model may try to generalize too literally. Two examples are minimum for the model to see shape variation (read-only vs. one with mutation, structured-print vs. plain-print). Three examples is the load-bearing minimum where the model has enough surface to pattern-match against the user's intent without the docstring growing into a tutorial. More than three is docstring bloat and reduces the model's pattern-extraction signal. The three examples chosen:
- **(Read-only, structured output):** list clip names on a named reel. This is the dogfood query that surfaced 23.1 — the docstring should explicitly enable it.
- **(Read-only, plain output):** print the current Flame version and OCIO config. Pattern-match shape for "look up something simple."
- **(Mutation with main_thread=True):** rename a single segment on a sequence. Pattern-match shape for "do something destructive" plus the explicit `main_thread=True` signal.

### §3.5 — Why the error-message fix preserves the existing `timeline` / `events` structures

`call_wrapper.py` already records per-attempt outcomes in `events` (kind="attempt", duration, result, status_code) and `timeline` (per-step facts). The trace is already truthful — the rendering at the user-visible surface is where the lie is. Rewriting the rendering to compose from the existing data, rather than introducing new state, keeps the fix narrow and the existing tests (timeline/events tests) green.

### §3.6 — Why no unit test for the regression query itself

Writing a hermetic unit test for "chat converges on Flame question with `flame_exec_python` available" is structurally impossible — it depends on the LLM's tool-selection behavior, the Flame hook being live, and the model's stability across runs. The unit-test surface for 23.1 is:
- `flame_exec_python` unit tests: mock `bridge.execute`, assert the tool wraps the response shape correctly + handles BridgeConnectionError. ~3 tests.
- `call_wrapper.py` error-message tests: assert the rendered message names budget exhaustion vs single-attempt timeout correctly. ~2 tests.

The regression smoke is human-author-walked, same model as 20.1's UAT gates. The phase-close evidence is the walk transcript, not a `pytest` line.

### §3.7 — Why the v1.6-FRAMING.md edit is part of this phase

The framing artifact was drafted hours ago in this same conversation and explicitly named Phase 24's scope as including `flame_exec_python`. Phase 23.1 pulls that piece forward. Leaving the framing artifact stale by even one citation creates the kind of downstream drift the "ground specs in actual files" feedback memory exists to prevent. The 5-line footnote update lands in 23.1 alongside the code so the framing stays load-bearing.

## §4 — Verification Gates

23.1 closes when ALL of the following hold:

1. **Unit tests green.** New tests for `flame_exec_python` and the rewritten error-message path pass; existing unit suite passes unchanged. `pytest forge_bridge/tests/` (or equivalent) reports zero new failures.
2. **`fbridge mcp stdio` smoke.** The new tool appears in the tool list; `list_tools` confirms `flame_exec_python` registered with `_source=builtin` and `readOnlyHint=False`.
3. **Author-walk regression query passes on at least one of {portofino, flame-01}.** `fbridge chat "What are the clips on Reel 1"` converges within the 125s outer cap, the chat trace shows the model called `flame_exec_python` (not just `flame_list_desktop`), and the output is a list of clip names. Operator records the converged response in the close summary.
4. **Misleading-error case verified.** Reproduce the original failure mode (e.g., by deliberately running a query that hits the 120s server cap — possibly by removing `flame_exec_python` temporarily, or by sending a query that's genuinely hard). The CLI message now names budget exhaustion after a server 5xx, not "timed out after Ns."
5. **v1.6-FRAMING.md footnote landed.** Phase 24 scope updated to reflect `flame_exec_python` moved up to 23.1.
6. **STATE.md cursor updated.** v1.5 milestone status updated to reflect 23.1 closed; milestone-close evaluation now unblocked.

The phase does NOT close on green tests alone — Gate 3 (live author-walk) is load-bearing per the same principle that closed Phase 20.1: a v1.5 ship-blocker isn't closed by `pytest`, it's closed by an author at a workstation declaring the dogfood path works.

## §5 — v1.5 Milestone-Close Impact

23.1 is *the* gate that converts v1.5's milestone-close evaluation from "no, chat is broken" to "yes, the bridge is operable + legible enough for a non-author operator to independently succeed." Before 23.1, the framing of v1.5 Legibility is half-honest — the bridge has docs that say it works, and the bridge demonstrably doesn't on the second-simplest query.

After 23.1 closes, v1.5 milestone-close evaluation should be straightforward:
- INSTALL works (Phase 20, 20.1).
- GETTING-STARTED, RECIPES, TROUBLESHOOTING exist and have been authored against real production friction (Phases 21, 22, 23).
- The dogfood path that exposed the v1.5 limit case now works (23.1).
- The five user-facing surfaces are each independently testable to a non-author operator.

v1.5 closes on its merits, the planted seeds carry forward, v1.6 opens against the framing artifact, and the writer's-room convergence on §12 of v1.6-FRAMING can proceed without the time pressure of "users can't actually use bridge today."
