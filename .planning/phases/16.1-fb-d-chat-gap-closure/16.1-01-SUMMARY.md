# Phase 16.1 Plan 01: Backend-Aware Tool-List Scoping Filter Summary

**One-liner:** Backend-aware TCP probe filter with 5s asyncio.Lock cache, 7-tool in-process classification, 503 fallback, and 3 structured-log field extensions fixing the 49-tool chat hang (Bug C).

## What Was Built

### Task 1.1 — `_tool_filter.py` + unit tests

New module `forge_bridge/console/_tool_filter.py` with:

- `_IN_PROCESS_FORGE_TOOLS: frozenset[str]` — hardcoded 7 in-process forge_* tool names (`forge_manifest_read`, `forge_tools_read`, `forge_list_staged`, `forge_get_staged`, `forge_approve_staged`, `forge_reject_staged`, `forge_staged_pending_read`)
- `_BACKENDS: tuple` — single backend `("flame_bridge", "127.0.0.1", 9999)` — no :9998 (see classification section below)
- `_probe_backend(host, port) -> bool` — async TCP probe via `asyncio.open_connection`, 1.5s timeout, never propagates exceptions
- `_get_backend_reachability() -> dict[str, bool]` — 5s monotonic cache, `asyncio.Lock`-guarded
- `filter_tools_by_reachable_backends(tools) -> list` — main public API
- `_reset_for_tests()` — test affordance

New test file `tests/console/test_chat_tool_filter.py` with 15 tests (8 behavioral + 7 parametrized in-process coverage).

## Locked Per-Tool Routing Classification

The planner audit (verified against `grep -n 'name="forge_' forge_bridge/console/resources.py`) confirmed the CONTEXT D-01 prefix-only rule was incomplete. The classification implemented:

| Tool class | Backend | Filter rule |
|---|---|---|
| `flame_*` (29 tools) | Flame HTTP bridge :9999 | Drop when :9999 unreachable |
| Flame-dependent `forge_*` (20 builtin tools from `forge_bridge.tools.*`) | Flame HTTP bridge :9999 | Drop when :9999 unreachable |
| In-process `forge_*` (7 tools from `register_console_resources`) | Local SQLAlchemy + ManifestService | Always keep |
| `synth_*` | Synthesizer (in-process) | Always keep |

The `:9998` backend from CONTEXT D-01 is NOT probed. The audit confirmed:
- `forge_query_alternatives` (the only :9998 consumer via switch_grade.py) also calls `bridge.execute_*()` — it's Flame-dependent anyway
- No `forge_*` tools route ONLY to :9998 independently of Flame

**Delta from `<interfaces>` block:** None. The plan's per-tool classification is fully implemented as specified.

## Deviations from Plan

### Pre-existing out-of-scope failure

**`test_ui_chat_handler_renders_panel_template` (Bug A — TemplateResponse):** This test was failing before Plan 16.1-01. The failure is due to `ui_handlers.py` using the deprecated `TemplateResponse(name, ctx)` Starlette signature — exactly Bug A addressed by the Starlette migration plan (Plan 16.1-B or equivalent). This plan does NOT own this fix.

Verified pre-existing by: `git stash && pytest test_ui_chat_handler_renders_panel_template` → same failure on the base commit.

### chat_client fixture passthrough patch (Rule 2 — Missing correctness requirement)

The `chat_client` fixture in `test_chat_handler.py` was updated to also patch `filter_tools_by_reachable_backends` with a passthrough stub. This was required for correctness: without the patch, the real TCP probe runs in test context (no Flame on :9999) and existing tests began returning 503. The passthrough stub ensures the 11 baseline tests continue testing what they tested before (handler behavior, not filter behavior). New filter-specific tests provide their own patches that override the fixture's default.

### `tool_call_count=%d` count discrepancy in acceptance criteria

The plan acceptance criteria states `grep -c "tool_call_count=%d" forge_bridge/console/handlers.py` returns 3. The actual baseline (verified at commit `586af5f`) had `tool_call_count=%d` in only 1 log line (the success path). The timeout and loop_budget log lines never had `tool_call_count` — they only get `tools_offered_count` added by this plan. The Phase 16 contract is fully preserved (field not renamed, not removed), and `tools_offered_count` appears correctly in all 3 log lines.
