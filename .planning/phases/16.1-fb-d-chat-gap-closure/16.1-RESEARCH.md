### Reachability probe — Option A (recommended)

Reuse the existing `httpx.AsyncClient` + `asyncio.wait_for` pattern from `ConsoleReadAPI.get_health._check_flame_bridge` at `forge_bridge/console/read_api.py:215-238`:

```python
# forge_bridge/console/_tool_filter.py (NEW, recommended) — sketch only
import asyncio
import time
import httpx
from typing import Dict

# Tuple form of the prefix → backend map (single source of truth in this file).
_BACKENDS = (
    # (prefix, backend_id, host, port)
    ("forge_", "forge_bridge_ws",   "127.0.0.1", 9998),  # WS server, but TCP probe is sufficient
    ("flame_", "flame_http_bridge", "127.0.0.1", 9999),
)
_PROBE_CACHE_TTL_SEC = 5.0
_PROBE_TIMEOUT_SEC = 1.5

# Cache: backend_id -> (reachable: bool, expires_at_monotonic: float)
_cache: Dict[str, tuple[bool, float]] = {}
_cache_lock = asyncio.Lock()

async def _probe_backend(host: str, port: int) -> bool:
    """TCP-connect probe; return True if backend is reachable within timeout."""
    try:
        reader, writer = await asyncio.wait_for(
            asyncio.open_connection(host, port),
            timeout=_PROBE_TIMEOUT_SEC,
        )
        writer.close()
        await writer.wait_closed()
        return True
    except (OSError, asyncio.TimeoutError):
        return False

async def get_reachable_prefixes() -> set[str]:
    """Return prefixes whose backends are currently reachable, with 5s caching."""
    now = time.monotonic()
    reachable_prefixes: set[str] = {"synth_"}  # always reachable (in-process)

    async with _cache_lock:
        for prefix, backend_id, host, port in _BACKENDS:
            cached = _cache.get(backend_id)
            if cached is not None and now < cached[1]:
                if cached[0]:
                    reachable_prefixes.add(prefix)
                continue
            ok = await _probe_backend(host, port)
            _cache[backend_id] = (ok, now + _PROBE_CACHE_TTL_SEC)
            if ok:
                reachable_prefixes.add(prefix)
    return reachable_prefixes
```

**Why TCP-connect over HTTP-GET:**
- The forge-bridge server on `:9998` is a WebSocket endpoint, not HTTP. A `httpx.get()` would fail with a protocol error even when the server is up. TCP-connect tests "is something listening" — exactly what we want.
- Simpler than the existing `_check_flame_bridge` helper (which uses HTTP GET with a status-code threshold) — Flame bridge IS HTTP, so the planner could optionally use the same pattern for the `:9999` probe. **Recommendation:** TCP-connect for both, uniform shape.

**Cache approach — `time.monotonic()` keyed by backend ID:** matches the cache pattern in `forge_bridge/console/_rate_limit.py:175-212` (verified). Use `asyncio.Lock` to serialize cache mutations under concurrent chat requests; the probe itself is non-blocking inside the lock so contention is microseconds.

### Reachability probe — Option B (alternative; CONTEXT-acceptable)

Reuse `ConsoleReadAPI.get_health()` and parse `services.flame_bridge.status` and `services.ws_server.status`:

```python
# Inside chat_handler:

flame_ok = health["services"]["flame_bridge"]["status"] == "ok"
forge_ok = health["services"]["ws_server"]["status"] == "ok"
```

**Tradeoff:**
- **Pro:** zero new code; reuses an already-tested probe surface; no caching layer to maintain (`get_health()` already does sub-50ms steady-state).
- **Con:** `get_health()` checks 7 services, including DB and watcher. Cost per chat request is higher than a single TCP probe. On CHAT-04 latency budget (`<60s`) this is fine, but on a tight chat loop it's wasteful.
- **Con:** the health check key names (`flame_bridge`, `ws_server`) are an implicit coupling — if Phase 9's health surface ever renames a service, the chat filter silently breaks.

**Recommendation:** Option A. It's the simpler, more isolated, more performant path; the 5s monotonic-clock cache prevents flooding; the helper module is testable in isolation; CONTEXT D-02 explicitly calls option A "recommended."

### Stable tool-list ordering for deterministic re-runs

`mcp.list_tools()` returns tools in registration order — which is itself stable because `register_builtins(mcp)` runs at module import time and orders are deterministic in the source file (`registry.py:282+`). To slice the first N tools deterministically:

```python
def _build_tool_list(n: int) -> list:
    """Slice the real registered MCP tool list to the first N tools.

    Order is registration order (deterministic across runs because
    register_builtins runs at module import time).
    """
    from forge_bridge.mcp import server as _mcp_server
    all_tools = asyncio.run(_mcp_server.mcp.list_tools())
    # Sort alphabetically as a defense against future registration-order
    # drift; bisection re-runs sample the same tool set.
    return sorted(all_tools, key=lambda t: t.name)[:n]
```

**Recommendation:** sort alphabetically. The cost is one O(N log N) sort per measurement (negligible at N=49). The benefit is bisection is reproducible across days, machines, and code changes that reshuffle `register_builtins`.

### NEW pitfalls introduced by 16.1

#### Pitfall 6: Probe cache TTL race under concurrent chat requests
**What goes wrong:** Two chat requests arrive within the 5s probe-cache TTL. Both check the cache; both find it expired; both fire a TCP probe; both write to the cache. Because the probes are async, the second write may overwrite the first with a different result if the backend's reachability state flipped between the two probes (rare but possible under flaky network).
**Why it happens:** the cache lock is acquired around the entire critical section, but if the planner forgets the lock, async-context interleaving makes the cache inconsistent.
**How to avoid:** wrap the cache mutation in `asyncio.Lock` per the §2 sketch above. The lock holds for the duration of the probe (~1.5s max) — chat requests are at most 1/min/IP per CHAT-01 rate limit, so contention is irrelevant.
**Warning signs:** tests with `asyncio.gather(*[chat_handler(req) for _ in range(10)])` produce probe results that differ across requests. Add a `concurrent_probes` test in the new `_tool_filter` module's unit tests.

#### Pitfall 7: Boot-wiring smoke test triggers real `startup_bridge` connection attempt
**What goes wrong:** `_lifespan` calls `await startup_bridge()` at line 96, which tries to WS-connect to `:9998`. In a test harness with no `:9998` server, this either hangs (10s `wait_until_connected` timeout per `mcp/server.py:239`) or logs a noisy WARNING.
**Why it happens:** `_lifespan` is the boot sequence; bypassing pieces of it defeats the test.
**How to avoid:** patch `startup_bridge` and `shutdown_bridge` with `AsyncMock(return_value=None)` per the §4 sketch. The test still exercises the ConsoleReadAPI construction and llm_router wiring (the gap-target). Note: this is a tradeoff — patching out `startup_bridge` means the test does NOT catch a Bug B-class failure that originates in startup_bridge itself. Acceptable per CONTEXT D-07 #1 ("~10 LOC, single test").
**Warning signs:** boot-wiring smoke test hangs ~10s before passing — `startup_bridge` wasn't patched; the test is exercising too much of the real boot path. Add the patch.

#### Pitfall 8: UI-render smoke test depends on Jinja2 template inheritance not breaking
**What goes wrong:** the test asserts `id="health-strip"` is in every rendered page. If `shell.html`'s `{% include "fragments/health_strip.html" %}` ever moves to a different template, every test in the parameterized list fails together.
**Why it happens:** the `id="health-strip"` invariant is shared across all UI pages by inheritance, not by per-page assertion.
**How to avoid:** document the invariant in the test docstring with a pointer to Phase 10.1 D-39. Any future change to the shell template that drops the health strip should fail this test BEFORE deploy — that's the test working correctly.
**Warning signs:** developer changes shell.html to remove the health strip; UI-render smoke fails on ALL routes. Either restore the health strip or update the test sentinel (and update Phase 10.1 D-39 invariant if the change is intentional).

#### Pitfall 9: Strategy B chat E2E test assumes `qwen2.5-coder:32b` is loaded in VRAM
**What goes wrong:** the test runs against assist-01 with FB_INTEGRATION_TESTS=1, but Ollama hasn't preloaded the model. First call triggers a model load — adds 10-30s to first `complete_with_tools()` call. The 60s assertion fails not because of a regression but because of cold-start.
**Why it happens:** Ollama's default behavior is lazy-load on first request. `ollama ps` shows the model is "in cache" but not necessarily "in VRAM" until a request fires.
**How to avoid:** either (a) document operator preconditions: "before running the Strategy B suite on assist-01, run `ollama run qwen2.5-coder:32b 'warm'` to preload the model"; or (b) the fixture itself can fire a warm-up request before the actual test (cold-start surface area moves to fixture, not assertion). **Recommendation:** (a) — operator precondition is documented in `16.1-HUMAN-UAT.md` and matches the existing Phase 15 LLMTOOL-01 test pattern (which also assumes model is warm).
**Warning signs:** Strategy B test passes on second invocation (model now warm) but fails on first. Add a `@pytest.fixture(scope="session")` warm-up call.

### Bug D class — what to also guard against

The Phase 16 deploy bugs surfaced in this order: A (TemplateResponse) → B (LLMRouter wiring) → C (49-tool hang). A common thread runs through them: **automated tests bypassed the real boot/install paths.** Specifically:

- **Bug A:** Tests run against the dev env's pinned Starlette; never exercise the fresh-install dependency resolution.
- **Bug B:** Tests inject `llm_router=AsyncMock(...)` directly into `ConsoleReadAPI(...)`, bypassing `_lifespan`'s real construction.
- **Bug C:** Tests inject a small synthetic tool list, bypassing the real `mcp.list_tools()` registry.

**The Bug D class** is anything else where automated coverage substitutes a small mocked surface for the real shipped surface, and that substitution hides a real failure. Candidates:

- **Bug D-1: `app.state` middleware ordering** — chat handler reads `request.app.state.console_read_api._llm_router`. If a future middleware reorders `app.state` initialization (e.g., adds a wrapper around `build_console_app`), the chat handler may NOT see the router. The boot-wiring smoke (D-07 #1) catches this.
- **Bug D-2: Real-Ollama tool-schema validation** — the Strategy B chat E2E (D-07 #3) tests the WHOLE registry against live Ollama. If a future tool registration introduces a schema field that Ollama can't parse (e.g., a JSON Schema feature qwen2.5-coder rejects), Strategy B fails. This is exactly the Bug C class.
- **Bug D-3: `pip install -e .` with dependency drift** — the deploy bug surface. **No automated test catches this** at the source-tree level. The closest gate is the D-12 fresh-operator UAT (which is a manual operation on a fresh assist-01). The pin removal in D-09 is the long-term fix; in the short term, the human UAT is the gate.

**Recommendation:** the three guards in D-07 cover D-1, D-2, and D-3 by construction. No additional 16.1 scope needed. Document the Bug D class in the 16.1 SUMMARY so future gap-closure phases don't reinvent the analysis.
