## Implementation Decisions

All twelve decisions below are **locked** — derived directly from the 5 recommendations the user accepted across G-1..G-5. The planner sizes plans/tasks around them.

### G-1: Tool-list scoping policy for chat (the Bug C fix)

- **D-01 (locked):** Chat handler implements **backend-aware filtering** at request time. The handler probes which backends are reachable and drops the corresponding tool prefixes from the list it passes to `complete_with_tools()`. Concretely:
  - If projekt-forge bridge (HTTP `:9998` per existing topology) is unreachable → drop all `forge_*` tools (21 tools)
  - If Flame bridge (HTTP `:9999`) is unreachable → drop all `flame_*` tools (28 tools)
  - On assist-01 reality (no Flame, no projekt-forge bridge): chat sees only the manifest/registry/staged-ops tools that actually have working backends in the bridge process itself
  - Tools whose backend is the bridge process itself (manifest reads, staged-ops, health) are always available regardless of remote-backend reachability
- **D-02 (locked, planner picks shape):** Reachability probe approach. The planner picks one of:
  - **Option A (recommended):** Lightweight TCP/HTTP probe with a short cache TTL (e.g. 5s). Single probe per request → cached for next 5s → no per-request flooding of the upstream. Wired into `chat_handler` before `mcp.list_tools()`.
  - **Option B:** Use the existing health-check pathway (`forge://health` resource / `/api/v1/health`) as the source of truth. If health says backend is up, include its tools.
  - The choice MUST be documented in the plan with reasoning — both are acceptable; option A is simpler, option B reuses an existing surface. **No new dependency** in either case.
- **D-03 (locked, conditional escalation):** If the bisection (D-04 below) reveals the hang persists at small tool counts even with `forge_*` and `flame_*` removed (e.g. the model chokes on the ~10 manifest/staged-ops tools too), escalate to a hybrid policy: backend-aware filtering **plus** a hard cap (e.g. 15 tools) **plus** a static "core" priority list. The planner re-opens this gray area only if bisection forces it; default path is D-01 alone.

### G-2: Bisection scope and methodology

- **D-04 (locked):** Bisect by **tool count** first. Run `complete_with_tools()` on assist-01 against live `qwen2.5-coder:32b` with tool counts {5, 10, 15, 20, 25, 30, 35, 40, 49} (or a smarter binary-search variant — planner's call). Each measurement records: tool count, prompt tokens, completion tokens, elapsed seconds, stop reason, iteration count. Time-box: **30 minutes wall-clock for the bisection itself** (not counting test-harness setup). If the data shows a clean threshold (e.g. fast through 20, hangs at 25+), that's the answer and we move on.
- **D-05 (locked, conditional):** Escalate to **content-bisection** ONLY if count-bisection shows random-looking variance (e.g. fast at 49 minus tool X, slow at 49 minus tool Y). Content-bisection is "drop one tool at a time and time the call." Skip otherwise — a clean count threshold is sufficient signal for D-01's tool-list scoping policy to be the correct fix.
- **D-06 (locked):** Bisection lands as a deterministic Strategy-B-gated test in `tests/integration/test_chat_tool_list_threshold.py`. Gate: `FB_INTEGRATION_TESTS=1` plus the existing live-Ollama skipif gate (model present, daemon reachable). The test is parameterized over the count list and asserts: at the chosen scoping count, `complete_with_tools()` returns within the per-call budget with `stop_reason="end_turn"` (not `"loop_budget_exceeded"`) and non-zero `completion_tokens`. Future regressions rerun this test; it is the live signal that the chat surface is healthy on a real registry.

### G-3: Regression-test surface

- **D-07 (locked):** Three regression guards land in 16.1, each as part of the relevant fix plan (NOT a separate "tests-only" plan):
  1. **Boot-wiring smoke test** (always-on, ~10 LOC). Builds the console app via the existing `build_console_app(...)` helper, drives `_lifespan` via TestClient, asserts `app.state.console_read_api._llm_router is not None` after lifespan enters. Lives in `tests/console/test_lifespan_wiring.py` (new file). This catches the Bug B class.
  2. **UI-render smoke test** (always-on, ~20 LOC). Drives TestClient through every `/ui/*` route registered in `app.py`, asserts each returns 200 and contains the expected shell markers (the `id="health-strip"` invariant from Phase 10.1 D-39 is a natural sentinel). Lives in `tests/console/test_ui_handlers_render.py` (new file). This catches the Bug A class — any `TemplateResponse` signature drift fails this test before deploy.
  3. **Strategy B chat E2E test** (gated on `FB_INTEGRATION_TESTS=1` plus live-Ollama skipif). Drives the full chat path against real `mcp.list_tools()` and live Ollama; asserts a useful response within budget. Lives in `tests/integration/test_chat_endpoint_live.py` (new file — adjacent to the existing `test_chat_endpoint.py` Strategy-A tests). The test reuses the Strategy-B gate pattern from `tests/integration/test_complete_with_tools_live.py:49-66` (Phase 15 precedent).
- **D-08 (locked):** All three tests are wired into the default `pytest` discovery path. The Strategy B test self-skips when its gates aren't satisfied — no separate invocation, consistent with Phase 15's pattern. CI is unaffected (Strategy B tests don't run without `FB_INTEGRATION_TESTS=1`).

<specifics>
## Specific Ideas

- **Backend-aware filtering is THE choice because of assist-01's reality:** assist-01 runs forge-bridge alone. No Flame on `:9999`. No projekt-forge bridge on `:9998`. Every `forge_*` and `flame_*` tool the LLM picks would fail at execution anyway. The fix isn't just "make the chat fast" — it's "stop offering the model tools that route to dead backends." The user's preview file makes this explicit: "on bare assist-01 (no projekt-forge bridge on :9998, no Flame), every forge_* and flame_* tool the model picks would fail at execution anyway. CHAT-04 may need tool-list scoping — exposing only tools whose backends are reachable — before the artist UAT can succeed in principle."
- **The 30-minute bisection time-box is non-negotiable** because the planning value of the bisection is "what's the threshold and is the fix the right shape?", not "characterize the entire failure surface." If 30 minutes doesn't produce a clean threshold, the data is telling us content matters more than count, and we escalate via D-05.
- **Three guards (boot-wiring + UI-render + Strategy B chat E2E) all land in 16.1** because each catches a distinct bug class that automated coverage missed in Phase 16 deploy. Boot-wiring → Bug B class. UI-render → Bug A class. Strategy B chat E2E → Bug C class. Single phase, three small tests, three different failure modes covered.
- **TemplateResponse migration is mechanical and the pin is real tech debt** — keeping it in 16.1 avoids the "ship a fix, leave a known stale pin in pyproject.toml, route it to a v1.4.x patch milestone" anti-pattern. The migration is the only way to drop the pin; doing it now keeps Phase 16 closed cleanly.
- **The fresh-operator gate is non-negotiable for v1.4 close** — Phase 10.1 taught us that artist-facing surfaces need a hard gate, not a soft developer-as-operator UAT. Chat is the most artist-facing of all v1.4 surfaces; the gate has to be hard.

</specifics>
