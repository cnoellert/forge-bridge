## G-1: Tool-list scoping policy for chat

| Option | Description | Selected |
|--------|-------------|----------|
| (a) Backend-aware filtering | Drop `forge_*` if projekt-forge bridge unreachable; drop `flame_*` if Flame unreachable. Keeps registry whole; chat sees only what works. Best matches assist-01's "no Flame, no projekt-forge bridge" reality. | ✓ |
| (b) Static "core" tool set for chat | Pin a small allowlist (manifest reads, registry queries, staged-ops listing). Other tools surface in MCP-direct usage but never in chat. | |
| (c) LLM-assisted routing | Meta-tool that lets the model ask "which tools are available for X?" and pull schemas only for the relevant subset. | |
| (d) Hard cap on tool count | Pass at most N tools (10? 15?) with some prioritization rule. | |
| (e) Hybrid | Multiple of the above (e.g. backend-aware + hard cap). | |

**User's choice:** (a) Backend-aware filtering — recommendation accepted verbatim.

**Notes:** The discussion log preserves option (e) as a conditional fallback per D-03 — if bisection (D-04) reveals the hang persists with `forge_*` and `flame_*` removed, escalate to a hybrid policy with hard cap and static-core priority. The default path is D-01 alone.

| Option | Description | Selected |
