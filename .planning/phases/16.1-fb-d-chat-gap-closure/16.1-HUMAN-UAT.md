## Bug D — new structural failure

The artist-visible chat surface terminates after the LLM emits a tool call. The raw tool-call JSON is rendered to the user as if it were the assistant's terminal text response. There is no evidence of the tool-call execution loop iterating: no tool result, no follow-up LLM turn, no synthesized natural-language answer.

This is **distinct from Bugs A/B/C** that Phase 16.1 closed:

- Plans 01–04 successfully fixed the backend-aware tool-list filter, the TemplateResponse migration, the lifespan wiring guard, and the threshold bisection. All regression guards remain green. Those bugs are not regressing here.
- The remaining failure mode is in the tool-call execution / loop / rendering pathway — code that Phase 16.1 did not touch. Three plausible loci:
  - (a) `LLMRouter.complete_with_tools()` may be terminating after a single Ollama turn when the model emits a tool call rather than continuing the agentic loop (execute tool → re-prompt with result → repeat until `end_turn`)
  - (b) The chat handler may not be wired to actually execute the chosen tool against `mcp.call_tool()` and feed results back into the loop
  - (c) The UI may be receiving a structured tool-call response but rendering it verbatim as text instead of running through the agentic loop

**Strategy B test gap (recorded for follow-up):** the locked Strategy B chat E2E test at `tests/integration/test_chat_endpoint_live.py` PASSED in 2.02s on assist-01 even though the canonical prompt fails the artist UAT. The test asserts only structural shape (`stop_reason == "end_turn"`, content length ≥ 40, elapsed < 60s) — the raw tool-call JSON satisfies all three. Strengthening this assertion is part of the Phase 16.2 closure work.
