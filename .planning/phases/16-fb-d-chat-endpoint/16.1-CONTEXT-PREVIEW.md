### Bugs found in deploy (in order)

**Bug A — Starlette 1.0.0 incompat (FIXED, commit `cfc39f2`):**
- `pip install` resolved Starlette 1.0.0 (the dev env had an older version pinned by historical install).
- Starlette 1.0.0 removed the back-compat shim for `TemplateResponse(name, ctx)` deprecated since 0.30.
- Every UI page rendered `TypeError: unhashable type: 'dict'` from Jinja's template cache key (the dict context was being interpreted as the template name).
- Pinned `starlette<0.53` in `pyproject.toml`. assist-01 ran `pip install 'starlette==0.52.1'` to recover; future fresh installs honor the pin.

**Bug B — LLMRouter never wired into ConsoleReadAPI (FIXED, commit `60d28fa`):**
- `chat_handler` at `forge_bridge/console/handlers.py:511` short-circuits with 500 when `app.state.console_read_api._llm_router is None`.
- `ConsoleReadAPI(...)` was constructed at `forge_bridge/mcp/server.py:_lifespan` line 133 without the `llm_router=` kwarg, so the chat path never had a router.
- Wired `LLMRouter()` instantiation + `llm_router=` kwarg into `_lifespan`. `LLMRouter.__init__` is pure env-reading (no I/O at construction), safe to always-construct.
- The dev-machine "That's working" checkpoint approval during plan 16-05 was a misread: the error banner we explained as "no Ollama → 504 fallback" was actually this 500-internal-error path. We never read the dev server logs.

**Bug C — chat_handler hangs on full MCP tool list (DEFERRED, the reason for 16.1):**
- With A+B fixed and Ollama healthy (qwen2.5-coder:32b loaded, `ollama run "say hi"` returns in 1.4s), `complete_with_tools()` hangs the full 120s wall-clock with `prompt_tokens_total=0` / `completion_tokens_total=0`.
- 1-tool synthetic test via `complete_with_tools()` returns in 2.3s. So the native `ollama` Python client + tool-calling are fundamentally working.
- `mcp.list_tools()` returns **49 tools** (21 `forge_*` + 28 `flame_*`). Passing those to `complete_with_tools()` reproduces the hang.
- Bisection between 1 and 49 tools **not yet performed** — that's a 16.1 task.

## Open questions for `/gsd-discuss-phase 16.1`

These are the gray areas that the discuss-phase agent should drive to a decision:

### G-1: Tool-list scoping policy for chat
- Current behavior: `chat_handler` calls `mcp.list_tools()` and passes ALL 49 tools to `complete_with_tools()`. This is the obvious thing to do but is the proximate cause of Bug C.
- Options to consider:
  - **Backend-aware filtering** — at request time, filter out `forge_*` tools if projekt-forge bridge unreachable, filter out `flame_*` if Flame unreachable. Keeps the registry whole; chat just sees what works.
  - **Static "core" tool set for chat** — define a small subset (manifest reads, registry queries) that's always exposed to chat regardless of backend availability. Other tools surface in MCP-direct usage but not chat.
  - **LLM-assisted tool routing** — meta-tool that lets the LLM ask "which tools are available for X?" and only fetches schemas for the relevant subset.
  - **Hard cap on tool count** — pass at most N tools (N=10 or 15?), with some prioritization rule.
- Which policy? What's the prioritization rule? How does the artist's question about "synthesis tools" actually get answered — is there a tool, or is the manifest a resource the LLM reads?

### G-2: Is the hang count or content?
- Need a bisection: 5 tools, 10, 20, 30, 40, 49 → where does it tip from "fast" to "hang"?
- If clean count threshold (e.g., works at 20, hangs at 30) → tool-schema bloat is the issue, fix is trimming.
- If random tools cause hangs (e.g., works with 49 minus tool X) → specific schema content issue, fix is repairing that schema.
- This is a 30-minute synthetic-test loop, not a planning question — but the FINDING shapes the right fix.

### G-3: How do we make sure this doesn't happen again?
- Add a Strategy B integration test (`FB_INTEGRATION_TESTS=1` gated) that exercises `chat_handler` end-to-end with the real `mcp.list_tools()` registry + live Ollama, asserting a useful response. Mocked-router-only coverage missed all three deploy bugs.
- Add a TestClient + `_lifespan` smoke test that asserts `app.state.console_read_api._llm_router is not None` after boot. Single test, ~10 LOC.

### G-4: Starlette migration
- Migrate all `forge_bridge/console/ui_handlers.py` callers from `TemplateResponse(name, ctx)` → `TemplateResponse(request, name, ctx)`. ~8 mechanical sites.
- Drop the `starlette<0.53` pin once migration verified.
- Standalone task or part of 16.1?

### G-5: Re-attempt CHAT-04 on assist-01
- Once 1–4 are done, the artist UAT runs again. What's the canonical re-test prompt? Same ("what synthesis tools were created this week?") or revised?
- What's the threshold for "useful answer" — does a 504 timeout still trigger 16.2, or is the bar just "produces *any* coherent prose"?
