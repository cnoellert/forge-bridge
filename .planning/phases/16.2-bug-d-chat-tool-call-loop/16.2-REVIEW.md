**Reviewed:** 2026-04-27
**Depth:** standard

## Summary

Phase 16.2's Bug D salvage is well-scoped, defensive, and thoroughly tested.
The `_try_parse_text_tool_call` helper correctly mirrors the existing
structured-parse path (line ~415), never raises, and respects the established
"NEVER raises — return None on any parse failure" contract documented in
its own docstring. The salvage hook in `OllamaToolAdapter.send_turn()` is
guarded by both `not tool_calls` AND `text` (so it cannot regress an already
healthy turn), correctly clears `text` after consumption to prevent
double-emission, and is contained entirely inside `OllamaToolAdapter` —
`AnthropicToolAdapter` is untouched and shares no code path with the
salvage helper.

The captured fixture in `test_ollama_adapter.py` matches the recorded
JSON in `16.2-CAPTURED-OLLAMA-RESPONSE.json` byte-for-byte. The
`_BUG_D_TERMINAL_JSON_RE` regex in the integration test is surgical and
will not false-positive on natural prose, numeric responses, or
markdown-fenced code blocks. The negative-case companion test
(`test_plain_text_terminal_response_not_misclassified`) is a real
guard, not a tautology.

The findings below are minor — coverage gaps in the salvage helper's
input space (markdown-fenced JSON, trailing prose after the JSON object,
`ref` collision when the salvaged call is later folded into a turn that
also produced structured calls) and one duplicated-docstring lint. None
are blocking; all are recommendations for hardening.

## Warnings

### WR-01: Markdown-fenced or prose-prefixed Bug D variant slips past salvage AND past the regex guard

**File:** `forge_bridge/llm/_adapters.py:142-170` and `tests/integration/test_chat_endpoint_live.py:95`

**Issue:** Both the salvage helper and the integration regex require the
content to start with `{` (after `.strip()` / `^\s*` respectively). The
captured assist-01 fixture happens to start with a bare `{`, but
qwen2.5-coder is also known to wrap tool-call JSON in a markdown fence
(```` ```json\n{...}\n``` ````) or to prepend a sentence ("Here you go: ...").
Either variant would:

1. Slip past `_try_parse_text_tool_call` (returns `None` because
   `stripped.startswith("{")` is false), and then
2. Slip past `_BUG_D_TERMINAL_JSON_RE` in the integration test (no `^{`
   match), so the integration test would *pass* on a regression that
   still presents the exact same operator-visible Bug D behavior.

This is a defense-in-depth concern, not a known regression — the captured
fixture is the only documented Bug D shape and the helper handles it
correctly. But the bug class is "qwen2.5-coder emitted a tool call as
text"; restricting both detector and salvager to a single textual
prefix means a closely related variant from the same model would
re-open Bug D with no test signal. CONTEXT D-07 explicitly rejects
*natural-prose detection*, but a markdown fence or a single-sentence
prefix is still mechanical/structural — the same shape, different
wrapper.

**Fix:** Either accept the narrow scope explicitly (document that the
fix only covers the bare-JSON variant and that markdown-fenced /
prose-prefixed variants are deferred), OR widen the salvage prefilter
to scan for the first `{` in the string and attempt to parse a JSON
object from that point. A minimal version:

```python
# Cheap pre-filter: must contain a JSON object before paying json.loads cost.
brace = stripped.find("{")
if brace == -1:
    return None
try:
    parsed = json.loads(stripped[brace:])
except (json.JSONDecodeError, ValueError):
    # JSON object may have trailing prose; try a raw_decode pass.
    try:
        parsed, _ = json.JSONDecoder().raw_decode(stripped[brace:])
    except (json.JSONDecodeError, ValueError):
        return None
```

The integration regex should mirror whatever scope the helper accepts —
they must agree on what counts as "Bug-D-shaped" or the test stops
being a regression guard for the helper.

If the team prefers to defer this, add an explicit comment in the
helper noting "covers bare-JSON variant only; markdown-fenced and
prose-prefixed variants intentionally deferred to a future phase."

### WR-02: Salvaged tool-call `ref` is hard-coded to `"0:{name}"`, can collide with structured tool calls in mixed turns

**File:** `forge_bridge/llm/_adapters.py:167, 484-488`

**Issue:** The salvage helper hard-codes `ref=f"0:{name}"` because, per
the inline comment, "idx is always 0 — the salvage path emits one tool
call per turn." That is true *today* because the salvage path only fires
when `not tool_calls` (the structured field was already empty). But the
guard at line 484 (`if not tool_calls and text`) and the helper itself
are loosely coupled — a future contributor who relaxes the guard (e.g.,
"also salvage when the model emitted N structured calls but appended
a text-shaped one") would silently produce `ref="0:foo"` on the
salvaged call while the structured path also produced a `ref="0:bar"`
on its first call. Ollama's append_results uses ORDER-based matching
(verified in `test_order_preserved_across_results`), so this would
surface as misordered tool results, not a hard error — exactly the
class of bug Phase 16.2 is closing.

This is a hardening concern, not a current bug. The current invariant
is correct; it's the *evolvability* of the invariant that is fragile.

**Fix:** Either compute `ref` from the caller's context (pass `idx` in)
or assert the precondition explicitly so a future change fails loudly:

```python
# In send_turn, after the existing structured-parse loop:
if not tool_calls and text:
    salvaged = _try_parse_text_tool_call(text)
    if salvaged is not None:
        # _try_parse_text_tool_call returns ref="0:{name}"; this is only
        # safe because the structured path produced zero tool_calls. If
        # this guard is ever relaxed, recompute ref with the correct idx.
        assert not tool_calls, "salvage ref collision risk"
        tool_calls.append(salvaged)
        text = ""
```

Or, more cleanly, compute the ref at the call site:

```python
if not tool_calls and text:
    salvaged = _try_parse_text_tool_call(text)
    if salvaged is not None:
        # Reassign ref using the correct index (always 0 here, but explicit).
        salvaged = _ToolCall(
            ref=f"{len(tool_calls)}:{salvaged.tool_name}",
            tool_name=salvaged.tool_name,
            arguments=salvaged.arguments,
        )
        tool_calls.append(salvaged)
        text = ""
```

The dataclass is mutable (line 85, `@dataclass` not `@dataclass(frozen=True)`),
so direct field reassignment also works — but rebuilding is more honest
about intent.

### IN-01: `_try_parse_text_tool_call` `args` defaulting silently swallows malformed-but-recognizable arguments

**File:** `forge_bridge/llm/_adapters.py:157-165`

**Issue:** When `parsed["arguments"]` is a string that fails `json.loads`,
or is some other non-dict type, the helper silently substitutes `{}`.
This mirrors the structured path at lines 466-470 (which also silently
falls back), so the behavior is intentional and consistent — but it
means an LLM that emits *almost* the right shape (e.g., `arguments`
as a JSON-like string with a syntax error, or as a list) will be
salvaged into a tool call with empty arguments, which the tool will
likely reject. The model then sees a generic error and may not
self-correct.

This is not a bug — the existing structured path has the same property
and the duplicate `is_error=True` repeat-call detection in
`router.py:464` will eventually break the loop. Worth noting that the
log signal is silent: there's no WARNING that the salvage path
discarded malformed arguments. A debug-level log line here would help
operators triage future model regressions.

**Fix:** Optional — add a debug log when the args fallback fires:

```python
if isinstance(args, str):
    try:
        args = json.loads(args)
    except (json.JSONDecodeError, ValueError):
        logger.debug(
            "_try_parse_text_tool_call: dropped malformed string arguments "
            "for tool=%r; using empty dict",
            name,
        )
        args = {}
```

### IN-03: Negative-case companion test verifies behavior, not internals — could miss a regression where helper raises on prose

**File:** `tests/llm/test_ollama_adapter.py:331-351`

**Issue:** `test_plain_text_terminal_response_not_misclassified` asserts
`resp.tool_calls == []` and `resp.text.startswith("The synthesis tools")`.
This is the right end-to-end assertion, but it would *also* pass if a
future refactor of `_try_parse_text_tool_call` started raising on
unparseable text and the exception was swallowed by `OllamaToolAdapter`
in some new try/except. The helper's own contract ("NEVER raises") is
load-bearing — a direct unit test on the helper would lock that contract.

**Fix:** Optional — add a direct test on the helper:

```python
from forge_bridge.llm._adapters import _try_parse_text_tool_call

class TestTryParseTextToolCall:
    def test_returns_none_on_plain_prose(self):
        assert _try_parse_text_tool_call("Hello world") is None

    def test_returns_none_on_empty_string(self):
        assert _try_parse_text_tool_call("") is None

    def test_returns_none_on_json_array(self):
        assert _try_parse_text_tool_call('["a", "b"]') is None

    def test_returns_none_on_dict_without_name(self):
        assert _try_parse_text_tool_call('{"foo": "bar"}') is None

    def test_returns_none_on_name_not_string(self):
        assert _try_parse_text_tool_call('{"name": 42}') is None

    def test_never_raises_on_garbage(self):
        # Property: must not raise for any string input
        for s in ("{", "}", "{{{", '{"name":', "\x00\x01"):
            assert _try_parse_text_tool_call(s) is None
```

This complements the existing adapter-level test and pins the
"NEVER raises" contract directly.

### IN-04: `_BUG_D_TERMINAL_JSON_RE` `re.IGNORECASE` flag is unnecessary

**File:** `tests/integration/test_chat_endpoint_live.py:95`

**Issue:** The regex `r'^\s*\{\s*"name"\s*:'` has no alphabetic
characters that vary by case. The `re.IGNORECASE` flag has no effect
on the literal `"name"` (lowercase) and is misleading — it suggests the
regex would match `{"NAME": ...}` or `{"Name": ...}`, but those are
*not* the Bug D shape (the model emits exactly `"name"`, lowercase, per
the OpenAI/Ollama function-call schema). Removing the flag matches the
spec more precisely.

**Fix:** Optional — drop the flag:

```python
_BUG_D_TERMINAL_JSON_RE = re.compile(r'^\s*\{\s*"name"\s*:')
```

If the intent *was* to catch `"Name"` / `"NAME"` variants, document
that explicitly — but the captured fixture and the OpenAI tool-call
spec both use lowercase, so the broader match would be loosening the
guard for no clear reason.
