## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `_lifespan` startup → `get_async_session_factory()` | The factory is constructed from environment (`FORGE_DB_URL`); env-source is trust boundary. |
| `ConsoleReadAPI.get_staged_ops` → `StagedOpRepo` | The session_factory is invoked per call; no untrusted input crosses this boundary at this layer (filter values are validated upstream by handlers/tools in Wave 2). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-02-01 | I (Information disclosure) | `_lifespan` startup logs | mitigate | `get_async_session_factory()` is silent on startup; if it ever throws (e.g., malformed `FORGE_DB_URL`), the existing top-level `_lifespan` exception handler catches and logs `type(exc).__name__` only per Phase 8 LRN — never `str(exc)` (could include credentials in connection string). Verify by reading `_lifespan` outer try/except. |
| T-14-02-02 | D (Denial of service) | Per-call `async with self._session_factory()` | mitigate | Each call opens + closes a session; pool sizing is owned by the SQLAlchemy engine (default pool: 5). For v1.4 staged-ops volume (dozens/day per CONTEXT.md), pool exhaustion is not a real threat. v1.5 connection-pool tuning is a deferred concern. |
| T-14-02-03 | E (Elevation) | Lifecycle bypass via direct `session_factory` use | accept | Anyone with a `session_factory` reference can `session.add(DBEntity(entity_type='staged_operation', ...))` directly, bypassing `StagedOpRepo`. This is the FB-A repo contract (D-08): the rule is enforced by code review, not at runtime. FB-B handlers and tools are audited to use `StagedOpRepo` exclusively (Plans 14-03, 14-04 verification). The threat is real but accepted — preventing it would require a runtime guard that adds complexity disproportionate to the risk for a single-tenant local-first system. |
| T-14-02-04 | T (Tampering) | `session_factory` returned by `get_async_session_factory()` is shared across `ConsoleReadAPI`, `build_console_app`, and `register_console_resources` | accept | All three consumers use the factory to open NEW sessions per call (`async with factory() as session`); the factory itself is a stateless callable. No shared state is mutated; threat is structurally absent. |
</threat_model>
