```python
    def set_storage_callback(self, fn: Optional[StorageCallback]) -> None:
        """Register (or clear with None) a single best-effort storage callback.

        The JSONL log is source-of-truth; the callback is a best-effort mirror.
        A failing callback is logged at WARNING level and never disrupts the
        JSONL append.

        The callback may be sync (returns None) or async (returns Awaitable[None]).
        Dispatch mode is detected once here via inspect.iscoroutinefunction and
        cached — per-record dispatch does not re-inspect.

        An async callback requires record() to be called from a running event loop.
        When no loop is present at dispatch time, asyncio.ensure_future raises
        RuntimeError which is caught and logged as a warning.

        Args:
            fn: Callable taking an ExecutionRecord and returning None or an
                Awaitable[None]. Pass None to clear a previously-set callback.
        """
        if fn is None:
            self._storage_callback = None
            self._storage_callback_is_async = False
            return
        self._storage_callback = fn
        self._storage_callback_is_async = inspect.iscoroutinefunction(fn)
```

                        exc_info=True,

            return True
        return False

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ExecutionLog.record() → storage callback | Untrusted-function boundary: callback code lives in the consumer (e.g., projekt-forge), not in forge-bridge. A failing callback must never corrupt forge-bridge state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | DoS (Denial of Service) | `ExecutionLog.record()` sync callback path | mitigate | Wrap sync callback call in `try/except Exception` with `logger.warning(..., exc_info=True)`; callback errors never propagate into the record() caller and never roll back the JSONL write. Per D-05. |
| T-06-02 | DoS (Denial of Service) | `ExecutionLog.record()` async callback path | mitigate | `asyncio.ensure_future` + `add_done_callback(_log_callback_exception)` — fire-and-forget with exception-logging via done-callback. If no running loop, `RuntimeError` is caught and logged as warning; record() still returns successfully. Per D-05/D-06. |
| T-06-03 | Tampering | Callback mutating the ExecutionRecord payload | mitigate | `@dataclass(frozen=True)` on ExecutionRecord — consumer code cannot mutate fields after construction. Field drift between JSONL payload and callback payload prevented by building dataclass once and using `asdict(record)` for the JSONL write. |
| T-06-04 | Information Disclosure | Callback receives `raw_code` (may contain secrets if user executed sensitive Python) | accept | Documented behavior per D-03: callback receives the FULL record. Consumers are responsible for filtering/redacting in their callback. JSONL already stores raw_code on disk since Phase 3; the callback surface does not expand the exposure. |
| T-06-05 | Repudiation | Silent swallowing of callback failures makes storage gaps invisible | mitigate | All exception paths call `logger.warning(..., exc_info=True)`. Operators grepping logs for "storage_callback" see failures. Grep-able phrases: "storage_callback raised" (sync + async-done), "storage_callback scheduled outside event loop" (async no-loop). |
</threat_model>
