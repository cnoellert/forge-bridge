    ```python
    def _derive_tool_status(tool: "ToolRecord") -> str:
        """D-40 status derivation for the Tools-view Status column.

        Mirrors the manifest-view _filter_manifest_entries precedent at
        lines 326-357 — derived server-side from ToolRecord fields so the
        template stays thin. Returns one of two values in Phase 10.1:

          - "active":  synthesized tool with a code_hash AND observation_count > 0
          - "loaded":  everything else visible in get_tools() — builtin tools,
                       synthesized tools with no observations, synthesized tools
                       with a missing code_hash.

        The "quarantined" status does NOT appear here by design: quarantined
        tools are removed from ManifestService by the watcher's removal-mirror
        path (forge_bridge/learning/watcher.py:278-293) after
        ProbationTracker.quarantine() (forge_bridge/learning/probation.py:71-88)
        fires, so they are never in the get_tools() stream. Locked by
        tests/test_tool_quarantine_surface.py. If that test fails, this
        function and forge-console.css's chip variant set must be re-planned.
        """
        origin = getattr(tool, "origin", None) or (
            tool.get("origin") if isinstance(tool, dict) else None
        )
        observation_count = getattr(tool, "observation_count", None)
        if observation_count is None and isinstance(tool, dict):
            observation_count = tool.get("observation_count", 0)
        code_hash = getattr(tool, "code_hash", None)
        if code_hash is None and isinstance(tool, dict):
            code_hash = tool.get("code_hash")

        if origin == "synthesized" and code_hash and (observation_count or 0) > 0:
            return "active"
        return "loaded"
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ConsoleReadAPI.get_tools() → `_derive_tool_status` → template dict | In-process; input is a `ToolRecord` instance (frozen dataclass with known-shape fields). No untrusted input crosses this boundary. |
| Template render → browser | Jinja2 autoescape is on by default in Starlette's `Jinja2Templates` (confirmed by D-14 provenance rendering contract); `tool.name`, `tool.namespace`, `tool.status`, `tool.origin`, `tool.code_hash`, `tool.synthesized_at` are all escaped before insertion. `status` is a server-constant string (`"active"` or `"loaded"`) produced by a pure function — no attacker-controlled input reaches it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10.1-07 | T (Tampering) | `status-chip--{{ tool.status }}` interpolation | mitigate | `tool.status` is server-computed by `_derive_tool_status`, which returns a literal from a hard-coded two-value set (`"active"`, `"loaded"`) — never attacker-controlled. Even if an untrusted ToolRecord somehow reached the template, Jinja2 autoescape would HTML-escape the value into the class attribute, blocking attribute-injection attacks. |
| T-10.1-08 | I (Information Disclosure) | Demoted telemetry in Name cell (code_hash[:8], obs count, namespace) | accept | All three values are already visible in Phase 10's Tools view (as primary columns). Demotion does not change exposure — it changes visual weight. No data shown that wasn't shown before. |
| T-10.1-09 | S (Spoofing) | Status chip label visual spoofing | mitigate | `.status-chip--active` uses green; `.status-chip--loaded` uses neutral. Labels read as the literal status word. A screenshot-based phishing attempt would need to mimic the chip CSS — outside this phase's threat envelope (localhost-bound read-only surface per v1.3 milestone scope). |
| T-10.1-10 | I | Jinja autoescape coverage of new template expressions | mitigate | `{{ tool.name }}`, `{{ tool.namespace }}`, `{{ tool.status }}`, `{{ tool.origin }}`, `{{ tool.code_hash[:8] }}`, `{{ tool.synthesized_at }}`, `{{ tool.observation_count }}` all use the `{{ }}` expression form which is autoescape-covered. The `class="status-chip--{{ tool.status }}"` attribute interpolation is also autoescape-covered (HTML attribute context). No `\|safe`, no `Markup`, no raw HTML injection. |
