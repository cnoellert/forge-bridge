# Phase 10.1 Plan 06: Chip-click shell-duplication fix + Playwright regression — Summary

Ships the gap-closure fix for the 2026-04-24 pre-UAT smoke-check blocker (chip click produced shell-inside-shell via `htmx.ajax()` JS path that Plan 10.1-04's nav-swap regression did not cover). Adds the companion Playwright regression test so the bug cannot recur on either the JS-ajax path OR the hx-boost path. Renames the `Active synth` preset chip label to `Synth only` so the user-facing text matches its `origin:synthesized` filter grammar. Along the way, resolves a latent URL-push bug: `pushUrl: true` is a no-op in `htmx.ajax()` options for htmx 2.0.10 — replaced with explicit `history.pushState()` so filtered URLs are actually bookmarkable.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `pushUrl: true` is a no-op in htmx 2.0.10 `htmx.ajax()` options — replaced with explicit `history.pushState()`**

- **Found during:** Task 2, first pre-commit test run against the post-fix state.
- **Issue:** The Task 2 regression test asserts `page.url.endswith("/ui/tools?origin=synthesized")` as a belt-and-suspenders that the pushUrl contract still holds after the swap-target fix. On first run, this assertion failed — the URL remained `/ui/tools` (no query string). Investigation via `grep -oE '[a-zA-Z]{3,}[Uu]rl' forge_bridge/console/static/vendor/htmx-2.0.10.min.js | sort -u` shows only `badResponseUrl` and `validateUrl` — the `pushUrl` token does not exist in htmx 2.0.10's minified source; only the `hx-push-url` HTML attribute is parsed. So `pushUrl: true` passed as an option to `htmx.ajax()` was silently ignored, both pre-fix AND post-fix.
- **Why it blocks correctness:** Filter URLs must be bookmarkable and back-navigable — that's the user-visible contract of the "chip click → filtered Tools view" interaction. The pre-fix code had the same latent bug; it was masked because the shell-duplication was more disruptive (and the URL was pushed as a side effect of the innerHTML swap in some browsers depending on how htmx parses the response's `HX-Push-Url` response header — which our handlers don't emit).
- **Fix:** Added explicit `history.pushState({}, '', url)` immediately before each `htmx.ajax()` call in `submit()` (line 96) and `clearAndReset()` (line 106). Removed the no-op `pushUrl: true` option. Browser-native `pushState` always pushes the URL regardless of htmx's internal option handling.
- **Impact:** Task 2 assertion passes; user URL contract is correct; pre-existing tests remain green (the `TestClient`-based tests don't exercise browser history).
- **Files modified:** `forge_bridge/console/templates/fragments/query_console.html`
- **Commit:** `3f96dcb`

**2. [Rule 3 — Blocking] FAIL→PASS cycle used `git checkout` round-trip instead of `git stash push/pop`**

- **Found during:** Task 2, Step 1 of the verification cycle.
- **Issue:** The plan's exact Step 1 text is `git stash push -m "10.1-06 FAIL-check" forge_bridge/console/templates/fragments/query_console.html`. This captures the working-tree diff for a file against the index. In this worktree, the Task 1 fix (`714518d`) and the Rule 1 auto-fix (`3f96dcb`) are both committed — `query_console.html` has no working-tree modification — so `git stash push` reports `No local changes to save` and fails to create a stash entry.
- **Why it blocks:** Without a mechanism to put the pre-fix buggy content on disk, the Task 2 verification cycle cannot be executed, and the SUMMARY cannot record the FAIL evidence that the plan's Task 2 action explicitly requires.
- **Fix:** Used `git checkout d3fe650 -- forge_bridge/console/templates/fragments/query_console.html` → run test (FAIL) → `git checkout HEAD -- forge_bridge/console/templates/fragments/query_console.html` → run test (PASS) round-trip. Functionally equivalent: both mechanisms put the pre-fix bytes on disk, let the test execute against them, then restore HEAD. The round-trip leaves no residue (`git status --short` clean apart from the new untracked test file; `git stash list` unchanged).
- **Precedent:** Plan 10.1-04's SUMMARY §Deviations §1 adopted exactly this mechanism for the same reason (Plan 10.1-02's fix to `shell.html` was upstream of 10.1-04's cycle, mirroring the situation here). The planning-artifact-vs-reality reconciliation is the same.
