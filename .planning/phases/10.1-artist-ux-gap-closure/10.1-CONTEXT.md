## Implementation Decisions

All six items below are **locked** — derived directly from `10-UAT.md` "Decision" section. Planner treats each as a required outcome and sizes plans/tasks around them.

### Nav swap contract (the shipping render bug)

- **D-37 (locked):** The `hx-boost="true"` + `hx-target="#view-main"` + `hx-swap="innerHTML"` combination on `<nav class="top-nav">` in `shell.html` line 7 is the defect. Fix one of two ways — planner picks which, and justifies in the plan:
  - **Option A (minimal, recommended):** Remove `hx-target="#view-main"` and `hx-swap="innerHTML"` from the nav element. Leave `hx-boost="true"` + `hx-push-url="true"`. `hx-boost` then does its default whole-body swap: it fetches the URL, diffs `<body>`, replaces it. The shell, health strip, and view all come from the same new server response — no duplication.
  - **Option B (architectural):** Keep the `#view-main` target but make `/ui/*` full-page handlers content-negotiate on the `HX-Request` header. If `HX-Request: true`, return only the `{% block view %}` content (a new "view-only" template variant). If absent, return the full shell+view page. Heavier — adds a new template inheritance path — but preserves partial-swap performance.
- **D-38 (locked):** UI-SPEC.md `## Interaction Contracts` §"htmx swap targets" table row for "Nav link click (hx-boost)" is updated to reflect whichever option is chosen in D-37. No other rows in that table are touched.
- **D-39 (locked):** Whichever option is chosen, the end-to-end behavioral contract is: after clicking any top-nav link, the rendered DOM contains exactly one `.top-nav` element and exactly one element with `id="health-strip"`. Verified in the new Playwright test (D-43).

### Tools table artist legibility (D-35/D-36 qualitative fail)

- **D-40 (locked):** The Tools table gains a leading **Status** column (immediately after Name, before any telemetry). Values are rendered as one of three CSS-styled chips:
  - `active` — neutral/positive chip (default for `origin ∈ {builtin, synthesized}` with `observation_count > 0`, not quarantined)
  - `quarantined` — warning/amber chip (any tool with a quarantined marker — planner verifies exact source-of-truth field via `ManifestService.snapshot()` + `ToolRecord`)
  - `loaded` — neutral chip (builtin or synthesized with `observation_count == 0`, not quarantined)
  - Chip label text is the status word itself; no icon-only variant. Chip reuses palette tokens already in `forge-console.css` (see UI-SPEC §Color — `Status: ok` / `Status: warn` / `Status: error` palette entries map to chip variants; planner picks the mapping).
- **D-41 (locked):** Column header relabel + telemetry demotion. Primary columns on first paint:
  - `Name` (unchanged — tool name, mono, primary)
  - `Status` (new — chip per D-40)
  - `Origin` (unchanged label OR relabel to "Type" — planner picks; if relabeled, map synthesized→"Synthesized", builtin→"Built-in")
  - One artist-friendly "freshness" column: relabel `Synthesized` → `Created` (keep ISO timestamp; humanize to "2h ago" / "yesterday" in a later phase — not 10.1).
  - Developer telemetry (`Code hash`, `Obs count`, raw `Namespace`) MUST NOT appear as primary column headers. Planner picks one of:
    - **Option A (recommended):** Render them as a secondary 12px/400 row underneath the tool `Name` in the same cell, visually demoted (grey/secondary text color).
    - **Option B:** Hide them entirely from the list and surface them on the drilldown page (`tools/detail.html`) only. The drilldown already shows full provenance, so duplication is optional.
- **D-42 (locked):** Preset chip row above the Tools table gets a one-line caption directly above it: `"Click a chip to narrow the list"` (exact wording — or equivalent 4-8 word hint; planner picks, does not over-engineer). Chip row is visually separated from the query-console input with ≥`sm` (8px) vertical gap. No change to the chip tokens or labels themselves — D-09 chips already shipped.

### Regression guard

- **D-43 (locked):** A new in-browser test exists that drives a real browser (Playwright preferred; planner may pick an equivalent if it justifies why Playwright is wrong) and asserts the D-39 DOM invariant after a nav click. Test must:
  - Boot a fixture server on an available port (reuse the Phase 10 fixture-server pattern from `tests/test_ui_js_disabled_graceful_degradation.py`).
  - Load `/ui/tools`, click a `.nav-link` (any of Executions / Manifest / Health / Chat).
  - Assert: `document.querySelectorAll('.top-nav').length === 1` AND `document.querySelectorAll('#health-strip').length === 1` after swap settles.
  - Fail when run against the 2026-04-23 pre-fix `shell.html` (the test is useless if the pre-fix state doesn't fail it — the planner includes a step to verify failure on the broken state before committing the fix).
  - Be picked up by the default test command (`pytest` discovery or an equivalent `playwright test` invocation documented in `pyproject.toml`/`README`). If Playwright is a new dev dependency, added to `pyproject.toml` `[project.optional-dependencies]` (a `dev` / `test-e2e` extra, not the base install — Phase 10 D-32 asset budget doesn't apply to dev deps but conservatism still wins).

### Dogfood re-UAT

- **D-44 (locked):** Re-UAT operator is fresh (not `CN/dev`, not `ET/tester` from the 2026-04-23 record). Same fixture state as the 2026-04-23 run: 5 tools (1 quarantined; 3 synthesized with distinct `synthesized_at` timestamps). Criterion: operator identifies the 3 most-recently-synthesized tools and their status within 30 seconds. Timing is measured this run (2026-04-23 UAT bailed qualitatively before timing completed).
- **D-45 (locked):** Re-UAT record lives at `.planning/phases/10.1-artist-ux-gap-closure/10.1-UAT.md` and mirrors the format of `.planning/phases/10-web-ui/10-UAT.md` (fixture state, operator ID, duration, outcome, observations, decision). Phase 10.1 verification does not pass until this record is committed with `Outcome: PASS`.

<specifics>
## Specific Ideas

- **The render bug is the highest-leverage fix.** The UAT observation explicitly notes the bug "compounded the problem on the Tools page itself". Fixing it first is cheap (one-line template change under Option A) and may or may not materially improve comprehensibility — but the Status column + header relabel are still required. Treat D-37 as wave-1, everything else as wave-2 or later.
- **Option A (remove hx-target/hx-swap) is strongly recommended over Option B.** It is one line of template change, reduces the UI-SPEC.md swap table to a simpler contract, and matches what `hx-boost` is for in htmx 2.0.10. Option B is correct but adds a template-inheritance split for zero artist-visible benefit. The planner should pick Option A unless they discover a fragment-swap case on the nav that breaks under whole-body swap (there is no such case in the Phase 10 codebase).
- **Status is derivable, not a new field.** `ManifestService.snapshot()` already exposes `origin`, `observation_count`, and whatever quarantine signal the pipeline carries (planner confirms by reading `manifest_service.py` + `read_api.py`). The Status column is a pure template/presentation change — no `ConsoleReadAPI` contract change, no `ToolRecord` change, no MCP resource change. If the quarantine signal isn't surfaced by the read path, that's a hidden gap the planner flags — it's a read-API fix, not a template fix.
- **The in-browser test's specific assertion matters more than the tool.** The assertion is "one nav, one health strip after click". A contrived test that passes both before AND after the fix is worthless. The planner includes a verification step: run the new test against the pre-fix `shell.html` state and confirm it fails, before applying the template change.
- **Column relabel is the cheapest artist-UX win.** "Synthesized" → "Created", "Obs count" hidden/demoted, "Code hash" hidden/demoted. Three grep-and-replace edits in `tools_table.html` + `list.html` header row. No behavior change, no data change, no API change — pure rendering. If this alone passes the re-UAT, great; it's the safest lever.
- **Playwright is the right tool.** The UAT.md observation is explicit: "None of plan 10-02 / 10-03 tests exercised the in-browser swap; `TestClient` hits handlers in isolation." Playwright's `page.locator().count()` is a two-line assertion; the fixture-server helper already exists in `tests/test_ui_js_disabled_graceful_degradation.py`.
- **Re-UAT operator is fresh, not ET/tester.** Learning from the 2026-04-23 run: a second UAT with the same person would be contaminated by their prior session. Planner's verification plan includes a note to the developer: line up a new operator before the re-UAT task can execute.

</specifics>
