<objective>
Fix the topology framing in `docs/INSTALL.md`. The doc currently reads as "single-machine operator install" with Flame + Postgres + Ollama colocated as host-local prerequisites. Reality: forge-bridge has always supported a remote Ollama via `FORGE_LOCAL_LLM_URL` (verified: `forge_bridge/llm/router.py:199`). The realistic deployment topology — and the topology the user actually runs — is: operator host runs Flame + bridge + Postgres locally; Ollama runs on a SEPARATE LLM service host on the same network because Flame already saturates the workstation's GPU and RAM. Single-machine is supported by the code but is the exception that requires substantial dedicated GPU/RAM headroom beyond Flame's needs — it is NOT the default and should not be presented as such.

Discovered during Phase 20 Track A pre-walk fixture prep: assist-01, which CONTEXT.md D-02 named as the reference operator workstation, is actually an LLM service host (Ollama + qwen2.5-coder:32b only; no Postgres, no Flame, no bridge). The non-author UAT could not start because the fixture's actual shape did not match what INSTALL.md presupposed. The doc — not the fixture — is the artifact that drifted from reality. INSTALL.md was authored against a single-host mental model; the code has always supported (and the realistic Flame-operator deployment requires) a multi-host one.

This is a doc-only fix. The code is fine — `FORGE_LOCAL_LLM_URL` already accepts any base URL (`http://localhost:11434/v1` is just the default and only sensible when the operator has dedicated GPU/RAM headroom). `FORGE_DB_URL` already accepts a remote Postgres URL. No code, no migrations, no tests need to change. The Plan 20-01 version-consistency test must continue to pass after the edit.

Output: a refreshed `docs/INSTALL.md` whose preamble, "Before you start" deps split, "Topology / network reachability" subsection, Step 5 env-var prose, and reference appendix all frame the realistic deployment as multi-host (operator workstation + separate LLM service host) with single-machine treated as the hardware-rich exception. Includes a small "LLM service host" appendix explaining the minimum install for the box that just runs Ollama. The Track B carveout (existing) is updated with one sentence acknowledging it also typically uses a remote Ollama.

Per CONTEXT.md D-04, this is an in-flight Phase 20 follow-up plan: ≤1 plan worth of work, single file, no architectural change. After this lands, Plan 20-05's checkpoint resumes against the corrected doc on a real operator-workstation host (a Flame workstation that points at the existing assist-01 Ollama service).
