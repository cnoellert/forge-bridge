#### Postgres setup (manual, against Autodesk's :7533)

The bootstrap's role+db creation logic uses bare `psql -c "..."` which connects as the OS
user `cnoellert`. Autodesk's `pg_hba.conf` rejects that — it only allows the `flame_db`
superuser. Walker performed manual setup:

```

$ sudo -u flame_db /opt/Autodesk/pgsql-17/bin/psql -p 7533 postgres <<'SQL'
```

#### Investigation: launchd EX_CONFIG = 78 with empty log dir

```
$ sudo launchctl print system/com.cnoellert.forge-bridge | grep -E 'state|last exit|pid' | head -6
    state = spawn scheduled
    last exit code = 78: EX_CONFIG

$ ls -la /var/log/forge-bridge/
drwxr-xr-x   2 root  wheel    64 May  1 22:02 .   # empty + owned by root:wheel mode 755

$ ls -la /usr/local/bin/forge-bridge*
-rwxr-xr-x  1 root  wheel  786 May  1 22:02 /usr/local/bin/forge-bridge-daemon
-rwxr-xr-x  1 root  wheel  619 May  1 22:02 /usr/local/bin/forge-bridge-server-daemon

$ xattr /usr/local/bin/forge-bridge-daemon   # no quarantine attribute
$ plutil -lint /Library/LaunchDaemons/com.cnoellert.forge-bridge*.plist
/Library/LaunchDaemons/com.cnoellert.forge-bridge.plist: OK
/Library/LaunchDaemons/com.cnoellert.forge-bridge-server.plist: OK

$ sudo /usr/local/bin/forge-bridge-server-daemon &   # MANUAL — works!
[INFO] forge-bridge server v0.1.0 starting...
[INFO] Database tables verified.
[INFO] Registry loaded: 6 roles, 2 relationship types.
[INFO] Server ready. Press Ctrl+C to stop.
$ sudo lsof -iTCP:9998 -sTCP:LISTEN
COMMAND     PID  USER   FD   TYPE   DEVICE SIZE/OFF NODE NAME
python3.1 86524 root    9u  IPv4 ...      TCP *:distinct32 (LISTEN)   # 9998 bound
```

**Root cause:** launchd opens `StandardOutPath`/`StandardErrorPath` BEFORE forking. With
`UserName=cnoellert` in the plist + `/var/log/forge-bridge/` owned `root:wheel` mode `755`,
`cnoellert` cannot write there → launchd bails with EX_CONFIG=78 → wrapper never runs →
empty log dir. The bootstrap script had `chown root:wheel /var/log/forge-bridge` — wrong
owner for a user-context daemon.

**In-flight fix #3 (commit `cb687f8`):** chown the log dir to `${SUDO_USER}:${SUDO_USER_GROUP}`
so the daemon (running as that user via plist `UserName=`) can create logs. Same dynamic
group resolution as Gap 2's chown fix.
