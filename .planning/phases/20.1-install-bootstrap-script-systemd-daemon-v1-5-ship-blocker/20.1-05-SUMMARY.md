# Phase 20.1 Plan 05: daemon_state Sub-check for forge doctor Summary

**One-liner:** OS-branched `_check_daemon_state()` in `forge_bridge/cli/doctor` probes systemd (Linux) or launchd (macOS) for daemon liveness and adds a single `daemon_state` row to the `forge-bridge console doctor` table.

## What Was Built

Added `_check_daemon_state()` to `forge_bridge/cli/doctor.py` (lines 345-410) and wired it into `doctor_cmd()` after `_check_disk_space()`. The function branches on `sys.platform`:

- **Linux path**: runs `systemctl is-active forge-bridge-server` and `systemctl is-active forge-bridge`; maps `active/active` → `ok`, one active → `warn`, neither → `fail`.
- **macOS path**: runs `launchctl print system/com.cnoellert.forge-bridge-server` and `launchctl print system/com.cnoellert.forge-bridge`; parses `state = X` line from each; maps `running/running` → `ok`, one running → `warn`, neither → `fail`.
- **Other platforms** (Windows, BSD): returns `warn` with informative fact cell — no probe attempted.

Both paths use `timeout=2` and catch `(FileNotFoundError, subprocess.TimeoutExpired)` — T-11-01 discipline enforced: `f"probe-failed:{type(exc).__name__}"` in the fact cell, never `str(exc)`.

## Integration Test Subprocess Stubbing (Deviation)

**[Rule 1 - Bug] Existing CliRunner tests failed after wire-in on machines without daemons installed**

- **Found during:** Step D (full test_cli_doctor.py regression run)
- **Issue:** `doctor_cmd()` now calls `_check_daemon_state()`, which invokes real `launchctl`/`systemctl`. On macOS without daemons installed, `launchctl` returns `returncode=113`, mapping to `fail` status, breaking `test_all_ok_exit_0` and `test_sidecar_dir_missing_warns`.
- **Fix:** Added `monkeypatch.setattr(_doctor_module.subprocess, "run", ...)` stub to `_setup_writable_dirs` (used by 9 existing integration tests) and to `test_sidecar_dir_missing_warns` directly. Stub returns `stdout="state = running\n", returncode=0` on macOS (or `stdout="active\n"` on Linux) so daemon_state → ok in CliRunner-level tests.
- **Files modified:** `tests/test_cli_doctor.py` (stub added to two locations)
- **Commit:** `a10ba6f`

This is the correct fix: the 3 new unit tests for `_check_daemon_state` use `@patch("forge_bridge.cli.doctor.subprocess.run")` directly, so they are not affected. The existing integration tests needed isolation from the real subprocess.
