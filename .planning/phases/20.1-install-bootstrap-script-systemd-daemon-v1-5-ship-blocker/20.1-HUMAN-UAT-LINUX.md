**Gap surfaced (the structural one):**
- **Gap 4 (route to 20.1-08, root cause):** `forge-bridge.service` runs
  `python -m forge_bridge` in FastMCP **stdio transport mode** (the FastMCP default,
  per `forge_bridge/mcp/server.py:348` `mcp.run()`). Combined with `StandardInput=null`,
  FastMCP's stdio loop reads `/dev/null` → EOF → exits cleanly with code 0. Because
  the exit is clean, `Restart=on-failure` does NOT respawn. The Console + chat endpoint
  + Read API are co-hosted on `:9996` via FastMCP **lifespan** (per CLAUDE.md) — they
  die when FastMCP exits. Net effect: surfaces 2/3/4 (Web UI, chat, Read API) are
  unreachable under the systemd daemon model. Phase 20 manual install path worked
  around this with `tail -f /dev/null | python -m forge_bridge` (infinite empty stdin).
  Phase 20.1 dropped that workaround in favor of `StandardInput=null`, which is the
  opposite of what FastMCP-stdio needs. **Fix:** add `--transport` flag to
  `forge_bridge/__main__.py` + thread through `forge_bridge/mcp/server.py:main()` to
  `mcp.run(transport=...)`. Use `sse` or `streamable-http` for daemon mode (long-running
  ASGI server, no stdin EOF dependency). Update systemd + launchd ExecStart to pass
  the flag. Preserve `stdio` as the default for Claude Desktop config compatibility.
  Add a regression test that asserts `forge-bridge.service` stays `active (running)`
  for ≥10s and survives an MCP client connect/disconnect cycle.
- **Gap 5 (related to Gap 4):** `:9996` squatter from a stale manual
  `python -m forge_bridge` (PID 925928) was free to take the port because the systemd
  unit cleanly exits and stops binding. Once Gap 4 is fixed, the systemd-spawned
  process holds `:9996` continuously and a manual squatter would EADDRINUSE on its own
  start, surfacing the conflict at the right time. No separate fix needed if Gap 4
  is closed.

## Outcome

**Status: FAIL**

`forge-bridge.service` does not stay running under the Phase 20.1 systemd unit configuration.
Root cause is Gap 4: FastMCP stdio transport + `StandardInput=null` cause the service to exit
cleanly within ~1s of start, taking the Console + chat endpoint + Read API on `:9996` down with
it. The unit's `Restart=on-failure` does NOT trigger because the exit is clean (code 0).
Three of the five user-facing surfaces (Web UI, chat, Read API) are unreachable. The daemon
model that Phase 20.1 set out to ship does not hold.

**Reasoning:**
- D-19 requires "all 5 reachable surfaces green" for PASS-with-deviations. Three surfaces FAIL.
- Gap 4 root cause is structural, not configuration drift; cannot be patched at the unit level
  without an upstream code change to add a non-stdio transport flag.
- Per D-21 budget assessment, the fix is bounded (one new flag, two unit edits, one test) —
  routed as in-flight plan **20.1-08** (not a 20.1.1 fractal).

**Routing decision:** Spin plan 20.1-08 to ship a `--transport` flag on `python -m forge_bridge`
(default `stdio` for Claude Desktop compat; daemon path uses `sse` or `streamable-http`).
Update systemd + launchd units to use the daemon-mode flag. Add a regression test that
asserts service persistence after start. Re-walk Steps 6-8 on flame-01 against the fix.
Then run the macOS walk on portofino against the corrected launchd plist.
