## Wrapper Script Design

### forge-bridge-server-daemon

```
set -a; . /etc/forge-bridge/forge-bridge.env; set +a
exec ${FORGE_PYTHON:-...conda.../python} -m forge_bridge.server
```

The `set -a; . FILE; set +a` triple is the only portable POSIX equivalent of systemd's `EnvironmentFile=`. Without `set -a`, sourced `KEY=VALUE` lines stay local to the wrapper shell and Python never sees them.

### forge-bridge-daemon (MCP+Console wrapper)

Same env-source pattern, plus the ordering gate:

```bash
PORT="${FORGE_BRIDGE_PORT:-9998}"
for i in $(seq 1 30); do
    if nc -z localhost "$PORT" 2>/dev/null; then break; fi
    sleep 1
done
exec ${FORGE_PYTHON:-...conda.../python} -m forge_bridge
```

This 30-iteration loop replicates what `Requires=forge-bridge-server.service` gives us for free on Linux. At 30s timeout the wrapper exec's anyway and Console hits Phase 07.1 graceful-degradation (WARN, not ERROR).

## Phase 20 Gap #11 macOS Parity

**The gap:** launchd has no `Requires=` analog. On Linux, `forge-bridge.service` sets `Requires=forge-bridge-server.service`, meaning:
1. systemd starts the bus first
2. if the bus dies, systemd also kills the Console

**The macOS solution:** The `forge-bridge-daemon` wrapper probes `:9998` with `nc -z` in a 30s loop before `exec`-ing. This handles the startup-ordering case. The "bus death cascades" behavior is not replicated (launchd would just respawn Console into the Phase 07.1 degradation path), which is acceptable — the operator will see the bus as stopped in `launchctl print system/com.cnoellert.forge-bridge-server` and restart it.

## /var/log/forge-bridge/ Pre-creation Requirement for P4

**Critical:** launchd opens `StandardOutPath` file descriptors BEFORE invoking `ProgramArguments`. If `/var/log/forge-bridge/` does not exist when `launchctl bootstrap` runs, launchd silently aborts the job with `last exit code = 78` (ENXIO) and writes no logs. P4 `install-bootstrap.sh` MUST create this directory before running `launchctl bootstrap`.

## Modern launchctl Syntax

P4 must use the bootstrap/bootout idiom, not the deprecated load/unload:

```bash
sudo launchctl bootstrap system /Library/LaunchDaemons/com.cnoellert.forge-bridge.plist   # install
sudo launchctl bootout    system /Library/LaunchDaemons/com.cnoellert.forge-bridge.plist   # remove
sudo launchctl kickstart -k system/com.cnoellert.forge-bridge                              # restart after env edit
```

## Foundation for P4 (install-bootstrap.sh)

P4 `install-bootstrap.sh` macOS path must:
1. Create `/var/log/forge-bridge/` (mode 755 root:wheel) BEFORE `launchctl bootstrap` — Pitfall 2
2. Copy `packaging/launchd/*.plist` to `/Library/LaunchDaemons/`
3. Copy `packaging/launchd/forge-bridge-*-daemon` to `/usr/local/bin/` with mode 755
4. Run `substitute_placeholders()` on plists + wrappers (replacing `__SUDO_USER__` with `$SUDO_USER`)
5. Run `sudo launchctl bootstrap system /Library/LaunchDaemons/com.cnoellert.forge-bridge-server.plist`
6. Run `sudo launchctl bootstrap system /Library/LaunchDaemons/com.cnoellert.forge-bridge.plist`
