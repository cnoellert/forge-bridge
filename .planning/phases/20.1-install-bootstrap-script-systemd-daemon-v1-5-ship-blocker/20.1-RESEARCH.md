# Phase 20.1: Install Bootstrap Script + Systemd Daemon (v1.5 ship blocker) - Research

**Researched:** 2026-05-01
**Domain:** Bash install script + systemd units (Rocky/RHEL 9) + launchd plists (macOS 13+) + Postgres bootstrap (initdb / pg_hba auth-method alignment) + env-file template + `forge doctor` daemon-state sub-check
**Confidence:** HIGH for systemd/launchd/pg_hba mechanics; HIGH for INSTALL.md reshape; MEDIUM for the exact Rocky-9 PG13 vs PG16 password_encryption-default reality (Phase 20 walk = md5; vendor docs say PG13 default = md5, PG14.3+ default = scram-sha-256 — handle by probe, not assume); HIGH for fresh-VM contract.

## TL;DR for the planner

Copy these straight into PLAN tasks. Every claim has a source tag — verify, don't re-derive.

1. **Two systemd units, two launchd plists. The bus comes up first.**
   - Linux: `forge-bridge-server.service` (`:9998` bus, `Type=simple`, `StandardInput=null`, `Restart=on-failure`, `EnvironmentFile=/etc/forge-bridge/forge-bridge.env`) AND `forge-bridge.service` (`:9996` MCP+Console, `Requires=forge-bridge-server.service` + `After=forge-bridge-server.service` + `Wants=postgresql.service` + `After=postgresql.service`, same EnvironmentFile, same Restart). [VERIFIED: Phase 20 HUMAN-UAT gap #11 + freedesktop systemd.unit(5)]
   - macOS: `com.cnoellert.forge-bridge-server.plist` + `com.cnoellert.forge-bridge.plist`, both in `/Library/LaunchDaemons/`, both `RunAtLoad=true` + `KeepAlive={SuccessfulExit=false}` (restart on crash, not on clean exit). Ordering enforced via the wrapper script in `forge-bridge.service` (sleep-and-probe `:9998` before exec). launchd has no `Requires=` equivalent. [VERIFIED: Apple BPSystemStartup + launchd.plist(5)]
2. **Two-layer plist envelope on macOS (because `EnvironmentFile=` has no launchd equivalent).** Each plist's `ProgramArguments` invokes a thin wrapper at `/usr/local/bin/forge-bridge-daemon` (or `forge-bridge-server-daemon`). The wrapper does `set -a && . /etc/forge-bridge/forge-bridge.env && set +a`, then `exec /Users/<sudo_user>/miniconda3/envs/forge/bin/python -m forge_bridge[.server]`. This is the operator-recommended pattern and beats hardcoding env in the plist (which forces an `install-bootstrap.sh` re-run on every env edit). [VERIFIED: lucaspin.medium.com + naiyerasif.com + Apple developer forums]
3. **`pg_hba.conf` auth method = whatever `SHOW password_encryption` says.** Rocky 9 stock `postgresql-server` is PG13 → cluster initdb defaults to `md5`. RHEL/upstream PG14.3+ ships `scram-sha-256` as the initdb default. Probe the cluster, don't assume. The flip pattern: idempotent `sed -i.bak -E "s/^(host\s+all\s+all\s+(127\.0\.0\.1\/32|::1\/128)\s+).*$/\1${AUTH}/g" "$PG_HBA"` then `sudo -u postgres psql -c "SELECT pg_reload_conf();"`. [VERIFIED: postgresql.org/docs/14/release-14.html + Phase 20 HUMAN-UAT gap #6]
4. **All-in-one Postgres bootstrap (Linux):** `command -v psql >/dev/null || dnf install -y postgresql-server postgresql-contrib` → `[ -d /var/lib/pgsql/data/base ] || postgresql-setup --initdb` → `systemctl enable --now postgresql` → `until pg_isready -h localhost -p 5432; do sleep 1; done` → probe `password_encryption` → flip `pg_hba` if needed → reload → `psql -tAc "SELECT 1 FROM pg_roles WHERE rolname='forge'" | grep -q 1 || psql -c "CREATE USER forge WITH PASSWORD 'forge';"` (idempotent) → `psql -tAc "SELECT 1 FROM pg_database WHERE datname='forge_bridge'" | grep -q 1 || psql -c "CREATE DATABASE forge_bridge OWNER forge;"` → `cd "$REPO_ROOT" && alembic upgrade head`. Closes Phase 20 gaps #1, #3, #4, #5, #6. [VERIFIED: Phase 20 HUMAN-UAT log + postgresql.org/docs/current/auth-pg-hba-conf.html]
5. **macOS Postgres branch (D-04 rare path):** `if ! command -v psql && ! brew list postgresql@16 >/dev/null 2>&1; then brew install postgresql@16 && brew services start postgresql@16; fi`. Operator-confirmed all Mac Flame workstations canonically have Postgres pre-installed (via Postgres.app or Homebrew) — this branch fires on operator-friendly fresh-VM only. The script does NOT auto-install Homebrew (D-06).
6. **Env-file at `/etc/forge-bridge/forge-bridge.env`, mode `0640`, owner `root:$SUDO_USER`.** systemd reads it via `EnvironmentFile=/etc/forge-bridge/forge-bridge.env`. macOS reads it via the wrapper script. Single source of truth. Format: `KEY=VALUE`, no `export`. [VERIFIED: freedesktop systemd.exec(5) — line-format compatibility + integration-points note in 20.1-CONTEXT.md]

7. **Interactive prompt = `read -rp "...: " VAR`.** `FORGE_LOCAL_LLM_URL` is NOT secret (URL with possibly internal hostname); the `-r` flag is correct (don't interpret backslashes), but `-s` is wrong (operator wants to see the URL they're typing for typo verification). Pattern: `read -rp "Ollama base URL [http://localhost:11434/v1]: " FORGE_LOCAL_LLM_URL; FORGE_LOCAL_LLM_URL="${FORGE_LOCAL_LLM_URL:-http://localhost:11434/v1}"`. [VERIFIED: linuxopsys.com + memory feedback_secret_paste_pattern.md — secret pattern preserved for any future password reads but doesn't apply here]

8. **`forge doctor` gets ONE new sub-check named `daemon_state`** (not two). Internal OS branching via `sys.platform`. Linux: `subprocess.run(["systemctl", "is-active", "forge-bridge-server"], capture_output=True)` + same for `forge-bridge`. macOS: `subprocess.run(["launchctl", "print", "system/com.cnoellert.forge-bridge-server"], capture_output=True)` + parse `state = running`. Status: `ok` if both daemons up, `warn` if one degraded (legible from cli output), `fail` if both down. One row keeps the doctor table clean; granular detail goes in the `fact` cell. [VERIFIED: Phase 20 doctor table format + Phase 20.1-CONTEXT.md D-13]

9. **macOS log dir creation:** `install-bootstrap.sh` MUST `mkdir -p /var/log/forge-bridge && chown root:wheel /var/log/forge-bridge && chmod 755 /var/log/forge-bridge` BEFORE `launchctl bootstrap` — plist `StandardOutPath` won't auto-create the directory and the daemon will silently fail to boot with launchctl print showing `last exit code = 78` (ENXIO).

10. **Modern launchctl syntax:** use `launchctl bootstrap system /Library/LaunchDaemons/com.cnoellert.forge-bridge.plist` (not deprecated `launchctl load`) and `launchctl bootout system /Library/LaunchDaemons/com.cnoellert.forge-bridge.plist` (not `launchctl unload`). macOS 13+ requirement — `load`/`unload` work but emit deprecation warnings. [VERIFIED: alansiu.net 2023 + Homebrew/homebrew-services PR #112]

### Pattern 1: systemd two-unit cascade with ordering

**What:** Two independent units; the Console unit `Requires=` the bus unit and `After=` it. systemd handles cascade-start, restart-on-failure, and `journalctl -u <unit>` log streaming for free.

**When to use:** Linux daemon orchestration where one process logically depends on another being up first. Gold-standard pattern. [VERIFIED: oneuptime.com/blog 2026-03-04 + funwithlinux.net]

**Example:**

`__SUDO_USER__` is replaced at install time by `install-bootstrap.sh` via `sed -e "s|__SUDO_USER__|${SUDO_USER}|g"`. The conda env path is also captured at install time (`CONDA_BASE=$(conda info --base 2>/dev/null || echo "/home/${SUDO_USER}/miniconda3")`).

**Why `Requires=forge-bridge-server.service` IS hard:** the WS bus IS critical to the Console process — Phase 07.1 graceful-degradation only covers the warning path. If the bus dies, the Console should die too and let systemd restart both. `Requires=` is correct here.

**Why `KeepAlive={SuccessfulExit=false}` not `KeepAlive=true`:** under bare `KeepAlive=true`, if the daemon ever exits cleanly (exit code 0), launchd still respawns it. For long-running daemons that's fine, but if a future `--help`-like clean-exit code path slips in, you get a respawn loop. `SuccessfulExit=false` says "respawn ONLY on crash (non-zero exit)," which is the durable contract. [VERIFIED: launchd.info + tjluoma/launchd-keepalive]

**Why a wrapper script (not env in plist):** launchd has no `EnvironmentFile=` analog. Hardcoding env in the plist's `EnvironmentVariables` dict means every env edit requires re-running `install-bootstrap.sh` to regenerate the plist (and `launchctl bootout` + `bootstrap` to reload). The wrapper-script pattern means the operator edits `/etc/forge-bridge/forge-bridge.env` and `launchctl kickstart -k system/com.cnoellert.forge-bridge` reloads — same workflow as `systemctl restart` on Linux. Operator parity wins. [VERIFIED: lucaspin.medium.com + naiyerasif.com 2024-12]

### Pattern 3: Postgres bootstrap with version-aware auth alignment

**What:** Probe the cluster's `password_encryption` GUC, align `pg_hba.conf` to match, reload. Idempotent on re-run.

**When to use:** Linux fresh-VM or fresh-cluster setup. macOS Homebrew/Postgres.app installs ship pg_hba defaulting to `trust` for `localhost` — they don't need this flip; the script must skip when `host all all 127.0.0.1/32 trust` is the existing rule.

**Example:**

```bash
# Source: VERIFIED — postgresql.org/docs/14/release-14.html (PG14 default change),
# Phase 20 HUMAN-UAT gap #6 (Rocky 9 PG13 cluster), postgresql.org/docs/current/auth-pg-hba-conf.html

probe_pg_auth() {
    # Print the cluster's password_encryption: 'md5' or 'scram-sha-256'.
    sudo -u postgres psql -tAc "SHOW password_encryption;" 2>/dev/null | tr -d ' '
}

resolve_pg_hba() {
    # Resolve pg_hba.conf path from the cluster itself — varies by version + install method.
    sudo -u postgres psql -tAc "SHOW hba_file;" 2>/dev/null | tr -d ' '
}

align_pg_hba() {
    local auth="$1"          # 'md5' or 'scram-sha-256'
    local pg_hba; pg_hba="$(resolve_pg_hba)"

    if [ -z "$pg_hba" ] || [ ! -f "$pg_hba" ]; then
        echo "ERROR: cannot resolve pg_hba.conf path" >&2
        return 1
    fi

    # Check if already aligned — idempotency wins.
    if grep -qE "^host\s+all\s+all\s+127\.0\.0\.1/32\s+${auth}\b" "$pg_hba"; then
        echo "[forge-bridge] pg_hba already aligned to ${auth}"
        return 0
    fi

    echo "[forge-bridge] aligning pg_hba auth method to ${auth}"
    # Backup, then edit lines matching the localhost rules.
    sudo cp "$pg_hba" "${pg_hba}.bak.$(date +%s)"
    sudo sed -i.tmp -E \
        "s|^(host[[:space:]]+all[[:space:]]+all[[:space:]]+(127\.0\.0\.1/32|::1/128)[[:space:]]+).*|\1${auth}|g" \
        "$pg_hba"
    sudo rm -f "${pg_hba}.tmp"

    sudo -u postgres psql -c "SELECT pg_reload_conf();" >/dev/null
}

bootstrap_pg() {
    # 1. Detect: package present? cluster initialized? daemon running?
    if ! command -v psql >/dev/null 2>&1; then
        sudo dnf install -y postgresql-server postgresql-contrib
    fi

    if [ ! -d /var/lib/pgsql/data/base ]; then
        sudo postgresql-setup --initdb
    fi

    sudo systemctl enable --now postgresql

    # 2. Wait for readiness (pg_isready is the canonical probe).
    for i in $(seq 1 30); do
        if pg_isready -h localhost -p 5432 >/dev/null 2>&1; then break; fi
        sleep 1
    done

    # 3. Align auth method.
    local auth; auth="$(probe_pg_auth)"
    if [ -z "$auth" ]; then
        echo "ERROR: cannot probe password_encryption — Postgres unreachable" >&2
        return 1
    fi
    align_pg_hba "$auth"

    # 4. Idempotent role + db creation.
    if ! sudo -u postgres psql -tAc "SELECT 1 FROM pg_roles WHERE rolname='forge'" | grep -q 1; then
        sudo -u postgres psql -c "CREATE USER forge WITH PASSWORD 'forge';"
    fi
    if ! sudo -u postgres psql -tAc "SELECT 1 FROM pg_database WHERE datname='forge_bridge'" | grep -q 1; then
        sudo -u postgres psql -c "CREATE DATABASE forge_bridge OWNER forge;"
    fi

    # 5. Run migrations against the default URL hardcoded in alembic.ini.
    cd "$REPO_ROOT" && alembic upgrade head
}
```

**alembic.ini handling per CONTEXT.md `code_context`:** the script does NOT rewrite `alembic.ini` — it runs migrations against the default URL `postgresql+psycopg2://forge:forge@localhost:5432/forge_bridge`, which is exactly the role+db pair `install-bootstrap.sh` just created. Operators with non-default credentials use `alembic -x url=...` per the existing INSTALL.md `Note on Alembic + custom credentials` section. (Re-architecting alembic.ini to read FORGE_DB_URL is a v1.6+ scope item; not 20.1.)

### Pattern 5: `forge doctor` `daemon_state` sub-check (one row, OS-branched)

**What:** One new check function in `forge_bridge/cli/doctor.py` that branches on `sys.platform` and emits a single doctor row.

**Example:**

```python
# forge_bridge/cli/doctor.py — append after _check_disk_space()
# Source: VERIFIED — Phase 20 HUMAN-UAT doctor table format (lines 117-141 of 20-HUMAN-UAT.md)

import subprocess
import sys


def _check_daemon_state() -> dict:
    """D-13: Probe systemd (Linux) or launchd (macOS) for daemon liveness.

    Returns ONE row in the doctor table — the fact cell carries granular detail
    so we don't double the table footprint with one row per unit.
    """
    name = "daemon_state"
    if sys.platform == "linux":
        results: dict[str, str] = {}
        for unit in ("forge-bridge-server", "forge-bridge"):
            try:
                r = subprocess.run(
                    ["systemctl", "is-active", unit],
                    capture_output=True, text=True, timeout=2,
                )
                results[unit] = (r.stdout or "").strip() or "unknown"
            except (FileNotFoundError, subprocess.TimeoutExpired) as exc:
                # T-11-01: surface exception class name only, never str(exc)
                results[unit] = f"probe-failed:{type(exc).__name__}"

        all_active = all(v == "active" for v in results.values())
        any_active = any(v == "active" for v in results.values())
        status = "ok" if all_active else ("warn" if any_active else "fail")
        fact = ", ".join(f"{u}={s}" for u, s in results.items())
        try_msg = "" if all_active else "sudo systemctl status forge-bridge && journalctl -u forge-bridge -n 50"
        return {"name": name, "status": status, "fact": fact, "try": try_msg}

    if sys.platform == "darwin":
        results: dict[str, str] = {}
        for label in ("com.cnoellert.forge-bridge-server", "com.cnoellert.forge-bridge"):
            try:
                r = subprocess.run(
                    ["launchctl", "print", f"system/{label}"],
                    capture_output=True, text=True, timeout=2,
                )
                if r.returncode != 0:
                    results[label.split(".")[-1]] = "not-loaded"
                    continue
                # Parse `state = running` line from `launchctl print` output.
                state = "unknown"
                for line in (r.stdout or "").splitlines():
                    if "state =" in line:
                        state = line.split("=", 1)[1].strip()
                        break
                results[label.split(".")[-1]] = state
            except (FileNotFoundError, subprocess.TimeoutExpired) as exc:
                results[label.split(".")[-1]] = f"probe-failed:{type(exc).__name__}"

        all_running = all(v == "running" for v in results.values())
        any_running = any(v == "running" for v in results.values())
        status = "ok" if all_running else ("warn" if any_running else "fail")
        fact = ", ".join(f"{k}={v}" for k, v in results.items())
        try_msg = "" if all_running else "sudo launchctl print system/com.cnoellert.forge-bridge && tail -50 /var/log/forge-bridge/console.log"
        return {"name": name, "status": status, "fact": fact, "try": try_msg}

    # Other platforms (Windows, BSD) — skip with informative warn.
    return {
        "name": name,
        "status": "warn",
        "fact": f"daemon_state probe not implemented for sys.platform={sys.platform!r}",
        "try": "no-op — supported only on linux + darwin in v1.5",
    }
```

### Anti-Patterns to Avoid

- **Putting env in plist `EnvironmentVariables` dict directly.** Forces `install-bootstrap.sh` re-run + `launchctl bootout/bootstrap` cycle for every env edit. Wrapper script wins.
- **`launchctl load` / `launchctl unload`.** Deprecated since macOS 11 (Big Sur); emits warnings; lacks proper error reporting. Use `bootstrap` / `bootout` with `system` domain. [VERIFIED: alansiu.net + Homebrew/homebrew-services PR #112]
- **`Requires=postgresql.service` (hard).** Cascades a Postgres restart into a forge-bridge restart. Phase 8 STORE-06 explicitly designed JSONL-authoritative + SQL-mirror precisely to keep forge-bridge running through Postgres outages. `Wants=` + `After=` is correct.
- **Hardcoding `User=cnoellert` in unit files committed to git.** Use `__SUDO_USER__` placeholder; `install-bootstrap.sh` substitutes at install time.
- **Bash 4-only constructs in `install-bootstrap.sh`.** macOS portofino ships bash 3.2 (Apple stopped updating; license issue). Use POSIX-compatible `[ "$x" = "$y" ]` not `[[ "$x" == "$y" ]]` where possible — though `#!/usr/bin/env bash` + portofino has bash 5.x via Homebrew, the system bash is 3.2 and `sudo` may pick that up. Test on stock macOS bash via `/bin/bash --version` first.
- **`read -s` for the LLM URL.** Hides the URL the operator is trying to type — typo-prone. `read -rp` (visible) is correct for non-secret input.
- **Editing `/etc/sudoers` to drop the `sudo` requirement.** Out of scope. Operator runs `sudo ./install-bootstrap.sh` once; daemons run as `$SUDO_USER` after that.
- **Missing `mkdir -p /var/log/forge-bridge` before `launchctl bootstrap`.** Plist `StandardOutPath` won't auto-create the dir; daemon fails to load with cryptic error. The script MUST create it (and `chown root:wheel`) BEFORE bootstrap.
- **Using `Type=forking` for the systemd unit.** The Python daemon doesn't double-fork; `Type=simple` is correct (the default). [VERIFIED: freedesktop systemd.exec(5)]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Process supervision (start at boot, restart on crash) | bash `nohup` + pid-file + cron-watchdog | systemd unit (Linux) / launchd plist (macOS) | These are exactly what the OS ships an init system to solve. Phase 20 gap #12 ("how do I stop the server?") is the canonical "we tried to hand-roll lifecycle and it bit us." |
| Env-file loading | bash `source ~/.bashrc` per terminal | systemd `EnvironmentFile=` / wrapper-script `set -a; . file; set +a` | systemd handles this natively; the wrapper-script pattern reaches macOS parity. Re-implementing `EnvironmentFile=` in the daemon's Python entry-point is wasted code. |
| Postgres readiness gate | `nc -z localhost 5432` polled in a sleep loop | `pg_isready -h localhost -p 5432` | `pg_isready` ships in `postgresql-contrib`, knows about cluster startup phases (e.g., `accepting connections` vs `recovering from WAL`), returns deterministic exit codes. [VERIFIED: postgresql.org/docs/current/app-pg-isready.html] |
| pg_hba.conf parsing | bash regex on the file content | `SHOW password_encryption` + `SHOW hba_file` from psql | psql tells us authoritatively where the file lives and what auth method the cluster uses. Don't re-derive from version detection. |
| Idempotent role/db creation | bash `if grep -q "..." /var/lib/pgsql/...` | `psql -tAc "SELECT 1 FROM pg_roles WHERE rolname='...'"` | psql is the canonical interface; pg_roles/pg_database are stable views. |
| systemd unit ordering across multiple processes | bash sleep-and-poll | `Requires=` + `After=` directives | systemd was built for this; reinventing it in shell is exactly the architectural mistake we caught in Phase 20 gap #11. |
| launchd ordering across multiple plists | install one big plist that forks both processes | wrapper script with sleep-and-probe gate | launchd really does lack `Requires=`; the sleep-and-probe pattern is the operator-recommended workaround. Don't fork-from-one-plist (loses individual restart semantics). |
| Distro detection | `uname -a | grep -i centos\|rocky\|rhel` | `[ -f /etc/redhat-release ]` + `[[ "$(uname -s)" == "Darwin" ]]` | The OS ships exactly the file we need. uname-grepping is fragile across kernel versions. |
| Doctor sub-check process probe | `pgrep -f "python -m forge_bridge"` | `systemctl is-active` + `launchctl print` | The OS init system is the source of truth for "is my daemon up." pgrep can find an orphaned process the OS thinks is dead. |

**Key insight:** The Phase 20 walk's deepest failure (gap #11 — "single process boots all four surfaces" doc lie) was the result of trying to hand-roll process orchestration in narrative prose. systemd and launchd exist precisely so we don't have to. The 20.1 architectural choice is "let the OS init system own the lifecycle" — every other 20.1 decision flows from that.

## Common Pitfalls

### Pitfall 1: pg_hba.conf flip lands BEFORE the cluster is ready

**What goes wrong:** Script runs `postgresql-setup --initdb` then immediately tries `SHOW password_encryption;` over psql — gets connection refused because postgres daemon isn't started yet (or is starting).

**Why it happens:** No readiness gate between `systemctl enable --now postgresql` and the first psql probe.

**How to avoid:** `until pg_isready -h localhost -p 5432; do sleep 1; done` between daemon start and first SQL probe. Cap the loop at 30 iterations.

**Warning signs:** `psql: error: connection to server on socket "/var/run/postgresql/.s.PGSQL.5432" failed: No such file or directory` or `Connection refused` on first probe.

### Pitfall 2: macOS plist `StandardOutPath` log directory missing

**What goes wrong:** `launchctl bootstrap` returns 0 but the daemon never appears in `launchctl list`. `launchctl print system/com.cnoellert.forge-bridge` shows `last exit code = 78`. No log line written anywhere.

**Why it happens:** `/var/log/forge-bridge/` doesn't exist. launchd opens the StandardOutPath file before the wrapper script runs — file open fails with ENXIO and launchd silently aborts the job.

**How to avoid:** `sudo mkdir -p /var/log/forge-bridge && sudo chown root:wheel /var/log/forge-bridge && sudo chmod 755 /var/log/forge-bridge` BEFORE `launchctl bootstrap`. Add the same dir-creation idempotently to the wrapper script as belt-and-suspenders.

**Warning signs:** `launchctl print` shows non-zero `last exit code` and no log file appears.

### Pitfall 3: `EnvironmentFile=` permissions block daemon read

**What goes wrong:** systemd reports `Failed to parse environment file: Permission denied`. Daemon refuses to start.

**Why it happens:** File created with mode `0600` root:root; systemd reads as the operator user (`User=__SUDO_USER__`) and can't read.

**How to avoid:** `chmod 0640 /etc/forge-bridge/forge-bridge.env && chown root:${SUDO_USER} /etc/forge-bridge/forge-bridge.env`. Group-readable by operator's primary group.

**Warning signs:** `journalctl -u forge-bridge` shows `Failed to parse environment file: ... Permission denied`.

### Pitfall 4: conda env path captured stale on a re-run after env was reinstalled

**What goes wrong:** `install-bootstrap.sh` first run captures `/home/cnoellert/miniconda3/envs/forge/bin/python` and writes it into the systemd unit. Operator later removes + recreates conda env (now living elsewhere via `--prefix`). Daemon fails: `No such file or directory`.

**Why it happens:** systemd unit's `ExecStart=` is a literal path captured at install time, NOT a symbolic resolution.

**How to avoid:** Ship `FORGE_PYTHON=/path/to/python` in the env file. systemd unit reads `ExecStart=${FORGE_PYTHON} -m forge_bridge`. Re-running `install-bootstrap.sh` re-writes the env file with the current resolution. (`bash -c 'echo $FORGE_PYTHON'` in the unit is forbidden; systemd does NOT shell-expand `ExecStart=` — must be an absolute path.) Per [systemd.exec(5)] `ExecStart=` does support `$VAR` expansion from `EnvironmentFile=`. [VERIFIED: freedesktop systemd.exec(5) — "Variables in $VAR style"]

**Pattern:**
```
EnvironmentFile=/etc/forge-bridge/forge-bridge.env
ExecStart=${FORGE_PYTHON} -m forge_bridge.server
```
Env file:
```
FORGE_PYTHON=/home/cnoellert/miniconda3/envs/forge/bin/python
```

**Warning signs:** Daemon ExecStart fails with `No such file or directory` after env was rebuilt.

### Pitfall 5: The wrapper script and the plist disagree on which user

**What goes wrong:** Plist `UserName=cnoellert`. Wrapper script (run as cnoellert) `exec /Users/cnoellert/miniconda3/envs/forge/bin/python ...`. But operator's actual login is `chris` and conda lives at `/Users/chris/miniconda3/...` — daemon hangs on file-not-found.

**Why it happens:** `install-bootstrap.sh` captures `$SUDO_USER` to substitute `__SUDO_USER__` placeholders, but if the placeholder substitution misses one file, you get a name mismatch.

**How to avoid:** ONE substitution helper in `install-bootstrap.sh`:
```bash
substitute_placeholders() {
    local file="$1"
    sudo sed -i.bak \
        -e "s|__SUDO_USER__|${SUDO_USER}|g" \
        -e "s|__CONDA_BASE__|${CONDA_BASE}|g" \
        -e "s|__REPO_ROOT__|${REPO_ROOT}|g" \
        "$file"
    sudo rm -f "${file}.bak"
}
```
Run on every file copied to `/etc/`, `/Library/`, `/usr/local/bin/` after the cp.

**Warning signs:** `journalctl -u forge-bridge` or `tail /var/log/forge-bridge/console.log` shows `python: No such file or directory`.

### Pitfall 6: Forgetting `daemon-reload` after copying systemd units

**What goes wrong:** `systemctl start forge-bridge` returns "Unit not found" immediately after `cp packaging/systemd/*.service /etc/systemd/system/`.

**Why it happens:** systemd caches its unit graph; new files require `systemctl daemon-reload`.

**How to avoid:** Always pair the cp with `daemon-reload`:
```bash
sudo cp packaging/systemd/*.service /etc/systemd/system/
sudo systemctl daemon-reload
```

**Warning signs:** `systemctl status forge-bridge` reports "Unit forge-bridge.service could not be found."

### Pitfall 7: `launchctl bootstrap` requires root for system-domain plists

**What goes wrong:** `launchctl bootstrap system /Library/LaunchDaemons/com.cnoellert.forge-bridge.plist` returns "Bootstrap failed: 5: Input/output error."

**Why it happens:** system-domain operations require sudo. The operator may have run `install-bootstrap.sh` itself with sudo but `launchctl bootstrap` is a separate invocation; if it lacks the `sudo` prefix it fails.

**How to avoid:** Inside `install-bootstrap.sh`, always `sudo launchctl bootstrap system ...` (even when the script itself was invoked with sudo — explicit sudo on each launchctl call is defensive). Alternatively, document "run with `sudo`" at the top of the script and abort if `[ "$EUID" -ne 0 ]`.

**Warning signs:** macOS-only — bootstrap exits with `5: Input/output error`.

### Pitfall 8: Python `Type=simple` daemon exits cleanly when stdin closes

**What goes wrong:** systemd unit boots, journalctl shows the startup banner, then `Process exited with status code 0` and the unit goes to "inactive (dead)" within seconds.

**Why it happens:** FastMCP stdio transport reads from stdin; when stdin closes (because systemd doesn't connect a TTY), it interprets that as "client disconnected, shut down." This is exactly Phase 20 gap #10.

**How to avoid:** `StandardInput=null` in the unit body. Sends an EOF-immediately stream that FastMCP's stdio handler can ignore — but the actual fix is that `python -m forge_bridge` already handles this case via the lifespan task — verify in P5.

**Warning signs:** unit goes "inactive" within 1-2s of `systemctl start`. journalctl shows clean startup banner with no error.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `launchctl load -w /path/to.plist` | `launchctl bootstrap system /path/to.plist` | macOS 11 (Big Sur, 2020) deprecated; macOS 13+ enforces | Modern syntax surfaces clearer errors. Use bootstrap/bootout in 20.1. [VERIFIED: alansiu.net 2023] |
| Postgres `password_encryption = md5` (PG13 initdb default) | Postgres `password_encryption = scram-sha-256` (PG14.3+ initdb default) | PostgreSQL 14.3 release (2022) | Rocky 9 still ships PG13 → cluster initdb produces md5. RHEL 10 / PG16 produces scram-sha-256. Probe at runtime, don't assume. [VERIFIED: postgresql.org/docs/14/release-14.html] |
| Bash `read -s VAR` (no prompt arg) | `read -rsp "prompt: " VAR` (combined read-prompt-silent) | Long-standing best practice; surfaced in user memory `feedback_secret_paste_pattern.md` 2026-04 | Prompt-less `read -s` lets the user paste a secret on the command line (visible in history). Combined `-rsp` is correct for secrets. NOT applicable to FORGE_LOCAL_LLM_URL (visible URL). |
| systemd `Type=forking` for "real" daemons | `Type=simple` for foreground processes | systemd v240+ (2019) | Python's `python -m foo` is a foreground process. `Type=simple` is the modern default. Use it. [VERIFIED: freedesktop systemd.exec(5)] |
| Hardcoding env in plist `EnvironmentVariables` dict | Wrapper script sources `/etc/forge-bridge/forge-bridge.env` | Operator-recommended pattern surfaced ~2020 | Wrapper-script pattern reaches systemd EnvironmentFile parity on macOS. [VERIFIED: lucaspin.medium.com] |

**Deprecated / outdated:**
- `~/.launchd.conf` and `/etc/launchd.conf` — removed in macOS 10.10 Yosemite. Don't use.
- `launchctl setenv` from a login script — replaced by `EnvironmentVariables` in plist or wrapper-script approach.
- `pg_hba.conf` `password` (cleartext) auth method — never use; md5 or scram-sha-256.

### Known Threat Patterns for {Bash install scripts + systemd/launchd}

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Shell-history secret leak via `read -s VAR` (no prompt) | Information Disclosure | Use `read -rsp "prompt: " VAR` with length-check (memory feedback). 20.1's only prompt is non-secret (URL); standard `read -rp` applies. |
| pg_hba.conf misalignment opens cluster to wrong-method auth | Spoofing / Tampering | Probe `password_encryption` from psql; align method exactly. Anchored regex on `host all all 127.0.0.1/32` line. Backup file before edit. |
| Symlink attack on env file write | Tampering | `mkdir -p /etc/forge-bridge` (idempotent); `cp` template (refuses to follow symlink with default options); `chown` + `chmod` after. |
| Daemon runs as root by default | Privilege escalation | Explicit `User=$SUDO_USER` in unit + `UserName=$SUDO_USER` in plist. NEVER omit. |
| Hardcoded credentials in tree | Information Disclosure | Env-file template ships with empty `ANTHROPIC_API_KEY=` and default `forge:forge` Postgres pair (low-trust local-only credential per Phase 8 contract). NO secrets in `packaging/`. |
| Path traversal in dynamic plist generation | Tampering | install-bootstrap.sh uses `${SUDO_USER}` directly (validated as non-empty at script entry); no operator-supplied path components flow into the plist. |
| systemd unit injection via `$SUDO_USER` containing newlines | Tampering | bash variable expansion is single-line; sed substitution treats matched group as literal. Validate `$SUDO_USER` matches `^[a-z][a-z0-9_-]*$` before substitution. |
| launchctl print outputs unsanitized values into doctor row | Information Disclosure | `_check_daemon_state` only matches the `state = X` line; ignores other content (no leak of file paths or env values). |
