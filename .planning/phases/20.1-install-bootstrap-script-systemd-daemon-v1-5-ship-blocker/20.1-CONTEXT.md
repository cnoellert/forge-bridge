# Phase 20.1: Install Bootstrap Script + Systemd Daemon (v1.5 ship blocker) - Context

**Gathered:** 2026-05-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Transform forge-bridge's install procedure from prose-as-imperatives into a script + daemon model — `scripts/install-bootstrap.sh` + two systemd units (Linux) + two launchd plists (macOS) + `/etc/forge-bridge/forge-bridge.env` + reshaped `docs/INSTALL.md` — so an operator (or author-walking on a fresh VM) can complete the install end-to-end without re-deriving the topology each time. Closes the v1.5 milestone ship-blocker gap surfaced by Phase 20 Track A UAT (13 documented gaps; gap #11 is a substantive doc lie about single-process launch).

**Forcing function:** if a fresh-VM walk on flame-01 (Linux) AND a fresh-state walk on portofino (macOS) both don't reach all reachable surfaces clean, the milestone doesn't ship. Phase 20.1 is the v1.5 ship gate, not optional polish.

**Five surfaces** (parity with Phase 20):
- Web UI on `:9996/ui/`
- CLI `forge-bridge`
- HTTP `/api/v1/chat`
- MCP server (`python -m forge_bridge`, stdio default)
- Flame hook on `:9999` (skipped on macOS by default; opt-in via `--with-flame-mac`)

**Out of scope** (locked):
- New `docs/GETTING-STARTED.md`, surface map, projekt-forge consumer-pattern explainer (Phase 21)
- Daily-workflow recipes (Phase 22)
- TROUBLESHOOTING.md (Phase 23) — 20.1 only adds new `forge doctor` sub-checks for daemon state, not full failure-mode docs
- Apt/Debian/Ubuntu Linux distro support
- Auto-installing Homebrew on macOS (operator-trust concern)
- Multi-machine / multi-user / auth (Phase 20 D-09; v1.6+)
- Public `curl | bash` distribution form for `install-bootstrap.sh` (script lives in-tree only in 20.1)

<decisions>
## Implementation Decisions

### Service ownership + identity (D-01..02)
- **D-01:** System service in `/etc/systemd/system/` (Linux); `/Library/LaunchDaemons/` on macOS. Parity. Both managed via root-owned unit/plist files. User-level `systemctl --user` and `~/Library/LaunchAgents/` rejected — survives logout, mirrors operator mental model ("you sudo edit `/etc` files"), and `journalctl -u forge-bridge` works without `--user`.
- **D-02:** `User=` on Linux units = `$SUDO_USER` (operator's login user, captured at install time). macOS launchd plists `UserName=` = same. Dedicated `forge-bridge` system user is v1.6+ scope (would break the JSONL-authoritative `~/.forge-bridge/` invariant from Phase 8 STORE-06 and force a path migration).

### Postgres install scope (D-03..04)
- **D-03:** Linux (Rocky/RHEL) — **all-in-one**. Script runs `dnf install -y postgresql-server postgresql-contrib` if package missing, then `postgresql-setup --initdb` if cluster uninitialized, then pg_hba auth-method alignment (probe `SHOW password_encryption`, set pg_hba accordingly), then `systemctl enable --now postgresql`, then role+db creation via `psql -c "CREATE USER forge WITH PASSWORD 'forge'; CREATE DATABASE forge_bridge OWNER forge;"` (idempotent), then `alembic upgrade head`. Closes Phase 20 gaps #3, #4, #5, #6.
- **D-04:** macOS — detect Homebrew `postgresql@16` OR Postgres.app. Install Homebrew variant via `brew install postgresql@16 && brew services start postgresql@16` if neither present. **Operational reality (user-confirmed):** all Mac Flame workstations canonically have Postgres pre-installed; the install branch is the rare edge case but must work for parity.

### Distro / OS support (D-05..06)
- **D-05:** Linux Rocky/RHEL primary. Debian/Ubuntu/apt explicitly out of scope for 20.1.
- **D-06:** macOS launchd plists ship in 20.1 (not deferred to a hypothetical Phase 20.2). Pre-reqs on macOS: Python 3.11 + git + Homebrew. Script detects Homebrew presence; if missing, emits friendly error pointing at https://brew.sh and exits non-zero. Does NOT auto-install Homebrew (operator-trust + curl-pipe-bash ceremony deferred to v1.6+ if ever).

### Partial-install flag matrix (D-07..09)
- **D-07:** Flag surface = `--track-b` + `--no-postgres` + `--mcp-only` + `--with-flame-mac` + `--non-interactive`. **All flags must work in 20.1** (no "planted but unimplemented" lazy paths). `--no-postgres` planted now per user signal of expected v1.5–v1.6 use; `--mcp-only` is a real deploy shape user has already used (chat-only / MCP-only).
- **D-08:** macOS auto-applies `--track-b` and emits a clear log line: "macOS detected → skipping Flame hook install (override with `--with-flame-mac` if you have Mac-Flame)". `--with-flame-mac` opts the rare Mac-Flame artist back in.
- **D-09:** UAT test matrix (4 sane combos): (a) Linux full (everything); (b) Linux `--mcp-only` (no Postgres, no Flame); (c) macOS default (auto-track-b); (d) Linux `--no-postgres`. Planner picks coverage strategy — minimum required is (a) + (c) for both gates per D-18; (b) and (d) as smoke-checks.

### Env-file edit UX (D-10..11)
- **D-10:** Interactive prompt for `FORGE_LOCAL_LLM_URL` only — the operator's one decision ("where does my LLM run?"). All other env vars stay at template defaults. The artist is not put into `vi` for one line.
- **D-11:** `--non-interactive` flag for unattended re-runs (CI / re-installs). Env file at `/etc/forge-bridge/forge-bridge.env` on **both Linux and macOS** (parity; macOS does have `/etc/`). Permissions: `0640 root:$SUDO_USER` so daemons can read but the file isn't world-readable (contains URLs that may include hostnames worth not leaking).

### Verification / `forge doctor` integration (D-12..13)
- **D-12:** Script auto-runs `forge-bridge console doctor` at the end of install (after `systemctl start` / `launchctl load` succeeds). Emits PASS/FAIL/PARTIAL with the doctor table inline so the operator sees install-success-or-failure in one screen. Non-zero exit on FAIL; zero on PASS or PARTIAL (PARTIAL = degraded-tolerant warns, system runs).
- **D-13:** `forge doctor` gets new sub-checks for daemon state — **Linux:** `systemctl is-active forge-bridge-server` + `systemctl is-active forge-bridge`. **macOS:** `launchctl print system/com.cnoellert.forge-bridge-server | grep state` + `launchctl print system/com.cnoellert.forge-bridge | grep state`. Single sub-check name `daemon_state` with branching internal logic (one row in the doctor table, OS-dispatched). Phase 23 DIAG-05 parity work pulls forward into 20.1 because the daemon model needs the verification path to land together.

### Logs + lifecycle docs (D-14..15)
- **D-14:** Linux logs via `journalctl -u forge-bridge -f` (console + MCP) and `journalctl -u forge-bridge-server -f` (bus). Replaces Phase 20 gap #12 ("how do I stop the server?") via `systemctl stop forge-bridge` / `restart` documentation.
- **D-15:** macOS logs to `/var/log/forge-bridge/server.log` + `/var/log/forge-bridge/console.log` via plist `StandardOutPath` / `StandardErrorPath`. Operator can `tail -f` the file directly (no `log stream` ceremony). Plist permissions handle log-file rotation as a planner concern (newsyslog or no rotation in 20.1; logrotate-on-mac is v1.6+).

### UAT logistics + acceptance gates (D-16..19)
- **D-16:** Author-walk fires **immediately** on both Linux (flame-01) and macOS (portofino). No artist recruiting window. Both walks happen as part of phase execution.
- **D-17:** Discipline contract = **fresh VM / throwaway box** for both walks. No prior conda env, no prior Postgres, no prior shell history. The script must handle a true cold start — Phase 20's flame-01 had postgresql package pre-installed (cluster uninitialized), and gap #3 only surfaced from that incidental state. A fresh VM forces every branch of `install-bootstrap.sh` to fire honestly. Walker (cnoellert) honors the same verbatim copy-paste / verbatim transcription / no-silent-route-around discipline as Phase 20 D-02.1.
- **D-18:** **Both walks are gates.** Phase 20.1 doesn't close until BOTH the flame-01 Linux walk AND the portofino macOS walk report clean (zero gaps surface, all reachable surfaces green, `forge doctor` PASS or PARTIAL). One-of-two close is rejected.
- **D-19:** Outcome semantics — author-walk PASS = phase closes **PASS-with-deviations** (matching Phase 20 D-02.1 honesty), and 20.1 plants `.planning/seeds/SEED-PHASE-20.1-ARTIST-UAT-V1.6+.md` for future real-artist verification. Closing 20.1 the same way Phase 20 closed (clean PASS off author-walk) would re-burn the gate; the seed is honest.

### Carried forward from Phase 20 (locked, do not re-litigate)
- **D-20 (= Phase 20 D-04):** Aggressive in-flight gap-fix policy. Any gap that blocks either walk gets a code-fix plan added to 20.1 in-flight. "Document the workaround" is not acceptable for install-blocking gaps.
- **D-21 (= Phase 20 D-05):** Decimal-phase fractal. 20.1 IS the invocation of D-05 from Phase 20; if a single 20.1 gap balloons beyond ~1 plan during execution, spin Phase 20.1.1 / 20.2 rather than overgrow.
- **D-22 (= Phase 20 D-13/14):** Minimum-version + reference-version model. Reference versions captured during 20.1's two walks (Linux fresh-VM Postgres/Python/conda/Ollama-on-assist-01; macOS portofino brew/postgresql/Python/conda).
- **D-23 (= Phase 17 D-30):** Decoupled-commit purity. Structural changes (script creation, units/plists creation, env-file template) ship in separate commits from value flips and INSTALL.md reshape.

### Claude's Discretion (planner picks based on implementation reality)
- Exact systemd ordering directives beyond `Requires=`/`After=` (`Wants=`, `BindsTo=`). Default to `Requires=` + `After=` from the candidate; tighten if behavior surfaces a gap.
- launchd plist `KeepAlive` / `RunAtLoad` / `WorkingDirectory` directives. Recommend `RunAtLoad=true`, `KeepAlive=true` (with `SuccessfulExit=false` if needed to avoid restart-loop on `--help`-style exits), planner verifies on portofino.
- `pg_hba.conf` flip method — sed-in-place vs full file rewrite. Idempotency wins (re-runs must be safe).
- Whether the env-file's interactive prompt uses bash `read -p` directly or a polished `select` / `whiptail` / `dialog` UX. Bash `read` is fine if the prompt is unambiguous.
- Plan numbering inside 20.1 (likely 5–7 plans across script + units/plists + env-file + doc reshape + doctor sub-checks + two HUMAN-UAT walks).
- Whether the new `daemon_state` `forge doctor` sub-check is one row with internal branching or two rows (one per unit/plist). One row is cleaner; two rows is more granular for triage. Planner picks.
