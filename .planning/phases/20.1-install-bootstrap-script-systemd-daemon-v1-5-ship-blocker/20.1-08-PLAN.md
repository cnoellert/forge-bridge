# Plan 20.1-08 — MCP HTTP transport for daemon mode + service-persistence regression test

## Objective

Close the structural gap surfaced by the flame-01 Linux walk: `forge-bridge.service` runs
FastMCP in stdio transport mode with `StandardInput=null`, which causes FastMCP to read
EOF on `/dev/null`, exit cleanly with code 0, and take the Console + chat + Read API on
`:9996` down with it via the FastMCP lifespan co-host. `Restart=on-failure` does not
respawn (clean exit), and `:9996` becomes unbound, leaving three of five user-facing
surfaces unreachable.

This plan adds a `--transport` flag to `python -m forge_bridge` that selects between
`stdio` (existing default — preserves Claude Desktop config compatibility), `sse`, and
`streamable-http` (the latter two run a long-running uvicorn server on `:9997` that
does not exit on stdin EOF). The systemd unit + launchd plist + wrapper switch to
`--transport streamable-http --mcp-port 9997`, drop `StandardInput=null`, and tighten
`Restart=` to `always`. A new packaging regression test spawns the daemon-mode invocation
as a subprocess, polls liveness for 3 seconds, and asserts the process is still bound —
catching any future regression where the daemon exits prematurely.

## Tasks

### Task 1 — Add --transport + --mcp-port flags + thread to FastMCP run()

**Files:** `forge_bridge/__main__.py`, `forge_bridge/mcp/server.py`, `tests/test_cli_main.py`

1. In `forge_bridge/__main__.py`, extend the root callback signature with two new Typer
   options matching the existing `--console-port` precedence pattern (flag > env > default):

   ```python
   transport: str = typer.Option(
       "stdio",
       "--transport",
       help="MCP transport: stdio | sse | streamable-http. Default stdio preserves "
            "Claude Desktop / Claude Code config compatibility. Daemon mode (systemd / launchd) "
            "should use streamable-http.",
   ),
   mcp_port: Optional[int] = typer.Option(
       None,
       "--mcp-port",
       help="Port for FastMCP to bind under sse / streamable-http transports "
            "(default 9997, or $FORGE_MCP_PORT). Ignored under stdio.",
       envvar=None,
   ),
   ```

   Validate `transport in {"stdio", "sse", "streamable-http"}`; raise typer.BadParameter
   on anything else (the FastMCP runtime would do the same, but a Typer-level check
   gives a cleaner CLI error).

2. Resolve port precedence (flag > FORGE_MCP_PORT env > default 9997) before forwarding
   to `mcp_main(transport=transport, port=resolved_port)`.

3. In `forge_bridge/mcp/server.py:main()`, accept the parameters and configure FastMCP:

   ```python
   def main(transport: str = "stdio", port: int = 9997) -> None:
       logging.basicConfig(...)
       if transport != "stdio":
           mcp.settings.port = port
       mcp.run(transport=transport)
   ```

4. New tests in `tests/test_cli_main.py` (create the file if it doesn't exist; mirror the
   pattern in `tests/test_cli_doctor.py`):

   - `test_transport_flag_stdio` — `forge-bridge --transport stdio --help` succeeds
     (smoke test of flag parsing; do not actually boot the server)
   - `test_transport_flag_sse` — same for sse
   - `test_transport_flag_streamable_http` — same for streamable-http
   - `test_transport_flag_invalid` — `--transport bogus` exits non-zero with a CLI error
   - `test_mcp_port_env_override` — `FORGE_MCP_PORT=12345 forge-bridge --transport stdio --help`
     does not error (env is read but stdio ignores port)
   - `test_mcp_port_flag_precedence` — flag wins over env

   Use Typer's `CliRunner` and monkeypatch `forge_bridge.mcp.server.main` to a no-op so
   the tests do not actually start the MCP server.

### Task 2 — Update systemd + launchd + env-file template; tighten packaging tests; add daemon-persistence regression

**Files:** `packaging/systemd/forge-bridge.service`, `packaging/launchd/com.cnoellert.forge-bridge.plist`, `packaging/launchd/forge-bridge-daemon`, `packaging/forge-bridge.env.example`, `tests/test_packaging.py`

1. Update `packaging/systemd/forge-bridge.service`:
   - Change ExecStart from `... python -m forge_bridge` to `... python -m forge_bridge --transport streamable-http --mcp-port 9997`
   - Remove the `StandardInput=null` line (no longer needed; FastMCP under streamable-http
     does not read stdin)
   - Change `Restart=on-failure` to `Restart=always` (a clean FastMCP exit during shutdown
     should still trigger systemd respawn so the service self-heals if anything crashes the
     uvicorn loop)
   - Keep `Requires=forge-bridge-server.service` and `RestartSec=5` unchanged

2. Update `packaging/launchd/com.cnoellert.forge-bridge.plist` AND/OR
   `packaging/launchd/forge-bridge-daemon` (whichever owns the python invocation per Wave 3
   — likely the wrapper script):
   - Append `--transport streamable-http --mcp-port 9997` to the python invocation
   - Verify the plist's `KeepAlive={SuccessfulExit: false}` is still set; if the daemon
     wrapper exits cleanly during normal shutdown, launchd's KeepAlive=false-on-success
     prevents respawn (matches systemd Restart= semantics, opposite polarity but consistent
     in intent: respawn on failure, don't respawn on intentional stop)

3. Update `packaging/forge-bridge.env.example` to add an 8th key:

   ```bash
   # ── MCP HTTP transport port ─────────────────────────────────────────────────
   # Port for FastMCP to bind under streamable-http / sse daemon transports.
   # Read by forge-bridge.service and com.cnoellert.forge-bridge.plist via env file.
   # Ignored under stdio transport (Claude Desktop / Claude Code stdio invocations).
   FORGE_MCP_PORT=9997
   ```

4. Update `tests/test_packaging.py`:
   - Tighten the existing `test_systemd_facade_python_invocation` (or add if missing) to
     assert `--transport streamable-http` AND `--mcp-port 9997` both appear in the unit's
     ExecStart line
   - Replace `test_systemd_facade_standard_input_null` with
     `test_systemd_facade_no_stdio_input_null` (forbidding `StandardInput=null` for
     forge-bridge.service since the daemon no longer needs it). The bus unit
     forge-bridge-server.service keeps its `StandardInput=null` — still asserted by its
     own existing test.
   - Update launchd tests to assert the daemon flags appear in whichever file (plist or
     wrapper) owns the python invocation
   - Update `test_env_template_keys` (or equivalent) to assert all 8 keys including
     `FORGE_MCP_PORT`
   - **New test `test_daemon_persistence`**:
     ```python
     import socket, subprocess, time
     def _free_port():
         s = socket.socket(); s.bind(("127.0.0.1", 0))
         port = s.getsockname()[1]; s.close(); return port
     def test_daemon_persistence():
         """forge-bridge.service must NOT exit cleanly within 3s of start (Phase 20.1 walk gap)."""
         port = _free_port()
         proc = subprocess.Popen(
             [sys.executable, "-m", "forge_bridge",
              "--transport", "streamable-http", "--mcp-port", str(port)],
             stdout=subprocess.PIPE, stderr=subprocess.PIPE,
             stdin=subprocess.DEVNULL,
         )
         try:
             time.sleep(3)
             assert proc.poll() is None, (
                 "forge-bridge daemon exited within 3s — stdio-EOF regression "
                 "(Phase 20.1 flame-01 walk gap, plan 20.1-08)"
             )
         finally:
             proc.terminate()
             proc.wait(timeout=5)
     ```
     Mark as `@pytest.mark.integration` if the suite uses that marker; ensure subprocess
     test infrastructure works in CI (the launching python must be in a venv that has
     forge_bridge installed in editable mode — pytest's own python).

5. Atomic commit: `feat(20.1-08): switch forge-bridge daemon to streamable-http + add persistence regression test`
