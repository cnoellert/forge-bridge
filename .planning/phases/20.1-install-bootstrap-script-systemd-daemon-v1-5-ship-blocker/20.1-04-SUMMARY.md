## Postgres Bootstrap Auth Alignment

Script handles both Rocky 9 default (PG13, `md5`) and PG14.3+ (`scram-sha-256`):

- `probe_pg_auth()` queries `SHOW password_encryption` from the live cluster
- `align_pg_hba()` checks `grep -qE "^host...127.0.0.1/32...$auth"` before touching the file (idempotent)
- Backs up `pg_hba.conf` to `.bak.$(date +%s)` before sed (T-20.1-17 lockout prevention)
- Reloads via `SELECT pg_reload_conf()` (no restart needed)

macOS Homebrew Postgres ships with `trust` for localhost — pg_hba alignment skipped on Darwin.

## Idempotency Contract

Every mutating step uses probe-then-act:

| Step | Idempotency guard |
|------|-------------------|
| `dnf install postgresql-server` | `command -v psql` check |
| `postgresql-setup --initdb` | `[ -d /var/lib/pgsql/data/base ]` check |
| env file copy | `[ ! -f /etc/forge-bridge/forge-bridge.env ]` check — **operator edits preserved across re-runs** |
| CREATE USER forge | `SELECT 1 FROM pg_roles WHERE rolname='forge'` check |
| CREATE DATABASE forge_bridge | `SELECT 1 FROM pg_database WHERE datname='forge_bridge'` check |
| launchd plists | `bootout ... || true` before `bootstrap` (idempotent reload) |

Re-running the script on an already-installed machine is a no-op per D-21 + RESEARCH.md Open Question 3.

## Security Controls Applied (threat_model mitigations)

| Threat | Mitigation in script |
|--------|---------------------|
| T-20.1-15: $SUDO_USER injection via sed | `^[a-z][a-z0-9_-]*$` regex validation at script entry, before any sed call |
| T-20.1-16: env file world-readable | `chmod 0640` + `chown root:$SUDO_USER` on every run |
| T-20.1-17: pg_hba rewrite without backup | `cp pg_hba.conf pg_hba.conf.bak.$(date +%s)` before sed |
| T-20.1-18: postgresql-setup --initdb twice | Guard: `[ ! -d /var/lib/pgsql/data/base ]` |
| T-20.1-21: script run as non-root | `[ "$EUID" -eq 0 ]` assertion at entry |
| T-20.1-22: launchd log dir race | `mkdir -p /var/log/forge-bridge` + `chown/chmod` BEFORE `launchctl bootstrap` |

## D-12 Doctor Auto-run

`verify_doctor()` runs `sudo -u "$SUDO_USER" "$CONDA_PYTHON" -m forge_bridge.cli console doctor`:
- Runs as operator account (not root) — doctor checks operator-account state
- Uses `$CONDA_PYTHON` directly (avoids root PATH not having the binary)
- Exit code maps: 0 = PASS/PARTIAL → script exits 0; non-zero = FAIL → script exits 1
